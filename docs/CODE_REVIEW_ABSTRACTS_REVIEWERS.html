Code Review — Abstracts & Reviewers

Production-readiness audit · 10 July 2026 · EA-SYS main @ 101916e
Four independent review passes: submission→decision lifecycle · RBAC / info-leaks · concurrency & integrity · drift & failure-visibility.

How to read this. Findings are grouped by theme; where two independent passes hit the same defect that is noted. Everything already tracked in docs/ROADMAP.md's "Reviewer/submitter lifecycle audit" (June 26) was excluded: REVISION_REQUESTED author lock, COI-blocks-write-not-read, mean-score visibility to submitters. The feedbackOnly dead-code suspicion was re-examined and is REPORTED here (M10) because the finding grew: the entire "reviewer added feedback" notification path silently vanished, and CLAUDE.md still advertises it. Top-severity findings (H1–H6, H8, H9) were independently re-verified in source by the lead reviewer — quoted lines confirmed byte-for-byte.

1. Executive summary

The Sprint-B redesign (per-reviewer AbstractReviewSubmission rows, the requiredReviewCount gate, chair override, the /my-reviews portal) is structurally sound, and several hard things are done right: the review-submission upsert is DB-native atomic; the pool-membership array goes through the row-locked updateEventSettings helper everywhere; unassign genuinely preserves submissions (SET NULL verified in schema); the 300-word and 20-co-author caps hold on every write path; and the June 26 COI gates are intact on all three submit paths.

Four structural patterns explain nearly every finding:

  1. Status is writable OUTSIDE the gated service. changeAbstractStatus centralizes the review gate, terminal guard, audit, and notifications — but the PUT's field-update branch writes status blindly (H1), the create POST accepts any birth status (H2), and even the service itself writes without a conditional claim (H4).
  2. The decision gate counts rows, not reviews. Empty submissions satisfy it (H5), pre-revision submissions satisfy it after a resubmit (M1), phantom submissions on DRAFTs count (H6), and the count is read outside any transaction (M4).
  3. The blind-review posture is one-sided. The submissions GET carefully strips reviewer identity for authors — but the decision email names every reviewer next to their notes (H9), MEMBER gets the full who-said-what view (M7), and pool reviewers see each other's scores (L2).
  4. Review submission never got its service. The ~120-line pipeline is triplicated across REST + two MCP executors with live drift — including an admin-auth difference and, on the write side, a missing org bind that lets another org's admin inject scores (H7, H3); MCP assignment silently drops the COI flag that has been an enforcement input since June 26 (H8).

3. Status integrity — writes outside the gated service

High   H1 — The PUT's field-update branch writes status with zero transition validation: an ACCEPTED or WITHDRAWN abstract can be flipped back to SUBMITTED or DRAFT — by roles as low as REVIEWER, or even a REGISTRANT with event access

src/app/api/events/[eventId]/abstracts/[abstractId]/route.ts:48, 345

status: z.enum(["DRAFT", "SUBMITTED", "UNDER_REVIEW", "ACCEPTED", "REJECTED", "REVISION_REQUESTED", "WITHDRAWN"]).optional(),
...
...(data.status && { status: data.status }),   // field branch — no prior-state check

Failure scenario

SUBMITTED/DRAFT targets are neither review statuses nor terminal, so they skip changeAbstractStatus entirely (and its WITHDRAWN terminal guard) and land in the blind field write. Role guards fence only review statuses and the SUBMITTER branch — other roles with event access (REVIEWER; a REGISTRANT whose linked registration grants access via buildEventAccessWhere) pass. A PUT of {status:"SUBMITTED"} on an ACCEPTED/WITHDRAWN abstract reverses the decision, re-stamps submittedAt (isSubmission fires), re-emails the speaker a fresh submission confirmation, and re-notifies admins; {status:"DRAFT"} on an ACCEPTED abstract hides it from every organizer/reviewer list (draft-visibility filter) while returning edit rights to the author. Fix shape: the field branch must never write status — every status write routes through the service.

High   H2 — Abstract CREATE accepts any birth status: a submitter can mint a born-ACCEPTED abstract with one crafted request

src/app/api/events/[eventId]/abstracts/route.ts:20, 37, 158

const abstractStatusSchema = z.nativeEnum(AbstractStatus);
...
status: abstractStatusSchema.default("SUBMITTED"),
...
if (session.user.role === "REVIEWER") { return ... 403 }   // only REVIEWER blocked from POST

Failure scenario

The PUT path's careful gates (review-status 403 for submitters, the review-count gate, the chair-override audit, the speaker notification) apply only to transitions, never to birth status. A self-service SUBMITTER POSTs { status: "ACCEPTED" } — the row is born accepted with zero reviews, no submittedAt (stamped only for SUBMITTED), no chair-override audit, no email trail; organizer statistics count it as a decision. Fix shape: restrict create status to DRAFT | SUBMITTED.

High   H4 — changeAbstractStatus itself is check-then-act: racing decisions resurrect a WITHDRAWN abstract and send the author contradictory accepted+rejected emails

src/services/abstract-service.ts:140-147, 179-196

if (abstract.status === "WITHDRAWN" && newStatus !== "WITHDRAWN") { ... return ABSTRACT_WITHDRAWN }
...
updated = await db.abstract.update({ where: { id: abstractId }, data: { status: newStatus, ... } });

Failure scenario

Both guards run on a snapshot; the write is unconditional on id. Organizer A PUTs WITHDRAWN while organizer B PUTs ACCEPTED: both reads see UNDER_REVIEW, both pass the terminal guard (the window spans B's aggregates query), A commits WITHDRAWN, B commits ACCEPTED — the "terminal" state is exited and the author receives BOTH emails, each computed against its own stale previousStatus. House fix: conditional claim (updateMany where { id, status: previousStatus }, count 0 = lost race) inside one transaction — which also fixes M4 for free, and the MCP path inherits it via the shared service.

4. The decision gate — counts rows, not reviews

High   H5 — The requiredReviewCount gate is satisfiable by completely EMPTY review submissions — including ones an admin mints for themself with no pool membership and no audit trail

…/submissions/route.ts:11-16, 173-174 · src/services/abstract-service.ts:157

const submissionSchema = z.object({
  criteriaScores: ...optional(), overallScore: ...optional(),
  reviewNotes: ...optional(), recommendedFormat: ...optional(), confidence: ...optional(),
});                                                    // EVERY field optional → POST {} inserts a row
...
if (!forceStatus && aggregateResult.aggregates.count < requiredCount) { ... }   // count = rows

Failure scenario

Event requires 2 reviews. A reviewer clicking through the form without scoring — or an ADMIN/ORGANIZER (allowed to submit with no pool membership or assignment) posting {} — creates an all-null submission; the count reaches 2 and ACCEPTED passes the gate with zero actual scores and no chair-override record (unlike forceStatus, the sanctioned + audited bypass). Fix shape: the gate counts only submissions carrying at least an overallScore or non-empty criteriaScores; the schema should reject fully-empty POSTs.

High   H6 — DRAFT (and WITHDRAWN) abstracts leak into the reviewer portal, can be ASSIGNED, and can be SCORED — and those phantom submissions count toward the decision gate

src/app/api/my-reviews/route.ts:36-51 · …/submissions/route.ts POST (no abstract-status check) · …/reviewers/route.ts POST (assigns onto DRAFTs)

db.abstract.findMany({
  where: { event: { status: { not: "CANCELLED" }, settings: { path: ["reviewerUserIds"], array_contains: userId } } },
  // ← no abstract.status filter; DRAFTs + WITHDRAWNs render as PENDING work

Failure scenario

Both abstracts LIST routes deliberately route through abstractListStatusFilter so drafts are never shown to reviewers — the portal query forgot the filter, so a pool reviewer sees (title + status) and reviews an author's unfinished draft. The submissions POST never checks abstract status, so DRAFT/WITHDRAWN/already-decided abstracts are all scoreable, and those submissions satisfy H5's gate. Fix shape: status filter in the portal feed; the submissions POST refuses unless status ∈ {SUBMITTED, UNDER_REVIEW, REVISION_REQUESTED}; assignment POST refuses DRAFT/WITHDRAWN.

Medium   M1 — Decisions can rest on reviews of pre-revision content, and NEEDS_UPDATE misses content-only edits

src/services/abstract-service.ts:152-169 · src/app/api/my-reviews/route.ts:129

After REVISION_REQUESTED → resubmit (which re-stamps submittedAt), the old submissions still fully satisfy the ACCEPT/REJECT gate — nothing compares submission.updatedAt to abstract.submittedAt at decision time. Separately, the portal staleness flag keys on submittedAt, so an organizer editing content via the field-only PUT (bumps only updatedAt) never flips NEEDS_UPDATE — contradicting its documented intent.

Medium   M4 — The review-count gate is a TOCTOU: count read outside any transaction, status written afterwards

src/services/abstract-service.ts:152-181

computeSubmissionAggregates accepts a tx parameter precisely for consistent reads — the service never passes one. A submission cascade-deleted between the count and the write (e.g. the reviewer's User account deleted, onDelete: Cascade) lets ACCEPTED commit below the gate with a stale reviewCount in the audit row. Folds into H4's transaction fix.

5. Blind review — one-sided anonymity

High   H9 — The status-change email NAMES each reviewer next to their notes — the API strips identities for authors; the email hands them over

src/lib/abstract-review.ts:182 → src/lib/abstract-notifications.ts (speaker email) · contrast …/submissions/route.ts:83-102 (anonymized author view)

return withNotes.map((s) => `— ${s.reviewerName}:\n${s.reviewNotes}`).join("\n\n");   // consolidateReviewNotes
// vs the submissions GET speaker view: reviewer identity deliberately stripped

Failure scenario

Every REJECTED / REVISION_REQUESTED email with 2+ notes tells the author exactly who wrote each criticism (single-note panels leak by elimination). Real-world harm on a medical-conference platform: reviewer harassment, damaged relationships, chilled honest reviews — the exact things the GET's anonymization exists to prevent. One of the two intents is wrong; as shipped, the API hides what the email reveals. Fix shape: strip attribution in the email (join notes without names), or make attribution an explicit per-event choice.

Medium   M7 — MEMBER (read-only, "sponsor-side stakeholders") receives the FULL per-reviewer view: identities, notes, per-criterion scores, confidence

…/submissions/route.ts:71-108 (GET — isOrgMember passes MEMBER; returns un-stripped aggregate.submissions)

The author view is correctly stripped; the org-member view is not role-tiered. Restrict the who-said-what view to ADMIN/ORGANIZER (and assigned reviewers per L2's decision); MEMBER gets the anonymized aggregates.

6. Cross-surface security & drift

High   H3 — Cross-org review-score injection: the submissions POST authorizes admins by BARE ROLE with an unscoped event lookup

…/submissions/route.ts:149-182 (POST) — contrast the org-safe GET in the same file

db.event.findFirst({ where: { id: eventId }, ... })     // ← no organizationId / buildEventAccessWhere
...
const isAdmin = session.user.role === "ADMIN" || ... "ORGANIZER";
if (!isEventReviewer && !existingAssignment && !isAdmin) { ... 403 }

Failure scenario

An ORGANIZER of org B posts a score against any org A abstract id — passes isAdmin, upserts a submission, and that score feeds the mean + the decision gate of another org's peer review. The sibling GET binds event.organizationId === session.user.organizationId; the write path forgot it. Same class as this week's activity-route and invoice-create IDORs.

High   H7 — The review-submission pipeline is TRIPLICATED (REST + 2 MCP executors) with live behavioral drift — including an admin-auth difference

…/submissions/route.ts:200-295 vs src/lib/agent/tools/abstracts.ts:539-655 (submit) vs :711-816 (on-behalf)

Verified drift

  • REST lets admins score without pool/assignment; MCP submit_abstract_review has no admin term — same organizer succeeds in the dashboard, gets NOT_A_REVIEWER via MCP.
  • REST can clear notes with an empty string; MCP silently keeps the old notes.
  • REST Zod requires integer scores; the executors accept floats (currently re-guarded only by the MCP transport's own Zod).
  • REST rejects DUPLICATE_CRITERION_ID; the executors have no such check.

~120 lines × 3 copies of exactly the logic class abstract-service.ts was created to centralize. Fix shape: extract submitAbstractReview() into the service (auth matrix + COI + validation + weighted-score + upsert + audit), all three callers delegate — which also closes H3 in one place.

High   H8 — MCP assign_reviewer_to_abstract advertises conflictFlag but silently drops it — and COI has been ENFORCEMENT, not metadata, since June 26

src/lib/agent/register-mcp-tools.ts:368-372 (schema promises it) vs src/lib/agent/tools/abstracts.ts:306-318 (executor never reads it)

Failure scenario

An admin telling claude.ai/n8n "assign Dr X, flag the conflict of interest" gets success: true, no flag is written, and the conflicted reviewer's score silently counts toward ACCEPT/REJECT — the exact outcome the June 26 COI fix exists to prevent. The MCP path also can't toggle COI on an existing assignment (REST can). CLAUDE.md's claim of REST↔MCP parity here is false.

Medium   M8 — Reviewer-assignment target userId is looked up GLOBALLY with no org/role bind — assigning the abstract's own author enables self-review; arbitrary users get emailed abstract details

…/reviewers/route.ts:69-77, 184-193 + MCP abstracts.ts:325-328 (identical unbound lookup)

db.user.findUnique({ where: { id: validated.data.userId }, ... })   // any user, any org, any role

The assignment then grants that user reviewer-auth on the abstract (the existingAssignment path in the submissions POST) — so assigning the abstract's own speaker.userId launders a self-review into the gate. Also fires notifyReviewerAssigned to arbitrary addresses with the abstract title + event name. Fix shape: target must not be the abstract's author, and must be in the event reviewer pool or an org team role.

Medium   M9 — In-app agent: 5 abstract executors undiscoverable + the one status tool it has advertises reviewNotes the executor discards and omits force

src/lib/agent/tools/abstracts.ts:903-999 (9 defs) vs :1001-1016 (14 executors) · update_abstract_status def :979-993

assign/unassign_reviewer_to_abstract, submit_abstract_review, admin_submit_review_on_behalf, get_abstract_scores are wired + MCP-registered but absent from ABSTRACT_TOOL_DEFINITIONS — the in-app agent can't call them (the June session-tools gap class, again). And the def's reviewNotes param is never read by the executor — notes an agent believes went to the author vanish with success: true — while force (supported + suggested by the INSUFFICIENT_REVIEWS error) is missing from the schema, so schema-strict clients can't pass it.

7. Data-integrity & workflow gaps

Medium   M2 — Criteria are mutable/deletable mid-round with no protection: orphaned score keys, mixed weight bases in the mean, and no sum-to-100 enforcement anywhere

…/review-criteria/[criterionId]/route.ts:50-88 · src/lib/abstract-review.ts:31, 43-49 · …/submissions/route.ts:210-229

const weightedOnTen = scores.reduce((sum, c) => sum + (c.score * c.weight) / 100, 0);  // assumes Σweight = 100
weight: z.number().int().min(1).max(100),   // per-criterion bound only — the helper's "enforced by the route schema" comment is FALSE

Weights summing to 130 clamp all top scores to 100 (compression); summing to 50 cap everyone at 50. overallScore is computed with the weights current at submit time and never recomputed, so reweighting mid-round averages incomparable bases into the mean that drives decisions. Deleting a criterion leaves orphaned ids in criteriaScores JSON (still surfaced in perCriterion with no name) and hard-rejects any reviewer mid-form with INVALID_CRITERION_ID — losing everything they typed. Fix shape: warn/refuse weight-sum ≠ 100; refuse delete when submissions reference the id (the certificates template-delete 409 pattern).

Medium   M3 — allowAbstractSubmissions and abstractDeadline gate only the PUBLIC REGISTRATION — not abstract creation, submission, or resubmission

src/app/api/public/events/[slug]/submitter/route.ts:98-116 (the only checks) — neither setting referenced in the abstracts POST/PUT

Any submitter who registered before the CFP closed can keep creating and submitting abstracts indefinitely; flipping the toggle off only stops new account creation. Fix shape: the same two checks in the create POST + the DRAFT→SUBMITTED / resubmit branches (organizer/admin exempt).

Medium   M5 — Abstract create has zero double-submit defense: duplicate abstracts + duplicate emails on a double-click

src/app/api/events/[eventId]/abstracts/route.ts:223-243

No unique on (eventId, speakerId, title), no dedup check — unlike every other create surface (registration, speaker). Double-click → two SUBMITTED twins, two confirmation emails, two admin notifications, double reviewer effort. The PUT resubmit double-fires too when the client omits expectedUpdatedAt.

Medium   M6 — The PUT's status branch bypasses the optimistic lock and applies field updates in a second, non-atomic write

…/abstracts/[abstractId]/route.ts:264-314 vs the field-only branch's STALE_WRITE guard at :332

"Set track + ACCEPTED" clobbers a concurrent editor's changes with no 409, and if the follow-up field write throws, the decision + speaker email have already fired while the caller sees a 500 claiming nothing succeeded. Fold into H1's restructure (single transaction, lock honored on both branches).

Medium   M10 — The "reviewer added feedback" notification path silently vanished in Sprint B; feedbackOnly is dead code and CLAUDE.md still advertises the feature

src/lib/abstract-notifications.ts:61, 85-90, 131-134 — zero callers pass feedbackOnly; no notification helper is called from the submissions POST or either MCP submit executor

The speaker "Reviewer Feedback Received" email is unreachable, and admins are no longer notified when a review lands (only on status transitions). CLAUDE.md's "Email sent when reviewer adds notes/score without status change" claim is stale. Decide: re-wire the notification from the (to-be-extracted, H7) submission service, or delete the branch + correct the doc.

Medium   M11 — Silent 4xx sweep: ~20 unlogged rejection paths across the domain

PUT guards (SUBMITTED_LOCKED, WITHDRAW_NOT_ALLOWED, forceStatus/review-status 403s, Track/Theme 404s, PRESENTATION_TYPE_REQUIRED), the service-error passthrough (only INSUFFICIENT_REVIEWS logs), submissions 403/400s, reviewers + review-criteria 404s, abstracts POST 403/404s

11. What checked out clean

  • The review-submission upsert is DB-native atomic (compound-unique where, scalar-only payload) — no first-submission race.
  • Pool membership writes all go through the row-locked updateEventSettings helper with in-lock dedupe.
  • FKs match the code's claims: unassign preserves submissions (SET NULL verified in schema); abstract delete cascades reviewers + submissions.
  • Submitter ownership holds on the detail GET (404, non-enumerating), the PUT submitter branch (DRAFT-only, no review statuses, no force), my-profile (ownership by construction), and the LIST (own abstracts only, drafts hidden from non-owners).
  • COI gates intact on all three submit paths; draft saves stay silent (no emails/notifications); the 300-word + 20-co-author caps hold on every write path with no MCP bypass.
