Testing Strategy — mocked units vs real-DB integration

EA-SYS · Assessment prepared July 23, 2026 · ea-sys@0.4.25 on main

Status: ASSESSMENT — no code changed. Written because the question came up: the unit suite mocks the database (vi.mock("@/lib/db")) — should it use a real local DB instead? Short answer: don't replace the 3,718 wholesale — but do add a targeted real-DB integration layer for the paths where mocks demonstrably let bugs through. This doc lays out why, and a concrete plan, so it's a considered decision.

1 · Where we are today — it's already three tiers, not one

The codebase is not "all mocked." It already runs real databases in two of its four test tiers:

TierCommandDBCount / speedWhat it proves
Unit npm run test (vitest) mocked vi.mock("@/lib/db") 3,718 tests · ~4.2 s Logic: validation, pricing, mapping, business rules, and that the code calls the DB with the right args.
e2e npm run test:e2e (Playwright) real local ea_sys_test ~70 specs · minutes Full user flows through the real app + real Postgres. workers: 1 — shared seeded DB.
Tenancy npm run test:tenancy real Docker Postgres + pgbouncer ~25 tests · CI service container RLS isolation against a real pooler in transaction mode. Separate vitest project, fileParallelism: false.
Migration replay CI job real empty Postgres every deploy · GATING The full migration chain applies cleanly against a fresh DB + schema-diff.

Of the 264 unit-test files, ~164 mock @/lib/db; ~100 are pure logic (no DB at all — validators, formatters, calculators). The pure ones gain nothing from a real DB.

2 · The real problem: mock drift lets bugs through

The weakness isn't philosophical — it has bitten this codebase repeatedly. A mocked test asserts "the code called db.x.update with these args," not "this actually works against Postgres." So a test can be green while the feature is broken. Documented instances:

  • MCP bulk-email HTML escaping (A1, July 16): the parity test "mocked executeBulkEmail and asserted only the call shape" — it passed while every agent/n8n bulk email rendered HTML as literal source code, audience-wide, with success: true.
  • Contacts sync (H4, July 13): "a test that only checked 'syncToContact was called' would have passed against the bug" — the fix note explicitly re-instates the lesson: assert the effect, not the call.
  • The class in general: Prisma semantics that mocks can't reproduce — interactive $transaction atomicity, SELECT … FOR UPDATE row locks, updateMany predicate-guard return counts, @@unique race → P2002, cascade behaviour. These are exactly the primitives the money + concurrency code relies on, and exactly what a mock returns "whatever you told it to."

Note the mitigation already in the culture: several fix notes call for "effect-level" tests. But an effect-level test against a mock still can't prove a real row lock serializes two writers — only a real DB can.

4 · Recommendation — Option A, and why it's low-risk here

The decisive fact: the real-DB-vitest pattern already exists in this repo — tests/tenancy/ is a second vitest project (vitest.config.tenancy.ts) with a globalSetup that does prisma db push + seed against a real Postgres, runs fileParallelism: false, and is wired into CI as its own service-container job. An integration layer is a copy of that shape pointed at ea_sys_test — not new infrastructure.

And the scope is small because the gap is narrow: only the DB-semantics paths need a real Postgres. Everything else (the ~100 pure-logic files + the ~164 mocked-logic files where the DB is incidental) stays fast and mocked. You keep the 4-second loop and get real-Postgres proof exactly where mocks lie.

5 · The plan for Option A (when approved)

5.1 Tooling — reuse ea_sys_test, no new dependency

CandidateCall
Local ea_sys_test + CI Postgres service container chosen — the e2e suite already proves this exact path (real Prisma → real Postgres). Zero new deps. Matches the tenancy/e2e CI job shape we already run.
@electric-sql/pglite (embedded WASM Postgres) No server, fast-ish, real PG semantics — but Prisma support is via a newer driver adapter; would need to verify our schema (gen_random_uuid(), extensions, enums) loads cleanly. Deferred: a new-dep risk for no benefit over the DB we already run.
testcontainers (Postgres per run) Most faithful, heaviest. Unnecessary while ea_sys_test + a CI service container already give real Postgres.

5.2 Isolation — truncate-between-tests, not outer-transaction-rollback

The usual "wrap each test in a transaction and roll back" trick does not work for our target paths: they are exactly the code that opens its own interactive $transaction + SELECT … FOR UPDATE, and Prisma can't nest a rollback-only outer transaction around code that manages its own. So: truncate the touched tables between tests (fast on a small DB) + build each test's fixtures explicitly. fileParallelism: false (same as the tenancy project) avoids cross-file races. This is slower per-test than mocks — which is why it stays a small, separate suite.

5.3 What to cover FIRST — the money + concurrency guards

The paths where a mock gives false confidence and a real DB earns its keep:

  • payment-service — refund optimistic lock (refundedAmount conditional updateMany), credit-note cap under SELECT … FOR UPDATE, cancel-refund-before-cancel.
  • registration-service + registration-seat-db — claimSeat / claimSeats soldCount < quantity guard, promo usedCount release/claim, the @@unique duplicate-registration race.
  • accommodation-service — the atomic overbooking guard (bookedRooms < totalRooms updateMany).
  • invoice-service.createPaidInvoice — the reuse-or-promote under the registration row lock (the M1 "two PAID invoices mint" guard).
  • registration-serial — the atomic counter upsert (the P2024/collision fix).

That's ~6–10 service files → an estimated 40–60 integration tests, each proving a guard actually serializes/rejects against real Postgres (e.g. fire two concurrent claims, assert exactly one wins). Contrast: 3,718 to migrate under Option B.

5.4 Wiring

# package.json

# CI: new job with a postgres:16 service container, mirroring the
# tenancy/migration-replay jobs. NON-GATING first; promote to gating
# after >=5 green runs (the repo's own promotion rule).

5.5 Effort

~1 focused session to stand up the harness (config + globalSetup + truncate helper + CI job) and convert the first money-path service as the reference pattern. Then incremental — one service per sitting. No big-bang; the mocked suite is untouched throughout.

6 · Why NOT Option B (the numbers)

  • 4.2 s → minutes. 3,718 tests each seeding/truncating real rows destroys the instant feedback loop that makes the suite worth running on every save.
  • Flakiness + no parallelism. Shared DB state means ordering dependencies; the e2e suite already runs workers: 1 for exactly this reason. That's fine for 70 specs, ruinous for 3,718.
  • Total rewrite. Every mockDb.x.findFirst.mockResolvedValue(...) becomes "insert real rows first" — a different authoring model, ~164 files.
  • Most of it is wasted. A Zod validator, a price calc, a token formatter gains nothing from a real Postgres. You'd pay the full cost to cover code that was never at risk.

7 · Decision

Recommended: Option A — keep the fast mocked suite, add a small test:integration layer (real ea_sys_test, copy of the tenancy-harness shape), cover the money + concurrency guards first, wire a non-gating CI job that promotes to gating after ≥5 green. Incremental, reversible, extends a proven pattern.

This doc changes nothing on its own. When you want it built, the §5 plan is ready to execute.
