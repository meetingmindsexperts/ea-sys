The three mistakes (read this first)

This domain has one job: keep RoomType.bookedRooms equal to the number of accommodations that actually hold a room. That's an invariant — a statement that must be true before and after every operation. Almost every HIGH finding is the same invariant broken a different way.

Mistake 1 — Deciding from a stale read ("check-then-act")

You read a row, make a decision in JavaScript, then write. Between the read and the write, someone else changed the row. Your decision was based on a fact that is no longer true.

// ❌ check-then-act
const acc = await db.accommodation.findFirst({ where: { id } });   // reads status = CONFIRMED
if (acc.status !== "CANCELLED") {                                   // ...but another tab just cancelled it
  await tx.roomType.update({ data: { bookedRooms: { decrement: 1 } } }); // releases a room already released
}
// ✅ decide INSIDE the write — let the database check the precondition
const claimed = await tx.accommodation.updateMany({
  where: { id, status: { not: "CANCELLED" } },   // the condition is part of the write
  data:  { status: "CANCELLED" },
});
if (claimed.count === 0) return; // someone else got here first — do NOT touch the counter

The database evaluates where and applies data as a single atomic step. Nobody can sneak in between. count === 0 is how you learn you lost the race.

Mistake 2 — Moving a counter without a floor

Every release in this domain is a bare { decrement: 1 }. If a bug (or a race) releases twice, the counter goes negative. And negative is worse than wrong — available = totalRooms − bookedRooms now reports more rooms than exist, and the booking guard bookedRooms < totalRooms happily admits them. A counting bug silently becomes physical overbooking.

// ✅ guarded release — can never go below zero (this codebase already does this for seats:
//    see releaseSeat() in src/lib/registration-seat-db.ts)
await tx.roomType.updateMany({
  where: { id: roomTypeId, bookedRooms: { gt: 0 } },
  data:  { bookedRooms: { decrement: 1 } },
});

Mistake 3 — Trusting an id from the URL

A URL like /api/events/:eventId/hotels/:hotelId/rooms/:roomId has three ids and the user controls all of them. Checking that one of them belongs to you is not enough — you must check the whole chain: event belongs to my org → hotel belongs to that event → room belongs to that hotel. Break any link and an attacker swaps in an id from someone else's data.

Why this domain is worse than the others: the hotels/rooms routes predate the logging + IDOR sweeps that were applied elsewhere. The same directory contains a route that gets the security check right and one that doesn't — see H1. When you fix a class of bug, fix it in every file, not just the one in the ticket.

High H1 — Cross-tenant IDOR: the hotel is never bound to the event SHIPPED 7fcb256

src/app/api/events/[eventId]/hotels/[hotelId]/rooms/[roomId]/route.ts — GET :46, PUT :106, DELETE :208

All three handlers verify the event belongs to your org, then look up the room type like this:

The bug 
const event = await db.event.findFirst({
  where: { id: eventId, organizationId: session.user.organizationId! },  // ✅ event is org-checked
});
if (!event) return NextResponse.json({ error: "Event not found" }, { status: 404 });

const roomType = await db.roomType.findFirst({
  where: { id: roomId, hotelId },   // ❌ hotelId comes from the URL and is NEVER tied to `event`
});

Why it's wrong: the chain is broken in the middle. We proved your event is yours, then looked up a room using a hotelId nobody validated. The query happily matches a hotel belonging to a completely different organization.

The proof that this is drift and not a decision: the sibling file in the same folder does it correctly.

The correct pattern — already in the codebase, one file away 
// hotels/[hotelId]/rooms/route.ts  (the list/create route)
const hotel = await db.hotel.findFirst({
  where: { id: hotelId, eventId },   // ✅ hotel is bound to the event, which is bound to the org
});
if (!hotel) return NextResponse.json({ error: "Hotel not found" }, { status: 404 });

Failure scenario

An admin of org B calls /api/events/{their-own-event}/hotels/{orgA-hotel}/rooms/{orgA-room}. The event check passes (it is their event). The room lookup matches org A's row. GET dumps org A's room type including every booking with the full registration and attendee (guest names, emails, entry barcodes). PUT lets them rewrite org A's prices and capacity. DELETE removes the room type.

Fix

const roomType = await db.roomType.findFirst({
  where: { id: roomId, hotel: { id: hotelId, eventId } },   // full chain in one query
});

💡 What a junior dev should learn

Authorization is a chain, and it's only as strong as its weakest link. When a URL nests ids (/events/:a/hotels/:b/rooms/:c), every id must be provably reachable from the one you authorized. A useful habit: for each id in the route, ask out loud "what stops the user from putting someone else's id here?" If the answer is "nothing", you have an IDOR (Insecure Direct Object Reference).

Also: this is rated HIGH, not BLOCKER, only because the platform runs a single organization today. The bug is already written — it becomes exploitable the day a second org exists. Latent security bugs are still security bugs; they're just waiting for a config change to go live.

High H2 — Entry barcodes leak through the booking endpoints SHIPPED 7fcb256

accommodations/[accommodationId]/route.ts GET :41 & PUT response :249 · hotels/[hotelId]/rooms/[roomId]/route.ts GET :53

The bug 
const accommodation = await db.accommodation.findFirst({
  where: { id: accommodationId, eventId },
  include: {
    registration: { include: { attendee: true, ticketType: true } },  // ❌ the ENTIRE Registration row
    ...
  },
});
return NextResponse.json(accommodation);   // ❌ no redaction of any kind

Why it's wrong: include: { registration: true } returns every column of that row — including qrCode (the entry barcode) and dtcmBarcode (the Dubai compliance credential). Those are physical-access credentials: whoever holds one can print a badge and walk through the door.

On July 11 we shipped barcode-visibility.ts precisely to stop this, and applied it to the registration list + detail endpoints. MEMBER — a read-only, sponsor-side observer role — is deliberately excluded from the roles allowed to see barcodes. But the accommodation endpoints were never updated, so they hand the same credentials straight back.

Failure scenario

A MEMBER opens any booking's detail (or walks rooms/[roomId] once per room type) and harvests every booked guest's entry barcode. The registrations page carefully strips these for them; the accommodation page gives them away.

Fix

import { canViewEntryBarcode, redactBarcodeFields } from "@/lib/barcode-visibility";

const payload = canViewEntryBarcode(session.user.role, isApiKey)
  ? accommodation
  : redactBarcodeFields(accommodation);
return NextResponse.json(payload);

Better still: stop using include: { registration: true } and select only the fields the booking UI actually renders (name, email, ticket type).

💡 What a junior dev should learn

include is a security decision, not a convenience. include: { registration: true } means "ship every column of that table to the client, forever, including columns that don't exist yet." The safe default is select — an explicit allow-list. When someone later adds a sensitive column (a barcode, a password reset token, a salary), select keeps it private automatically; include leaks it automatically.

Second lesson: when you build a protection, grep for every place it should apply. The barcode fix was correct — it just wasn't applied everywhere the data flows. A protection with a hole is only as good as the hole is small.

High H3 — One request, two corrupted counters (room change + cancel) SHIPPED 71ce307

accommodations/[accommodationId]/route.ts :188-222

The PUT handles a room-type change and a status change in the same transaction. Both touch bookedRooms. They don't know about each other.

The bug 
// A) room-type change — runs whenever roomTypeId differs. No status check!
if (data.roomTypeId && data.roomTypeId !== existing.roomTypeId) {
  await tx.roomType.update({                       // release the OLD type
    where: { id: existing.roomTypeId },
    data: { bookedRooms: { decrement: 1 } },
  });
  await tx.roomType.updateMany({ ... });           // claim the NEW type
}

// B) status change — always passed the OLD roomTypeId
await applyRoomStatusTransition(tx, {
  prevStatus: existing.status,
  nextStatus: data.status ?? existing.status,
  roomTypeId: existing.roomTypeId,                 // ❌ the OLD type, even though (A) just moved the booking
});

Why it's wrong — two independent defects:

  1. Block (A) ignores status. A CANCELLED booking already gave its room back. Changing its room type releases the old type again (a room it doesn't hold) and claims a new one (a room a cancelled booking shouldn't have).
  2. Block (B) uses the stale room id. If you change the room type and cancel in one save, (A) releases the old type and claims the new one, then (B) releases the old type a second time.

Failure scenario (no race needed — one click)

Operator opens a booking, picks a different room type, sets status to Cancelled, hits Save once. Result: old room type bookedRooms −2 (for a single room it held), new room type +1 that nothing will ever release. The old type now under-counts (→ oversell) and the new type over-counts (→ falsely reports "no rooms available").

Fix

Compute the counter movement once, from the (before → after) pair, instead of letting two blocks each move it. That is exactly the shape of planSeatTransition() that already exists for registrations:

// one function, one decision, one move
const plan = planRoomTransition({
  prev: { status: existing.status,          roomTypeId: existing.roomTypeId },
  next: { status: data.status ?? existing.status,
          roomTypeId: data.roomTypeId ?? existing.roomTypeId },
});
// plan = { release?: roomTypeId, claim?: roomTypeId }  — at most one of each
if (plan.release) await releaseRoom(tx, plan.release);  // guarded, never below 0
if (plan.claim)   await claimRoom(tx, plan.claim);      // atomic capacity guard

💡 What a junior dev should learn

Two pieces of code that both move the same counter will eventually disagree. The fix is not "add a status check to block (A)" — that's patching a symptom and leaves the next edge case (room change + reinstate) to be discovered in production. The fix is to make it structurally impossible: one function takes the whole before-state and after-state, and returns the single net movement.

Notice the codebase already learned this lesson once — planSeatTransition() was written after the exact same class of bug leaked ticket soldCount. When you fix a bug, look for its siblings.

High H4 — Deleting a guest silently loses their room forever SHIPPED ce7ae60

prisma/schema.prisma :1390 · registrations/[registrationId]/route.ts :566 · speakers/[speakerId]/route.ts :423

The bug 
// schema.prisma
model Accommodation {
  registration   Registration? @relation(..., onDelete: Cascade)   // ❌
  speaker        Speaker?      @relation(..., onDelete: Cascade)   // ❌
}

The registration DELETE route carefully releases the ticket seat, releases the promo-code usage, and cleans up the attendee. It says nothing about accommodation — because it doesn't have to: Postgres deletes the booking row automatically via the cascade.

Why it's wrong: the database can delete the row, but it has no idea a counter in another table was tracking it. The booking vanishes; bookedRooms stays incremented. Forever.

Failure scenario

A room type has 10 rooms, all booked. An admin deletes one registrant (duplicate signup, test data, a no-show being cleaned up). The Accommodation row disappears; bookedRooms stays at 10. The room type now reports sold out with 9 real bookings and an empty room, and every future booking attempt fails with NO_ROOMS_AVAILABLE. Repeat during event cleanup and the whole hotel block silently jams. There is no recovery path except hand-written SQL — no reconciliation script exists for bookedRooms.

Fix

Two options, and the right answer is probably both:

  1. Release the room explicitly in the delete transaction (before the cascade fires), the same way the seat and promo counters already are.
  2. Write a reconcile-bookedrooms.ts script (dry-run by default, like reconcile-soldcounts.ts) that recomputes every counter from row-truth, so drift that already happened can be repaired.

💡 What a junior dev should learn

onDelete: Cascade deletes rows, not side effects. A cascade is invisible — it happens in the database, so nothing in your TypeScript runs. Any counter, aggregate, external booking, or file on disk that the deleted row was responsible for is silently orphaned.

Rule of thumb: if a row's existence is counted somewhere else, it must not be cascade-deleted — either delete it explicitly in application code, or use onDelete: Restrict so the database forces you to deal with it. Denormalized counters are fast, but you own their correctness by hand — the DB won't help you.

High H5 — Two cancels release two rooms (check-then-act on status) SHIPPED 71ce307

accommodations/[accommodationId]/route.ts :96-106 & :218-244 · MCP tools/accommodations.ts :468

The route reads the booking before the transaction, and passes that snapshot's status into the counter logic. Inside the transaction, the row is written with a bare { id } predicate:

The bug 
const existing = await db.accommodation.findFirst({ where: { id, eventId } }); // status: CONFIRMED

await db.$transaction(async (tx) => {
  await applyRoomStatusTransition(tx, {
    prevStatus: existing.status,          // ❌ a snapshot from before the tx — may already be stale
    nextStatus: data.status,
    roomTypeId: existing.roomTypeId,
  });
  await tx.accommodation.updateMany({
    where: { id: accommodationId,
             ...(data.expectedUpdatedAt && { updatedAt: ... }) },  // ❌ the guard is OPTIONAL
    data:  { status: data.status, ... },
  });
});

There is an optimistic-lock token (expectedUpdatedAt) — but it's optional ("during rollout"), and omitting it only logs a warning. With no token, the write has no precondition at all.

Failure scenario

Two browser tabs both show the booking as CONFIRMED. Both click Cancel. Both read prevStatus = CONFIRMED, both take the "active → cancelled" branch, and both decrement bookedRooms. One booking, two rooms released. The mirror case (two reinstates) claims two rooms for one booking. The June-29 atomic claim protects capacity — it does not protect idempotency.

Fix

// make the status itself the precondition — the DB decides who wins
const claimed = await tx.accommodation.updateMany({
  where: { id: accommodationId, status: existing.status },  // ✅ "only if it's still what I read"
  data:  { status: nextStatus, ... },
});
if (claimed.count === 0) throw new Error("STALE_WRITE");    // someone beat us — counter untouched
// only NOW move the counter

💡 What a junior dev should learn

An optional safety check is not a safety check. If a guard only works when the caller remembers to pass a token, then every caller who forgets (or every attacker who omits it) bypasses it. Make the precondition part of the write itself so it can't be skipped.

And note the ordering: claim the row first, then move the counter. If you move the counter first and the row-claim then fails, you've already corrupted the count (the transaction rolls it back here — but only because it's inside one; get that wrong and you leak).

High H6 — DELETE releases a room that was already released SHIPPED 71ce307

accommodations/[accommodationId]/route.ts :321-350

The bug 
const accommodation = await db.accommodation.findFirst({ where: { id, eventId } }); // pre-tx read

await db.$transaction(async (tx) => {
  if (accommodation.status !== "CANCELLED") {          // ❌ decision from the stale snapshot
    await tx.roomType.update({ data: { bookedRooms: { decrement: 1 } } });
  }
  await tx.accommodation.delete({ where: { id } });    // ❌ no status precondition
});

Failure scenario

Tab A cancels the booking (room released, bookedRooms −1). Tab B — opened earlier, still showing CONFIRMED — deletes it. B's snapshot says "not cancelled", so the transaction decrements again. One held room, two releases. Combined with the missing floor (M1), the counter goes negative and the room type starts advertising more rooms than the hotel has.

Fix

await db.$transaction(async (tx) => {
  // delete conditionally on the status we believe, and learn whether we were right
  const gone = await tx.accommodation.deleteMany({
    where: { id, status: { not: "CANCELLED" } },
  });
  if (gone.count === 1) {
    await releaseRoom(tx, accommodation.roomTypeId);   // guarded release
  } else {
    await tx.accommodation.deleteMany({ where: { id } }); // it was cancelled — delete, don't release
  }
});

💡 What a junior dev should learn

Same lesson as H5, in a different costume — which is the point. "Read a row, branch on it, then write" is a bug pattern, not a code pattern. Once you can see it, you'll spot it everywhere: it's the same shape as the overbooking bug this domain already fixed, the same shape as the ticket-seat leak, and the same shape as the double-send bug fixed in the email worker last week.

The generic cure has a name: make the write conditional on the state you assumed, and treat count === 0 as "my assumption was wrong — undo nothing, change nothing."

High H7 — ~30 failure paths log nothing DEFERRED

all 6 route files · src/services/accommodation-service.ts :109-199

EA-SYS has a hard rule: every failure path must log. This domain predates the sweeps that enforced it elsewhere. Verified silent returns include the business rejections that operators actually hit:

  • "Cannot delete hotel with existing bookings" (400) — hotels/[hotelId]/route.ts:218
  • "Cannot delete room type with existing bookings" (400) — rooms/[roomId]/route.ts:225
  • "Total rooms cannot be less than booked rooms" (400) — rooms/[roomId]/route.ts:131
  • "No rooms available" (400) and "Event not found" (404, the cross-org denial) — throughout

Worse, the REST create path logs none of the service's ten rejection codes — the route maps result.code to an HTTP status and returns, with no apiLogger call. The MCP twin of the same operation is logged (runTool auto-warns on error results), so the same failure is visible when an agent causes it and invisible when a human does.

Failure scenario

During a live event, room assignments start failing. Ops checks /logs and finds… nothing. Was it sold out? A duplicate assignment? A cross-org 404? The exact signal you need is the one that was never written. (An attacker probing hotel/room ids for the H1 IDOR is likewise invisible.)

Fix

if (!result.ok) {
  apiLogger.warn({ msg: "accommodation:create-rejected", code: result.code,
                   eventId, userId: session.user.id });
  return NextResponse.json({ error: result.message, code: result.code },
                           { status: HTTP_STATUS_FOR[result.code] });
}

💡 What a junior dev should learn

A 4xx is a conversation with the user; a log line is a conversation with the person debugging at 2am. Returning the right status code is not the same as handling the failure. If the only record of a rejection is an HTTP response that has already been thrown away by the browser, then the failure effectively never happened — and you will not be able to answer "why did this break?" the next morning.

Cheap rule: every return … status: 4xx gets a warn; every 500 gets an error. It takes 10 seconds to write and saves hours.

High H8 — The UI silently swallows errors (the button just… does nothing) DEFERRED

(dashboard)/events/[eventId]/accommodation/page.tsx :189, :219, :236

The bug 
const res = await fetch(url, { method: "POST", body: ... });
if (res.ok) {                       // ❌ no else
  toast.success("Hotel saved");
  refresh();
}
// a 400 / 403 / 500 falls off the end of the function. Nothing is shown. Nothing is logged.

Failure scenario

The operator clicks Delete hotel and confirms. The server correctly refuses with 400 "Cannot delete hotel with existing bookings" — a message that would have told them exactly what to do. The UI shows nothing at all. The hotel simply doesn't disappear. The operator clicks again, then reports "the delete button is broken."

Same in the assign-room dialog: a failed registrations fetch is mapped to [] (res.ok ? res.json() : []), so the picker cheerfully says "No registrations for this event yet" for an event with 800 registrants.

Fix

const res = await fetch(url, { method: "POST", body: ... });
if (!res.ok) {
  const data = await res.json().catch(() => ({}));
  toast.error(data.error ?? "Failed to save hotel");   // show the server's actual reason
  console.error("hotel-save-failed", res.status, data);
  return;
}
toast.success("Hotel saved");
refresh();

💡 What a junior dev should learn

fetch() does not throw on 400 or 500. This is the single most common misunderstanding in frontend code. await fetch() resolves successfully for an HTTP error — only a network failure rejects. So try/catch around a fetch catches almost nothing, and if (res.ok) { … } with no else silently discards every server-side error.

The tell-tale symptom is a user saying "the button does nothing". That sentence almost always means: the server said no, and the frontend didn't listen. Note the same file handles this correctly in handleStatusUpdate — so one developer knew, and the pattern didn't spread.

Med M1 — Every release is unguarded: the counter can go negative SHIPPED 71ce307

accommodation-rooms.ts :29 · accommodations/[accommodationId]/route.ts :200, :342

All three release sites use a bare { decrement: 1 } with no bookedRooms > 0 floor — unlike releaseSeat() in registration-seat-db.ts, which guards exactly this. This is the amplifier: it's what turns every double-release above from "off by one" into bookedRooms = −1, which makes available exceed totalRooms and lets the bookedRooms < totalRooms guard admit more bookings than there are rooms. Fix: guarded updateMany({ where: { id, bookedRooms: { gt: 0 } } }).

💡 Lesson

Clamp your invariants at the boundary. A counter that represents a physical quantity (rooms, seats, stock) can never be negative in the real world, so it must never be negative in the database. Guarding the decrement means a counting bug stays a reporting bug instead of escalating into a double-booking bug.

Med M2 — Guest count is validated on create, never on edit DEFERRED

accommodations/[accommodationId]/route.ts :16

createAccommodation() enforces GUEST_COUNT_EXCEEDS_CAPACITY. The PUT's Zod schema only checks min(1) — so you can edit a booking to 6 guests in a 2-person double, or move a 4-guest booking into a single-occupancy room. Price is recomputed; capacity is not re-checked. The hotel receives a rooming list that doesn't fit in the rooms.

💡 Lesson

An invariant enforced only at create is not an invariant. Rows don't stop existing after they're created — they get edited. Every rule that makes a row valid must be re-checked on every path that can change it (REST PUT, MCP, bulk import, admin script). Put the rule in one shared function and call it from all of them.

Med M3 — There is no status state machine on the server DEFERRED

accommodations/[accommodationId]/route.ts :18 · tools/accommodations.ts :430

Both the REST enum and the MCP set accept any → any transition. Only the dashboard's buttons constrain the flow, and buttons are not security. An agent told "mark all bookings checked out" can push a CANCELLED booking to CHECKED_OUT — and because the applier sees "was cancelled, will be active", it re-claims a room for a booking nobody is coming to. CHECKED_OUT → PENDING resurrects a completed stay.

💡 Lesson

If your status has arrows in the design doc, it needs a table in the code. A Zod enum validates that a value is a status; it says nothing about whether it's a legal next status. Model the allowed transitions explicitly (const NEXT: Record<Status, Status[]>) and reject the rest — otherwise the UI is your only enforcement, and the API is a way around the UI.

Med M4 — Lowering totalRooms is check-then-act SHIPPED 71ce307

rooms/[roomId]/route.ts :106→:138 · MCP tools/accommodations.ts :243

Read bookedRooms (line 106), compare in JS (line 131), write unconditionally (line 138) — no transaction, no predicate. Med M5 — A lost create race returns an opaque 500 instead of a clear 409 DEFERRED

accommodation-service.ts :174 (check) → :212 (create) → :260 (catch)

The "already has a room" check is a pre-transaction read, so two concurrent creates for the same person both pass it. The @unique on registrationId correctly stops the loser (data stays clean, the counter rolls back) — but the catch block doesn't recognise Prisma's P2002, so it falls through to code: "UNKNOWN" with a raw "Unique constraint failed…" string. The caller can't tell "already booked" from "server exploded", and the MCP auto-pivot hint never fires. Fix: map P2002 to REGISTRATION_HAS_ACCOMMODATION / SPEAKER_HAS_ACCOMMODATION.

💡 Lesson

A unique constraint is a feature, not just a safety net — catch it and translate it. The DB is the only thing that can truly enforce "one booking per guest" under concurrency, so let it. But a raw constraint error is a leaky implementation detail; convert it into the domain error you'd have returned if the check had caught it. The happy path and the race path should be indistinguishable to the caller.

Med M7 — Hotel/room CRUD awaits the audit write with no catch DEFERRED

hotels/[hotelId]/route.ts :147, :230 · rooms/route.ts :156 · rooms/[roomId]/route.ts :159, :237

A transient failure on the audit log insert — which happens after the hotel was already deleted — returns 500 "Failed to delete hotel" to a client whose hotel is deleted.
