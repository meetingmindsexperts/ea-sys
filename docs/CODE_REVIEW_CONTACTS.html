Verdict

The foundations are sound: cross-tenant isolation is clean everywhere (every contact lookup is findFirst({ where: { id, organizationId } }); the MCP layer injects organizationId from the validated API key and never trusts tool input; wrong-org ids 404 without leaking existence). The May-18 write-guard fix held — every write path runs denyReviewer. CSV export escapes formula injection. The email-change race is correctly mapped to a 409.

The problems cluster in three places, and each is an instance of a principle this codebase already knows:

  1. The guard was put on writes, and reads were forgotten. denyReviewer gates every contacts mutation and nothing gates the reads — so the whole org CRM, private notes included, is exportable by roles that should never see it (H1). This is the same class as the check-in review's H7/H8 (barcodes were returned to everyone because the guard sat on the POST).
  2. One writer skipped the normalization every other writer applies. Five paths lowercase the email; the dashboard create path doesn't, and the unique index is case-sensitive — so the system mints duplicate people, and the EU mirror silently drops one of each pair (H2).
  3. The CRM goes stale silently. Agent-driven speaker edits never reach the Contact (H4), bulk agent imports produce name+email husks (M3), imports into an event never record the event on the contact (M6), and the EU mirror never learns about a rename, merge or delete — so a typo'd email lives on forever as a live-looking marketing record (H3).

How to read this document

Each finding has: the issue (what the code does), failure scenario (concrete inputs → concrete damage — if you can't write this, it isn't a finding), and suggested fix. The green “What to learn” box states the general engineering principle the bug is an instance of; they're collected in § Lessons at the bottom.

Deliberate decisions — NOT findings

These were verified as intentional and excluded from the report. Don't "fix" them:

  • Enrich-only sync — syncToContact never clears a populated Contact field with null/""/undefined (June 29 decision).
  • Contact email is immutable via the general PUT (400 EMAIL_IMMUTABLE), and the dedicated PATCH /email deliberately does not cascade to Speaker/User/Registration — Contact is a CRM snapshot, and propagating from it would re-introduce the identity drift the email-change feature eliminated.
  • The central-sync read-modify-write window (cross-source lost write between our GET and POST) — documented and accepted in docs/CONTACTS_CENTRAL_SYNC.md.
  • PII leaving to the EU Supabase mirror — an explicit, signed-off data-sharing decision. (How it flows — key handling, PII in logs — remained in scope; see M11.)
  • A person with two email addresses is two contacts — the store is keyed @@unique([organizationId, email]); known and accepted.
  • The mirror re-creating a row deleted out-of-band while the person still exists in EA-SYS — that is the definition of a self-healing mirror.

HIGH

High H1 — The entire org contact book is readable and exportable by ONSITE, MEMBER, and internal-domain REGISTRANT

src/app/api/contacts/export/route.ts:6-31 · contacts/route.ts:34 (list GET) · contacts/tags/route.ts:6 · contacts/[contactId]/route.ts (detail GET). Authorization is getOrgContext only — no denyReviewer, no role check. Found independently by two review angles.

The issue

// export/route.ts — the whole org CRM, one GET, no role gate
const contacts = await db.contact.findMany({
  where: { organizationId: ctx.organizationId },
  orderBy: { createdAt: "desc" },
  select: { title, firstName, lastName, email, organization, jobTitle,
            specialty, registrationType, bio, phone, tags, notes },  // full PII, every contact
});

denyReviewer is applied to every contacts write and to none of the reads. getOrgContext returns a valid context for any session user carrying an organizationId — and per the RBAC model that includes ONSITE (per-event desk temps), MEMBER (read-only sponsor-side observers), and internal-domain REGISTRANTs (org attached since the June-16 rule, but not staff). src/proxy.ts explicitly skips /api/*, so the API layer is the authoritative gate — and it isn't gating.

Failure scenario

A temp desk worker is hired for one event and given an ONSITE account (deliberately scoped to that event's registrations). They call GET /api/contacts/export and receive a CSV of the entire organization's CRM — every attendee and speaker across every event, with email, phone, bio and the organizer's private notes. Nothing is audited, so there is no record it happened, and nothing is rate-limited, so it can be pulled repeatedly. The same applies to a MEMBER (a sponsor-side observer) and to any internal-domain registrant.

Suggested fix

  • Gate the four read routes to org staff (SUPER_ADMIN / ADMIN / ORGANIZER + API keys), the way barcode-visibility.ts gates door credentials — a dedicated predicate, not a finance/`denyReviewer` reuse.
  • Audit the export (who, when, how many rows) and rate-limit it.
  • Owner decision needed: should MEMBER see the contact book at all? It is finance-capable by design, but a full cross-event CRM with private notes is a data-minimization question, separate from ONSITE/REGISTRANT (who clearly should not).

Actual fix — SHIPPED a0744ef (July 14, 2026)

Owner decision: staff + MEMBER may read; ONSITE / REGISTRANT / REVIEWER / SUBMITTER may not. New src/lib/contact-visibility.ts is the boundary — canViewContacts(role, isApiKey) + denyContactAccess(ctx) (which logs the refusal itself, so no call site can forget). It is deliberately its own predicate rather than a reuse: denyReviewer is a write guard that blocks MEMBER (who is allowed here), FINANCE_ROLES includes ONSITE (who is not), and BARCODE_ROLES is the exact inverse of what we want. Fails closed on an unknown role. Wired into all four read routes; the export additionally writes an EXPORT audit row (who, when, row count, role, IP) and is rate-limited to 10/hr/org. Two previously-silent rejections (the duplicate-email 409 and the detail 404) now log. +51 tests: the truth table incl. the fail-closed cases, every role × every read route, "a blocked role never reaches the database", the audit row, and the 429.

What to learn. A write-guard is not an access-control model. When you add a guard to the mutations of a resource, ask immediately: who can read this, and is reading it harmful? A bulk export endpoint is the highest-value object in any CRM — it deserves a stricter gate than the rows it exports, plus an audit trail. This exact class was already fixed once in this codebase (check-in H7/H8: barcodes were readable by everyone because the guard sat on the POST). One domain learning a lesson does not teach the others.

High H2 — REST contact create never normalizes the email → duplicate people, and the mirror silently drops one of each pair

src/app/api/contacts/route.ts:14 (schema — no transform), :150-167 (dup-check + create use the raw string). Every other writer normalizes: agent/tools/contacts.ts:42, lib/contact-sync.ts:73, contacts/import/route.ts:88, contacts/import-eventsair/route.ts:95, lib/email-change.ts (normalizeEmail). Found independently by two review angles.

The issue

const existing = await db.contact.findUnique({
  where: { organizationId_email: { organizationId: ctx.organizationId, email } },  // raw, case preserved
});
...
const contact = await db.contact.create({
  data: { organizationId: ctx.organizationId, ...rest, email },  // no .trim().toLowerCase()
});

The @@unique([organizationId, email]) index is case-sensitive. Five of six write paths lowercase before touching it; the dashboard create path does not — so its own duplicate check is bypassable by casing, and it can insert a row that collides logically (but not physically) with an existing one.

Failure scenario

  1. An organizer adds John@Hospital.com from the dashboard → row A.
  2. John later registers for an event → syncToContact lowercases to john@hospital.com → the dup check misses row A → row B is created. Two contacts for one person, each accumulating half of his data, half of his event history, half of his tags.
  3. Downstream, contacts-central-sync.ts:167-171 dedups the outbound payload by lowercased email with last-write-wins — so one of the two rows never reaches the EU mirror at all, silently: no log, no counter, no failure.

Suggested fix

One line: run the create schema's email through the existing normalizeEmail() (src/lib/email-change.ts) like every other path. While there, map the create's P2002 race to a 409 instead of the current 500 that echoes the raw Prisma message as detail. A one-time case-collapse backfill will be needed for rows already minted.

Actual fix — SHIPPED a0744ef (July 14, 2026)

The create schema now trims + lowercases before validation (z.preprocess), so the dup-check and the insert both see the canonical value — and a pasted trailing space is accepted rather than 400'd (a plain .transform() after .email() would have rejected it). normalizeEmail() itself wasn't reused because it returns string | null, which would have made a required field nullable. The P2002 race now maps to a 409 and no longer echoes the Prisma message. +5 tests (lowercased dup-check, lowercased persist, case-variant 409, race 409 with no detail leak).

Still open

No backfill has been run — any case-variant duplicate pairs already in production remain. This fix is forward-looking (it stops new ones). A one-time case-collapse/merge script is tracked in ROADMAP.

What to learn. If a value has an invariant (“emails are stored lowercase”), the invariant must be enforced at the boundary of the data, not in each writer — a shared normalizer called by convention will eventually be forgotten by exactly one caller, and that caller is enough to break it. The strongest version is a database-level guarantee (a citext column or a functional unique index on lower(email)), because then no code path can get it wrong. The weakest version is what we have: six writers, five of whom remembered.

High H3 — The central mirror never learns about a rename, merge, or delete — orphaned identities live forever

src/lib/email-change.ts:35-44 (repointOrgContactEmail — updates or deletes the Contact) · src/app/api/contacts/[contactId]/route.ts:226 (DELETE) · src/lib/contacts-central-sync.ts — contains no DELETE and no rename handling; the only mutation is the POST upsert at :293-297.

The issue

// email-change.ts — EA-SYS side handles both cases correctly…
if (collision) { await tx.contact.delete({ where: { id: old.id } }); return "merged"; }
await tx.contact.update({ where: { id: old.id }, data: { email: newEmail } });

// contacts-central-sync.ts — …but the mirror is email-keyed and APPEND-ONLY
const postRes = await fetch(`${base}/rest/v1/${table}?on_conflict=email`, { method: "POST", ... });

The mirror's primary key is the email address, and the sync only ever upserts. Nothing communicates “this email is no longer a person” to the target.

Failure scenario

jane@hosptial.com (typo) is created and syncs to the mirror with her full events_attended, tags, and ea_synced = true. An organizer fixes the typo via the email PATCH → EA-SYS now holds jane@hospital.com. The next incremental tick pushes a new mirror row. The old row is never touched again — and the mirror feeds mailchimp_* columns, so the typo'd address remains a live-looking, EA-maintained marketing record forever. Jane now exists twice downstream. The nightly reconcile does not heal this: it only upserts, never prunes. Merges and deletes orphan rows identically. There is also a snapshot race: if a tick reads its batch and the rename commits mid-tick, the tick re-POSTs the old email and refreshes its last_updated, making the stale row look freshly maintained.

Suggested fix

On updated / merged / deleted, tell the mirror: PATCH the old-email row (e.g. ea_synced = false, or an ea_merged_into pointer), or at minimum emit an operator-visible log line so the mirror can be cleaned. Deletes-are-kept may be defensible (“the mirror is a historical superset”) — the rename case is a defect regardless, because both rows look like current people.

What to learn. An append-only mirror keyed on a mutable field is not a mirror — it is an accumulator. Before you build a sync, enumerate every operation the source supports (create, update, rename-the-key, merge, delete) and decide what each one means on the target. “We only ever upsert” quietly encodes the decision that keys are immutable, and that decision is wrong the first time someone fixes a typo.

High H4 — MCP update_speaker syncs name + email only, while the REST PUT syncs the full profile

src/lib/agent/tools/speakers.ts:328-334 vs src/app/api/events/[eventId]/speakers/[speakerId]/route.ts:318-341.

The issue

// MCP executor — the entire sync payload
syncToContact({
  organizationId: ctx.organizationId,
  eventId: existing.eventId,
  email: updated.email,
  firstName: updated.firstName,
  lastName: updated.lastName,
}).catch(...)

// REST PUT — threads title / role / additionalEmail / organization / jobTitle /
//            phone / photo / city / country / bio / specialty / registrationType

update_speaker accepts and persists title, bio, organization, jobTitle, phone, city, country, specialty and photo — but sends none of them to the Contact store. And because syncToContact is enrich-only, a name+email payload against an existing contact is effectively a no-op: the call succeeds, logs nothing, and changes nothing.

Failure scenario

An n8n workflow (or a Claude conversation) updates 40 speakers' affiliations and phone numbers before a congress. All 40 Speaker rows are correct. Zero of the org Contact records are updated — and since the EU mirror enriches its scalars from Contact, none of it propagates downstream either. The CRM and the marketing mirror quietly hold last year's affiliations, with no error anywhere.

Why this existed — the instructive part

This is the same bug the team already found and fixed for registrations on July 13. The fix comment still sits in the code: registration-service.ts:1394 — “Full-merge Contact sync (REST's shape — MCP used to sync name+email only)”. That fix worked because the registration update was extracted into a service both callers share. The speaker update never was — only createSpeaker got a service — so REST PUT and the MCP executor remain two mirrored implementations, and they drifted exactly as the house rule predicts.

Suggested fix

Extract speaker-service.updateSpeaker() and delegate both callers — the template is registration-service.updateRegistration(), sitting in the next file. (Minimum viable: copy the REST sync shape into the executor with a drift-guard comment — but that just re-arms the same trap.)

Actual fix — SHIPPED 5849962 (July 14, 2026)

The extraction, not a patched payload. New speaker-service.updateSpeaker() owns the optimistic-locked write, the tag mirror onto the registration facet, the decline cascade, the full contact sync, the audit row and the stats refresh. Both callers are now thin: REST keeps session auth, denyReviewer, EMAIL_IMMUTABLE, Zod, the org-scoped event lookup and the HTTP status map; MCP keeps loose-input coercion, its org-scoped speaker lookup and the JSON-RPC response shape.

Extra drift found and fixed while extracting (all in the MCP copy, none of it in the original finding): its audit row recorded status only — now full before/after snapshots, so the Activity timeline can render diffs; the write is now bound to {id, eventId} so a mis-scoped call can't touch a sibling event's speaker; stats now refresh for the speaker's own event (it used ctx.eventId); and photo is coerced instead of blind-cast (input.photo as string | null let a non-string reach Prisma raw). The REST audit write became fire-and-forget-with-logged-catch (M13 class) and its event-not-found 404 now logs.

+20 service tests, and the regression guard asserts the EFFECT, not the call — a test that merely checked “syncToContact was called” would have passed against the bug. All 2666 pre-existing tests (including every REST speaker-route test) pass unchanged through the delegation, which is the real evidence the extraction preserved behavior. No MCP schema change → no package bump.

Still open — the same class, one file over

M3 (MCP create_registrations_bulk syncing name+email only) is the identical bug in the bulk path, and bulk paths are deliberately NOT service-extracted (different mechanics: per-row error capture, createMany). It needs its own fix.

What to learn. When you fix a drift bug, fix the duplication that allowed it, not just the instance — otherwise you have proven the mechanism works and left it loaded. And note what made this bug invisible: an enrich-only sync means a wrong-but-well-formed payload fails silently and successfully. Any code path whose failure mode is “succeeds and does nothing” needs a test that asserts the effect, not the call.

Medium M1 — syncToContact is check-then-act; the P2002 loser is swallowed, dropping an event association and all enrich data

src/lib/contact-sync.ts:105-137 (race) and :156-162 (the swallow).

const existing = await db.contact.findUnique({ where: compositeKey, select: { eventIds: true } });
if (existing) {
  const eventIds = existing.eventIds.includes(data.eventId) ? existing.eventIds : [...existing.eventIds, data.eventId];
  await db.contact.update({ where: compositeKey, data: { ...optional, eventIds } });
} else {
  await db.contact.create({ data: { ...optional, eventIds: [data.eventId] } });
}
...
} catch (err) {
  apiLogger.warn({ msg: "Contact sync failed", ... });   // the eventId + enrich data are GONE
}

Failure scenario

A brand-new email registers for Event A (public register) at the same moment an admin adds them to Event B. Both findUnique return null; both create; the loser hits P2002 on the unique index and lands in the catch as a warn. Its eventId and every enrich field are dropped, and nothing ever re-fires the sync for that pair — so the Contact permanently lacks Event B, and the mirror's events_attended / event_speciality / event_type lack it too (the nightly reconcile recomputes from Contact.eventIds, which is itself wrong — so the reconcile cannot heal this one). The update/update variant loses an event the same way: both read [X], one writes [X,A], the other [X,B], last writer wins.

Suggested fix

Retry-once on P2002 by falling through to the update branch, and make the array append atomic (eventIds: { push } with a dedup-on-read, or a guarded array_append … WHERE NOT eventIds @> …).

What to learn. “Failures here are non-blocking” is a contract about not breaking the caller — it is not a licence to manufacture failures and then swallow them. A check-then-act against a unique index will lose races; wrapping it in a catch converts a race into permanent, invisible data loss. And note the second-order trap: a “self-healing” reconcile heals nothing when it recomputes from the very field the race corrupted.

Medium M2 — Audit coverage is 2 of 9 mutation paths; DELETE writes nothing

Only contacts/[contactId]/email/route.ts:98 and MCP update_contact (tools/contacts.ts:153) write an AuditLog row.

MutationAudited?
REST POST /api/contacts (create)❌
REST PUT /api/contacts/[id]❌ (MCP's equivalent does audit — inconsistent)
REST DELETE /api/contacts/[id]❌ ← destructive, unrecoverable, no trace
REST PATCH /contacts/bulk-tags❌ (the registrations + speakers twins are audited)
CSV import · EventsAir import❌ · ❌
PATCH /contacts/[id]/email✅ before/after + ip
MCP create_contact · update_contact❌ · ✅

Failure scenario

A contact vanishes from the CRM. “Who deleted it, when, and what did it contain?” is unanswerable — there is no audit row and the row itself is gone. Same for “who bulk-retagged 500 contacts last Tuesday” (tags drive email cohorts and certificate eligibility) and “who imported these 3,000 rows”.

Suggested fix

Audit rows on DELETE (snapshot the row in changes) and bulk-tags first; then create/import/PUT for consistency with the MCP path.

What to learn. Auditing isn't a compliance checkbox — it's the difference between “we can answer the question” and “we cannot”. The rule of thumb: any operation whose effect cannot be reconstructed from the resulting state must write an audit row. A DELETE is the purest example, because the evidence is precisely what it destroys.

Medium M3 — MCP create_registrations_bulk syncs name+email only, and its “parity” comment is now false

src/lib/agent/tools/registrations.ts:973-982 vs src/services/registration-service.ts:803-828 (~20 fields).

The comment claims parity with the single-create executor — true only against the pre-service version. The single path now delegates to registration-service, whose sync threads title, role, organization, jobTitle, phone, city, state, zip, country, bio, specialty, memberId, studentId…

Failure scenario

An agent bulk-imports 100 delegates from a sponsor's spreadsheet. Their organization, job title and country are persisted to the Attendee rows (they even appear in the confirmation emails) — but the 100 resulting CRM contacts are name+email husks. Same root cause as H4: a copy that didn't move when the original did.

What to learn. A comment asserting “parity with X” is a liability the moment X changes — it makes the next reader stop checking. If two things must stay in sync, encode it in code (one shared function) or in a test, never in prose.

Medium M4 — Tag normalization drift: 2 of 5 writers skip normalizeTag, and every filter is exact-match

✅ REST POST/PUT/bulk-tags (z.string().transform(normalizeTag)) · ✅ MCP update_contact (tools/contacts.ts:118) · ❌ MCP create_contact (tools/contacts.ts:72 — raw, un-normalized, un-capped) · ❌ CSV import (import/route.ts:112 → parseTags trims only).

Failure scenario

An n8n workflow creates contacts tagged committee. The dashboard filter (and the ?tags=Committee API pull that feeds the Webflow people-sync) is exact-match — so those contacts are invisible to it. This is precisely the case-sensitivity pain already hit once in the EA-SYS → Webflow sync; the tag writers, not the readers, are where it originates.

What to learn. Same principle as H2, different field: normalize at the boundary or not at all. A “normalize on write, exact-match on read” design is only safe if every writer normalizes — and the way to guarantee that is to make the un-normalized path impossible, not to remember.

Medium M5 — Two “import contacts” features with opposite semantics on existing rows; CSV skips are unitemized

contacts/import/route.ts:126-129 — createMany({ skipDuplicates: true }) (existing rows untouched) vs contacts/import-eventsair/route.ts:145-161 — enrich-upsert + eventIds append (the org-standard semantics).

Failure scenario

An organizer receives a richer contact list from a society and re-imports the CSV to fill in missing phone numbers and job titles. Nothing happens to any existing contact — every row is silently skipped — and the response reports only a count (itself derived from two racing count() queries, see L5), never which emails were skipped. The operator believes the enrichment worked.

What to learn. Two features with the same verb in the UI must not have opposite semantics under it. And a bulk operation that skips items owes the operator an itemized account of what it skipped — a bare count is indistinguishable from success.

Medium M6 — Neither import-contacts → event path appends the event to Contact.eventIds

events/[eventId]/registrations/import-contacts/route.ts — zero syncToContact calls and zero audit writes · events/[eventId]/speakers/import-contacts/route.ts — audits, but also never touches Contact.eventIds.

Failure scenario

A contact is imported from the CRM into Event X as a registration. Every other participation path appends the event id to Contact.eventIds — this one doesn't. Since contacts-central-sync.ts:113 derives events_attended from Contact.eventIds, that person's attendance at Event X never appears in the EU mirror, and the contact-detail event history undercounts — permanently, unless some later edit happens to re-sync them.

What to learn. Skipping the field-level sync here is defensible (the Contact is the source of truth for the fields). But “this path doesn't need the sync” was applied to the whole call — including the one part that is new information (the person now attends this event). When you skip a shared side-effect, enumerate what it does; you're usually skipping more than you meant to.

Medium M7 — REST ⇄ MCP update_contact field drift, including a round-trip-breaking limit mismatch

contacts/[contactId]/route.ts:19-39 vs agent/tools/contacts.ts:96-122.

(a) A contact's state/zipCode arrive wrong via a sync. They are uneditable from the dashboard — those columns aren't in the REST create or update schema at all. Medium M8 — 13 post-auth 4xx paths return without logging

contacts/route.ts:155 (409 duplicate) · [contactId]/route.ts:53 (404), :145 (EMAIL_IMMUTABLE), :175 (404), :216 (404), :222 (403 protected) · [contactId]/email/route.ts:59, 68, 73, 86, 123 · bulk-tags/route.ts:51 · import/route.ts:39.

House rule: every failure path logs. These 13 return silently, so a support question (“my import did nothing”, “I get a 409 and don't know why”) leaves no trace in /logs. This is the contacts instance of the same cluster already deferred from the communications review. One apiLogger.warn per branch.

What to learn. A 4xx is not “nothing happened” — it's the system refusing a real user's real intent. If it isn't logged, you cannot answer the only question that matters afterwards: how often, and to whom?

Medium M9 — bulk-tags has no rate limit and no cap on contactIds

contacts/bulk-tags/route.ts:18 (guarded but un-limited), :55-72 (unbounded $transaction).

Every sibling write is limited — create 50/hr, update 100/hr, email-change 30/hr, import 10/hr. This one isn't, and its contactIds array is validated only with .min(1) (no max), then executed as one update per id inside a single transaction. A caller can pass thousands of ids and loop freely; and because the route is also un-audited (M2), the damage is invisible afterwards.

What to learn. When you add a limit to a family of endpoints, the one you forget is the one that gets abused. Bulk endpoints deserve two limits: rate (calls per window) and size (items per call).

Medium M10 — SSRF: downloadExternalPhoto fetches an EventsAir-supplied URL with no allowlist

src/lib/storage.ts:214-224, reached from contacts/import-eventsair/route.ts:128-130 with contact.photo.url straight off the EventsAir API response.

export async function downloadExternalPhoto(externalUrl: string) {
  if (externalUrl.startsWith("/uploads/")) return externalUrl;
  if (externalUrl.includes(".supabase.co/storage/")) return externalUrl;
  const res = await fetch(externalUrl, { signal: AbortSignal.timeout(10000) });  // no scheme/host allowlist

Failure scenario

The only constraints are content-type ∈ {jpg,png,webp} and size < 500 KB — which bound exfiltration but not a blind SSRF. A crafted EventsAir dataset makes the box issue GETs to http://169.254.169.254/… (instance metadata) or internal services; status and timing are observable through the error logs. Trust here is only “the org's own EventsAir account” — but that is still external, attacker-influenceable data reaching a server-side fetch.

Suggested fix

An allowlist before the fetch: https only, deny RFC-1918 / link-local / metadata IPs — the same shape as the hardened certificates/pdf-loader.ts guard written for the certificates BLOCKER.

What to learn. “The URL comes from a trusted integration” is not a security boundary — the integration's content is still attacker-influenceable data. Any server-side fetch() of a URL you did not construct yourself needs an allowlist. We have already written this guard once in this codebase; the lesson is to reuse it, not to re-derive it per domain.

Medium M12 — Both central-sync jobs are exposed to the P3 pooler stall — the mirror can stop silently

worker/jobs/contacts-central-sync.ts:24, contacts-central-reconcile.ts:24, worker/lib/advisory-lock.ts:21-35, 72-74, 126.

The known P3 caveat is that pg_try_advisory_lock and pg_advisory_unlock are separate statements through the transaction pooler and can land on different backends. The existing note argues this hasn't bitten us because a single worker never contends with itself — but that only covers the double-run half. The other half needs no second runner:

  1. Acquire lands on backend A → true; the tick runs.
  2. The unlock lands on backend B → no-op. (pg_advisory_unlock on a lock you don't hold returns false silently — it does not throw, so the worker:advisory-unlock-failed line never fires.)
  3. The lock is now stuck on A's long-lived pooled connection. Every subsequent tick's acquire lands elsewhere → false → worker:skip-tick-locked at debug level → skipped, with no JobRun row.

Failure scenario

The incremental mirror stops. Nobody notices — the only trace is a debug-level line, and skips deliberately don't record a JobRun. If the reconcile's lock wedges the same way, the designed safety net for every LOW in this report stops too, and the EU mirror quietly diverges from EA-SYS indefinitely.

Suggested fix

Durable: point the worker at DIRECT_URL (session mode) as the P3 note already prescribes. Interim: log skip-tick-locked at info for these two jobs, or use pg_advisory_xact_lock inside a transaction (which pins one backend under transaction pooling). Observable signature to check first: gaps > 37 min (job 1007) / > 24 h (job 1008) in JobRun.

What to learn. A lock you cannot observe is a lock you cannot trust. Two lessons here: (1) an abstraction (“advisory locks give us singletons”) that is only true under an assumption (“one session”) must state the assumption at the call site, because the assumption is invisible; (2) the failure mode of a silent skip is a job that appears healthy while doing nothing — always give “I did no work” a louder voice than you think it deserves.

Verified clean — checked, not a bug

  • Cross-tenant IDOR on [contactId] (GET/PUT/DELETE/email/bulk-tags): every lookup is findFirst({ where: { id, organizationId } }); a wrong-org id 404s without leaking existence.
  • MCP org binding: organizationId is injected from the validated API key, never from tool input; update_contact binds contactId to the org before mutating.
  • The May-18 write-guard fix held — denyReviewer is present on POST/PUT/DELETE/ bulk-tags/import/email-change.
  • CSV formula injection on export: every cell (including tags.join(",")) runs through escapeCsvCell.
  • CSV import insert race: createMany({ skipDuplicates: true }) is a single atomic INSERT … ON CONFLICT DO NOTHING — no check-then-act (the suspected bug isn't there).
  • Missing updatedAt bumps: false alarm — every Contact mutation uses Prisma update/upsert, which sets @updatedAt; no raw SQL touches the table. All mutations are therefore visible to the incremental sync filter.
  • The 45-min lookback vs 37-min cadence: 8 minutes of margin; all Contact writes are short interactive transactions and web+worker share one box (no clock skew). Effectively unreachable, and the nightly reconcile covers it.
  • Two email-changes racing: the PATCH maps the losing P2002 to a 409, and the speaker/ registration variants run repointOrgContactEmail inside a transaction that rolls the whole cascade back.
  • Central-sync service-role key: header-only; only response bodies are logged. Not leaked.
  • Contact photo stored-XSS: rendered via next/image → plain <img src>; not exploitable. (The import fetch is the real angle — M10.)

Lessons for a junior developer — the principles behind these bugs

  1. A write-guard is not an access-control model. Guarding mutations says nothing about who may read. In a CRM, the export endpoint is the crown jewels — it deserves the strictest gate and an audit trail. (H1)
  2. Enforce invariants at the data boundary, not in each writer. Six writers, five of whom remembered to lowercase, is a broken invariant. The database can enforce it; convention cannot. (H2, M4)
  3. An append-only mirror keyed on a mutable field is an accumulator, not a mirror. Before building a sync, enumerate every source operation — create, update, rename-the-key, merge, delete — and decide what each means on the target. (H3)
  4. When you fix a drift bug, delete the duplication that caused it. Otherwise you've proven the mechanism works and left it loaded. (H4, M3)
  5. Beware code whose failure mode is “succeeds and does nothing.” An enrich-only sync fed the wrong payload logs nothing and changes nothing. Test the effect, not the call. (H4)
  6. “Non-blocking” is not a licence to manufacture failures. A check-then-act against a unique index will lose races; a catch around it converts a race into permanent invisible data loss. (M1)
  7. Audit anything whose effect can't be reconstructed from the resulting state — a DELETE most of all, because the evidence is what it destroys. (M2)
  8. Comments asserting parity are liabilities. If two things must stay in sync, encode it in code or a test — never in prose. (M3)
  9. Two APIs writing the same row share one schema with two spellings. An asymmetric field cap makes a record read-only by accident. (M7)
  10. “It comes from a trusted integration” is not a security boundary. Any server-side fetch() of a URL you didn't construct needs an allowlist — and we already wrote that guard once. (M10)
  11. “We never log PII” must hold on the exception path — which is exactly where nobody looks. Identifiers in a query string end up in every error and proxy log. (M11)
  12. Give “I did no work” a loud voice. A silently-skipped job looks identical to a healthy one. (M12, L1)
