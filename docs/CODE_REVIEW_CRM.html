Verdict

The CRM's structural skeleton is genuinely good — better than most of the older domains were at first review. The centralized route boundary (requireCrmRead/Write/Delete in crm-route.ts) makes an unguarded or un-rate-limited route structurally hard to write; every cross-org exploit attempt failed (all id paths bind organizationId, including nested ids); the conditional-claim pattern is used correctly on the transitions that matter (stage move, close, task complete, reminder claim); there is exactly one CrmActivity writer; the crm→core import boundary is ESLint-enforced; and CSV export, token escaping, and the sponsor-email "narrow-never-widen" audience math all held up under attack.

The findings cluster in four shapes: (1) a fixed bug class not propagated to its siblings (the products seed-race fix exists one file away from two seeders that still have it, and the products reject() logging helper exists one file away from eight services that don't log rejections); (2) name-keyed semantics (Won/Lost stage matching by string) that let a deal's stage and status silently diverge and corrupt every report; (3) money math that isn't currency-aware; and (4) soft-delete that state machines don't check.

The four recurring shapes (read this before the findings)

Most of what follows is not 33 unrelated bugs — it is four mistakes, each repeated:

  1. A fix is not done until it's applied to every sibling of the pattern. The seed-race fix (@@unique + skipDuplicates) shipped for CrmProduct; the two sibling seeders (CrmPipelineStage, CrmEmailTemplate) still race (H1). The reject() logging helper shipped for the product service; eight sibling services still return silent rejections (M5). When a review fix lands, grep for the pattern, not the file.
  2. Names are not semantics. closeDeal and terminalStatusFor key the WON/LOST outcome on a stage being literally named "Won"/"Lost". The moment an org renames, deletes, or adds a terminal column, stage and status diverge and the reports lie (H3). Load-bearing semantics belong in a column, not a string match.
  3. Soft-delete is only real if the state machines check it. Every write claim binds { id, organizationId } — none binds archivedAt: null, so an archived deal can be dragged, closed, and reopened through the API (M1).
  4. A null total is honest; a fabricated one is not. The module already knows this — sumDealProducts returns null on mixed currencies — but the deal-value aggregates (reports, board columns, rep leaderboard) sum AED and USD together and print "$" (H2).

High Open H1 — Pipeline-stage and email-template seeders can double-seed: no unique constraint, so skipDuplicates is a no-op

src/crm/services/pipeline-service.ts:82–114 · src/crm/services/crm-email-template-service.ts:59–82 · prisma/schema.prisma (both models carry only @@index([organizationId, sortOrder]))

The code

// pipeline-service.ts — ensurePipelineStages
const existing = await db.crmPipelineStage.findMany({ where: { organizationId }, ... });
if (existing.length > 0) return existing;
await db.crmPipelineStage.createMany({
  data: DEFAULT_PIPELINE_STAGES.map(...),
  skipDuplicates: true,          // ← skips only rows that violate a UNIQUE constraint.
});                              //   CrmPipelineStage has none. It skips nothing.
// "Two concurrent first-loads can both try to seed. Losing that race is harmless" ← FALSE:
// with no unique constraint there is no loser. Both writers succeed.

ensureCrmEmailTemplates is the same shape and doesn't even pass skipDuplicates. Verified in the schema and in migrations 20260714120000 / 20260715160000: both create a plain CREATE INDEX, not a unique one.

Failure scenario

A brand-new org's first CRM visit fires parallel requests (two tabs, board + reports SSR + client hydration, or two users on day one). Both GET /api/crm/pipeline-stages calls see 0 rows, both insert 8 stages → a 16-column board with two "Won" and two "Lost" terminal columns. Corruption then compounds: closeDeal finds its landing column with findFirst by name, so won deals split arbitrarily across two identical columns; and planPipelineReconciliation consumes each canonical name once, so the second "Won" becomes an orphan — running the reconcile script moves its WON deals into "New" and deletes the column. Templates: two of each seeded template; a CRM user edits one copy and the compose picker keeps offering the stale twin. The existing "seed-once idempotency" tests test sequential re-calls — they cannot catch this, because the bug only exists when the count === 0 fast-path is passed concurrently.

Suggested fix

One additive migration: @@unique([organizationId, name]) on CrmPipelineStage and CrmEmailTemplate, plus skipDuplicates: true on the template seed — literally the 20260715180000 products fix applied to the two siblings it skipped. The name is already the load-bearing key (closeDeal and the reconcile planner both match by name), so uniqueness is semantically required anyway — and it also blocks a manual second "Won" via createStage (see L-list). Ship before the CRM migrations go to prod so no dedup backfill is ever needed.

What to learn: skipDuplicates is not a dedup mechanism — it is a unique-constraint conflict handler, and without a constraint it is dead code that makes the read-then-insert race look handled. And when a review fixes a bug class ("seed can double-seed"), the fix isn't finished until every code path with the same shape is swept — grep for the pattern (count === 0 → createMany), not the file.

High Open H2 — Reports and board column totals silently mix currencies and label everything USD

src/app/api/crm/reports/route.ts:51–77 · src/app/(dashboard)/crm/reports/page.tsx:42–45 · src/crm/lib/crm-types.ts:349–354 (sumStageValue) · src/crm/components/deal-board.tsx:136

The code

// reports/route.ts — every aggregate:
_sum: { dealValue: true }        // grouped by stageId / status / ownerId — NEVER by currency

// reports/page.tsx
function money(v: number | null): string {
  return v === null ? "—" : formatDealValue(v, "USD") ?? "—";   // ← hardcoded dollars
}

// crm-types.ts — board column totals, same hole
export function sumStageValue(deals: CrmBoardDeal[], currency = "USD"): string | null {
  ...
  return formatDealValue(total, currency);   // callers pass nothing → "USD"
}

Failure scenario

Per-deal currency is a first-class input (the create dialog has a currency select; the catalog seeds AED prices while a new deal defaults USD). Org holds an AED 500,000 Abbott deal and a USD 50,000 Pfizer deal → reports show "Open value $550,000" — a fabricated number (~$186k real). The win/loss KPIs, the per-rep leaderboard (and its sort order — reps are ranked by the raw mixed sum), and every board column total are wrong the moment a second currency exists. The module already solved this exact class for line items: sumDealProducts returns null on mixed currencies and the UI prints "— (mixed currencies)". The deal-value aggregates got no equivalent treatment.

Suggested fix

Group every aggregate by currency as well; when a bucket contains >1 currency, return null + a "mixed currencies" marker (mirror sumDealProducts) or emit per-currency rows. At minimum, thread the deals' actual (uniform) currency into formatDealValue instead of the "USD" literal, and make sumStageValue return null when its column mixes currencies.

What to learn: the instant a money field carries a sibling currency column, every SUM() over it is only meaningful per currency. "—" (unknown) is a fact; "$550,000" (AED+USD added and stamped with a dollar sign) is fiction that finance will act on. Prefer honest nulls — the codebase already established this rule in sumDealProducts; extend rules like this to every aggregate the day they're invented.

High Open H3 — Won/Lost is keyed on the stage NAME; divergence puts deals in the wrong (or no) report bucket, and a routine drag silently reopens a won deal

src/crm/services/deal-service.ts:502–516 (terminalStatusFor) · :425–441 (closeDeal) · :329–343 (moveDealStage) · src/app/api/crm/reports/route.ts:84–94 + src/crm/lib/reports.ts:41–48

The code

// terminalStatusFor — outcome derived from the NAME
const n = stageName.trim().toLowerCase();          // "won"/"closed won" → WON, "lost"/... → LOST
                                                   // anything else → null → status "OPEN"
// closeDeal — landing column found by NAME
const terminal = await db.crmPipelineStage.findFirst({
  where: { organizationId, isTerminal: true,
           name: { equals: status === "WON" ? "Won" : "Lost", mode: "insensitive" } } });
...(terminal ? { stageId: terminal.id } : {}),     // no "Won" column → card stays where it is

// moveDealStage — EVERY move unconditionally rewrites status + stamps
status: terminalStatusFor(toStage.name, toStage.isTerminal) ?? "OPEN",
...closeStamps(...)                                // non-terminal target → wonAt/lostAt WIPED

Failure scenario — three verified consequences

  • (a) Close with no "Won" column ⇒ double-counted. The org renamed/deleted "Won" (the API allows both — POST /pipeline-stages accepts any name with isTerminal: true, and DELETE on an empty Won column succeeds). The Won button sets status: WON but leaves the card in e.g. Negotiation. The pipeline report computes openValue from non-terminal stage buckets with no status filter — so this deal's value appears in both "Open value" and "Won". Fictional pipeline money.
  • (b) Drag into a custom terminal stage ("Signed & Done") ⇒ counted nowhere. terminalStatusFor returns null → status is written OPEN; the stage is terminal so it's excluded from open counts; the deal is neither WON nor LOST. A rep's closed business vanishes from all four KPIs and the leaderboard.
  • (c) An innocent drag reopens a divergent WON deal. Take the deal from (a) — WON, parked in Negotiation. A colleague tidies the board and drags it to Proposal: the claim passes (it checks only stageId, never status), status flips WON→OPEN, and wonAt is wiped. July's won-revenue quietly shrinks; the only trace is a generic STAGE_MOVE activity row. (Dragging out of a terminal column reopening is a documented decision; two ordinary columns reopening a deal is not — the user had no idea the card was closed.)

Suggested fix

Stop deriving the outcome from a string: add a terminalOutcome (WON | LOST | null) column on CrmPipelineStage, seeded onto Won/Lost (module is not deployed — clean migration). closeDeal lands on the outcome-mapped stage (and refuses, or returns a meta.stageUnchanged warning, when none exists rather than minting a divergent deal); moveDealStage only rewrites status/stamps when the move actually crosses a terminality boundary, and records an explicit REOPENED activity whenever it clears a wonAt/lostAt. Refuse deleting the last WON-mapped stage. (Related: M10 — lostReason is not cleared on these paths either.)

What to learn: when a comment says a name is "LOAD-BEARING, not decorative" (pipeline-service.ts:55), that's the design telling you the semantic is stored in the wrong place. A rename should never be able to change what a state machine does — put the semantic in a typed column and let the name be decoration. And a state transition must be conditioned on the state it changes: moveDealStage claims on stageId but rewrites status — the claim guards the wrong field.

High Open H4 — Renaming a company/contact to a taken name/email returns an unlogged HTTP 500

src/crm/services/company-service.ts:278–285 · src/crm/services/crm-contact-service.ts:258–265 · src/crm/lib/crm-route.ts:168 (UNKNOWN: 500)

The code

// company-service.ts — updateCompany's catch
if (err instanceof Prisma.PrismaClientKnownRequestError && err.code === "P2002") {
  return { ok: false, code: "UNKNOWN",                      // ← a business conflict...
           message: "Another company already uses that name",
           meta: { conflict: "name" } };                     //   ...mapped to HTTP 500, unlogged
}

Failure scenario

A completely ordinary action — editing "Cleveland Clinic Foundation" to "Cleveland Clinic" when that account already exists, or fixing a CRM contact's email to one already on file — hits the @@unique and returns HTTP 500 (STATUS_BY_CODE maps UNKNOWN → 500, whose own comment says a business rejection must never land on 500: "defaulting to 500 would page us for it"). It is also the only rejection in either function with no log line — the P2002 early-return sits above the generic update-failed error log, so nothing reaches /logs. It corrupts 500-rate monitoring and violates the "every failure path logs" rule. Note the create paths handle P2002 correctly (race → re-read winner → reuse); only the two update paths are wrong.

Suggested fix

Dedicated codes NAME_TAKEN / EMAIL_TAKEN mapped to 409 in STATUS_BY_CODE, logged at warn (the message text is already right — only the code and the log are missing). The edit dialogs already toast the server message, so the UI improves for free.

What to learn: UNKNOWN is for the unforeseen. The moment you can write a human-readable message for a failure ("another company already uses that name"), it is a known business rejection and deserves its own code, its own 4xx status, and a warn log. A P2002 on an update is a user action, not a server fault.

Med Open M1 — Archived records are fully mutable: no state-machine claim includes archivedAt: null

deal-service.ts:280 (update), :332–343 (move), :434–442 (close), :612 (addDealContact) · crm-product-service.ts:233 (addDealProduct) · task-service.ts:232/:283 (complete/reopen) · company/contact update paths

Failure scenario

Admin A archives a deal; colleague B's board hasn't refetched and B drags the card. The claim checks only { id, organizationId, stageId } — it succeeds: the archived deal's stage, status and wonAt/lostAt are rewritten (dragging it out of a terminal column even reopens an archived closed deal). B gets a 200, the card vanishes on refetch, and a later restore surfaces state nobody consciously chose — the only clue is a STAGE_MOVE activity row after the ARCHIVE row. An archived OPEN deal can likewise be closed WON. The board disabling drag on the archived view is a UI courtesy; the API and any api-key caller go straight through.

Suggested fix

Add archivedAt: null to the conditional-claim where of every state-changing write (stage move, close, complete, reopen, add contact/product), and extend each path's existing zero-row disambiguation read to return a distinct DEAL_ARCHIVED-style 409. Whether plain field edits on archived rows stay allowed is a product call — state machines should not.

What to learn: a soft-delete flag is part of the record's state, so it belongs in every conditional claim exactly like the state being transitioned. "The UI can't reach it" is never the guard — the API is.

Med Open M2 — MEMBER can read rival deal money out of free-text notes, bypassing the structured redaction

src/app/api/crm/notes/route.ts (GET is requireCrmRead, returns raw body) · companies/[companyId]/route.ts + contacts/[crmContactId]/route.ts (return notes fields) · crm-roles.ts:81–96 (the stated goal)

Failure scenario

The whole reason canViewDealValues is narrower than canViewFinance is documented in crm-roles.ts: a sponsor-side MEMBER "must not be able to read every RIVAL sponsor's deal value off the board." The structured redaction is airtight (dealValue stripped, value filter dropped so it can't be binary-searched, CSV omits the column). But MEMBER passes requireCrmRead, deal ids are visible on the board, and GET /api/crm/notes?dealId=<rivalDealId> returns a rep's note verbatim — "Abbott came back at AED 480k, they'll sign if we hold at 500k." Same leak through CrmCompany.notes / CrmContact.notes on the detail GETs (redactForCaller strips financial keys; prose passes through).

Suggested fix (owner decision)

(a) Treat the negotiation log as money-adjacent: gate note reads + the notes fields to canViewDealValues (consistent with the stated goal), or (b) explicitly accept prose leakage and document it next to the crm-roles.ts comment so the boundary's real shape is honest. (a) is the consistent choice.

What to learn: key-based redaction protects fields, not facts. If a fact is sensitive enough to strip from a column, assume it also lives in prose next door — the boundary must cover the channel (notes, activity payloads, exports), not just the schema key.

Med Open M3 — Sponsor/deal email blast: no double-submit idempotency

src/app/api/crm/sponsor-email/send/route.ts:47 · src/crm/services/sponsor-email-service.ts:272–368

Failure scenario

The send is synchronous in the request (batches of 25, attachments up to 10 MB base64 — slow). The only repeat-guard is the 10/hr rate bucket, which happily admits both halves of a double-click, a browser retry of a slow POST, or an operator pressing Send again mid-flight → all ~40 sponsors get the prospectus twice, with doubled PROSPECTUS_SENT History rows. A mid-send crash has the same recovery trap: "send again" re-emails the delivered half. The event bulk-email pipeline got exactly this fix (comms H2: content-hash + 2-minute window; comms H1: emailedKeys resume) — the CRM path got neither.

Suggested fix

Copy the comms-H2 dedup: best-effort content+audience hash with a ~2-minute window returning the prior result (no schema needed); optionally skip recipients with a PROSPECTUS_SENT activity / EmailLog row in the same window on a re-run.

What to learn: a rate limit is abuse control, not idempotency. Any fan-out send needs an answer to "what happens when this exact request arrives twice?" — and the codebase already has the reference implementation to copy.

Med Open M4 — Record edits: no optimistic lock, and the snapshot → write → re-read triple can misattribute changes in the History log

deal-service.ts:271–295 · same shape in company-service.ts:245–273, crm-contact-service.ts:225–253, task-service.ts:181–205

The code

const before = await db.crmDeal.findFirst({ ... });                       // (1)
await db.crmDeal.updateMany({ where: { id, organizationId }, data });     // (2)
const deal = await db.crmDeal.findUniqueOrThrow({ where: { id } });       // (3)
const fieldChanges = diffFields(before, deal, DEAL_DIFF_KEYS);            // ← diff of a racy re-read

Failure scenario

Users A and B edit the same deal. A shared field is silent last-write-wins with no 409 (no expectedUpdatedAt, unlike the registration PUT). Worse for the audit trail: if B's write commits between A's steps (2) and (3), A's diff records B's change under A's actorId in CrmActivity — the change log, whose entire purpose is "who changed what", attributes the wrong author; the mirror-image window hides B's change entirely.

Suggested fix

Wrap (1)–(3) in a $transaction (kills the misattribution window), or better, adopt the house expectedUpdatedAt conditional claim so a concurrent edit is a 409. Cheapest correct option: compute the diff from before + the submitted patch, not a re-read.

What to learn: three sequential awaits are three transactions. If a derived artifact (a diff, an audit row) is built from reads on both sides of a write, another writer can sit in between — derive it from data you already hold, or hold the whole triple in one transaction.

Med Open M5 — Silent business rejections across 8 services; crmErrorResponse doesn't log either

deal-service.ts (:165, :170–172, :239, :250–252, :257–259) · company-service.ts (:111–113, :224–226, :237–239) · crm-contact-service.ts (:109–114, :192–208, :217–219, :318–320) · task-service.ts (:107, :157, :174, :177–178) · note-service.ts (:102, :179, :184–186) · crm-email-template-service.ts (:105–107, :151–165) · pipeline-service.ts (:140) · sponsor-email-service.ts (:231–249, :132–134, :197–199) · crm-route.ts:171–181

Failure scenario

The route layer is clean (every safeParse failure logs via zodErrorResponse), but crmErrorResponse() does not log — so any service rejection the service doesn't log itself leaves no trace. The products review's H1 added a reject() helper that logs before every return { ok: false } — to the product service only. Every other service still has unlogged rejections, including fully reachable ones: the sponsor-email EVENT_NOT_FOUND/DEAL_NOT_FOUND 404s and all of validateSend (TOO_MANY_ATTACHMENTS, ATTACHMENT_TOO_LARGE…); the *_REQUIRED codes are reachable past Zod with whitespace-only strings; NO_FIELDS with a bare {} PATCH. Internal drift too: createTask logs its relation failure, updateTask doesn't. Directly contradicts src/crm/README.md §4 ("every failure path logs").

Suggested fix

The structural option: one warn line inside crmErrorResponse() (code + message + route context) — it catches everything any service ever misses, exactly the same move as gate-owned refusal logging. Optionally also lift reject() into a shared helper for services that want richer context.

What to learn: logging at each rejection site is a per-developer discipline; logging at the boundary the rejections all flow through is a structural guarantee. When one choke point exists (crmErrorResponse), put the rule there — then forgetting becomes impossible instead of likely.

Med Open M6 — Frontend silent failures: every list page renders a failed fetch as an empty state; three mutations fail with no toast

deals/page.tsx:78 · companies/page.tsx:51–52 · contacts, tasks (:46), products, templates, reports pages (all ignore isError) · create-company-dialog.tsx:51–63 · deal-detail-body.tsx:106–117 (add follow-up) · company-detail-body.tsx:170–172 ("They're different" dismiss) · use-crm-api.ts:213–236, :272–281 (no onError)

Failure scenario

A failed GET /api/crm/deals (session lapse, 500, network) renders as "No deals yet" — the sales team concludes the pipeline is empty. This is the exact class fixed in core for the registrant portal on June 26; only the deal detail page handles isError. Mutations: creating a company (a 429 from the 100/hr cap is realistic), adding a deal follow-up task, and dismissing a payer-review flag all await mutateAsync with no try/catch and no hook-level onError — the button does nothing, no toast, no log. This is drift within the module: the deal/contact/template dialogs all handle errors correctly.

Suggested fix

Per list page: branch on isError → a "Couldn't load — your data is safe, try again" card with a refetch button (copy the registrant-portal fix). Per mutation: try/catch + toast.error in the three handlers (or onError in the two hooks).

What to learn: an error state rendered as an empty state is the most expensive UI bug per byte — it doesn't just hide the failure, it asserts a false fact ("you have no deals"). { data = [], isLoading } destructuring is the tell: the = [] default is what converts the error into the lie.

Med Open M7 — Dead contactId field on the deal schemas: accepted, validated, silently discarded

src/app/api/crm/deals/route.ts:16 · src/app/api/crm/deals/[dealId]/route.ts:13 · deal-service.ts:40–77 (CreateDealInput/UpdateDealInput have no such field)

Failure scenario

Both Zod schemas declare contactId: z.string().min(1).optional().nullable(), but the service input types don't carry it — the spread into createDeal(...) drops it. No dialog sends it (verified), so the victim is the supported source: "api" caller: create a deal with its primary contact → 201, no link, no log — and that person is then skipped by the next sponsor-prospectus blast (audience = deal contacts only). Leftover of the pre-CrmDealContact-join design; the matching docs/CRM_STATUS.html §4 row is stale on both columns (the model moved to the join table, and a people-on-deal picker now exists). Same class as the MCP conflictFlag H8: an advertised param with a no-op executor.

Suggested fix

Either wire it through addDealContact after create, or delete the field from both schemas (and fix the CRM_STATUS row). Deleting is the honest minimum; wiring is a ~10-line nicety.

What to learn: a validated-but-unconsumed schema field is an API contract lie — worse than no field, because the caller gets a 201 and believes it worked. When a design migrates (scalar FK → join table), grep the Zod schemas for the old shape before calling it done.

Med Open M8 — Pipeline-stage hard DELETE is write-gated: the RBAC gradient inverts

src/app/api/crm/pipeline-stages/[stageId]/route.ts:15–16 (requireCrmWrite)

Failure scenario

ORGANIZER may not archive a single deal (denyCrmDelete), but may hard-delete a pipeline column — org-wide config, a real db.crmPipelineStage.delete, in a module whose README §3.6 headline is "There is no hard delete." Deletion is blocked while deals exist (Restrict FK + STAGE_HAS_DEALS), so no data loss — but the privilege gradient is inverted: the destructive, org-wide, irreversible action sits at a looser gate than the reversible, single-record one.

Suggested fix

One-line: requireCrmDelete on the stage DELETE (create/reorder staying write-gated is defensible).

What to learn: audit gates by blast radius, not by HTTP verb or entity type. "Config" deletion is still deletion — when a module defines a narrower delete tier, every destructive verb in the module must sit at or above it.

Med Open M9 — MIME header injection via attachment name/contentType in core buildRawMime

src/lib/email.ts:485, :488, :502, :504 · src/crm/services/sponsor-email-service.ts (validateSend checks count/size only)

The code

// email.ts — buildRawMime, interpolated raw:
`Content-Type: ${att.contentType || "application/octet-stream"}; name="${att.name}"`,
`Content-Disposition: attachment; filename="${att.name}"`,

Failure scenario

A name containing " or CRLF, or a contentType containing CRLF, breaks the MIME part framing and can inject arbitrary headers/parts into the outgoing message. The sender is an authenticated CRM writer (self-inflicted-adjacent), but the recipients are external sponsors at fan-out scale, under the org's branded sender — and this is the first path where a user supplies arbitrary attachment names/types. Core, pre-existing issue shared by every attachment path (quote PDF, agreement .docx use fixed names); best fixed once in email.ts. (Subject-header injection is already mitigated — encodeRfc2047 base64-encodes control characters. Verified.)

Suggested fix

In buildRawMime: strip CRLF and " from (or RFC 2047/2231-encode) att.name; validate contentType against a type/subtype regex. Optionally an attachment-type allowlist on the sponsor blast specifically (see L12).

What to learn: anything interpolated into a protocol header (MIME, HTTP, SMTP) is an injection surface exactly like HTML — same reflex, different escaper. Validate at the protocol boundary so every future caller inherits the fix.

Med Open M10 — Stale lostReason survives a drag-reopen; a WON deal exports with a "Lost reason"

deal-service.ts:439 (closeDeal clears it) vs :338–342 (moveDealStage never touches it) · export/route.ts:109

Failure scenario

Deal closed LOST ("budget cut") → the sponsor comes back → card dragged out of Lost (reopened) → later dragged into Won. Result: status: WON, lostReason: "budget cut" — printed in the CSV export and the deal sidebar. closeDeal handles the reason; closeStamps only handles timestamps, and the move path forgot the third field.

Include lostReason: null in the move payload whenever the computed next status is WON or OPEN (mirroring closeDeal); folds naturally into the H3 rework.

Med Open M11 — A task due today shows Overdue all day; report date ranges are server-TZ dependent

src/app/(dashboard)/crm/tasks/page.tsx:52 · src/crm/lib/deal-filters.ts:34–38

Failure scenario

dueAt is stored as UTC midnight of the picked date; the page renders new Date(t.dueAt) < now → a task due July 16 turns red "Overdue" at 00:00 UTC on July 16 itself (04:00 Dubai) — the entire due day is reported as missed. (Contrast buildTaskDueRange, which correctly treats to as inclusive end-of-day.) Separately, endOfDay in the deal filters uses server-local setHours(23,59,59) while parseDate yields UTC midnight — symmetric only because prod runs UTC; and "won in July" is a UTC July for a Dubai org (a deal won 22:00 UTC Jul 31 = 02:00 Aug 1 Dubai lands in July). The codebase already has event-time.ts TZ helpers to borrow.

Suggested fix

Compare against end-of-day of dueAt for the Overdue badge; build filter boundaries in UTC (or the org TZ) explicitly rather than server-local.

What to learn: a date (no time) compared against a timestamp is always a boundary decision — decide "due through the end of that day" once, in a helper, and never call bare < now on a date-derived value.

Med Open M12 — Moving a task's due date does not move its reminder

src/crm/services/task-service.ts:164–171

Failure scenario

The service re-arms (remindedAt = null) only when remindAt itself is edited. The only create surface (deal follow-up) sets remindAt = dueAt — so an edit that pushes dueAt out a week (API today; edit UI is a documented pending item) leaves the reminder at the old date: it either already fired (none will come at the new due date) or fires a week early. The product promise on the deal page is "you'll get an email when it's due" — a due-date edit silently breaks it.

Suggested fix

When dueAt changes and the caller didn't pass remindAt: if remindAt equaled the old dueAt, shift it to the new one and clear remindedAt.

What to learn: when field B is derived from field A at creation ("remind when due"), an update to A must decide what happens to B — silence is a decision too, and usually the wrong one.

Lessons index — what a developer should take from this review

  1. Fix the pattern, not the file (H1, M5). Two review-shipped fixes — the seed-race unique constraint and the reject() logging helper — each stopped at the one service the finding named, while identical siblings sat one file away. After any review fix: grep for the shape.
  2. Semantics live in columns, not names (H3, L5). If renaming a row changes what a state machine does, the design is storing meaning in a string. The comment "this name is load-bearing" is the bug report.
  3. Claims must guard the fields they change (H3c, M1). moveDealStage conditions on stageId but rewrites status; nothing conditions on archivedAt. A conditional claim is only as good as its where.
  4. An honest null beats a fabricated number (H2). The module invented this rule itself (sumDealProducts) and then didn't apply it to the aggregates finance will actually read.
  5. Known failures get codes, not UNKNOWN (H4, L4). If you can phrase the message, you know the case — give it a 4xx and a warn line. UNKNOWN → 500 is for the genuinely unforeseen.
  6. Log at the choke point (M5). Per-site logging discipline decays; one warn line in crmErrorResponse makes silent rejections structurally impossible — the same move as gate-owned refusal logging.
  7. Error ≠ empty (M6). { data = [], isLoading } silently converts a failed fetch into "you have no deals". Every list needs the isError branch.
  8. Redaction must cover channels, not just keys (M2). Money stripped from dealValue walks straight out through notes prose.
  9. A validated-but-unconsumed schema field is a contract lie (M7). The 201 tells the caller it worked. When a model migrates, sweep the Zod schemas for the old shape.
  10. Fan-out sends need idempotency, not rate limits (M3). "What happens when this exact request arrives twice?" must have a better answer than "everyone gets two emails".
  11. $transaction is not isolation (L1, M4). Aggregates take no locks under READ COMMITTED; three awaits are three transactions. Comments claiming otherwise teach the next reader a false pattern — the accommodation review's lesson, recurring.
  12. Gate by blast radius (M8). An org-wide hard delete at a looser gate than a single-record soft archive is an inverted privilege gradient, whatever the entity is called.
