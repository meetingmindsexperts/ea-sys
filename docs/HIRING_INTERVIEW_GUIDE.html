EA-SYS — Hiring & Interview Guide

Screening a co-maintainer for a production-live Next.js / Prisma event platform. Internal use.

Context for the interviewer. EA-SYS runs real Meeting Minds Group events — registrations, payments, check-in, certificates, live webinars. It is currently single-owner. The #1 reason to hire is bus factor: a second person who can deploy a fix, read the logs, and reason about the system during a live event. So screen hardest for judgment and discipline on a live system, not algorithm trivia. You won't ask everything here — pick ~8–10 questions weighted toward Section 1, then run the two hands-on exercises (Section 7). The exercises are the highest-signal part of the whole process.

1. Production judgment on a live system weight highest

Predicts whether they'll be safe to give deploy + prod-DB access. This section matters more than all the others combined.

You need to add a column to a table that's being written to constantly by live traffic. Walk me through how you ship that with zero downtime.
Listen for: additive/nullable first, backfill as a separate step, expand-then-contract, careful deploy ordering, idempotent migration. Red flag: "I'd just run the migration," or renaming/locking a hot column in one step.
It's 9pm during a live webinar with 3,000 attendees and something's broken. What are the first five things you do?
Listen for: check logs/monitoring first, reproduce + scope before changing anything, prefer the reversible fix, communicate status, don't cowboy a deploy. Red flag: jumps straight to editing prod code.
Tell me about a time you broke production. What happened, and what did you change afterward?
Listen for: ownership + a concrete process change. Red flag: "never broke anything" (not honest), or blames others.
When is the right move to NOT fix a bug immediately?
Listen for: understands blast radius, risk vs. reward, mid-event change freezes, deferring to a safer window.

2. Full-stack technical (Next.js / React / TypeScript)

Difference between a server component and a client component in the Next App Router — and how do you decide which to use?
A dashboard page is slow and hits several DB queries. How do you find and fix the bottleneck?
Listen for: parallelize independent queries, select only needed fields, indexes, N+1 awareness, measure before optimizing.
How do you keep server-only code (DB clients, secrets, Node modules) out of the client bundle — and what goes wrong if you don't?
How do you handle an external API call that might hang or fail (Stripe, an email provider, a webhook)?
Listen for: timeouts, bounded retries, idempotency, isolating the failure, logging it.

3. Database / Prisma / Postgres

How do you make a webhook handler idempotent so a retry doesn't double-charge or double-create?
Listen for: unique constraints, "check state before acting," dedupe keys, upserts.
Two requests try to book the last seat at the same time. How do you prevent overselling?
Listen for: atomic conditional update / transaction with a guard predicate. Red flag: read-the-count-then-write (a race).
When would you reach for a transaction, and what's the cost of holding one too long?
Listen for: connection-pool pressure, lock contention, keeping transactions short.

4. Ops & debugging on-call backup test

How comfortable are you SSH-ing into a Linux box, reading nginx + app logs, and restarting a Docker service? Walk me through diagnosing "the site is down."
A user says "I clicked the button and nothing happened, and there's nothing in the logs." How do you debug a failure that leaves no trace?
Listen for: understands client-side / browser-level failures, network tab, console, reproducing locally. High-signal question.
How do you think about backups and disaster recovery? What would you want in place before you'd feel safe running someone's production system?

5. Security & sensitive paths

How do you stop user A from reading or editing user B's data via the API?
Listen for: server-side ownership/org-scoping on every query, not trusting client-supplied ids (IDOR awareness). This system lives and dies on it.
Some roles must never see financial data. How would you enforce that so it can't leak through any endpoint?
Listen for: centralized guard, fail-closed, redaction at the response layer, defense in depth.
Where do secrets belong, and where should they never end up?
Red flag: casual about committing secrets / putting them in client code.

6. Working style & discipline culture fit

This codebase has strong rules — no silent error catches, every failure logs, full test/lint/typecheck before every push, additive-only migrations. How do you feel working inside conventions you didn't write?
Listen for: respects existing patterns first, improves incrementally. Red flag: wants to rewrite / "modernize" everything on day one.
How do you get up to speed on a large codebase you didn't build?
Listen for: reads docs/conventions, asks questions, makes small reversible changes first.
How do you communicate when you're blocked or unsure — vs. when you'll just decide and proceed?
What does "done" mean to you before you call a task finished?
Listen for: tested + verified, edge cases handled, not just "it compiles."

7. Hands-on exercises highest signal

Verbal answers are cheap. These reveal ~80% of what you care about. Do at least the second one.

A — Take-home (2–3 hrs · pay for it if you can)

Give a small, isolated slice: "Add a new field to an entity end-to-end — schema + migration, API validation, a UI control, and a test." You're not grading cleverness. You're watching whether they:

  • write the migration safely (additive/nullable, idempotent);
  • validate input and handle empty/error cases;
  • log failures instead of swallowing them;
  • follow the surrounding style rather than inventing their own;
  • actually test + verify before calling it done.

B — Live code-reading walkthrough (30 min · screenshare)

Open one real EA-SYS file cold and ask them to explain what it does, what could go wrong, and what they'd change. Pick something with real subtlety — an API route with auth + a transaction, or a webhook handler. Tests comprehension, judgment, and humility far better than trivia.

8. Scoring rubric

Score each 1–5; weight by the right column. Hire on the weighted total, but treat a 1–2 in Production judgment or Discipline as a near-automatic no, regardless of the rest.

DimensionWhat a 5 looks likeWeight
Production judgmentReflexively reversible/additive; logs failures; thinks blast radius first30%
Take-home qualitySafe migration, validated, tested, matches house style25%
Full-stack + DB skillSolid Next/Prisma/Postgres; idempotency + race awareness20%
Ops / debuggingComfortable on the box + logs; debugs silent failures10%
Discipline & communicationRespects conventions; clear about blocked vs. proceeding10%
Security instinctsIDOR/org-scoping + secrets handled without prompting5%
How to weight the whole thing: production judgment + the take-home are ~55–60% of the decision. Algorithm trivia is nearly irrelevant for this role — you need someone safe and conscientious on a live system far more than someone who can invert a binary tree on a whiteboard.
