Code Review — Check-in & Badges

Production-readiness audit · 10 July 2026 · EA-SYS main @ 2d1fb56
Four independent review passes: live-day correctness · RBAC & credential exposure · concurrency under door load · drift & failure-visibility.

Severity is calibrated lower than the sessions review, deliberately. None of the credential findings here are reachable by an arbitrary, unauthenticated caller the way B1/B2 were this morning. Every barcode leak requires either an org-attached account or knowledge of an unguessable cuid. That is why this report has 0 BLOCKERs — not because entry barcodes are unimportant, but because the exposure is insider-scoped. The door-day findings (H1–H4) are the ones that will actually cost you on the morning of day one.
Reviewer disagreements, resolved. Two passes contradicted each other twice; I verified both in source. (1) Double-scan: one pass called it "correctly idempotent", another called it a race. Both are right — sequential re-scans are caught by the gate, concurrent ones are not (H3). (2) DTCM import: one pass called it "per-row fault-isolated". It is not — the file contains exactly one try { (the outer handler); the continue statements are validation skips, not exception handling (M3).

1. Executive summary

The June unification genuinely held: checkedInAt is written in exactly one place (check-in.ts:106) — no raw status flip, bulk update, or registration PUT touches it — and both REST handlers plus the MCP tool commit through the same executeCheckIn, so the audit/notification fan-out is fire-and-forget with a logged catch on all three. Barcode rendering is properly centralized in renderBarcodePng across all five callers. Five of the six CLAUDE.md claims about this domain check out.

What the review found clusters into four themes:

  1. The admission gate and the badge filter disagree about who is attending. A sponsor-paid (INCLUSIVE) delegate is admitted by checkInGate and silently dropped from the badge PDF (H1). The "no payment due" rule exists canonically as NO_PAYMENT_DUE_STATUSES and the badge route re-implements it, wrong.
  2. There is no undo. checkedInAt: null is never written anywhere in src/. Flipping status back to CONFIRMED leaves the timestamp set, and the gate then refuses that person forever (H2). CLAUDE.md advertises "check attendees in (+ undo)". That capability does not exist.
  3. The door path has never met concurrency or scale. The check-in commit is an unconditional update by id with no where: { checkedInAt: null } claim (H3), and "Print All" is unbounded in N with unbounded bwip-js fan-out on the same box that serves the scanner (H4). Both are exactly what you would predict of code that has never had a load test — which the readiness audit already says it hasn't.
  4. The entry barcode is a physical-access credential travelling in ordinary read payloads. qrCode and dtcmBarcode are returned via include: on both registration GETs, and neither is in FINANCIAL_KEYS — so redactFinancialFields, the only redaction pass that runs for non-finance roles, never strips them (H6–H8). Same class as this morning's Zoom-host-URL leak; narrower blast radius.

3. Door-day failures (fix these before the next event)

High   H1 — Sponsor-paid (INCLUSIVE) and pay-at-desk (UNASSIGNED) attendees are admitted by check-in but get no badge, and the error message blames them for not paying

…/registrations/badges/route.ts:78-83, 86 vs src/lib/check-in.ts:72 vs registration-enums.ts:72-77

// badges route — re-implements "no money due", wrong:
const isComplimentary = r.paymentStatus === "COMPLIMENTARY" ||
  Number(r.ticketType?.price ?? 0) === 0 ||
  (r.pricingTier && Number(r.pricingTier.price) === 0);
return r.paymentStatus === "PAID" || isComplimentary;   // INCLUSIVE, UNASSIGNED, REFUNDED all dropped

// the canonical set it should have used:
NO_PAYMENT_DUE_STATUSES = ["PAID", "COMPLIMENTARY", "INCLUSIVE", "REFUNDED"]

// the gate that actually admits them — blocks ONLY these two:
if (!isComplimentary && (reg.paymentStatus === "UNPAID" || reg.paymentStatus === "PENDING")) …

Door scenario

A pharma sponsor block (Abbott/Pfizer contract → paymentStatus = INCLUSIVE, real price on the ticket) arrives. The scanner admits every one of them. "Print All" silently omits every one of them. If the operator selects only that sponsor block, they get a 400 reading "No paid or complimentary registrations found" — which tells the staff these VIPs haven't paid, when in fact the sponsor paid months ago and the badge filter simply drifted from the gate. This is the exact cross-caller-drift class the June unification was meant to kill, still live in the badge path.

High   H2 — There is no undo. Nothing in src/ ever clears checkedInAt, so an "undone" check-in permanently locks the attendee out of the scanner — and CLAUDE.md advertises the feature

Verified: grep "checkedInAt: null" src/ → no matches. The general registration PUT references checkedInAt zero times. Both check-in handlers (POST by id, PUT by scanned code) check in; neither undoes.

Door scenario

Staff scan the wrong person. To "undo" they flip the registration back to CONFIRMED — the only affordance that exists. checkedInAt stays set. When that person legitimately arrives and is scanned, checkInGate hits if (reg.checkedInAt) → ALREADY_CHECKED_IN and refuses them. Simultaneously the desk's attendance tile (keyed on status === "CHECKED_IN") shows them as not in. The staff see a flat contradiction — "the counter says he's not in, the scanner says he already is" — with no way out from any UI. CLAUDE.md's ONSITE description explicitly promises "check attendees in (+ undo)."

Fix shape

An explicit undo action that clears checkedInAt and status together, audited — not a bare status flip. It belongs in check-in.ts next to executeCheckIn, so all callers share it.

High   H3 — The check-in commit has no conditional claim, so two stations scanning the same badge concurrently both write: the true first-entry time is clobbered and audit + admin notifications duplicate

src/lib/check-in.ts:76 (gate read) → :114 (unguarded write). Schema: checkedInAt DateTime? — no unique, no partial index; nothing at the DB level enforces "check in once".

// :76 — the "already?" test is an in-memory read
if (reg.checkedInAt) { return { code: "ALREADY_CHECKED_IN", … }; }
// :114 — the commit carries NO predicate
: await db.registration.update({ where: { id: args.registrationId }, data, include });

Interleaving

Station A reads checkedInAt: null → gate passes. Station B reads it before A commits → gate passes. A writes 12:00:00.100. B unconditionally overwrites with 12:00:00.140. Result: the real arrival time is lost, two CHECK_IN audit rows are written, and two "Attendee Checked In" notifications fire. The scanner page's 2-second debounce is a per-tab useRef — it does nothing across two desk stations, and nothing across two tabs. Fix is one line: updateMany({ where: { id, checkedInAt: null } }), treating count === 0 as ALREADY_CHECKED_IN.

High   H4 — "Print All" is unbounded in N with unbounded barcode-render fan-out, and it runs on the same box that serves the live scanner

…/registrations/badges/route.ts:57-59 (no take), :161 (unbounded Promise.all), :191 (sync draw loop), Buffer.concat in-request

const where = all ? { eventId, status: { not: "CANCELLED" }, attendanceMode: { not: "VIRTUAL" } } : …;
const allRegistrations = await db.registration.findMany({ where, … });        // no take
await Promise.all(registrations.map(async (reg) => { … await renderBarcodePng(barcodeText); … }));

Load scenario

renderBarcodePng calls bwip-js.toBuffer — CPU-bound rasterization. At 2000 registrations this schedules 2000 simultaneous rasterizations, then draws 2000 A4 pages synchronously and accumulates the whole multi-MB PDF in memory before responding. One HTTP request pins the Node event loop on the swapless t3.large that also serves the check-in POST/PUT and the Stripe webhook receiver. Someone in the office clicks "Print All (1800)" during the arrival rush and the door scanner stalls; the operator retries, doubling the load. This is the concrete content of the readiness audit's "bulk badge Print All is fragile", and it rhymes with INC-001 (the on-box build that froze the box). Fix: a take cap, bounded concurrency (p-limit), and ideally move the render off the request path.

Not a problem

Per-row robustness is fine — a null qrCode is guarded at :166 and :237. The problem is scale, not bad rows.

High   H5 — The unknown-barcode scan — the single most valuable line to trace at a live door — leaves zero server trace

…/check-in/route.ts:147-149 (404, no log), :42 & :121 (event-not-found 404 = the ONSITE cross-event denial), badges/route.ts:86, import/barcodes 400s at :56, :63, :72, :76

if (!registration) {
  return NextResponse.json({ error: "Invalid code — not found" }, { status: 404 });  // no apiLogger
}

The gate rejections do log (check-in:qr-rejected), but the surrounding 4xx do not. During a live check-in you cannot answer "why did that badge not scan?" from /logs. Worse, an ONSITE cross-event denial — the precise adversarial case the July-7 isolation tests defend — 404s silently, leaving no audit trail that someone tried.

High   H6 — FINANCIAL_KEYS omits qrCode/dtcmBarcode, so the one redaction pass that runs on both registration GETs never strips the physical-access credential

src/lib/finance-visibility.ts (verified: no qrCode, no barcode) · applied at registrations/route.ts:276 and …/[registrationId]/route.ts:229

This is the amplifier that turns H7 and H8 from "an attendee list" into "a stack of door credentials". redactFinancialFields already fires for non-finance roles on exactly the two routes that return the barcodes. Adding them to a credential-redaction path — or switching those GETs to an explicit select that omits them for non-desk roles — closes H7 and H8 together, in one place.

High   H7 — The registrations LIST GET has no denyReviewer and returns every attendee's entry barcode + DTCM code to any org-attached caller — including MEMBER (the read-only "sponsor-side stakeholder" role) and internal-domain REGISTRANTs

…/registrations/route.ts — GET authorizes via getOrgContext + buildEventAccessWhere; the denyReviewer at :306 guards the POST. The registration query uses include: (not select), so all scalars — qrCode, dtcmBarcode — are returned.

Exploit

getOrgContext requires an organizationId, so an external attendee gets 401 — this is insider-scoped, which is why it is HIGH and not a blocker. But the org-attached set includes MEMBER (documented as "leadership / auditors / sponsor-side stakeholders") and, per the internal-domain rule, any MM-Group staffer who registered as an ordinary attendee. Either can call GET /api/events/{id}/registrations once and receive every attendee's qrCode — enough to print or clone a badge and walk anyone through the door of a paid CME conference — plus the full PII list and every DTCM compliance code.

High   H8 — Two IDOR paths to another person's entry barcode: the detail GET is row-unscoped, and the registrant self-service barcode route lets an ONSITE temp cross events

…/registrations/[registrationId]/route.ts (GET uses bare auth(); denyReviewer is on the PUT at :256; row lookup is { id, eventId }, not user-scoped) · src/app/api/registrant/registrations/[registrationId]/barcode/route.ts:43-51

// detail GET — any registration in an event you can reach
db.registration.findFirst({ where: { id: registrationId, eventId }, include: { attendee: true, … } })

// registrant barcode route — the non-registrant branch is ORG-wide, not per-event
...(isRegistrant ? { userId: session.user.id }
                 : { event: { organizationId: session.user.organizationId! } })

The detail GET is reachable by an external REGISTRANT (bare auth(); buildEventAccessWhere admits them via registrations.some.userId), and the row is bound only to the event — so any registration id in that event yields that person's qrCode. The barcode route's else branch org-scopes instead of routing through buildEventAccessWhere, so an ONSITE temp assigned to Event A can pull a scannable entry-barcode PNG for Event B in the same org — the same class as the July-7 BLOCKER, on a route that fix didn't touch.

Honest mitigation

Both need a target registrationId, which is an unguessable cuid — so neither is a spray-and-pray attack. H7 is what defeats that mitigation: it hands an insider the full id list. (The registrant branch itself is correctly userId-bound — a registrant cannot fetch another registrant's barcode.)

4. Gate policy & operator friction

Medium   M1 — No desk override for the payment gate: a PENDING attendee (Stripe webhook lag) is refused at the scanner and the queue stalls

src/lib/check-in.ts:68-74 — checkInGate accepts only allowCancelled; there is no allowUnpaid.

Someone paid online four minutes ago and the webhook hasn't landed; they are PENDING. The scanner says "Cannot check in — payment required." The only remedy is to leave the scanner flow, open the registration, record payment, and re-scan — with a queue behind them. The agent path has an override; the physical door does not.

Medium   M2 — The gate admits FAILED and REFUNDED registrations (it enumerates only UNPAID/PENDING) — and the desk has no reactivation for CANCELLED

src/lib/check-in.ts:64-74

The opposite failure mode to M1: a declined card (FAILED) or an already-refunded attendee scans in with no friction. Meanwhile allowCancelled is passed only by the MCP tool, so a cancelled-then-repaid attendee is refused at the door with no on-scanner reinstate. Worth an explicit decision: is "admit, sort the money later" the intended door policy? If yes, say so in the gate. If no, tighten it. Right now the policy is an accident of which statuses got enumerated.

Medium   M3 — DTCM import has no per-row try/catch: one P2002 aborts the request with a generic 500 and discards the import report, leaving a partially-applied import

…/import/barcodes/route.ts:120-137 (TOCTOU dup-check + unguarded update), :148 (outer catch). Verified: the file contains exactly one try {.

const existing = await db.registration.findFirst({ where: { dtcmBarcode: barcode, NOT: { id: registration.id } } });
if (existing) { errors.push(…); continue; }
await db.registration.update({ where: { id: registration.id }, data: { dtcmBarcode: barcode } }); // no try/catch

Each update auto-commits, so rows processed before the throw stay written. The operator gets { error: "Import failed" } and no idea which rows landed — they must re-diff by hand, 1–2 days before a Dubai-compliance deadline. Re-running the same CSV is idempotent (the dup-check excludes the row's own id), so recovery is safe — but only if they know to re-run.

Medium   M4 — DTCM import's email lookup picks an arbitrary registration when a person holds two in the event

…/import/barcodes/route.ts:105-113 — findFirst({ where: { eventId, attendee: { email } } })

Attendee rows are shared across registrations in this codebase (the orphan-attendee reuse path), so a compliance barcode can be stamped on the wrong registration. Prefer the registrationId column; when only email is available and it matches more than one row, report the ambiguity rather than silently choose.

Medium   M5 — The "live attendance counter" is not live: a successful scan never invalidates the registrations cache

…/check-in/page.tsx:79-80 — verified: the page contains zero useQueryClient / invalidateQueries references

The counter derives from the React Query cache (staleTime: 5min), and refetchOnWindowFocus never fires because the operator stays in the same window scanning. The one number the door supervisor watches to gauge arrival flow sits frozen for up to five minutes. barcode-import-dialog.tsx:67 correctly invalidates after import; the scanner was missed.

Medium   M6 — A venue network drop mid-scan is nearly invisible: no failure beep, no Recent-Scans row, and the input clears as if it worked

…/check-in/page.tsx:176-179

} catch (err) { console.error("[check-in] scan failed", err); toast.error("Network error"); }
finally { setScanning(false); setManualInput(""); inputRef.current?.focus(); }   // dropped scan, no requeue

Every 4xx and success path pushes a coloured row into Recent Scans and plays a beep. A fetch rejection — the #1 real venue failure — pushes nothing and plays nothing. In a loud hall an operator scanning fast relies on the beep and the list; silence reads as "processed". The attendee walks in un-checked-in, with no durable record to reconcile afterwards. This is the "no onsite offline resilience" gap, concretely. Also: if (…scanning) return silently drops any scan that arrives while a request is in flight — including a different person's.

Medium   M7 — Concurrent allowCancelled overrides double-increment seat and promo counters

src/lib/agent/tools/registrations.ts:391-412 → check-in.ts:109-114 → registration-seat-db.ts:104-129

claimSeat is correctly atomic and capacity-guarded, so two reactivations racing for the last seat resolve to one CAPACITY_EXCEEDED. But because the row commit still carries no status/checkedInAt predicate (H3's gap), two concurrent overrides on the same cancelled registration — with capacity available — both pass the in-memory gate, and each does +1 soldCount and +1 promo.usedCount. One registration consumes two seats and two promo redemptions. H3's conditional claim closes this too.

5. Lows

Low   L1 — The scan lookup ORs across two independent unique domains with no deterministic ordering

…/check-in/route.ts:132-139 — where: { eventId, OR: [{ qrCode }, { dtcmBarcode: qrCode }] }

qrCode and dtcmBarcode are each @unique, but independently — a string could be reg A's qrCode and reg B's dtcmBarcode. With OR and no orderBy, findFirst returns an arbitrary row, so the scan could admit the wrong person. qrCode is a generated cuid and dtcmBarcode is externally supplied, so a real collision is astronomically unlikely — but an explicit two-step lookup (qrCode first, then DTCM) removes the hazard for free.

Low   L2 — DTCM import silently overwrites an existing, different barcode on the same registration

…/import/barcodes/route.ts:120-134 — the dup guard checks other rows (NOT: { id: registration.id }) but never whether this row already holds a different code

A CSV with a wrong barcode replaces a good compliance barcode with no errors[] entry and no skip. Recoverable by re-running a corrected CSV, but invisible in the import summary.

Low   L3 — MCP check_in_registration short-circuits the shared gate, leaving its ALREADY_CHECKED_IN branch dead for that caller

src/lib/agent/tools/registrations.ts:357 — returns { alreadyCheckedIn: true, … } before checkInGate runs

Minor, but it slightly overstates "all three surfaces share one gate": one of the gate's three rules is bypassed by one caller, and the response shape diverges from REST's 400 { error: "Already checked in" }. If the double-check-in policy ever changes, check-in.ts alone won't cover the agent path — the drift shape the unification set out to prevent, in miniature.

Low   L4 — The registrant self-service barcode route has no rate limit

src/app/api/registrant/registrations/[registrationId]/barcode/route.ts

Bounded to self on the registrant branch. Combined with H8's org branch it permits unthrottled enumeration of barcode PNGs across the org. Worth a limit once H8 is fixed.

Explicitly NOT a finding: the desk check-in path has no rate limit, and that is correct. A hardware scanner legitimately fires many scans per second; the endpoint is authenticated and desk-gated; and the in-memory limiter is per-container and resets on deploy, so it could never provide a real guarantee. Throttling here would stall a busy door and buy nothing. The exposure on this path is idempotency (H3), not request volume — a conditional claim is the right control, not a limiter.

Phase 1 — Door correctness (H1 + H2 + H3 + M7) · commit 706ba17

  • H3 + M7 — race-safe commit. executeCheckIn now commits with a updateMany where { checkedInAt: null } conditional claim; a concurrent loser loads the already-checked-in row and returns it idempotently with no duplicate audit row / notification / clobbered timestamp. For the reactivation path the claim gates applyRegistrationTransition, so two concurrent allowCancelled overrides can't double-increment the seat + promo counters (M7).
  • All three callers (both REST handlers + MCP check_in_registration) already share check-in.ts, so H3/M7 are fixed once for everyone. +12 tests. Suite 2401 → 2413.

8. What checked out clean

  • The June unification held. checkedInAt is written in exactly one place (check-in.ts:106) — no raw status flip, bulk update, seed, or registration PUT writes it. Both REST handlers and the MCP tool commit through executeCheckIn.
  • Audit + notification cannot 500 a committed check-in, on all three paths — fire-and-forget with a logged catch. Fixed once, for everyone. The badge route's analytics write is likewise wrapped and still returns the PDF.
  • Barcode rendering is properly centralized in renderBarcodePng; all five callers (badges PDF, both barcode routes, email barcode) go through it. No duplication.
  • The scan lookup is event-scoped — a badge from another event scans to a clean 404. No cross-event check-in.
  • Sequential double-scan is idempotent (the gate returns ALREADY_CHECKED_IN with the original timestamp; it is not overwritten). Only the concurrent case races (H3).
  • The July-7 ONSITE desk fix holds on the five routes it touched: check-in POST/PUT, badges, and the registration create/edit all call denyReviewer(…, REGISTRATION_DESK_ALLOW) and route the event lookup through buildEventAccessWhere.
  • DTCM is never substituted for the entry barcode on badges or any barcode endpoint; the import is admin-only (bare denyReviewer), org-bound, and gated on Event.requiresDtcmBarcode, which exists. Virtual registrations are excluded from badges at the DB where.
  • Five of six CLAUDE.md claims verified true. The false one is "(+ undo)" — see H2.
