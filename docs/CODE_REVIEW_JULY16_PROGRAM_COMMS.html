2. The HIGH

High H1 — The M9 date guard runs on every General-tab save, so an event that already has an out-of-range session can't save ANY General setting

src/app/api/events/[eventId]/route.ts (the if (startDate || endDate || timezone) gate) · src/app/(dashboard)/events/[eventId]/settings/page.tsx (handleSaveGeneral)

// route: gate is PRESENCE in the body, not actual change
if (startDate || endDate || timezone) { ... sweep ALL sessions (no status filter) ... }

// settings page: General save ALWAYS sends the dates (and timezone via ...generalFormData)
body: JSON.stringify({
  ...generalFormData,
  startDate: fromDatetimeLocal(generalFormData.startDate),
  endDate: fromDatetimeLocal(generalFormData.endDate),
})

Failure scenario

M9's motivating population is events whose dates were narrowed before the guard existed — they already have orphaned out-of-range sessions. On any such live event, the organizer now edits the venue, or the tax rate, or renames the event → the General save 400s with "Cannot change the event dates: 1 session(s) would fall outside the new date range" although no date changed. Second scenario: the organizer does the right thing — CANCELS the day-3 session — then shortens the event; the guard still blocks, because CANCELLED sessions are counted even though the public agenda already excludes them. The only way out is hard-deleting the session (losing its record).

Fix

Run the sweep only when the effective window/timezone actually CHANGES (getTime() !== against the existing row), and exclude status: CANCELLED from the out-of-range check (decide DRAFT explicitly — recommendation: count DRAFT, it still renders in the dashboard).

What to learn: "the field is in the request body" is not "the field is changing". Full-form clients resubmit everything; a guard keyed on presence turns into a guard on every save. Gate mutations-with-side-conditions on a computed diff against the stored row, and always ask "which rows should this predicate ignore?" (here: cancelled sessions).

3. Mediums

Med M1 — Invitation sends rotate the agreement link BEFORE the send; a failed send kills the previously-delivered link

src/lib/bulk-email.ts (per-recipient mint inside generateEmailForRecipient) · src/lib/speaker-agreement.ts (mintSpeakerAgreementLink = delete-all + create)

const link = recipient.agreementAcceptedAt
  ? ""
  : await mintSpeakerAgreementLink(recipient.id, event.slug || event.id); // rotates NOW
// ... sendEmail happens later, and can fail

Failure scenario

A speaker received an agreement email last week (link valid 30 days). This week the organizer bulk-sends invitations — the default invitation template now carries {{agreementBlock}}, so every unsigned speaker is re-minted. One speaker's send bounces (SES transient): their old link is now dead and no new email arrived — clicking last week's button says "invalid or already used". Separately, the scheduled-emails worker processes due rows in parallel: an invitation and an agreement chaser due the same minute both mint for the same speaker, and whichever mints second invalidates the other email's link before it is even delivered. Pre-feature, invitation sends never touched agreement tokens — this is a new exposure.

Fix

Reuse-not-rotate for the soft path: mint only when no unexpired token exists (requires storing the raw token or deriving the link differently), or simplest — rotate only on the strict agreement type and have {{agreementBlock}} reuse the existing token when one is live, minting only when none exists.

What to learn: a "rotate on every send" token policy is only safe while exactly one email type can send it. The moment a second sender appears, rotation becomes cross-email interference — the invariant to design for is "the newest DELIVERED link works", not "the newest minted".

Med M2 — Nothing in the UI says an invitation now mints/rotates the agreement link and attaches the agreement document

speakers/[speakerId]/page.tsx + speaker-detail-sheet.tsx (invitation dialog copy) · components/bulk-email-dialog.tsx ("Invite speakers to your event")

Failure scenario

The organizer sent a formal Agreement email on Monday; on Thursday they send a friendly invitation chaser. The invitation silently rotates the agreement token (killing Monday's link — see M1) and attaches the personalized agreement PDF. Every dialog still describes the send as "a speaker invitation email with event details and a request to confirm participation". For an email that changes quasi-legal state, the description must say what the send actually does — the Agreement type's own description already meets this bar.

Fix

One sentence in the three dialogs: "Includes a Review & Agree link (rotates any previously-emailed agreement link) and attaches the personalized agreement when configured."

Med M3 — Bulk agreement sends give already-signed speakers a dead button; single-send gives them a working link

src/lib/bulk-email.ts (signed → link = "") vs speakers/[speakerId]/email/route.ts (agreement type mints unconditionally) · default speaker-agreement template hardcodes <a href="{{agreementLink}}">

Failure scenario

"Email All → Speaker Agreement" with no signed/unsigned filter includes signed speakers. They receive the full "confirm your participation by clicking the button below" copy with href="" — a button that goes nowhere — while the same speaker emailed individually gets a working link. Cross-caller behavioral drift inside the feature whose whole point was ONE implementation, and it misleads signed recipients about their own agreement state. (Pre-fix bulk was equally broken — literal {{agreementLink}} — so not a regression, but the paths now deliberately disagree.)

Fix

Pick one semantic: mint for signed speakers in bulk too (single-send parity — a re-send is an explicit operator act), or teach the agreement template to branch on signed state the way {{agreementBlock}} does. Recommendation: parity (mint), since the agreement TYPE is an explicit "send them the agreement" action.

Med M4 — Moving an abstract between topics of the SAME session fails with a false "assigned to another session"

src/services/session-service.ts (topics reconciliation loop) + SessionTopic.abstractId @unique

for (let i = 0; i < plan.length; i++) {           // payload order
  // create at i=0 assigns abstract A while kept topic t1 (updated later) still holds A
  //   → P2002 → whole tx rolls back → "Abstract is already assigned to another session"

Failure scenario

MCP update_session (the only topic-abstract writer) with [{title:"New", abstractId:"A"}, {id:"t1", abstractId:null}] where t1 currently holds A: the create runs before t1's update clears A → unique violation → the agent is told the abstract belongs to another session and goes hunting for an assignment that doesn't exist. A swap between two kept topics fails in any order. Clean rollback (no corruption), but a legitimate operation is impossible and the error text is wrong on both counts.

Fix

Two-phase within the tx: first null abstractId on every kept row that is losing/moving one, then apply assignments. And special-case the error message for the same-session collision.

Med M5 — SpeakerDocument replace race → generic 500 + a permanently orphaned file on disk

src/app/api/events/[eventId]/speakers/[speakerId]/documents/route.ts (fs.writeFile BEFORE the transaction)

Failure scenario

Two staff replace the signed agreement concurrently. Loser paths: both see no previous → B's create hits the partial unique index (P2002, thrown in-tx) → outer catch → 500 "Failed to upload document"; or both see the same previous → B's delete-by-id hits P2025 → 500. In both cases B's freshly-written file under public/uploads/speaker-docs/ is never unlinked (the only unlink covers the replaced file after a successful commit) — a disk orphan on every lost race or tx failure, and the operator gets no hint it was a race.

Fix

Catch P2002/P2025 → 409 "replaced concurrently — reload and retry", and unlink the just-written file in a tx-failure handler (or write the file only after commit: buffer → tx → write → row update is overkill here; unlink-on-failure is enough).

What to learn: when a filesystem write and a DB transaction must both happen, the failure of either must clean up the other. "Write file, then tx" needs unlink-on-tx-failure; "tx, then write file" needs a repair path for missing files. Pick one and handle its failure leg.

Med M6 — M10's "transactional" aggregate+create doesn't actually close the sortOrder race

src/app/api/events/[eventId]/tracks/route.ts · src/lib/agent/tools/sessions.ts (add_topic_to_session) — and the certificate-templates POST this pattern was copied from

await db.$transaction(async (tx) => {
  const max = await tx.track.aggregate({ _max: { sortOrder: true }, where: { eventId } });
  return tx.track.create({ ... sortOrder: (max._max.sortOrder ?? -1) + 1 ... });
});

Failure scenario

Prisma interactive transactions run READ COMMITTED on Postgres. Two concurrent creates both read max=5, both insert 6, both commit — a transaction with no row lock, no SERIALIZABLE, and no unique constraint serializes nothing. The observable impact is unchanged from pre-fix (a rare ordering tie), but the code and the commit now assert the race is closed, and the certificate-templates "reference pattern" has the identical flaw.

Fix

Honest options: @@unique([eventId, sortOrder]) + retry on P2002; or a SELECT … FOR UPDATE on the parent Event row; or accept the tie and correct the comments (ties are cosmetic — orderBy falls back non-deterministically). Whichever is chosen, fix the cert-templates comment too.

What to learn: wrapping read-then-insert in $transaction changes atomicity, not isolation. Under READ COMMITTED, two transactions still read the same max. Serializing requires a lock, a unique constraint, or SERIALIZABLE — a transaction alone is a no-op for this race.

Med M7 — Signed agreements download with no auth via the public /uploads catch-all

src/app/uploads/[...path]/route.ts (no auth() at all) serving /uploads/speaker-docs/{eventId}/{uuid}.pdf

Exposure scenario

The documents LIST route is properly RBAC-gated, but the bytes are served by the public catch-all — the only protection is the unguessable v4 UUID. Anyone who obtains a URL out-of-band (proxy/access logs, browser history on a shared machine, a screen-share, a Referer leak) fetches the signed legal document with no session. This is the repo-wide pattern (issued certificate PDFs and agreement .docx templates are served the same way), so it's an inherited posture, not a regression — but this commit added the most sensitive document class yet to it.

Fix

Either serve /uploads/speaker-docs/** through an authenticated streaming route that re-checks event access (and consider migrating certificates/agreements to it later), or record an explicit accepted-risk decision that signed agreements rely on UUID-obscurity like the rest of /uploads.

Med M8 — MCP update_session's declared schemas omit topics/sessionRoles: HTTP MCP silently drops them and reports success

src/lib/agent/tools/sessions.ts (executor handles topics incl. id; input_schema says "Does NOT touch topics or speakers") · src/lib/agent/register-mcp-tools.ts (Zod params: no topics)

Failure scenario

An n8n/claude.ai caller sends update_session { topics: [...] }. The HTTP registration's Zod shape strips the undeclared key; the call returns success: true; the topics are untouched — success-claimed-nothing-happened, agent flavor. Meanwhile the M2 commit message says "MCP update_session threads the id through", but that path is only reachable by the in-app agent (which bypasses the Zod shape), not the declared MCP surface. The description/schema/executor triple now disagrees three ways.

Fix

Either expose topics/sessionRoles/speakerIds in both the tool definition and the MCP Zod registration (+ package version bump for the client-cache hint) and update the descriptions — or delete the executor's dead branches and keep the "does not touch" contract true.

Med M10 — The timezone unification stopped at the agenda: five surfaces still show a different clock

speakers/[speakerId]/page.tsx (viewer-local toLocaleString() ×2 + SessionDetailSheet mounted without the timezone prop) · speaker-detail-sheet.tsx (viewer-local) · bulk-email.ts webinar vars (SERVER timezone — "06:00 AM UTC" for a 10:00 Dubai webinar) · speaker-agreement.ts {sessionDateTime} (Dubai-fixed formatDateTime, lands in the personalized agreement document) · dinner RSVP public page + console (viewer-local, no label) · (LOW) webinar console window

Failure scenario

One session can now show three different times on one screen: the agenda grid (event TZ), the speaker page's sessions list (viewer-local), and the SessionDetailSheet it opens (Dubai default). Worse, two attendee-facing producers of the same fact disagree with the fixed surfaces: webinar confirmation/reminder emails render the server's UTC clock, and the {sessionDateTime} merge field inside the quasi-legal agreement PDF is hardcoded GST. The RSVP invitee reads the dinner time in their home timezone with no label.

Fix

Same recipe as the agenda pass: thread event.timezone + formatTimeInTz/ formatDateInTz/tzLabel through the five surfaces (the webinar email + agreement merge field are the two that matter most — they're attendee/legal-facing).

What to learn: timezone fixes don't stay fixed by surface — they stay fixed by PRODUCER. Fixing where a fact is displayed while another code path still formats the same fact with a different clock just relocates the disagreement. Sweep by grep for the formatter (formatTime/formatDateTime/toLocaleString), not by page.

4. Lows

Low L1 — wallTimeInTzToDate's spring-forward doc claim is wrong

src/lib/event-time.ts

The comment says a nonexistent local time "resolves to the instant the clocks skipped to"; empirically a 02:30 New-York gap input resolves one hour before the gap (01:30 EST), and the direction varies by zone (Lord Howe resolves forward). Deterministic, round-trips visibly in the form, and Dubai (the default) has no DST — so impact is a rare silent ≤1h shift. Fix the comment; optionally normalize gap inputs.

The session sweep and the event.update are separate statements; a session created/retimed in the window (or racing the narrow) still produces the orphan M9 blocks. Best-effort is acceptable here — just don't describe the guard as airtight.

6. Lessons index

  1. Presence ≠ change (H1). Full-form clients resubmit every field; guards with side conditions must diff against the stored row.
  2. Token rotation is a cross-sender contract (M1/M2/M3). Adding a second sender to a rotate-on-send token turns rotation into interference; design for "newest delivered link works", and say so in the UI when a friendly-looking send mutates legal state.
  3. $transaction ≠ serialization (M6). READ COMMITTED lets two transactions read the same max. Locks, unique constraints, or SERIALIZABLE serialize; transactions alone don't.
  4. fs + DB need two-way cleanup (M5). Whichever side commits first, the other side's failure leg must undo it.
  5. Fix producers, not surfaces (M9/M10). Grep for the formatter/label map when unifying a concern — the sixth copy is always the one that keeps the bug alive.
  6. Tool schema, executor, and description must move together (M8). An executor capability the schema strips is worse than absent: callers get success for a no-op.
