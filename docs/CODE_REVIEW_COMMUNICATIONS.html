HIGH

H1 — no per-recipient send idempotency SHIPPED 5d03569

src/lib/bulk-email.ts (send loop) · ScheduledEmail (schema)

executeBulkEmail re-resolves recipients and re-sends to all of them on any re-run; the row stored only aggregate counts, no emailedAt/resume marker.

Failure scenario

A 500-recipient send is claimed PENDING→PROCESSING and gets ~200 in before the worker process is OOM-killed / the container restarts. The row sits PROCESSING; the 10-min sweep flips it to FAILED; the operator clicks Retry → the worker re-runs the whole 500-recipient audience with no dedup → the first ~200 delegates receive the email a second time.

Fix

Additive ScheduledEmail.emailedKeys String[] @default([]) (migration 20260711120000, ADD COLUMN IF NOT EXISTS → blue-green safe). executeBulkEmail gains alreadyEmailedKeys (skip these) + onBatchEmailed(keys) (report each sent batch); the worker passes row.emailedKeys in and appends each batch via { push }, so a retry resumes past everyone already delivered. Record failure is best-effort. Certificate sends are unaffected (own issue-or-reuse idempotency).

H2 — "Send now" enqueue is non-idempotent SHIPPED fadfc44

src/app/api/events/[eventId]/emails/bulk/route.ts:90

db.scheduledEmail.create with no dedup. A double-click or an HTTP retry of the 202 creates two identical rows and the worker drains both → the whole audience is emailed twice. The 20/hr limit is no defense (both fit under it; in-memory per-container, so ~2× during a blue/green deploy).

Fix

Best-effort guard: match a same-creator, same-content PENDING send-now row created in the last 2 min and return its jobId instead of enqueueing again. Value-equality only (never merges two genuinely-different sends); not bulletproof under true simultaneity — a unique idempotency key would be, at the cost of a column.

MEDIUM

M1 — stall-sweep vs. a long send SHIPPED 1e7c45e

src/lib/scheduled-emails-worker.ts

The 10-min stuck-sweep keys on updatedAt, but nothing refreshed it during a send (the only bump was the claim). A legitimately long send (>10 min — thousands of recipients / per-recipient PDF generation) was flipped to FAILED mid-flight; a retry then started a second concurrent send, and processRow's unconditional completion write clobbered the reclaimed row's state.

Fix

A 2-min heartbeat touches the PROCESSING row (no-op retryCount increment → @updatedAt) so the sweep only fires on a genuinely dead row; and the SENT / SENT-0 / FAILED writes became conditional updateMany(where:{id, status:PROCESSING}) — if we no longer own the row, count === 0 → log completion-superseded, skip the audit + admin notification (no clobber, no double-notify).

M2 — send-viability deferred to fire time SHIPPED fadfc44

emails/bulk/route.ts · emails/schedule/route.ts · scheduled-emails-worker.ts:200

"Send now" only shape-validated before enqueue, so every real check (untagged cert template, missing agreement template, unbuilt survey, deactivated custom slug) fired only at worker time — a green "queued" toast followed by a FAILED row a minute later. A worker comment still claimed the immediate route surfaces a synchronous 400 (untrue since the June-9 jobification).

Fix

Extracted the config checks from executeBulkEmail into a shared precheckBulkEmailViability() (one implementation, no drift) that the enqueue + schedule routes call synchronously → a real 4xx now. Stale comment corrected.

M3 — schedule-edit drops send-critical filter keys SHIPPED fadfc44

src/app/api/events/[eventId]/emails/schedule/[id]/route.ts:15

The PATCH filters schema knew only {status, ticketTypeId} and replaced the whole JSON, so editing a template/cert/survey/multi-value send stripped templateSlug / certificateTemplateIds / surveyExpiryDays / paymentStatus / ticketTypeIds / … → a fire-time FAIL or a silent audience-widen. Latent (the dialog never edits filters) but the route accepted it.

Fix

filters removed from the edit schema (Zod strips it → persisted filters untouched). Audience changes = cancel + re-create, which the dialog already tells the operator.

M4 — email-preview org-null collapse DEFERRED

src/app/api/events/[eventId]/email-preview/route.ts:51

The one communications route using ...(organizationId ? {organizationId} : {}) instead of the explicit organizationId! its five siblings use; the response renders a real registration's serialId + event name/venue. Not currently exploitable — denyReviewer blocks every org-null role before the query — but the isolation depends on the guard's block-set ordering, not the query. Fix: bind org explicitly for parity. Also: no rate limit (LOW).

M5 — recipient-count predicate triplicated DEFERRED

communications/page.tsx · registrations/page.tsx · speakers/page.tsx

countRegistrations/countSpeakers are hand-copied across three host pages, each independently mirroring executeBulkEmail's where. The speakers-page copy already omits the sessionRole filter the backend applies — count≠send, latent only because that surface doesn't yet pass sessionRoleFilter to the dialog. Fix: a shared matchesBulkEmailFilters() helper.

M6 — survey token re-mint breaks the first email's link SHIPPED 5d03569

src/lib/bulk-email.ts:1082

The survey-invitation token mint (deleteMany + create) lives in the per-recipient path; on a re-run it deleted the token the recipient already holds and issued a second, different link → the first email's link 404s. Closed by H1: the skip filter sits above the mint, so an already-emailed recipient's token is never touched on a re-run.

M7 — a bad filter value silently widens the audience, unlogged DEFERRED

src/lib/bulk-email.ts (parsePaymentStatusFilter / abstracts status as never)

A typo'd paymentStatus/status parses to empty → no predicate → the send goes to everyone (and a reminder re-admits CANCELLED). Unreachable via the current dialog (valid enum values only), reachable via MCP / API / a future tile; and it violates the every-failure-logs rule. Fix: reject + log an unparsable filter value instead of dropping it.
