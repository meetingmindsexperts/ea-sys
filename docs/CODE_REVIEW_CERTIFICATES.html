Code Review — Certificates

4-angle production review · July 13, 2026 · 1 BLOCKER / 5 HIGH / 9 MED / 7 LOW
Angles: issue lifecycle & correctness · security (RBAC / SSRF / injection) · concurrency & idempotency · drift / duplication / failure-visibility. Scope: all of src/lib/certificates/* (render, issue-worker, auto-issue, bundle, bulk-issue, deliver, eligibility, cert-context, survey-thankyou-sweep, pdf-loader, email-tokens), the 20 certificates/** routes, upload/pdf, the MCP certificate tools, the bulk-email.ts certificate branch, worker/jobs/cert-issue.ts + the legacy cron shim, and the Prisma models. Findings already tracked in docs/ROADMAP.md (June-3 deferred list, July bundling M7–M9, July-9 rework M1/L2/L3) were NOT re-reported.

Verdict — one BLOCKER, fixable in a contained change. The send/idempotency core is genuinely well-consolidated (findOrIssueCertificate + sendCertificateBundleEmail + renderBundleEmailContent are single-sourced, rejection logging is thorough). But the July-9 hardened PDF loader was bypassed by two callers that were never migrated onto it, and combined with a template URL field that is stored with zero validation, that composes into an arbitrary-file-read / SSRF exfiltration chain reachable by any ORGANIZER or API key (B1). Below that: a family of worker state-machine and double-email defects — the same classes the July-11 communications review fixed in the scheduled-emails worker, never mirrored into the cert worker.

How to read this document

Each finding has four parts: the issue (what the code does), failure scenario (concrete inputs → concrete damage — if you can't write this, it isn't a finding), suggested fix (what the reviewer proposed), and actual fix (what shipped, with the commit). The green “What to learn” box on each finding is written for a junior developer: the general engineering principle this bug is an instance of. The same principles are collected in § Lessons for a junior developer at the bottom.

BLOCKER

Blocker B1 — Unvalidated backgroundPdfUrl → arbitrary local file read + SSRF, with the duplicate route as a public exfiltration channel

Write side: certificates/templates/route.ts:48 + templates/[templateId]/route.ts:50 (z.string().max(500) — no shape validation) and src/lib/agent/tools/certificates.ts:35 (LOCAL_URL_RE = /^\/(uploads|certificates)\// — prefix-only, /uploads/../../.env passes).
Read side: src/lib/certificates/render.ts:33-47 (fetchBackgroundPdf) and templates/[templateId]/duplicate/route.ts:50-78 (fetchBackgroundBuffer) — both carry their own unguarded readFile/fetch copies that BYPASS the hardened src/lib/certificates/pdf-loader.ts.

The issue

CertificateTemplate.backgroundPdfUrl is persisted as any ≤500-char string. Two readers then consume it with no guard: render.ts does readFile(join(process.cwd(), "public", url)) for local paths (no traversal check) and a bare fetch(url) for everything else (no scheme/host allowlist, no timeout). The duplicate route's fetchBackgroundBuffer does the same — and then stores the raw fetched bytes via uploadCertificatePdf and returns the resulting world-readable /uploads/certificates/… URL on the clone.

Failure scenario

An ORGANIZER (or any MCP API key / OAuth grant) PATCHes a template: {"backgroundPdfUrl": "/uploads/../../.env"} — accepted by Zod. They POST /duplicate: join(cwd, "public", "/uploads/../../.env") resolves to the repo-root .env, its raw bytes are re-uploaded as the clone's “background PDF”, and the response hands back a public unauthenticated URL containing DATABASE_URL, NEXTAUTH_SECRET (which also encrypts the Zoom/EventsAir credentials), Stripe keys, and every API secret — full platform compromise from a non-admin dashboard role. The same field with http://169.254.169.254/… or http://localhost:3099/health turns the server into an internal-network probe (SSRF), exfiltrated the same way.

Why this existed — the instructive part

The July-9 review created pdf-loader.ts as “the SINGLE source of truth” with exactly these guards, and migrated the issue worker + resend route onto it. But render.ts and the (newer) duplicate route kept/added their own private copies. A consolidated safe helper protects nothing if callers can still hand-roll the unsafe version next to it.

Suggested fix

  • Validate on write: a shared validateBackgroundPdfUrl() — local must be /uploads/certificates/… with no .. segment; remote must be https on the Supabase allowlist. Wire into the REST POST + PATCH Zod and the MCP validatePdfUrl. Everything the system itself generates already satisfies this, so only attacker-shaped input is rejected.
  • Guard on read: delete both private fetch helpers' bodies; delegate to loadCertificatePdfBytes() (which also gains a 15s fetch timeout).

Actual fix

Shipped (2c32d21). validateBackgroundPdfUrl() added to pdf-loader.ts; both REST template routes reject invalid URLs via Zod refine with 400 INVALID_PDF_URL; the MCP validator delegates to the same function (the prefix-only LOCAL_URL_RE is gone); render.ts fetchBackgroundPdf and the duplicate route's fetchBackgroundBuffer now delegate to the guarded loadCertificatePdfBytes, which gained AbortSignal.timeout(15s). +11 tests. Package 0.4.19 → 0.4.20 (MCP validation behavior changed — clients reconnect).

What to learn: validate untrusted input at the write boundary AND guard at the read boundary (defense in depth) — either alone eventually gets bypassed. And when a security review consolidates a “single source of truth” helper, grep for every other caller of the same primitive (readFile, fetch) in the domain and migrate them too; otherwise the fix is documentation, not protection. Also note the amplifier: a read-only bug (file read) became full exfiltration because another feature (duplicate) republishes what it reads — always ask “where do these bytes go next?”.

HIGH

High H1 — Worker run-state transitions are unconditional update-by-id: Cancel is clobbered; a cancelled auto run resurrects and emails anyway

src/lib/certificates/issue-worker.ts — 14 certificateIssueRun.update({ where: { id } }) sites (render-complete :250/:508, send-complete :737, failRun :1028-1046, etc.) vs the cancel route's correctly guarded updateMany.

The issue

The cancel route atomically flips a run to CANCELLED, but every worker phase-completion write is a plain update({ where: { id } }) with no status predicate — last write wins.

Failure scenario

Operator clicks Cancel while the worker is inside the final render tick. Cancel commits CANCELLED; milliseconds later the worker's phase-complete write overwrites it with AWAITING_REVIEW (manual run — resurrected, sits sendable) or SENDING (auto run — the next tick emails the certificates despite the cancellation, with no operator in the loop). failRun can likewise overwrite CANCELLED with FAILED while replacing the per-item errors history.

Suggested fix

Every worker status transition becomes a conditional updateMany keyed on the expected prior status; when 0 rows are affected, log completion-superseded and skip the follow-on side effects. This is byte-for-byte the pattern the July-11 comms review applied to the scheduled-emails worker (commit 1e7c45e) — it was never mirrored here.

Actual fix

Shipped (bc06124). New transitionRunStatus() does a conditional updateMany keyed on the expected prior status; applied to all four phase-complete sites (legacy render, bundle render, send, reissue) + failRun (guarded to never overwrite a terminal run). A lost transition logs transition-superseded and skips its follow-on. +1 test (cancel-mid-tick skip).

What to learn: in any state machine shared by two writers (worker + operator route), a transition must be compare-and-swap — “move from RENDERING to AWAITING_REVIEW”, never “set to AWAITING_REVIEW”. updateMany({ where: { id, status: expected } }) returning count === 0 is your signal that someone else got there first; handle it explicitly. When one part of a codebase fixes a bug class, sweep sibling subsystems for the same shape.

High H2 — Missing/unfetchable background PDF silently issues + emails the instructional placeholder page as a real certificate

src/lib/certificates/render.ts:90-108 — both the “no background configured” and the “fetch failed” branches return renderPlaceholder(data), and renderCertificate is the render core for every REAL issue path (bulk-email certs, per-person Issue, auto-issue, worker phases), not just the preview endpoint.

Failure scenario

Organizer creates a template, sets tag + autoIssueOnSurvey, forgets the PDF upload → every survey completer is auto-emailed a page reading “No background PDF uploaded yet — configure this cert type…” as their certificate, with a real serial, recorded in IssuedCertificate, no review gate. Or: a transient storage failure ships the same placeholder, and findOrIssueCertificate's reuse rule re-attaches that placeholder PDF on every later resend. This contradicts the June-3 H4 policy (“fail the run hard rather than ship placeholders”), which only covered template-row deletion.

Suggested fix

renderCertificate throws when the background is missing/unfetchable unless an explicit allowPlaceholder flag is passed — set only by the preview endpoint, where the placeholder is a feature.

Actual fix

Shipped (bc06124). renderCertificate now THROWS on a real issue path when the background is missing/unfetchable; only the preview endpoint opts in via { allowPlaceholder: true }. Render errors are already caught per-item, so this surfaces as a per-recipient failure. +1 throw test.

What to learn: a graceful fallback that is correct for a preview is a silent data-corruption bug in production. When one function serves both, the safe default is fail loudly, with an explicit opt-in flag for the forgiving mode — never the other way round. Read the comment above the fallback: it says “so the preview shows something useful” — the code outgrew its comment, which is itself a smell worth checking.

High H3 — Manual run × auto-issue overlap double-emails the same certificate; the protection that made this a “verified false positive” was removed

src/lib/certificates/auto-issue.ts:322-336 (pending-item guard matches autoIssue: true runs only) · issue/route.ts:219-241 (mirror-image guard filters autoIssue: false) · issue-worker.ts:643-655 (the P2002 “fail-at-link” loser now proceeds to send).

Failure scenario

Operator starts a manual run for template T (tag committee) at 10:00; a committee member completes the survey at 10:01; the 10:02 auto-sweep finds no existing cert and no existing auto-run item (it never checks manual runs' pending items) → creates an auto run for (T, person). Both runs render (one mints, one reuses via findOrIssueCertificate) and both send phases email the same certificate — the auto run immediately, the manual run when the operator clicks Send. ROADMAP recorded this as a false positive because the item-level issuedCertificateId @unique made the loser fail-at-link and never email; the July bundle rework deliberately made that loser proceed (legacy-pointer-held-elsewhere), so the finding is live again.

Suggested fix

The auto-sweep's existing-item guard matches pending items of any non-terminal run covering (template, recipient) — not just autoIssue: true ones.

Actual fix

Shipped (bc06124). The auto-sweep guard now matches any NON-TERMINAL run (manual OR auto) covering (template, recipient), excluding FAILED/CANCELLED (so nobody is stranded) and render-failed items. +1 manual-run-suppression test.

What to learn: idempotency arguments are often load-bearing on some specific constraint. When you refactor away that constraint (here: the strict 1:1 cert-pointer link), every past “safe because of X” verdict that cited it silently expires. Keep a note of WHAT makes a race safe, so removing it forces the re-check — that's exactly what “verified false positive (because fail-at-link)” in a ROADMAP is for, if anyone re-reads it.

High H4 — Template duplicate silently drops role, cmeHours, autoIssueOnSurvey, autoIssueTag

templates/[templateId]/duplicate/route.ts:150-163 — the clone's create() copies name/category/background/textBoxes/sortOrder/email fields only.

Failure scenario

Duplicating a CME certificate produces a clone whose {{cmeHours}} renders blank; duplicating a role cert loses its {{role}}; duplicating an auto-issue template produces a clone that silently never auto-issues. No error anywhere — the organizer discovers it when attendees receive hour-less certificates.

Suggested fix

Copy role, cmeHours, autoIssueTag. Deliberately do NOT auto-enable autoIssueOnSurvey on the clone (a live auto-issuing clone would instantly double-issue the same tag audience); surface “auto-issue is off on the copy — re-enable when ready” in the response instead.

Actual fix

Shipped (05f0695). The clone now copies role/cmeHours/autoIssueTag; autoIssueOnSurvey is forced OFF (a live clone would instantly double-issue the source tag audience) and the response returns { autoIssuePaused } so the UI can prompt a re-enable. +3 route tests.

What to learn: hand-listing fields in a clone/copy operation creates a maintenance trap — every future column silently defaults on clones. Either copy by spread-and-omit ({ ...source, id: undefined, name: newName } shaped for the ORM) or add a test that fails when the model gains a field the clone doesn't handle. Also: when copying config that has side effects (auto-issue), think about whether “identical copy” is actually safe — sometimes the right copy is “same data, paused”.

High H5 — Dead /resend route + useResendCertificate hook: an orphaned 4th send pipeline that already drifted

certificates/issued/[certificateId]/resend/route.ts (449 LOC, zero callers) · src/hooks/use-api.ts:2249 (useResendCertificate, zero component callers — the card uses useReissueCertificate + useResendCertificateBundle).

The issue

The route hand-rolls the entire cover-email pipeline inline (token resolve → escape → wrapWithBranding → inlineCss → sendEmail) instead of the shared sendCertificateBundleEmail — and has already drifted: it doesn't set storeBody: true (breaking the “audit copy covers ALL cert email paths” contract), doesn't thread recipient names, and omits the attachment cap.

Suggested fix

Delete the route + hook. The reissue/resend-bundle paths supersede it.

Actual fix

Shipped (05f0695). Deleted the dead /resend route (449 LOC), the useResendCertificate hook, and its test.

What to learn: dead code is not neutral — a dead copy of a live pipeline is a drift bomb: the next person greps, finds it, “fixes” or resurrects it, and reintroduces every contract it missed. When a feature is superseded, delete the old path in the same change (git history keeps it). “Zero callers” is a five-minute grep and one of the highest-value checks in a review.

MEDIUM

Medium M1 — Global-unique serial + non-unique event code prefix: cross-event collision, misdiagnosed P2002

cert-context.ts:119-129 · schema: IssuedCertificate.serial @unique (global) while Event.code has no unique constraint and is auto-derived from the name.

Failure scenario

A recurring event (“OSH Monthly Meeting” → code OMM every month) produces identical serial prefixes, so event B's OMM-ATT-0001 collides with event A's. The P2002 fires on serial, but both catch blocks assume it's the recipient-uniqueness index: findOrIssueCertificate looks up a “winner” by recipient, finds none, rethrows (each retry burns another serial); issueSingleCertificate returns the flatly wrong ALREADY_ISSUED for a person who holds nothing.

Suggested fix

Discriminate the P2002 target (e.meta.target) in the catch blocks now (code-only); consider @@unique([eventId, serial]) as a follow-up schema change.

Actual fix

Shipped (a4b080d). New shared isSerialCollision() discriminates err.meta.target; both catch blocks surface a distinct serial-collision error instead of the wrong recipient-dup handling. The @([eventId, serial]) schema change is deferred to ROADMAP. +1 test.

What to learn: a table can have several unique constraints, and P2002 alone doesn't tell you which one fired — read error.meta.target before “handling” it, or your recovery path handles the wrong failure. And when building an identifier from parts, check the uniqueness of every part: unique(serial) is only as strong as the uniqueness of its prefix.

Medium M2 — Auto-issue sweep never checks registration status: a CANCELLED registration can be auto-issued and emailed a certificate

auto-issue.ts:147-166 (candidate query has no status predicate) — contrast eligibleForAttendance and the bulk-email guard, which both exclude CANCELLED.

Failure scenario

Cancel-after-survey: normally a ~3-minute window, but the retry backoff stretches it to hours (1+5+15+60+180 min) — a refund cleanup, duplicate removal, or the DECLINED-speaker companion cascade inside that window still results in a certificate of attendance emailed to a registration the org just cancelled.

Suggested fix

status: { not: "CANCELLED" } in the candidate query + a belt-and-braces skip in processRegistration.

Actual fix

Shipped (a4b080d). Added status: { not: "CANCELLED" } to the auto-issue candidate query. +1 test.

What to learn: when several code paths select “people who should get X”, the eligibility predicate must be shared or at least identical — grep for the sibling paths' WHERE clauses and diff them. A delayed/retried job also needs its guards re-checked at execution time, not just at enqueue time, because the world changes while it waits.

Medium M3 — Bulk-email certificate send with an explicit status filter bypasses the CANCELLED exclusion

src/lib/bulk-email.ts:791-798 — the CANCELLED exclusion applies only when no explicit status filter is set.

Failure scenario

An operator combines a stale “Cancelled” list filter with the certificate email type (or an MCP/n8n caller passes status: "CANCELLED") → every matching cancelled registration gets a real IssuedCertificate minted and emailed. Two clicks.

Suggested fix

For emailType: "certificate", force-exclude CANCELLED regardless of the explicit filter (or 400 INVALID_FILTER when the status list includes it).

Actual fix

Shipped (a4b080d). precheckBulkEmailViability now 400s INVALID_FILTER for a certificate send explicitly targeting CANCELLED. +1 test.

What to learn: a business invariant (“a cancelled reg never gets a cert”) must hold for every input combination, not just the default one. When a rule is implemented as “apply the guard unless the caller specified something”, the caller's input just became a bypass. Encode invariants unconditionally; let filters narrow, never widen.

Medium M4 — Survey auto-issue MULTI bundles ignore the editable certificate-bundle-delivery template

auto-issue.ts:379 (multi bundle → run cover = null) → issue-worker.ts:715 falls back to the HARDCODED SYSTEM_DEFAULT_*_MULTI constants, never calling loadBundleCoverEmailTemplate — the one send path that doesn't.

Failure scenario

A survey completer who is committee + speaker (2 certs → one bundle) gets the hardcoded default cover, ignoring the organizer's edited template. docs/CERTIFICATES.md explicitly claims the editable template covers “survey auto-issue bundles”.

Suggested fix

The worker send phase resolves its fallback cover via loadBundleCoverEmailTemplate(eventId) (per-run, cached per tick if needed) before the hardcoded constants.

Actual fix

Shipped (a4b080d). The worker send phase now resolves a multi-cert bundle cover via loadBundleCoverEmailTemplate before the hardcoded default, matching deliver.resolveResendBundleCover.

What to learn: when a configurable value replaces a hardcoded one, list every consumer of the old constant and migrate all of them — the constant should end up referenced only as the terminal fallback inside the loader itself, so no caller can accidentally skip the config.

Medium M5 — Legacy cron shim runs the cert sweeps with no advisory lock

src/app/api/cron/certificate-issues/route.ts:37-51 calls runAutoIssueSweep/runSurveyThankYouSweep/tickAllRuns directly, ungated — vs worker/jobs/cert-issue.ts:27 which wraps the same three in withJobLock.

Failure scenario

The crontab lines were disabled June 9 and the route is a rollback handle — but the moment it's re-enabled alongside the worker (the exact rollback scenario it exists for), both drain runs and sweep survey rows concurrently with no lock coordination, re-opening the double-email windows.

Suggested fix

Route the shim through the same advisory lock (or at minimum a prominent comment: “must stay disabled while the worker runs”). The lock is the honest fix — the shim's whole purpose is to be safe to turn on.

Actual fix

Shipped (a4b080d). Loud DEPRECATED-ROLLBACK-HANDLE guard comment + a runtime warn on every invocation. Full lock-sharing deferred (would couple src→worker and, per the P3 transaction-pooler limitation, wouldn't hold across backends anyway).

What to learn: a rollback path is production code — it runs precisely when things are already going wrong. If the primary path needed a lock, the fallback needs the same lock; “it's disabled right now” is a runtime state, not a safety property.

Medium M6 — Send-phase crash windows: emailedAt stamped after the send with no per-item claim; thank-you suppression keys on the legacy 1:1 pointer

issue-worker.ts:830-849 · survey-thankyou-sweep.ts:258-337.

The issue

A worker crash between “email sent” and “emailedAt stamped” re-emails that recipient next tick (no per-item conditional claim before the send). Separately, the thank-you sweep's cover-email suppression matches run items by issuedCertificateId — the legacy pointer a bundle item can legitimately lack — so in that branch the duplicate cover email is guaranteed, not just crash-window.

Suggested fix

Per-item conditional claim (updateMany where { id, emailedAt: null }) before the send — accepting the reversed trade-off (crash after claim = one missed email, recoverable) — and key suppression the same deterministic (templateIds × facets) way the send phase collects.

Actual fix

Deferred → ROADMAP. Needs the same at-least-once vs at-most-once decision the comms worker made; bundle it with H1's conditional-transition work if picked up together.

What to learn: “do the side effect, then record it” gives at-least-once (duplicates on crash); “record it, then do it” gives at-most-once (drops on crash). Neither is free — pick per side effect: for emails to humans, a rare missed email you can resend beats a duplicate you can't unsend.

Medium M7 — docs/CERTIFICATES.md documents the superseded per-category uniqueness; the doc contradicts itself

docs/CERTIFICATES.md:48-63 (claims @@unique([eventId, type, registrationId]) + a rationale heading defending it) vs prisma/schema.prisma (per-template uniqueness since migration 20260625140000) — later doc sections describe the new model correctly.

Actual fix

Fixed in docs (this review). docs/CERTIFICATES.md Models block now shows the per-template @ + a dated correction note; the old per-category rationale heading is rewritten.

What to learn: when you change a constraint, grep the docs for the OLD constraint string in the same commit. A doc that argues confidently for a superseded design is worse than no doc — the next engineer trusts the rationale section most.

Medium M8 — ROADMAP lists the abstractTitle XSS as open, but it's already fixed

docs/ROADMAP.md P1 #2 + June-3 deferred table vs email-tokens-resolver.ts:210 (escaped under escapeDynamic, set by every HTML caller).

Actual fix

Fixed in docs (this review). The stale abstractTitle XSS entry is struck from ROADMAP (already fixed in email-tokens-resolver.ts under escapeDynamic).

What to learn: a backlog is only useful if closing an item removes it. Doc-drift runs both directions — stale “open” items cause duplicated work just as stale “done” claims cause false confidence.

Medium M9 — MCP↔REST parity gap: no duplicate_certificate_template tool

src/lib/agent/tools/certificates.ts (list/create/update/delete/update_cme only).

Actual fix

Deferred → ROADMAP. Adding the tool owes a package bump + client reconnects; fold into the next MCP-changing deploy.

LOW

Low L1 — Legacy render path can overwrite an existing cert's PDF with one displaying a serial that exists on no DB row

issue-worker.ts:405-479 (renderAndStoreItem) — on P2002 it reuses the existing cert row (old serial) but still uploads the freshly rendered PDF with the NEW serial baked in. Reachable only for pre-2026-07-09 runs (empty templateIds); the bundle-path equivalent gets it right. Deferred — one more reason to delete the legacy pipeline (already a ROADMAP item).

What to learn: when recovering from a uniqueness conflict by adopting the winner's row, every derived artifact (here: the PDF) must be re-derived from the winner's data — reusing half of each is worse than either.

Low L2 — Auto-run partial-render failures leave failedCount in a state retry-failed “corrects” to zero

issue-worker.ts:600-616 vs runs/[runId]/retry-failed/route.ts — bundle partial failures bump failedCount without item-level errorMessage, so retry-failed finds zero failed items and resets the counter, erasing the only surface-level trace of a missed template. Deferred.

What to learn: keep aggregate counters derivable from row-level truth; a counter that can disagree with the rows it summarizes will eventually be “reconciled” in the wrong direction.

Low L3 — Issue route's concurrent-run guard is in a transaction but is not true mutual exclusion

issue/route.ts:211-278 — findFirst + create in a $transaction is not a serialization point under READ COMMITTED; two simultaneous Issues can both pass. Damage is bounded by per-recipient idempotency downstream. Deferred.

What to learn: wrapping check-then-act in a transaction does NOT make it atomic against concurrent transactions — you need a unique constraint, a row lock (FOR UPDATE), or an advisory lock for actual mutual exclusion.

Low L4 — Serial allocation outside the cert-create transaction (benign gaps)

cert-context.ts:119-129 — the counter upsert is atomic, but a failed render after allocation burns the number. Gaps are cosmetic; documented as accepted. Deferred (observation).

What to learn: sequence gaps are usually fine — chasing gapless sequences couples unrelated failures together. Know when NOT to fix something; write the acceptance down.

Low L5 — Stale caps/dates in comments and docs

MCP tool header says “5MB cap” (real cap 10MB, upload/pdf/route.ts:42); docs/CERTIFICATES.md header “Last updated 2026-06-02” above ~230 lines of July additions; the state-machine section says the tick is “every 60s” (worker is every 3 min). Docs phase.

What to learn: numbers in comments (caps, cadences, limits) drift the fastest — prefer referencing the constant (“see MAX_PDF_BYTES”) over restating its value.

Low L6 — revokedAt has no write path anywhere

Every send path refuses revoked certs (CERT_REVOKED), but no route, tool, or script sets revokedAt — and if a cert were revoked via SQL, the per-template unique index permanently blocks issuing a corrected replacement (no unrevoke/supersede flow). Observation; deferred until revocation is actually needed.

What to learn: guard machinery for an unreachable state is a design IOU — fine to keep, but write down that the feature is half-built so nobody assumes the workflow exists.

Low L7 — Issued cert PDFs are world-readable at unguessable UUID URLs

src/app/uploads/[...path]/route.ts serves everything under public/uploads unauthenticated with immutable caching. Capability-URL security (128-bit UUID) is an accepted trade-off today; revisit with signed URLs if certs ever carry sensitive data. Deferred.

What to learn: “unguessable URL” is a real security model (capability URLs) but it degrades with every place the URL is logged, cached, or forwarded — know that you chose it, and what would force an upgrade.

Lessons for a junior developer — the general principles behind these bugs

  1. A “single source of truth” is a claim you must enforce, not declare. The hardened PDF loader existed for 4 days while two unguarded copies sat beside it (B1). After consolidating, grep the domain for the raw primitives (readFile, fetch, sendEmail) and migrate or delete every remaining direct use.
  2. Validate at the write boundary AND guard at the read boundary. Either alone is one missed call site away from a hole. Stored data is still untrusted input to the code that reads it later.
  3. Ask “where do these bytes go next?” A file read is bad; a file read whose output is republished at a public URL (the duplicate route) is exfiltration. Severity comes from the full chain, not the single line.
  4. State machines need compare-and-swap transitions. “Set status to X” is a race with every other writer; “move from Y to X, and tell me if you couldn't” (updateMany + count check) is a transition. If a cancel button exists, every worker write must expect to lose to it.
  5. Fallbacks have a blast radius. A placeholder that is helpful in a preview is silent corruption in a production send (H2). Default to failing loudly; make forgiveness an explicit, caller-supplied flag.
  6. Idempotency proofs cite specific constraints — removing the constraint expires the proof. The double-email “false positive” was true only while the 1:1 pointer index existed (H3). When you delete a unique index or a foreign key, search for every argument that leaned on it.
  7. Clone operations rot. A hand-listed field copy silently drops every column added later (H4). Copy by spread-and-omit, or pin it with a test that fails when the model grows.
  8. Delete superseded code in the same change that supersedes it. A dead copy of a live pipeline (H5) is where drift hides and where the next engineer's grep goes wrong. Git history is the archive; the working tree is not.
  9. Know which constraint threw. P2002 carries meta.target; handling “a” uniqueness violation as “the” uniqueness violation produces confidently wrong recovery (M1).
  10. Invariants are unconditional. “Cancelled registrations never get certificates” must survive every filter combination and every entry point (M2, M3). If a rule only holds on the default path, it isn't a rule.
  11. At-least-once vs at-most-once is a choice you make per side effect. Stamp-then-send drops on crash; send-then-stamp duplicates on crash (M6). For human-visible emails, prefer the failure you can repair (resend) over the one you can't (unsend).
  12. Docs drift in both directions. A stale rationale section actively misleads (M7); a stale “open” backlog item gets re-fixed (M8). Update docs in the same commit as the behavior they describe.
