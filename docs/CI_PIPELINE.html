CI/CD Pipeline & GitHub-VM Capacity

What .github/workflows/deploy.yml does after the July 22, 2026 restructure, what runs on GitHub-hosted VMs vs the EC2 box, and the ranked menu of further work that could be offloaded to Actions. Companion to docs/ROLLBACK.md (rollback mechanics), docs/MULTI_TENANCY.md §0 (the tenancy harness this pipeline now runs), and docs/INCIDENTS.md INC-001 (why images build on GitHub, not the box).

Old shape: one serial ci job (~5–6 min) ran all of checks/tests/build in a row. The split (owner decision, July 22) trades a few extra billed runner-minutes + duplicated setup blocks for ~2–2.5 min less wall clock per deploy. Every gate that blocked image builds before still blocks them — build-push needs all three.

Accepted trade-offs of the split

  • +3–5 billed Actions minutes per push — each job pays its own ~40s checkout/Node/cache setup; a lockfile change triggers three concurrent npm ci runs.
  • Broken commits burn more minutes — previously a lint failure stopped the serial job before the expensive build step; now tests and build run to completion regardless.
  • Three duplicated setup blocks in the YAML to keep in sync (pattern pre-existed in sourcemaps/tenancy).

2. What runs where — the one fact to internalize

Every CI job — including the postgres/pgbouncer service containers — runs on GitHub-hosted throwaway VMs (runs-on: ubuntu-latest). They cost the EC2 box nothing: no CPU, no memory, no disk, no network. The box participates in exactly one step of the pipeline: the final deploy job SSHes in, checks out the pinned commit, and runs scripts/deploy.sh (which pulls the pre-built images — building on-box is only the emergency fallback, per INC-001). Service containers live for the duration of one job (~2 min) and are destroyed with the runner.

3. The tenancy job (first tenant of the service-container pattern)

LIVE Runs the tests/tenancy suite — 13 assertions against a real postgres:16 + edoburu/pgbouncer in transaction pool mode (mirroring the Supabase prod topology), two seeded tenants sharing an event slug, and the pilot Event RLS policy. This is the only automated proof that tenant A cannot read tenant B — RLS + pooler behavior is unprovable with mocked Prisma.

  • Non-gating by design: nothing needs: it, so a container flake can never block an emergency hotfix for a live event. A failure = red ✗ in the Actions tab only.
  • PENDING Promotion to gating after ≥5 consecutive green runs — a one-line change (add tenancy to build-push's needs). Track it; a permanently non-gating isolation suite is theater.
  • Local equivalent: docker compose --profile tenancy up -d → npm run test:tenancy.

4. Menu — what else GitHub VMs could do for us

Ranked by value to this repo. Nothing below is built; each is its own decision. Scheduled items belong in separate workflow files (that's the legitimate use of multiple .ymls — per-file schedule: triggers — not parallelism, which jobs already provide).

IdeaWhat / whyCost & shapeStatus
1. Fresh-DB migration replay Postgres service container + prisma migrate deploy against an empty DB — replays the entire migration chain from scratch. Prod only ever migrates incrementally, so a broken chain surfaces exactly when the two-silo platform instance gets its fresh database. This job makes that day boring. Directly protects the Phase-0 endgame. ~2 min; per-push (cheap) or per-migration-change via paths: filter. Non-gating → gating once trusted. RECOMMENDED NEXT
2. Playwright e2e in CI The e2e suite has never run in CI (local-only, with the 3113-port-drift friction). A CI job owns the whole stack: postgres container → db push + seed → dev server → Playwright. Closes the biggest testing gap; kills the local-port friction because CI starts its own server. ~6–8 min. Too slow for the deploy path — run daily on a schedule (own workflow file) or non-gating per-push. HIGH VALUE
3. Automated DR restore drill Monthly scheduled run of the quarterly-manual drill: pull latest S3 pg_dump, restore into an ephemeral postgres:17, row-count the 9 critical tables (scripts/dr-restore-drill.sh). Continuous proof that backups restore — the point of having them. ~5 min/month. Requires an IAM change: extend the GitHub OIDC role with read on the Singapore DR bucket + KMS decrypt (owner runs the AWS command — per the AWS instruct-don't-execute rule). HIGH VALUE
4. Weekly npm-audit report Scheduled workflow running npm audit, output to the run summary (or SES). Keeps the deliberate policy — audits never gate deploys — but makes the manual check automatic. ~1 min/week. Trivial. EASY WIN
5. Bundle-size guard Compare next build route sizes per push; warn on a big jump. Catches the accidental-heavyweight-import class (the Zoom-SDK-on-first-paint mistake). Piggybacks on the existing build job output; a small diff step. NICE
6. Schema-drift check prisma migrate diff asserting schema.prisma matches the migrations folder — catches a hand-edited schema that never got a migration. Seconds; could join the checks job. NICE

5. Quota math

Private repo → every job spends billed Actions minutes (Linux = 1×). Rough per-push spend after the split: checks ~2 + tests ~1.5 + build ~3.5 + tenancy ~2 + sourcemaps ~3 + build-push ~4.5 + deploy ~2.5 ≈ ~19 min/push (was ~16 pre-split). At 10 pushes/day ≈ 190 min/day ≈ ~4,000/month — check the org's plan quota (Free = 2,000, Team = 3,000, overage is billed per-minute) before adding per-push jobs; prefer schedule: triggers for anything heavy (e2e daily ≈ 210 min/month vs ~2,100 if per-push).
