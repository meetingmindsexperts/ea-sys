CRM — Build Status

What is built, what is wired to the rest of EA-SYS, and what is not.
Last updated: 15 July 2026 (product catalog + deal line items + editable CRM email templates + dedicated deal/account pages + CRM email + edit + soft-delete + change-log shipped) · HEAD WORKING · Plan: CRM_MODULE_PLAN.md

NOT DEPLOYED. Everything below is merged to main but not running in production. The migration has not been applied to the prod database, and no user can reach any of it. Deploy is a separate, explicit step (bash scripts/deploy.sh on the box).

The migration is additive + idempotent, and no existing code path touches the new tables, so deploying is low-risk — but it is a decision, not a formality.

CRM contacts are their own table (July 15). Business people we negotiate with — pharma reps, exhibitor sales, procurement — live in CrmContact, NOT in the event Contact store (HCPs: doctors, nurses). Reason: every Contact row is mirrored to the external HCP marketing list by contacts-central-sync (which has no where-clause), so a rep in Contact would be marketed to as a doctor. Separate tables make that leak structurally impossible, not merely prevented by remembering to filter.

A person who is genuinely both (a rep who also attends) is linked via CrmContact.contactId — one human, one record, two hats. And a deal now links to many contacts, each with a role on that deal (Primary / Procurement / Marketing / …), because a sponsorship is decided by the rep, marketing AND procurement — not one person.

5. What is DONE

Schema & migration DONE
  • 5 models — CrmCompany, CrmPipelineStage, CrmDeal, CrmTask, CrmNote — plus 4 enums and 2 nullable Contact columns.
  • Migration 20260714120000_add_crm_module: fully additive + idempotent (guarded CREATE/ADD), so the old container serves happily against it mid-blue-green-swap and a re-run is a no-op.
  • Verified column-by-column against prisma migrate diff --from-empty — all 63 columns, every index name and FK policy byte-match Prisma's canonical DDL.
  • FK policy protects money history: company/stage are RESTRICT (can't delete an account or a pipeline column that still holds deals); event/contact/owner are SET NULL (the deal outlives them).
  • CrmCompany.nameKey (trim+lowercase) carries the unique index, so the “Abbott vs abbott = two accounts” bug is structurally impossible rather than dependent on every writer remembering to normalize.
Services DONE
  • company — find-or-create; exact dupes blocked by the index, near-dupes created + flagged needsReview for a human.
  • pipeline — idempotent seeding; resolveStage() is the one place an untrusted stage id becomes trusted (always org-bound). Stages (Jul 15): New → Proposal → Negotiation → Contract Signed → Purchase Order → Invoice Sent → Won → Lost (Won/Lost keep those names — the close flow is name-bound). Existing orgs are brought to the new set by scripts/reconcile-crm-pipeline.ts (dry-run first; deals in dropped columns move to New), since ensurePipelineStages never mutates an org that already has a pipeline.
  • deal — the stage move is a conditional claim, not a write. It carries the stage the mover believed the card was in; if a colleague moved it first, the write is refused (409) instead of silently overwriting their decision.
  • task — the reminder stamp. Re-arming a reminder clears it (so a rescheduled reminder actually fires); completing does not (a cleared stamp on a due task would make the worker re-email).
  • note — author-only to edit; admins may delete but never rewrite. A note is a first-person account of a call held with no visible edit history.

The default write rate limit lives inside the write gate, so an unprotected CRM write is not expressible.

Worker — crm-reminders DONE

Every 5 minutes. Emails a task's owner when it falls due.

The row is CLAIMED before the email is sent, and that ordering is the design. Claim-then-send risks a reminder claimed but never delivered — logged loudly, and recoverable by a human seeing an OPEN overdue task. Send-then-claim risks emailing the same nag twice, or ten times if a slow send overlaps the next tick. A missed nudge beats a spammed inbox.

Registered on the Infra / Ops “Cron / Jobs” card, so it can't go silently unmonitored.

7. Honest notes

Three things were caught by guards, not by me — which is the single best argument for not skipping the Week-4 review:

  • The build caught a Node import (fs, via the logger) leaking into the client bundle. Had it not failed, the runtime symptom is “the button does nothing and there are no logs.”
  • An existing drift test caught the new cron missing from the monitoring roster — it would have run unwatched.
  • The owner's own review ("are you sure Week 1 is done?") found six gaps, including five mutations writing no audit row — the exact finding from the contacts review, reproduced verbatim weeks after writing it up.

Stabilization-first still applies. The CRM was parked on 13 July on the basis that no new domains should land until a few real conferences have run. It was un-parked the next day; that concern has not gone away, and it is carried into the build as a constraint: v1 stays behind the §10 scope fence, the module is namespace-bounded so it adds one place to look rather than eleven, and if a conference lands mid-build, the conference wins — the CRM schema is additive and unreferenced by any live path, so pausing costs nothing.

Deliberately not built (v1)

  • Two-way email inbox sync — integrate, don't build. Months of work to replicate; connect a real Freshsales/HubSpot via n8n if it's ever needed.
  • Lead scoring / workflow automation.
  • Hard delete for deals or companies — a deal is revenue history. A mistaken deal is closed LOST; a mistaken company is merged.
  • Multi-currency pipeline rollups — currency is stored per deal; report per-currency.
