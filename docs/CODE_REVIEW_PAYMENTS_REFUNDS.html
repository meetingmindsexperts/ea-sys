Code Review — Payments, Refunds & Credit Notes

Production-readiness audit · 10 July 2026 · EA-SYS main @ 420f139
Four independent review passes: money-flow lifecycle · concurrency & idempotency · RBAC / security · drift, duplication & failure-visibility.

How to read this. Findings are grouped by theme, not by which reviewer found them — the same root causes surfaced from several angles, and where two independent passes hit the same defect that is noted (it raises confidence). Each finding carries a severity, the exact code, and a concrete failure scenario. Everything already tracked in docs/ROADMAP.md from the partial-refunds and cancel-with-refund reviews (M1–M3, L1–L3, the staged-edits / generic-500 / auto-CN-divergence items) was deliberately excluded, except the queued "newest-payment-only" item, which is scoped precisely here as H6 because it moves real money. The highest-severity findings were independently re-verified in source by the lead reviewer (B1, H1, H2, H3, H4, H5, H9, H10, L1, L3 — quoted lines confirmed byte-for-byte).

1. Executive summary

The core designs are sound and several are genuinely careful: the credit-note cap is enforced inside the transaction under a real SELECT … FOR UPDATE row lock (and it holds through pgbouncer); checkout derives every price server-side — the client sends only a registrationId; the Stripe webhook verifies signatures on the raw body before any processing and re-validates metadata ids against the DB; the refund amount is server-capped against the remaining balance; duplicate Payment rows from webhook retries of the same session are structurally blocked by stripePaymentId @unique; and the seat/promo status transition is now consolidated in applyRegistrationTransition across all three callers — the known duplication has shrunk, not grown.

The risk concentrates in four structural patterns, and almost every finding below is an instance of one of them:

  1. The Stripe webhook is a second, ungoverned money-writer. It flips paymentStatus, mints documents, and reconciles refunds through hand-inlined logic that shares none of payment-service's gates or side-effect contract — so it pays cancelled registrations (H2), drops real settlements when the registration is already PAID via another channel (H1), never checks the amount paid against the amount owed (M8), and applies the opposite refund-communication policy to the same event depending on which button the organizer clicked (H11).
  2. One counter conflates two money sources. Registration.refundedAmount and the single payments[0] pick assume one payment from one source. Any registration with mixed Stripe + manual money breaks every refund path (H6), and the webhook's delta-reconcile silently miscounts against the mixed total (M4).
  3. Money state is written on the wrong side of slow network calls. The refund books the DB before Stripe moves money with no crash recovery (H4), the rollback can erase a refund that really happened (H5), checkout demotes a possibly-already-PAID row from a stale pre-Stripe read (H3), and cancel releases seats from a snapshot taken seconds before the transaction (M2).
  4. The finance read surface never got the ONSITE per-event fix. The July 7 remediation routed the five desk routes through buildEventAccessWhere — but every invoice/quote GET still org-scopes only, and ONSITE (and MEMBER, per the June 17 decision) is finance-capable (H10). The same unbound-id class also allows minting an invoice against another event's — or another org's — registration (H9).

One finding is graded BLOCKER because it double-charges real attendees in completely normal operation, with no race and no attacker: a sponsor-paid (INCLUSIVE) registration shows a live Pay Now button and checkout accepts the payment (B1).

3. Charging money — checkout & the Stripe webhook

Blocker   B1 — Sponsor-paid (INCLUSIVE) registrations show a live Pay Now, and checkout accepts it → double collection

src/app/api/public/events/[slug]/checkout/route.ts:90 · src/app/e/[slug]/my-registration/page.tsx:419-422

The codebase's own NO_PAYMENT_DUE_STATUSES (registration-enums) is PAID / COMPLIMENTARY / INCLUSIVE / REFUNDED — but both the checkout gate and the registrant portal check only the first two: 

// checkout/route.ts:90 — INCLUSIVE and REFUNDED sail through
if (registration.paymentStatus === "PAID" || registration.paymentStatus === "COMPLIMENTARY") {
  ... return NextResponse.json({ error: "Payment already completed" }, { status: 400 });

// my-registration/page.tsx — same two-status check drives the UI
const isPaid = reg.paymentStatus === "PAID";
const isComplimentary = reg.paymentStatus === "COMPLIMENTARY" || netPrice === 0;
const showPayment = !isPaid && !isComplimentary && reg.status !== "CANCELLED";

Failure scenario

Admin marks a $500 registration INCLUSIVE (sponsor paid a lump sum offline). The attendee opens /my-registration and sees an amber "Payment Required — Total Due $500" card with a working Pay Now button. One click → Stripe charges them → the webhook overwrites INCLUSIVE → PAID, destroying the sponsor attribution semantics and collecting the money twice (sponsor + attendee). No race, no attacker — this fires in normal operation the moment a sponsor-paid attendee logs in. REFUNDED registrations get the same live Pay Now (the re-pay's stranded-refundedAmount aftermath is deferred L1; the INCLUSIVE double collection is net-new). Fix shape: gate both surfaces on NO_PAYMENT_DUE_STATUSES.

High   H1 — Webhook idempotency is registration-level, not charge-level: a second real Stripe settlement is silently dropped from the books

src/app/api/webhooks/stripe/route.ts:68-72, 123-129, 179-217 — found independently by two review passes

The "idempotency" check answers "is this registration paid?", not "was this checkout session already processed?": 

// Idempotency: skip if already paid
if (registration.paymentStatus === "PAID") {
  apiLogger.info({ msg: "Stripe webhook: registration already paid, skipping", registrationId });
  return NextResponse.json({ received: true });
}
...
// in-tx recheck — same registration-level key
const current = await tx.registration.findUnique({ ... select: { paymentStatus: true } });
if (current?.paymentStatus === "PAID") return;

Failure scenario

Two ways a real charge vanishes: (a) two checkout sessions for the same registration (both legally created while UNPAID — the checkout route permits this; different payment intents, so stripePaymentId @unique doesn't apply). The user pays both; webhook 1 sets PAID + Payment row; webhook 2 hits the skip → the second settled charge gets no Payment row. (b) desk records a cash payment (the payments route wins its paymentStatus != PAID claim) while the registrant's Stripe checkout completes → the card charge is unrecorded. In both cases the money exists in Stripe but is invisible to paidTotal, refund caps, and finance exports — and a later charge.refunded for the orphaned intent hits payment.findUnique → not found → warn + 200, so it can't even be reconciled by refunding it. Extra wrinkle: when the in-tx recheck short-circuits (not the pre-check), execution falls through to the post-commit block, which still logs "Payment completed", notifies admins "Payment Received", and re-runs issuePaidRegistrationDocuments — whose sendPaymentDocumentsEmail is called unconditionally, so the attendee gets a duplicate documents email for money that was never recorded.

High   H2 — Webhook completes payment on a CANCELLED registration; cancel never expires the open Stripe session

src/app/api/webhooks/stripe/route.ts:53-152 · src/services/payment-service.ts:417-489

checkout.session.completed never inspects registration.status before flipping paymentStatus to PAID. The checkout route excludes CANCELLED only at session-create time, and cancelRegistration neither expires open Stripe sessions nor guards against later payment.

Failure scenario

Registrant clicks Pay Now (session created, paymentStatus → PENDING). Admin cancels the registration (PENDING ≠ PAID, so no refund step; seat + promo released). Registrant completes payment in the still-open Stripe tab (sessions live ~24h) → the row becomes status=CANCELLED, paymentStatus=PAID, a PAID invoice + receipt are minted and emailed, and money is collected for a seat that was already released — with nothing distinguishing it from a normal payment. The detail GET's CANCELLED handling (balanceDue: 0) further masks the anomaly. Fix shape: webhook refuses (or flags + notifies) when status === "CANCELLED"; cancel expires any open checkout session for the registration.

High   H3 — Checkout unconditionally flips paymentStatus → PENDING after the slow Stripe call: can clobber PAID → PENDING → UNPAID and re-open the pay path

src/app/api/public/events/[slug]/checkout/route.ts:167-171

// Update payment status to PENDING
await db.registration.update({
  where: { id: registrationId },
  data: { paymentStatus: "PENDING" },
});

Failure scenario

The PAID guard runs on a read taken before the network-bound stripe.checkout.sessions.create call. User has two tabs (or phone + laptop): tab A's session is paid and the webhook commits PAID + Payment row; tab B's request read the row as UNPAID before that commit, passes the guard, spends ~500ms creating session 2, then unconditionally writes PENDING — demoting a PAID registration. 24h later, checkout.session.expired for session 2 runs updateMany({ where: { paymentStatus: "PENDING" }, data: { paymentStatus: "UNPAID" } }) — a fully-paid registration with a PAID Payment row now shows UNPAID, gets payment-reminder chased, and can be charged a second time (the PAID guard now passes). Fix shape is the codebase's own pattern: a conditional claim — updateMany({ where: { id, paymentStatus: { notIn: ["PAID", "COMPLIMENTARY", ...] } } }).

Medium   M8 — Webhook flips PAID without reconciling amount_total against the amount currently owed

src/app/api/webhooks/stripe/route.ts:69-152

Checkout sessions live ~24h at a frozen price. If the registration is repriced in that window (promo removed, bulk type change re-stamping originalPrice, tier applied), the registrant pays the stale, cheaper session and the registration goes PAID for less than owed — with no warn log of the delta. Signature verification and metadata re-validation are correct; this is purely the missing amount comparison. A log-and-flag (not reject) of session.amount_total vs the current computed total closes the silent-underpayment window.

4. Refunding money — payment-service & charge.refunded

High   H4 — Refund is booked in the DB before Stripe moves money; a crash in between records a refund that never happened, with no recovery path

src/services/payment-service.ts:284-309

The optimistic-lock write commits refundedAmount = refundedAfter (and possibly paymentStatus: "REFUNDED") as a standalone statement, and only then calls Stripe: 

const locked = await db.registration.updateMany({
  where: { id: registrationId, paymentStatus: "PAID", refundedAmount: registration.refundedAmount },
  data: { refundedAmount: refundedAfter, ...(isFull ? { paymentStatus: "REFUNDED" as const } : {}) },
});
...
const refund = await stripe.refunds.create(
  { payment_intent: payment!.stripePaymentId!, amount: toStripeAmount(amount, currency) },

Failure scenario

updateMany commits → the process dies (deploy restart, OOM, any crash before the Stripe call returns) → Stripe never refunded, but the registration shows refundedAmount = X (or fully REFUNDED). Nothing recovers this: charge.refunded never fires (no Stripe refund exists), the reconciliation worker only covers invoices, and a full "refund" leaves paymentStatus = REFUNDED so a retry hits NOT_PAID. The attendee never gets their money, yet the books say they did. The Stripe-error rollback only covers the case where stripe.refunds.create returns an error — not a process death. This is the mirror image of deferred M3 and is not covered by it. Fix shape: persist a refund-intent record (per-attempt nonce) before the Stripe call and reconcile/sweep incomplete intents — which also fixes H5's key collision.

High   H5 — "Stripe succeeded but the SDK threw" + a fast charge.refunded webhook: the rollback erases a real refund permanently, and a retry can over-refund; the idempotency key can also wedge refunds for 24h

src/services/payment-service.ts:308, 311-316 · src/app/api/webhooks/stripe/route.ts:294-297

// route rollback — unconditional
} catch (stripeErr) {
  await db.registration.update({
    where: { id: registrationId },
    data: { refundedAmount: registration.refundedAmount, paymentStatus: "PAID" },
  })

// webhook delta claim
if (delta <= 0) {
  apiLogger.info({ msg: "charge.refunded: already reconciled, skipping", ... });
  return NextResponse.json({ received: true });
}

Failure scenario

Interleaving: route bumps refundedAmount 0→50 → stripe.refunds.create times out client-side but the refund WAS created at Stripe → Stripe fires charge.refunded (cumulative 50) → webhook reads refundedAmount = 50, delta = 0 → skips and 200s (Stripe never redelivers) → the route's catch then rolls refundedAmount back to 0 and paymentStatus to PAID. Final state: Stripe returned $50 to the attendee; the books say $0 refunded. The deferred M3 fix (conditional rollback) does not close this ordering — the webhook skipped without writing, so the rollback condition still matches. Follow-on: the organizer, seeing STRIPE_FAILED, retries with a different partial ($30) → new idempotency key → Stripe creates a second real refund → $80 returned on a $100 charge.

Key wedge (same root): the idempotency key refund-{paymentId}-{refundedAfter} encodes the cumulative total, but a rollback lets the same cumulative be re-reached by a different amount: refund $50 fails (key -50.00 stored at Stripe with the error) → rollback to 0 → refund $30 (ok) → refund $20 → refundedAfter = 50.00 → same key, different amount param → Stripe idempotency_error → every retry rebuilds the identical key; the last $20 is unrefundable via the product for ~24h. A per-attempt nonce persisted before the Stripe call (H4's fix) resolves both.

High   H6 — refundRegistration operates on payments[0] only: on a mixed Stripe + manual registration, every refund path is wrong (precise scoping of the queued "newest-payment-only" item)

src/services/payment-service.ts:237-238, 296-329 — the only refund site; it backs both /refund and /cancel { refund: true }

const payment = registration.payments[0];        // newest PAID payment
const isManualRefund = !payment?.stripePaymentId;
...
const refund = await stripe.refunds.create(
  { payment_intent: payment!.stripePaymentId!, amount: toStripeAmount(amount, currency) }, ...

Failure scenario

paidTotal correctly sums ALL PAID payments, but the refund executes against exactly one. Registration with Stripe $100 + manual $50 (paidTotal 150):

  • Manual payment newest: isManualRefund = true → a full $150 refund is merely "recorded" — the $100 Stripe charge is never reversed in Stripe. Via cancel-with-refund this is one click ("Cancel & refund 150"): the reg is cancelled, the response says manual: true, and the buried Stripe charge is on the admin to notice.
  • Stripe payment newest: the default full refund calls Stripe for $150 against a $100 payment intent → Stripe rejects → STRIPE_FAILED (502) → cancel-with-refund is permanently blocked. Even after a partial $100 succeeds, the Payment row stays PAID (only flipped on isFull of the registration), so the remaining $50 still routes to the exhausted intent and fails forever — the manual $50 is unrefundable through the product.

Scope-limited to multi-payment registrations (recovery captures, deposit + balance, desk top-ups) — but those are exactly the messy cases refunds get requested on. Fix shape: allocate the requested amount across payments (per-payment Stripe refunds.create capped at each charge, manual remainder recorded).

Medium   M4 — charge.refunded reconciles Stripe's Stripe-only cumulative against the mixed refundedAmount counter

src/app/api/webhooks/stripe/route.ts:276, 287-297

Registration.refundedAmount accumulates manual refunds too, but the webhook treats it as Stripe's cumulative. Reg with Stripe $100 + manual $50; a $50 manual refund is recorded (refundedAmount = 50); later the $100 Stripe charge is refunded in the Dashboard → webhook computes delta = 100 − 50 = 50, sets refundedAmount = 100 (truth: 150 refunded), leaves the reg PAID with "$50 remaining", and auto-mints a $50 credit note while $100 actually moved. A manual refund ≥ the Dashboard refund makes delta ≤ 0 → the Stripe refund is entirely unrecorded. Same root as H6 (one counter, two money sources); distinct site and failure. Durable fix: track Stripe-refunded per-payment (or a separate Stripe-cumulative column) rather than reconciling into the mixed total.

Medium   M3 — charge.refunded arriving before its Payment row exists is ACKed with 200 and permanently lost

src/app/api/webhooks/stripe/route.ts:267-270

if (!payment) {
  apiLogger.warn({ msg: "charge.refunded: no Payment record found", paymentIntentId });
  return NextResponse.json({ received: true });
}

Stripe does not guarantee event ordering. An operator refunds from the Dashboard moments after payment while checkout.session.completed is still mid-flight (it makes two synchronous Stripe API reads before its DB tx) → charge.refunded processes first → no Payment row → 200 → never redelivered. The reg stays PAID with refundedAmount = 0 forever; no sweep reconciles refunds. Returning 500 (→ Stripe retries) when the intent is unknown — or a refund-reconciliation sweep mirroring the invoice one — closes it. This same 200-on-not-found is what makes H1's orphaned second charge doubly unrecoverable.

Medium   M2 — cancelRegistration applies the seat/promo transition from a snapshot taken before the slow CN + refund phase: a concurrent type/tier change releases the wrong counter

src/services/payment-service.ts:421-434 → 475-489

The cancel transaction's claim (status: { not: "CANCELLED" }) only defends against a concurrent cancel. The prev passed to applyRegistrationTransition — ticketTypeId, pricingTierId, promoCodeId — comes from a read taken before the multi-second credit-note + Stripe refund phase. A detail-sheet PUT that changes the type from A to B during that window (releasing A, claiming B) is then followed by the cancel tx releasing A again from the stale snapshot (the soldCount ≥ 1 guard masks the double-release when A has other sales, silently under-counting) while B — the counter the row actually holds — is never released → B phantom-sells-out. Same staleness applies to promoCodeId. Fix: re-read (or FOR UPDATE) inside the transaction, as createCreditNote already does.

High   H11 — Out-of-band (Stripe Dashboard) refunds bypass the payment-service side-effect contract: no audit row, no admin notification — but they DO email the attendee, the exact opposite of the route-initiated policy

src/app/api/webhooks/stripe/route.ts:299-344

const { invoice: cn } = await createCreditNote({ ... });
await sendInvoiceEmail(cn.id);   // attendee gets an email

A refund executed from the Stripe Dashboard leaves zero AuditLog trail (there is no db.auditLog call anywhere in the webhook file) and no in-app admin notification — while refundRegistration writes REFUND_ISSUED and notifies admins. And the communication policy is inverted: payment-service explicitly documents "no automatic refund-confirmation email — the organizer communicates the refund manually", yet the webhook unconditionally emails the credit note. Whether the attendee hears about a refund depends on which button the organizer clicked. The webhook also calls raw createCreditNote instead of issueCreditNoteForRegistration, skipping the CREDIT_NOTE_ISSUED audit row. This is cross-caller drift in the one domain the services layer was built to own.

5. Documents of record — invoices, receipts, credit notes

High   H9 — Cross-org / cross-event IDOR on manual invoice creation: the body's registrationId is never bound to the event or org

src/app/api/events/[eventId]/invoices/route.ts:105 → src/lib/invoice-service.ts:140

// route — event is org-checked, but the registration id is passed through unbound
const invoice = await createInvoice({

Failure scenario

An ORGANIZER of org A posts a registrationId belonging to org B (or a different event in the same org). An invoice is minted under org A / event A's numbering, priced from org B's registration, and org B's registrant's name, email, and billing render in org A's invoice list, detail, and PDF. Also a within-org integrity bug: the invoice's eventId disagrees with registration.eventId. Notably the MCP path already does this right (tools/invoices.ts binds event: { organizationId } and derives eventId from the registration) — the REST route is the outlier. Single-org today makes the cross-tenant leg latent, but this is exactly the class the multi-tenancy impact doc warns about.

Medium   M1 — Post-payment document minting is check-then-act with no row lock: webhook / reconciliation-worker / manual paths can double-mint PAID invoices + receipts and double-email

src/lib/invoice-service.ts:238-259 (invoice check in-tx but unlocked) · :335-345 (receipt check entirely outside its tx) · payments/route.ts:268-271 (non-locking recovery count)

Unlike createCreditNote (which correctly takes SELECT … FOR UPDATE), there is no serialization and no @@unique([registrationId, type]). The webhook's detached post-commit block is slowed by two charges.retrieve round-trips; the reg is already PAID-with-Payment-and-no-invoice, so the reconciliation worker's next tick picks it up — both call the documents fan-out → two PAID invoices with distinct sequence numbers, two receipts, two "payment received" emails. Same race between manual capture and the worker. Related: the manual route's recovery branch guards with a non-locking tx.payment.count — two concurrent recovery clicks can both insert Payment rows, doubling paidTotal and therefore the refund/CN caps. Fix shape: the same row-lock pattern createCreditNote uses, or a partial unique index on (registrationId, type) for INVOICE/RECEIPT.

Medium   M5 — PAID invoice/receipt totals come from current pricing, never from the Payment actually captured; the promote-in-place path keeps fully stale amounts

src/lib/invoice-service.ts:238-259 (promote) · :224, 265-288 (mint) · charge source webhooks/stripe/route.ts:75-77

Two divergence windows: (a) admin pre-creates a SENT invoice at $500; a promo (−$100) is applied afterwards; the attendee pays $400 (checkout uses live pricing) → the invoice is promoted to PAID still stating $500 — the emailed PAID document overstates the charge. (b) the reverse: a session is created at price X, the reg is repriced before payment; the attendee pays X (session.amount_total) but the freshly minted PAID invoice's totals come from calcInvoicePricing on the new registration state — Payment.amount ≠ Invoice.total. Downstream, CN caps read the computed total and refund caps read the payments sum, so the mismatch propagates into both ceilings (the cap split itself is deferred M1 of the earlier review; the false PAID document is the finding here). Fix shape: PAID documents should be minted from — or at least reconciled against — the captured Payment.amount.

Medium   M10 — PDFs recompute tax and grand total from taxRate instead of printing the stored, reconciled taxAmount/total, defeating the partial-CN reconciliation

src/lib/pdf/document-layout.ts:429-433 (via credit-note-pdf.ts and invoice-pdf.ts) · reconciliation at invoice-service.ts:514-517

// createCreditNote deliberately reconciles the last component:

// drawTotals throws the reconciled figure away and re-derives:
const taxAmount = input.taxRate ? discountedSubtotal * (input.taxRate / 100) : 0;
const grandTotal = discountedSubtotal + taxAmount;

For a partial credit note with proportionally rounded components, the printed VAT line and "TOTAL CREDIT" can drift a cent from invoice.total — the number in the refund cap, the audit row, and the dialog the organizer confirmed. A financial document should print the frozen row, not re-run pricing. Related nit: none of the CN/refund math consults isZeroDecimalCurrency, so a JPY credit note can carry fractional yen components.

Medium   M9 — calcInvoicePricing is a fourth parallel totals implementation, unrounded, and there are three divergent round2 copies

src/lib/invoice-service.ts:112-128, 426 · src/lib/registration-financials.ts:54, 121-123 · src/services/payment-service.ts:20 · webhooks/stripe/route.ts:272

// invoice-service — no rounding at any step
const taxAmount = taxRate ? discountedPrice * (taxRate / 100) : 0;
const total = discountedPrice + taxAmount;

vs registration-financials.ts which rounds each step with Number.EPSILON — and the three local round2 copies differ on the EPSILON (so 1.005 rounds differently by call site). Concrete cost: createCreditNote's cap uses calcInvoicePricing's unrounded total while refundRegistration's remaining uses computeRegistrationFinancials().total — two "same" totals that can differ at the cent boundary, producing spurious CREDIT_LIMIT_EXCEEDED / INVALID_AMOUNT / "0.01 remaining" artifacts on tax-bearing amounts. This is the exact class of the July-7 payment-reminder bug. Fix: one totals function, one exported round2.

Low   L1 — Public document route serves the wrong invoice: orderBy: { status: "asc" } sorts by enum definition order, so DRAFT/SENT beat PAID — the opposite of the comment's claim

src/app/api/public/events/[slug]/registrations/[registrationId]/document/route.ts:105-111

// Prefer the post-payment Invoice when one exists. Ordered so a
// PAID row wins over a pending SENT row; ...

InvoiceStatus is declared DRAFT, SENT, PAID, OVERDUE, CANCELLED, REFUNDED; Postgres sorts enums by definition order, so asc ranks DRAFT and SENT before PAID. 6. The desk path — manual payment capture

src/app/api/events/[eventId]/registrations/[registrationId]/payments/route.ts:46-47, 210-211, 253-262, 354-372 — found independently by two review passes

Failure scenario

Total due $105; desk records an $80 bank transfer (partial captures are an intended flow — the detail GET's own comment says "a partial bank-transfer capture correctly leaves a balance"). Medium   M11 — issuePaidRegistrationDocuments's docblock claims it is "the single post-payment fan-out used by ALL payment channels" — but manual capture hand-mirrors the fan-out with different semantics

Fix: route manual capture through the helper; correct the docblock either way.

All of these gate with denyFinance — which passes ONSITE and MEMBER (both are in FINANCE_ROLES since June 17) — then scope by org only. An ONSITE temp assigned only to Event A can list, export, and download invoices and quote PDFs for any event in the org: full amounts, attendee PII, billing addresses, tax numbers, and the org's bank details. This is byte-for-byte the July 7 adversarial-review BLOCKER class ("every ONSITE-reachable route must route its event lookup through buildEventAccessWhere") — that fix covered the five desk routes but none of the finance GETs. The money writes in this domain are correct (payments uses buildEventAccessWhere; refund/CN/cancel/promo block ONSITE via denyReviewer).

High   H12 — REST PUT and MCP bulk_update_registration_status can flip paymentStatus: REFUNDED as a bare flag — no CN gate, no refundedAmount, no Payment flip — and the UI enums file claims the server excludes it

registrations/[registrationId]/route.ts:34 (z.nativeEnum(PaymentStatus)) · src/lib/agent/register-mcp-tools.ts:508-512 · claim at registration-enums.ts:58

The single-update MCP tool at least warns ("only flips the DB flag"); the bulk tool's description does not, and REST PUT accepts the full enum while the UI file asserts "Stripe-driven states (PENDING/REFUNDED/FAILED) are excluded" — a UI-only claim, unenforced server-side. An agent instruction like "mark these refunded" produces up to 200 regs labeled REFUNDED with refundedAmount = 0, Payment rows still PAID (so paidTotal still sums), no credit note, no REFUND_ISSUED audit — books that say money was returned when none moved, invisible to the gated-refund flow's "Refunded X of Y" display. Fix shape: reject (or route through the gated flow) REFUNDED/PENDING/FAILED on the admin-settable write paths, matching the documented subset.

Medium   M6 — MCP update_invoice_status allows arbitrary money-state flips with no state machine, no registration sync, and an undocumented PAID hazard

src/lib/agent/tools/invoices.ts:106-164 vs REST invoices/[invoiceId]/route.ts:56

const data: Prisma.InvoiceUpdateInput = { status: status as never };
if (status === "PAID") data.paidDate = new Date();

Any org API key or OAuth token can flip CANCELLED → PAID, PAID → DRAFT, or stamp an invoice PAID (with paidDate) against a registration still UNPAID — no Payment row, no receipt, off the promote-in-place path. The REST equivalent is deliberately narrower (only cancel / mark_overdue), so the MCP tool is a strict guard bypass relative to the dashboard; its description warns only about REFUNDED. Either warn for PAID too, or make PAID/backwards transitions refuse.

Medium   M7 — Refund and cancel have no rate limit, and refund omits denyFinance (guard asymmetry across the three sibling money writes)

refund/route.ts:42-43 · cancel/route.ts · contrast credit-notes/route.ts:54

Credit-notes has checkRateLimit (60/hr); refund and cancel — the two endpoints that actually move money through Stripe — have none, so a compromised organizer session can fire unbounded stripe.refunds.create calls. And the guard stack differs for no reason: refund runs only denyReviewer while credit-notes and cancel add denyFinance. Safe today by coincidence (denyReviewer blocks every non-finance role), but one { allow: … } refactor away from opening Stripe refunds to desk staff, and the inconsistency defeats the "denyFinance guards every money route" invariant reviewers grep for.

8. Failure visibility & documentation truth

Medium   M12 — Silent 4xx across the money surface: five payment-service rejection codes (including LOST_LOCK) and the routes' 401/404s log nothing; the auth guards never log a rejection

payment-service.ts:214, 217, 264, 292, 436, 439 · refund/route.ts:39, 57, 70-75 · credit-notes/route.ts:47, 73 · cancel/route.ts:41, 58 · auth-guards.ts:56-58, 74-81

REGISTRATION_NOT_FOUND, NOT_PAID, ALREADY_FULLY_REFUNDED, LOST_LOCK (the 409 you'd most want in /logs — "a refund is already in progress"), and ALREADY_CANCELLED return as error-values with no log in the service, and all three routes map !result.ok straight to HTTP with no apiLogger call. denyReviewer/denyFinance return bare 403s with no log, so a probing attacker enumerating money endpoints with a restricted role is invisible. The payments route is the domain's own gold standard (every branch logs) — the rest should match. Cheapest durable fix: log inside the guards themselves + one logging pass at the three route boundaries.

9. Suggested remediation phases

Grouped so each phase is one commit, independently shippable, testable, and small enough to review. Ordering favors "stops charging/refunding real money wrongly" first.
