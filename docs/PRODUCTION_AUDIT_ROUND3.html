The refund double-count (payment-service.ts:549-558) was found independently by the money and concurrency lenses and confirmed against source. It corrupts the money books in the normal path (no crash needed) and can permanently wedge a partially-refunded registration in REFUNDED, blocking the remaining refund forever. Verdict: the new July-2026 surface is well-built — every nested-id route is org-bound, the six deliberately-distinct visibility predicates are applied correctly, and no new XSS/SSRF/CSV-injection/enumeration was introduced (security lens: 0 blocker / 0 high / 0 med / 1 low). High R3-H1 — Route-initiated refunds double-count Registration.refundedAmount (normal path) and lose the refund entirely on a crash money + concurrency · VERIFIED at source

src/services/payment-service.ts:373-409, 467-558 · src/app/api/webhooks/stripe/route.ts:420-462 · src/lib/refund-reconciliation.ts:112-121

The pre-Stripe claim transaction books only Registration.refundedAmount + the RefundAttempt rows. The per-payment counter — the very field the webhook uses to dedup — is written after the Stripe call, outside that transaction, as an absolute SET from a start-of-call snapshot: 

// payment-service.ts:549-558 — written AFTER stripe.refunds.create, outside the claim tx
const paymentRefundedAfter = round2(Number(slice.payment.refundedAmount ?? 0) + slice.take);
await db.payment.update({
  where: { id: slice.payment.id },
  data: { refundedAmount: paymentRefundedAfter, ... },   // absolute SET from a stale read
});

The charge.refunded webhook's entire dedup for route-initiated refunds is that counter's delta, and it never inspects refund.metadata.refundAttemptId (which the service does set at line 478):

// stripe/route.ts:420-462
const delta = round2(cumulativeRefunded - Number(payment.refundedAmount));
if (delta <= 0) { /* "already accounted for" */ return; }
...
await db.registration.update({
  where: { id: payment.registrationId },
  data: { refundedAmount: { increment: delta } },   // NOT in a tx with the claim
});

Failure scenario A — normal path, no crash

Partial refund of 250 on a 500 payment. Claim tx books Registration.refundedAmount 0→250 without touching Payment. stripe.refunds.create succeeds and Stripe dispatches charge.refunded (cumulative 250) within the same second — routinely before the service reaches line 550. The webhook reads Payment.refundedAmount = 0 → delta = 250 → increments the registration 250→500 → isFull → flips the reg REFUNDED → mints a spurious 250 credit note. A half-refunded registration is now permanently REFUNDED; refundRegistration refuses NOT_PAID, so the real remaining 250 can never be refunded through the app.

Failure scenario B — crash window becomes permanent

Process dies between the Stripe call (line 473) and the counter write (line 550). The reconciliation sweep confirms the RefundAttempt SUCCEEDED but never bumps Payment.refundedAmount (refund-reconciliation.ts:112-121 touches only the attempt row). Stripe's retried charge.refunded (retries for days) is then guaranteed to double-count. For a full Dashboard refund with the reg roll-up crashing on a pooler blip, the opposite happens: Registration.refundedAmount = 0, PAID, no credit note — the books say the attendee still holds a paid registration while Stripe returned the money.

Related interleaving (same root cause — concurrent Dashboard refund)

Because line 550 is an absolute SET from a stale snapshot, a concurrent Dashboard refund that advances the Payment counter via the webhook is silently clobbered → the next charge.refunded re-applies its delta and mints a second phantom credit note.

Fix
Pre-book each slice's Payment.refundedAmount (guarded increment) inside the pre-Stripe claim transaction that books the registration counter (and un-book it in abortRemainder/the sweep rollback), so the webhook's delta ≤ 0 invariant holds before Stripe can dispatch. Make line 550 a guarded increment, not a snapshot SET. Additionally, have the webhook skip refunds whose metadata.refundAttemptId matches a known RefundAttempt.

High R3-H2 — A partially-captured registrant who clicks "Pay Now" is silently charged the full total again money

src/app/api/public/events/[slug]/checkout/route.ts:90-118 · src/lib/payment-reminder.ts:42 · src/app/e/[slug]/my-registration/page.tsx:425 · src/app/api/public/events/[slug]/payment-status/[registrationId]/route.ts:65

The July-10 H7 remediation deliberately keeps a partially-captured registration UNPAID so the balance stays chaseable — but no public surface nets the captured payments, and checkout charges the full computed total (totalPaid is hardcoded 0 in the reminder and the portal). The dashboard detail sheet does net PAID payments, so the organizer sees "balance due $200" while the attendee's reminder email, portal, and Stripe charge all say $500.

Scenario

$500 total; desk records a $300 bank-transfer deposit (reg stays UNPAID). A payment-reminder fires: "Amount due: USD 500.00 — Pay Now". Attendee pays; checkout creates a $500 session; the webhook records a second $500 Payment and flips PAID. Attendee over-pays $300. It is fully silent — the webhook divergence guard computes owedNow with totalPaid: 0 too, so no warn fires, and the duplicate-charge branch doesn't fire (the reg wasn't PAID). $800 collected for a $500 registration, zero alerts.

Fix
Thread Σ settled same-currency payments into the checkout's charged amount (total − priorCaptured, floor 0, reject if ≤ 0) and into the reminder / portal / payment-status totalPaid — or block self-service checkout when any settled Payment row exists (PARTIAL_PAYMENT_ON_FILE → contact organizer).

MEDIUM

Money integrity

Med R3-M1 — Manual capture counts refunded money as "captured" → a small deposit after a full refund flips PAID and emails false full-total documents

src/app/api/events/[eventId]/registrations/[registrationId]/payments/route.ts:178-182, 261-267

The capture route sums status IN (PAID, REFUNDED) without netting refundedAmount to answer "does money on hand cover what's owed" — but refunded money is no longer on hand. A $500 reg fully refunded then given a $100 desk deposit computes priorCaptured = 500, capturedAfter = 600 ≥ 500 → flips PAID and emails a PAID invoice/receipt asserting $500 settled with only $100 net held. Books overstate collections by $400 — the exact false-documents harm H7 was shipped to prevent.

Fix
Net per-payment: priorCaptured = Σ (amount − refundedAmount) (select refundedAmount), or exclude fully-REFUNDED rows and subtract partial refunds.

Med R3-M2 — Detail-sheet financials counts PAID-only while the whole money domain counts PAID+REFUNDED → a partially-refunded reg shows a phantom balance due

src/app/api/events/[eventId]/registrations/[registrationId]/route.ts:223-247

$500 paid as $300 Stripe + $200 desk; organizer refunds the $300 Stripe charge (that Payment → REFUNDED, reg stays PAID). The detail sheet computes totalPaid = 200 → the Billing tab reads "Refunded 300 of 200 · Outstanding 300" on a registration that owes nothing. The refund dialog's client cap credited − refunded goes negative; an operator trusting "Outstanding 300" chases or captures money the attendee doesn't owe (and R3-H2 means the portal would agree). The July-16 "one collected-truth" unification missed this reader.

Fix
Compute paidTotal from PAID+REFUNDED (the service rule) and balanceDue = total − paidTotal — a refund is not an un-payment.

Med R3-M3 — paidTotal/collectedTotal sum Payment.amount across mixed currencies

src/services/payment-service.ts:270-272, 726 · src/lib/invoice-service.ts:605-611

The manual-capture route explicitly permits foreign-currency captures and stores them — but the refund service, the July-16 collected-truth in createCreditNote, and cancel's settledSum all sum those rows unit-blind. A USD-500 event with a USD-500 Stripe payment + an AED-500 desk capture reads paidTotal = 1000; a "USD 1000" refund books 500 of AED money as USD, and the credit note document states a 1000 "collected total" that never existed in any single currency.

Fix
Filter settled sums to the registration's resolved currency (the manual route's exact rule); surface foreign-currency rows as not-refundable-via-this-flow.

Med R3-M4 — Partial manual captures have no duplicate guard at all — a double-click records the cash twice

src/app/api/events/[eventId]/registrations/[registrationId]/payments/route.ts:312-361

The 409 duplicate guard only fires when paymentStatus === "PAID" && payments > 0. The partial branch (!wasAlreadyPaid && !coversTotal) falls through with no lock, no claim, no dedup check. Two clicks of "Record $300 partial" both insert → $600 captured for $300 handed over.

Fix
Run the partial branch under the same FOR UPDATE lock with a short-window same-amount/method dedup, or require an idempotency token from the dialog.
