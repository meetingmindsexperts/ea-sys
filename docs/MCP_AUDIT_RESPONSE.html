Still open — critical Open

These are real gaps the audit correctly identified and today's expansion didn't address. All four should be closed in the next sprint.

1. update_abstract_status fails silently

Audit finding: Generic "Failed to update abstract status" error with no root cause exposed. Confirmed as still an issue in commit 3789256.

Probable cause: The error handler at event-tools.ts:1395 swallows the real Prisma error. Most likely triggers are (a) abstract already in a terminal state (ACCEPTED/REJECTED/WITHDRAWN), (b) invalid transition from DRAFT, or (c) the notifyAbstractStatusChange() helper throwing on missing speaker email.

Fix shape: catch the specific error, surface a structured response:

{
  "error": "Cannot update abstract status",
  "code": "INVALID_TRANSITION" | "ABSTRACT_TERMINAL" | "NOTIFICATION_FAILED",
  "message": "...human-readable cause...",
  "suggestion": "...actionable next step..."
}

Effort: 1–2 hours (trace actual Prisma error path, add branch, retest).

2. No accommodation CREATE via MCP

Audit finding: Only list_accommodations + list_hotels + create_hotel exist. Cannot book a room via Claude. The dashboard has an "Assign Room" dialog that POSTs to /api/events/[id]/accommodations — just needs MCP wrapping.

Fix shape: three new tools.

  • create_accommodation({ registrationId | speakerId, roomTypeId, checkIn, checkOut, guestCount?, specialRequests? })
  • update_accommodation_status({ accommodationId, status }) — PENDING → CONFIRMED → CHECKED_IN → CHECKED_OUT, or CANCELLED
  • list_room_types({ hotelId }) — to populate the UI for the above

Effort: 3–4 hours. The Prisma model + REST route already handle atomic bookedRooms counter adjustments; executors just need to replicate that via the existing helper.

3. No invoice CREATE / SEND via MCP

Audit finding: Only list_invoices exists. Cannot generate or dispatch invoices via Claude.

Available plumbing: src/lib/invoice-service.ts already builds PDFs (via pdfkit) and emails them. src/lib/quote-pdf.ts generates quote PDFs. The REST route at /api/events/[id]/invoices handles CRUD.

Fix shape:

  • create_invoice({ registrationIds[], invoiceType, dueDate?, notes? })
  • send_invoice({ invoiceId }) — emails the PDF
  • update_invoice_status({ invoiceId, status }) — DRAFT → SENT → PAID → OVERDUE → REFUNDED → CANCELLED

Effort: 4–5 hours. Heaviest lift is the create_invoice tool because of the invoice-numbering + line-item expansion logic. The rest are thin wrappers.

Note on Stripe refunds: The audit recommended linking invoices to payments with refund handling. We deliberately keep actual money movement dashboard-only. MCP tools can set paymentStatus: REFUNDED for book-keeping (documented in tool descriptions) but do NOT call Stripe. This is a safety rail, not a gap.

4. create_session missing time-range validation

Audit finding: No validation that endTime > startTime or that times fall within the event's date range. Confirmed still missing. update_session (added today) DOES validate — create_session was missed.

Fix: copy the validation block from update_session into createSession at event-tools.ts:771.

Effort: 30 minutes.

Still open — high priority High

5. No reviewer assignment / abstract scoring workflow

Audit correctly noted this. The MCP surface has create_review_criterion and list_reviewers but no way to assign a reviewer to a specific abstract, record their scores against criteria, or aggregate scores for a decision.

This is not just a missing MCP tool — the data model itself treats reviewers as event-level (Event.settings.reviewerUserIds JSON array), not per-abstract. To implement proper per-abstract review, we'd need a new AbstractReviewer join table + AbstractScore table (criterionId × reviewerId × abstractId → numeric score). That's a schema decision, not a 2-hour fix.

Effort: 10–15 hours including schema migration, REST route, MCP tools. Separate design discussion warranted.

6. Email template CRUD via MCP

Audit correct. Pre-built templates exist (list_email_templates works), but there's no tool to customize them. The REST route PUT /api/events/[id]/email-templates/[slug] exists — just not MCP-wrapped.

Effort: 3 hours. Straightforward wrap.

7. Bulk creates

Audit correct. Today we have bulk_update_registration_status but no create_speakers_bulk / create_registrations_bulk. Useful when pasting a CSV-like list to Claude and asking "create all these speakers".

Effort: 4 hours.

8. Duplicate detection audit was partly wrong (see below) but tightening wouldn't hurt

Our code DOES catch P2002 unique constraint on (eventId, email). But the error message currently returned doesn't clearly say "a speaker with this email already exists at id X, did you want to update instead?" — just returns "A speaker with email X already exists for this event". Could improve UX by returning the existing id so Claude can pivot to update_speaker automatically.

Effort: 1 hour across create_speaker, create_registration, create_contact.

9. Pagination inconsistency

Audit correct. Some tools accept limit, none support cursor/offset. Not urgent — the hard limit of 200 on most list tools caps memory usage.

Effort: 6–8 hours if done comprehensively. Defer.

What the audit got wrong Inaccuracies

Several findings in the audit don't match the current code. Listing them here so nobody chases fixes for non-problems.

  1. "No duplicate detection on create_speaker" — Wrong. The executor at event-tools.ts:677-688 catches the P2002 unique-constraint error on (eventId, email) and returns "A speaker with email X already exists for this event". The auditor's test ran twice with the same email but may not have read the second response carefully. Same protection exists on create_registration (via Attendee email uniqueness).
  2. eventType enum listed as "WEBINAR | CONFERENCE | SUMMIT | FORUM" — Wrong. Actual schema values are CONFERENCE | WEBINAR | HYBRID. SUMMIT and FORUM do not exist. Auditor inferred from sample event names, not the Prisma enum.
  3. Event status enum listed as "PUBLISHED, DRAFT, CANCELLED (inferred)" — Incomplete. Actual: DRAFT | PUBLISHED | LIVE | COMPLETED | CANCELLED.
  4. "list_contacts cannot search by tag" — Wrong. The tag parameter is supported and works with tags: { has: tag } at mcp-server-builder.ts:78.
  5. "No template variable interpolation in API — must pre-render HTML" for send_bulk_email — Wrong. The bulk email pipeline at src/lib/bulk-email.ts handles {{variable}} interpolation per-recipient using the template from emailType. Customers never pre-render.
  6. "Templates cannot be customized or created via API" — Partially wrong. The REST route PUT /api/events/[id]/email-templates/[slug] exists and works. The audit is right that there's no MCP tool wrapping it, which is legitimate (see item 6 above). But "via API" broadly includes REST, which works.
  7. "No audit logging — cannot track who changed what" — Wrong. The AuditLog Prisma model exists and is written on every mutation. Today's 22 new MCP tools all write with source: "mcp" so agent-initiated changes are distinguishable. Auditor didn't check the AuditLog table.
  8. "No role-based access — everyone has same permissions" — Partially wrong. Org-scoping IS enforced on every tool via organizationId from the API key / OAuth token. A user from org A literally cannot see org B's data. The audit is correctly flagging that within an org, per-role permissions are flat (admin vs organizer vs reviewer get the same MCP surface) — but the cross-org isolation claim is wrong.
  9. "Bulk email targeting too broad (all speakers OR all registrations)" — Partially wrong. send_bulk_email accepts recipientIds: string[] AND filters.status / filters.ticketTypeId. You can target specific people or a status-filtered subset. The tool description in the MCP registration may not be clear enough — that's a UX issue, not a missing feature.
  10. "No validation on stars field (accepts any number)" — Not a bug. Auditor noted stars=7 is accepted. This is intentional: the Burj Al Arab brands itself as a 7-star hotel. The field is a free-text integer by design, not a capped 1-5.

Intentionally deferred Out of scope

These audit recommendations were deliberately left out of today's work or earlier planning, not because they're unimportant but because they trade off badly against alternatives.

  • Hard deletes (delete_speaker, delete_session, delete_registration) — Soft deletes via CANCELLED status are safer via MCP. Claude making a destructive call is a risk we don't want to expose until we have tool-level permissions. Dashboard handles hard delete when needed.
  • Media upload via MCP — File uploads don't fit JSON-RPC transport well (would require base64 blobs inline). Dashboard-only is the right call.
  • Webhooks / real-time events — Valuable but substantial architectural lift (queue infrastructure, retry semantics, secret rotation). Not on the critical path.
  • Per-tool rate limiting — Current 100/hr at the MCP route level suffices. More granular limits can be added when we see actual abuse patterns.
  • Pagination overhaul — Cosmetic given the 200-row hard cap on list tools.
  • Advanced search syntax (date ranges, complex AND/OR) — search_event covers the 80% case. Complex queries are better asked as "use list_registrations with status=X then filter by Y" — Claude handles that naturally.

Answers to the audit's "Questions for Development Team"

  1. Is reviewer assignment intentionally missing, or oversight? — Neither. It's a schema trade-off: reviewers today are event-level (Event.settings.reviewerUserIds), not per-abstract. Pre-existing design that made sense at the product's early stage. The audit correctly flagged that the model needs upgrading — see Sprint B.
  2. Why does update_abstract_status fail? — Generic error wrapper swallows the real Prisma error. See critical item #1 for the fix. Root cause is likely invalid-transition or missing-speaker-email; the current code doesn't tell us which. Fix in Sprint A (A1).
  3. What's the intended accommodation booking flow? — Mixed: dashboard handles the assign-to-hotel-room UI with atomic bookedRooms counters; there's no external hotel-PMS integration. Sprint A3 exposes the same capability to MCP.
  4. Is invoice generation handled by a separate finance system? — No, it's in-app via invoice-service.ts (pdfkit + Brevo/SendGrid). Stripe handles payment capture + webhooks. Refund money-movement is deliberately dashboard-only (not MCP).
  5. Backup/DR strategy? — Out of scope for the MCP audit. Postgres runs on a managed provider with daily snapshots; Docker volumes for uploads. See docs/MUMBAI_SETUP.md for EC2 details.
