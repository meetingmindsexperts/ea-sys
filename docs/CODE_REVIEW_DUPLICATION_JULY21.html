1 · Why this audit exists

The services layer (April 2026) was created because MCP write tools had drifted from the REST routes doing "the same thing" separately — paid MCP registrations silently skipped the confirmation email; soldCount accounting diverged. The rule since then: a domain operation invoked from more than one entry point lives in exactly one place. This audit swept the repo for surviving violations and ranked them by whether the copies had already drifted (different behavior today) or were merely at risk.

3 · What shipped

Batch 1 — 814224aa · seat + promo accounting consolidated (finding 1)

src/lib/registration-seat-db.ts gains four shared helpers; all three hand-rolled call sites now route through them:

  • releasePromoUsage(tx, id, n) — guarded gte: n decrement, with a relative clamp fallback (release what's actually held; never negative, never an absolute set 0 — see review M1 below).
  • claimPromoUsage(tx, id, n) — the H6 re-claim on reactivation; updateMany so a hard-deleted promo row no-ops.
  • claimSeats(tx, counter, n) — guarded all-or-nothing bulk claim (soldCount <= quantity − n); claimSeat delegates to it. The bulk-type route's inline guard replaced by this (quantity now re-read inside the tx — strictly safer).
  • claimSeatsOverselling(tx, counter, n) — the bulk-reactivation allow-and-log policy, extracted from the MCP executor's duplicated tier/ticket branches; returns the oversell facts for the caller to warn-log.

Bug fixes riding the consolidation: the MCP bulk + REST delete promo decrements were unguarded (usedCount could go negative → a maxUses-capped code admits extra redemptions) — now guarded; and MCP bulk reactivation now re-claims promo usage (H6 symmetry — bulk cancel → reactivate → cancel used to double-release). +13 tests.

Batch 2 — 6b66649a · promo-code-service.createPromoCode (finding 2)

One create path (errors-as-values, 6 codes) owning: org-scoped event check, code normalization, discount rules (positive · percentage ≤ 100 · FIXED_AMOUNT requires currency), event-bound ticketTypeIds validation (REST used to silently link foreign ticket types), composite-key dup check + P2002 race mapping, maxUses clamps, and the CREATE_PROMO_CODE audit row tagged with source — the exact row the MCP path never wrote. REST keeps Zod + HTTP mapping (status codes unchanged; 409 body gains code); MCP keeps loose-input coercion + its response shape. +11 tests.

Batch 3 — 858a50cd · abstract-service.assignReviewer / unassignReviewer (finding 3)

The twice-drifted pair is now one implementation: org-scoped lookups, the H6 NOT_REVIEWABLE gate, upsert semantics (role/COI flip without unassign+reassign; idempotent noop), create defaults, ASSIGN/UPDATE/UNASSIGN audit rows (REST update audits now also carry previousRole/previousConflictFlag — MCP parity), and the create-only reviewer notification. Response bodies byte-compatible on both surfaces (incl. the MCP noop's conflictFlag). All 15 pre-existing route/COI tests pass unchanged through the delegation; +10 service tests.

4 · Post-ship adversarial review of the batches

An independent reviewer examined the three commits against pre-commit behavior, consumer code (use-api hooks, promo-codes panel, reviewers card, MCP contracts), and the services conventions. Verdict: 0 BLOCKER / 0 HIGH / 3 MED / 5 LOW.

M1  releasePromoUsage clamp fallback could erase a concurrent redemption  FIXED in-band
src/lib/registration-seat-db.ts · regression vs the single-statement release it replaced on the single-row path

The original clamp was an absolute set usedCount = 0 where usedCount > 0. Between the failed guarded decrement and the clamp, a concurrent public registration could commit a redemption (0→1) — the absolute set then erased it. Fixed: the fallback is now a relative guarded decrement of min(count, current), which leaves any concurrent increment intact while still flooring at 0. Test pins the fallback's write shape.

What to learn: in counter code, an absolute set is a different animal from a relative increment/decrement — only the relative form composes safely with concurrent writers under READ COMMITTED.
M2  REST reviewers 500 body lost its stable code + leaked raw error text  FIXED in-band
reviewers/route.ts + reviewers/[userId]/route.ts · contract/info-exposure regression

The service's UNKNOWN result carried err.message, and the route surfaced it as the top-level error (a Prisma error string in a user toast) without the historical ASSIGN_REVIEWER_FAILED/UNASSIGN_REVIEWER_FAILED code. Fixed: service UNKNOWN messages are static (raw error stays in the log), and both routes map UNKNOWN back to the exact pre-refactor 500 body.

What to learn: when a service absorbs a route's try/catch, the route's outer catch becomes unreachable for domain errors — the error-body contract must move into the result mapping, or it silently changes.
M3  MCP bulk promo re-claim rides the pre-existing read-outside-tx TOCTOU  DEFERRED → ROADMAP
src/lib/agent/tools/registrations.ts (bulk_update_registration_status)

The affected-rows read happens before the transaction and the final updateMany is unconditional on current status, so two concurrent identical bulk reactivations both re-claim promo usage (double increment → false exhaustion of a maxUses-capped code). The seat half of this TOCTOU is pre-existing and policy-accepted (oversell-allowed + logged); the promo half is new surface for the old race. Low likelihood (admin-only, low-concurrency path). Fix when the executor is next touched: condition the row update on status ≠ target or re-read inside the tx.

LOW findings

  • L1 — re-claim on never-released cancellations accepted-by-design: a registration cancelled before release-on-cancel existed re-claims on reactivation without a matching release — exactly the property the single-row H6 path already has; the guarded release self-heals downward on a later cancel.
  • L2 — promo audit-failure log downgraded error→warn FIXED in-band: restored to error per the standing owner rule (never downgrade alert levels without asking).
  • L3 — reviewer audit changes.source value changed "api"→"rest" noted: matches the services convention; nothing queries the old value; historical rows keep "api".
  • L4 — promo-usage consolidation incomplete DEFERRED → ROADMAP: promo-code-service.releaseExistingRedemption still hand-rolls the guarded single decrement (same op as releasePromoUsage(count=1), now with different clamp behavior); the maxUses-gated increments are semantically distinct and stay separate.
  • L5 — REST promo 404 message text changed ("Event not found" → "Event not found or access denied") — toast copy only.

Explicitly verified clean by the reviewer

  • claimSeatsOverselling is byte-equivalent to the old inline executor code, including log-field naming.
  • REST + MCP response contracts on all three surfaces byte-compatible (201/200/409/404 bodies, MCP shapes, the noop message templates).
  • MCP audit attribution works for API-key callers (mcp-remote is a real seeded User row — no FK failure).
  • H6 bulk↔single promo symmetry predicates match applyRegistrationTransition exactly; no double-release path.
  • Services conventions hold: errors-as-values, no next/server imports, service-owned side effects, source identity, every failure path logs.
