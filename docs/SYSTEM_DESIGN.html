🔒 Authentication & RBAC

JWT sessions, 7-role hierarchy, and 3-layer enforcement

Session Strategy

EA-SYS uses NextAuth.js v5 with a JWT strategy (no server-side session store). Tokens are signed with NEXTAUTH_SECRET and expire after 24 hours, forcing daily re-authentication.

ⓘ
Stale-role protection: The JWT callback re-validates the user's role from the database every 5 minutes. If an admin changes a user's role, the change takes effect within 5 minutes — not 24 hours.

CSRF Protection

Middleware validates Origin against Host for all mutations (POST/PUT/PATCH/DELETE). API-key or Bearer-token requests without an Origin header are exempt (non-browser clients).

    { error: "Invalid input", details: parsed.error.flatten() }, { status: 400 }
  );
  // ... business logic ...
}

Independent database calls are always parallelized with Promise.all():

// Good: parallel (1x latency)

  db.event.findFirst(...),

]);

Why Services Now?

The tipping point is external APIs. Today there are 2 caller surfaces (REST + MCP). When 3rd-party integrators arrive that becomes 3, and the drift problem compounds. Services make drift impossible by construction — every caller invokes the same function, so side effects can't silently diverge.

BenefitCost
Fixes drift bugs by constructionExtraction boilerplate (result types, error mapping) per service
Services are unit-testable (plain functions)Route handler shrinks but service file is new — net LOC roughly flat
External API route can reuse the same serviceChurn risk in a fast-moving repo — merge conflicts
Background jobs call the same domain codePre-committing to an abstraction before all callers are known

Locked-in Conventions (Phase 1)

  1. Result type: errors-as-values. { ok: true, <domain-key>, ...derived } | { ok: false, code, message, meta? }. TS compiler forces callers to narrow on result.ok before accessing the payload — forgetting the error branch is a compile error. Unexpected errors (DB crashes, bugs) still throw; known domain errors (validation, not-found, race) are values.
  2. Validation boundary. Services receive already-typed input (checkIn: Date, not string). Each caller parses at its own boundary (Zod on REST, manual on MCP) so validation isn't duplicated.
  3. Audit log. Service owns the write. Callers pass source: "rest" | "mcp" | "api", optionally requestIp. One audit shape regardless of caller.
  4. Directory. src/services/*.ts (top-level, sibling to lib/). Not nested under lib/ — services are a distinct architectural concern, not a utility grab-bag.
  5. Testing. Every service gets a __tests__/services/<name>.test.ts covering every error code + happy path + side-effect fan-out. Callers get thin integration coverage in e2e. Unit tests are the bulk of the coverage since services are pure functions.
  6. Review checkpoint. After each extraction, an independent review-agent verifies parity against the old inline paths with file:line citations. Services that fail review don't merge.

Guardrails

  • One service per PR. Keeps reviews tractable and allows pausing between extractions if something drifts.
  • No single-caller extractions. If a route is only called from one place and has no drift risk, extracting it creates a one-line pass-through — pure noise.
  • Services don't import from next/server. Enforced by ESLint rule to prevent HTTP concepts leaking into domain code.
  • Services don't know about sessions. Callers pass userId, organizationId, and source as plain fields.

function renderTemplate(template, variables, rawHtmlKeys?) {
  return template.replace(/\{\{(\w+)\}\}/g, (_, key) => {
    const value = variables[key];
    if (value === undefined) return `{{${key}}}`;  // Keep placeholder
    if (rawHtmlKeys?.has(key)) return String(value);  // Trust raw HTML
    return escapeHtml(String(value));  // XSS protection
  });
}

PKCE (S256 only)

export function verifyPkce(verifier: string, challenge: string): boolean {
  const expected = crypto
    .createHash("sha256")
    .update(verifier)
    .digest("base64url");
  // Timing-safe compare prevents timing attacks
  return crypto.timingSafeEqual(
    Buffer.from(expected), Buffer.from(challenge)
  );
}

🎥 Zoom Integration

Server-to-Server OAuth, in-memory token cache, encrypted credentials, Component View embed

Token Caching Algorithm

const tokenCache = new Map<string, CachedToken>();
const REFRESH_BUFFER = 5 * 60 * 1000; // Refresh 5 min before expiry

async function getZoomAccessToken(orgId) {
  const cached = tokenCache.get(orgId);
  if (cached && cached.expiresAt > Date.now() + REFRESH_BUFFER) {
    return cached.accessToken;  // Cache HIT
  }
  // Cache MISS → fetch new token from Zoom
  const token = await fetchAccessToken(credentials);
  tokenCache.set(orgId, token);
  return token.accessToken;
}
ⓘ
Why 5-minute buffer? Zoom tokens are valid for 60 minutes. Refreshing 5 minutes early prevents requests from failing with a token that expires mid-flight.

💳 Stripe Payments

Zero-decimal currency handling, idempotent webhooks, tax line items

Zero-Decimal Currency Algorithm

Stripe represents amounts in the smallest currency unit. For most currencies that's cents (1 USD = 100 units), but 16 currencies have no decimal subdivision:

const ZERO_DECIMAL = new Set([
  "bif", "clp", "djf", "gnf", "jpy", "kmf", "krw", "mga",
  "pyg", "rwf", "ugx", "vnd", "vuv", "xaf", "xof", "xpf",
]);

function toStripeAmount(amount, currency) {
  return ZERO_DECIMAL.has(currency)
    ? Math.round(amount)       // ¥500 → 500
    : Math.round(amount * 100); // $5.00 → 500
}

Idempotent Webhook Processing

The checkout.session.completed webhook uses a Prisma interactive transaction to prevent duplicate Payment records. If two webhook deliveries arrive simultaneously, only one succeeds:

await db.$transaction(async (tx) => {
  const existing = await tx.payment.findUnique({
    where: { stripePaymentId }
  });
  if (existing) return; // Already processed
  await tx.payment.create({ data: { ... } });
  await tx.registration.update({ ... paymentStatus: "PAID" });
});

Tax Calculation

Tax is computed at checkout time and sent as a separate Stripe line item (not automatic_tax). The formula: taxAmount = basePrice × taxRate / 100. Both amounts appear on the PDF proforma/invoice.

🧮 Abstract Scoring Algorithm

Weighted multi-criterion scoring with per-reviewer submissions

Data Model

AbstractReviewer (assignment) ├── role: PRIMARY | SECONDARY | CONSULTING ├── conflictFlag: boolean └── assignedBy: userId AbstractReviewSubmission (per-reviewer scores) ├── criteriaScores: { [criterionId]: score (0-10) } ├── overallScore: 0-100 (auto-computed) ├── reviewNotes: text ├── recommendedFormat: ORAL | POSTER | NEITHER └── confidence: 1-5

Weighted Score Computation

Each review criterion has a weight (summing to 100). The algorithm converts per-criterion 0-10 scores into an overall 0-100 score:

function computeWeightedOverallScore(scores: CriterionScore[]): number | null {
  if (!scores.length) return null;

  // Step 1: Weighted sum on 0-10 scale
  const weightedOnTen = scores.reduce(
    (sum, c) => sum + (c.score * c.weight) / 100, 0
  );

  // Step 2: Project to 0-100 scale
  return Math.round(Math.min(100, Math.max(0, weightedOnTen * 10)));
}
✅
Example: 2 criteria (Originality: weight 60, Methodology: weight 40)
Reviewer A scores: Originality = 8, Methodology = 7
Weighted on 10: (8×60 + 7×40) / 100 = 7.6
Overall: 7.6 × 10 = 76/100

📄 CSV Parser (RFC 4180)

Quoted field handling, multiline support, 5000-row limit

Core Parsing Algorithm

The parser implements a character-by-character state machine with an inQuotes flag:

function parseCSVLine(line: string): string[] {
  const fields = [];
  let current = "", inQuotes = false;

  for (let i = 0; i < line.length; i++) {
    const char = line[i];
    if (char === '"') {
      if (inQuotes && line[i + 1] === '"') {
        current += '"';  // "" → literal quote
        i++;
      } else {
        inQuotes = !inQuotes; // Toggle state
      }
    } else if (char === ',' && !inQuotes) {
      fields.push(current.trim());
      current = "";
    } else {
      current += char;
    }
  }
  fields.push(current.trim());
  return fields;
}

State Machine Transitions

State: OUTSIDE_QUOTES ├── char = '"' → Enter INSIDE_QUOTES ├── char = ',' → Push field, reset buffer └── char = other → Append to buffer State: INSIDE_QUOTES ├── char = '"' + next = '"' → Append literal '"', skip next ├── char = '"' + next ≠ '"' → Exit to OUTSIDE_QUOTES └── char = other → Append to buffer (commas, newlines allowed)

Multiline Support

splitCSVLines() pre-processes the raw text to merge physical lines that are inside quoted fields. This handles values like "Line 1\nLine 2" correctly — they stay as one logical CSV row.

Column Mapping

Headers are normalized to lowercase with spaces stripped. getField(fields, index) safely returns undefined for missing columns (index < 0), preventing out-of-bounds errors.

🔁 Contact Sync

Fire-and-forget upsert with zero-failure-propagation guarantee

Pattern: Fire-and-Forget

export async function syncToContact(data): Promise<void> {
  try {
    await db.contact.upsert({
      where: { organizationId_email: { organizationId, email } },
      update: { ...optional },  // Merge new data
      create: { ... },          // Or insert fresh
    });
  } catch (error) {
    apiLogger.warn({ err: error }); // Log, never throw
  }
}
ⓘ
Why fire-and-forget? Contact sync is a secondary effect of registration/speaker operations. A contact sync failure should never cause a registration to fail. The try/catch guarantees the caller is never affected.

🛡 Idempotency

Evidence-based audit: what protects against double-submits and retries

Verdict: PARTIAL — No dedicated table, relies on state checks

There is no IdempotencyRecord table in the schema. Instead, idempotency is achieved through a mix of Stripe SDK keys, transactional state checks, and conditional writes.

Conditional Write Pattern (Key Algorithm)

The sold-out and promo code guards use the same technique — a conditional updateMany that returns 0 rows if the guard condition fails:

// Inside $transaction — atomic capacity check
const updated = await tx.ticketType.updateMany({
  where: {
    id: ticketTypeId,
    soldCount: { lt: ticketType.quantity }, // Guard: still has capacity
  },
  data: { soldCount: { increment: 1 } },
});
if (updated.count === 0) throw new Error("SOLD_OUT");
⚠
Gap: No Idempotency-Key header handling on any API route. A client that double-submits a registration POST in <1s with different request IDs would rely solely on the email+event findFirst inside the transaction. If the first request hasn't committed yet, the second could also pass the check.

Smoke Test: Double-Submit Registration

Expected: Second request blocked by findFirst inside $transaction. Edge case: Under high concurrency with Supabase PgBouncer, the serialization level may not prevent both transactions from passing the findFirst. The P2002 catch on unique constraint is the last-resort safety net.

Where Transactions ARE Used (37 files)

Prisma $transaction is used extensively for intra-database atomicity:

The Only Compensating Action

The refund route is the only place with an explicit rollback pattern:

// 1. Optimistic lock: mark REFUNDED before Stripe call
await db.registration.updateMany({
  where: { id, paymentStatus: "PAID" },
  data: { paymentStatus: "REFUNDED" },
});

// 2. Call Stripe
try {
  refund = await stripe.refunds.create(..., { idempotencyKey: `refund-${payment.id}` });
} catch {
  // 3. Rollback on failure
  await db.registration.update({
    where: { id }, data: { paymentStatus: "PAID" },
  }).catch(...);
}

Saga Orchestration

Not implemented. Registration + payment + accommodation is a linear try/catch flow. There is no state machine tracking multi-step workflows.

Smoke Test: Stripe Fails After Seat Reservation

Stripe checkout is a separate route from registration. The seat (soldCount) is incremented during registration. If Stripe fails at checkout, the seat remains reserved. There is no automatic release — the organizer must manually cancel the registration or the attendee retries checkout.

📊 CQRS & Read Models

Evidence-based audit: query optimization for dashboards

Verdict: NOT IMPLEMENTED — All aggregates computed live

Zero materialized views in any migration file. Zero CREATE VIEW statements. The get_event_dashboard tool runs 14 parallel queries on raw tables for every call:

// src/lib/agent/tools/dashboard.ts — computed on every request
const [
  regByStatus, speakerByStatus, sessionCount,
  upcomingSessions, liveSessions, pastSessions,
  checkInCount, totalConfirmed, recentRegs,
  nextSession, abstractCount, ticketTypes,
  paymentStats, accommodationStats,
] = await Promise.all([
  db.registration.groupBy({ by: ["status"], _count: true, where: { eventId } }),
  db.speaker.groupBy({ by: ["status"], _count: true, where: { eventId } }),
  db.eventSession.count({ where: { eventId } }),
  // ... 11 more queries
]);
ⓘ
Current scale assessment: With Promise.all parallelization and composite indexes on [eventId, status] and [eventId, ticketTypeId], this is acceptable for events up to ~5K registrations. Beyond that, consider a pre-computed EventStats table refreshed on write or via cron.

What Mitigates the Gap

  • Composite indexes: [eventId, status] on Registration means groupBy is an index-only scan
  • Promise.all: 14 queries in parallel ≈ latency of the slowest single query
  • React Query caching: Client-side 5-minute stale time means the dashboard isn't hammered on every page visit
  • Select-only: Queries use _count and select: { id: true }, not full-row fetches

Smoke Test: Block EventsAir Domain

If EventsAir's domain is unreachable, every import request hangs for 30 seconds before the AbortController fires. During that window, the request thread is blocked. There is no circuit that opens after N failures to fail fast on subsequent requests.

⚠
Impact: A Zoom API outage during webinar provisioning would cause each provisioning attempt to hang 30 seconds. Since the AI agent and cron workers share the same process, a cascade of hung requests could exhaust the Prisma connection pool.

🛡 Bulkhead Isolation

Evidence-based audit: resource isolation between workloads

Verdict: IMPLICIT ONLY — No explicit boundaries

Connection Pool

Prisma defaults to 10 connections (hard-coded in driver). The DATABASE_URL in .env.example uses Supabase PgBouncer but does not set connection_limit. All workloads share one pool:

┌──────────────────────────────────────────────────────────┐ │ Next.js Process │ │ │ │ ┌─────────────┐ ┌─────────────┐ ┌─────────────────┐ │ │ │ Registration │ │ AI Agent │ │ Cron Workers │ │ │ │ API routes │ │ (5min max) │ │ (email, zoom) │ │ │ └──────┬───────┘ └──────┬──────┘ └───────┬─────────┘ │ │ │ │ │ │ │ └────────┬────────┴──────────────────┘ │ │ │ │ │ ┌────────▼────────┐ │ │ │ Prisma Client │ ← Single instance, 10 conn │ │ │ (globalThis) │ default pool │ │ └────────┬────────┘ │ │ │ │ └──────────────────┼────────────────────────────────────────┘ │ ┌────────▼────────┐ │ PostgreSQL │ │ (Supabase) │ └─────────────────┘
⚠
Risk: If an AI agent conversation hangs (5-minute maxDuration) and holds DB connections, a registration request competing for the same pool may be delayed or fail. No priority queue, no per-workload pool, no worker isolation.

No Queue System

Zero queue dependencies (bull, bullmq, bee-queue, amqp) in package.json. All work runs synchronously within request handlers. Long-running operations (bulk email, AI agent) block their request thread for the full duration.

What Partially Mitigates

  • Cron workers run as separate HTTP requests triggered externally (EC2 crontab), not in-process loops
  • Scheduled emails use DB-based polling with atomic PENDING → PROCESSING claim (prevents double-send)
  • n8n runs as a separate Docker container — its MCP calls to EA-SYS are external HTTP, not in-process
  • Agent SSE route sets maxDuration: 300 — prevents unbounded hangs
