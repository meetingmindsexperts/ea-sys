2. The theme: the public path got the hard things right, then trusted the easy things

The genuinely difficult parts of this module — the row-locked replace-all, the honest bulk-add counts, deadline enforcement existing server-side, token entropy — are all correct. What slipped is the boundary between "the form the invitee loaded" and "the world at submit time" (M1/M2/M3: deadlines passing, dinners appearing, stale tabs), and parity with protections built elsewhere after this module shipped (M6 anti-double-send, M8 renderMessageValue, M4/L9 buildEventAccessWhere). A module reviewed once is not reviewed forever: five of the twelve MEDs exist because the codebase moved (July 11 send-idempotency, July 16 message tokens, July 16 access-where sweeps) and nobody swept this domain along.

3. Findings in detail

Med M1 — A deadline-less dinner never closes

src/app/api/public/events/[slug]/rsvp/[token]/route.ts:102 (GET) + :150 (POST)
closed: d.rsvpDeadline ? d.rsvpDeadline.getTime() < now : false,          // GET
openDinners.filter((d) => !d.rsvpDeadline || d.rsvpDeadline.getTime() >= now) // POST

Failure scenario: rsvpDeadline is optional in the console (no asterisk), so the common case is blank — and then an invitee clicking their old link weeks after the gala can flip yes→no or add 20 guests, silently rewriting the historical attendance record the CSV and headcounts report. dinnerAt — the one date every dinner has — is never consulted. Combined with M10 (no audit trail) the rewrite is also invisible.

Fix: treat rsvpDeadline ?? dinnerAt (or dinner start + small grace) as the effective close in both the GET closed flag and the POST openIds filter.

What to learn: an optional config field needs a sensible default behavior, not just a nullable column. "No deadline" should mean "closes when the thing happens", not "open forever".

Med M2 — A just-closed dinner's answer is dropped with a false success

src/app/api/public/events/[slug]/rsvp/[token]/route.ts:152, 202
const accepted = parsed.data.dinners.filter((d) => openIds.has(d.dinnerId));
...
return NextResponse.json({ ok: true });   // form → "Your dinner RSVP has been recorded."

Failure scenario: the invitee opens the form at 11:50 PM; D1's deadline passes at midnight while the form sits open; they tick D1 = attending and submit at 12:05. The filter drops D1, the API 200s, the form shows the full thank-you. They believe they RSVP'd yes; three courses short at the gala. A deadline passing while a form is open is the normal deadline failure mode.

Fix: return ignoredDinnerIds (or 409 when a submitted dinner closed since load) and surface "D1 closed before you submitted" on the form.

Med M3 — A stale form still runs the destructive replace-all

src/app/api/public/events/[slug]/rsvp/[token]/route.ts:154–196
if (accepted.length === 0 && openIds.size === 0) { ...400... }   // only when EVERYTHING is closed
// accepted=[] with open dinners existing falls through to:
await tx.rsvpDinnerResponse.deleteMany({ where: { inviteId, dinnerId: { in: [...openIds] } } });
// …creates nothing, stamps RESPONDED, overwrites dietary

Failure scenario: tab A answered "attending D2" (a recently added dinner). Tab B holds a stale form from before D2 existed, showing only D1 — whose deadline has since passed. Tab B submits → accepted=[], openIds={D2} → the D2 "yes" is deleted, nothing replaces it, the invite reads RESPONDED. The route can't distinguish "explicit decline-all" (dinners: [] — a legitimate UI path) from "this payload said nothing about any open dinner".

Fix: when dinners.length > 0 && accepted.length === 0, reject 409 "your form is out of date — reload" instead of running the replace-all. (Scoping the delete to payload ids would reopen the ghost-attendance hole; reject is the right shape.)

What to learn: replace-all semantics need a freshness precondition. "The submit is the invitee's complete intent" is only true when the form they filled matches the world the server is about to replace.

Med M4 — GET /dinners: no role guard, hand-rolled org scope, zero callers

src/app/api/events/[eventId]/dinners/route.ts:20–40
// ← no denyReviewer
const event = await db.event.findFirst({
  where: { id: eventId, organizationId: session.user.organizationId! },  // ← not buildEventAccessWhere

Failure scenario: a CRM_USER (zero-events by design) or an ONSITE temp assigned only to Event A can curl any org event's dinner names/venues/deadlines — the July-7 ONSITE cross-event class, on the route the Round-1 H2 fix missed (it fixed the sibling roster GET). Sensitivity is low (no PII, no tokens) — and the route has no callers: the console reads dinners from the roster GET (per the B2 comment).

Fix: either delete the GET, or align it with the roster GET (denyReviewer + buildEventAccessWhere). Fixing it also fixes L9 for this route.

Med M5 — MEMBER reads the roster through the in-app agent

src/lib/agent/tools/dinner.ts (list_dinner_rsvps) · agent execute route allows MEMBER · isReadOnlyTool passes any list_*

Failure scenario: the Round-1 H2 fix put denyReviewer on the roster GET explicitly so MEMBER ("the read-only sponsor-side observer") cannot read the VIP guest list. But a MEMBER opens /events/[id]/agent and asks "who's coming to the gala?" — list_dinner_rsvps is list_-prefixed (passes the read-only gate), not in FINANCE_ONLY_AGENT_TOOLS, and returns names, emails, dietary notes and per-dinner responses. Tokens are not returned (verified), so this is an info-exposure inconsistency, not impersonation. The two surfaces must agree.

Fix: either add list_dinner_rsvps to a MEMBER-refused set in the agent execute route (the finance-tools pattern), or decide MEMBER may see the roster and update the H2 comment — owner call.

What to learn: a data-access decision enforced on ONE surface isn't a policy — every read path (REST, agent tool, MCP, export) must be swept when a "who can see this" call is made.

Med M6 — The send fan-out has none of the anti-double-send machinery

src/app/api/events/[eventId]/rsvp-invites/send/route.ts:143–186
for (const inv of invites) { ... await sendEmail({...}); ... }   // up to 500, inline, sequential

Failure scenario: send to 300 invitees; nginx/browser gives up at ~60s mid-loop → the caller sees a network error with unknown partial state; the natural retry re-mails everyone already mailed (no enqueue dedup, no emailedKeys resume, no abort — the exact class closed for bulk email June 9/July 11 and certificates July 16). A double-clicked dialog fires two full batches. No data corruption (sends never touch invite status — verified), but duplicate VIP invitations at scale.

Fix: route through the ScheduledEmail/worker enqueue path like the dashboard bulk sender, or minimally add a short-window content dedup + per-invite lastInvitedAt skip.

Med M7 — Preview shows neither the typed subject nor the typed message

src/app/api/events/[eventId]/email-preview/route.ts:123–129 · dinner console send dialog
const sampleVars = buildEventPreviewVariables(event, session.user, {
  ...(customMessage ? { message: customMessage } : {}),   // ← template uses {{personalMessage}}
});
const renderedSubject = renderTemplatePlain(eventTemplate.subject, sampleVars); // ← typed subject ignored

Failure scenario: organizer types a custom subject + "Dress code is black tie", clicks Preview → sees the default subject and the canned sample "We're excited to have you!" — a message they never wrote — while the real send uses their text. Preview ≠ send on both dimensions. (The custom-subject half likely affects other slugs' previews too — the route is shared.)

Fix: map customMessage → personalMessage for templates that use that token, and render customSubject ?? eventTemplate.subject.

Med M8 — Tokens typed into the Message box land literal (missed renderMessageValue wiring)

src/app/api/events/[eventId]/rsvp-invites/send/route.ts:133, 163

Failure scenario: the July-16 work made tokens typed into the free-text message resolve on all three other senders (bulk pipeline, speaker single-send, registration single-send). This route inserts personalMessage raw with no renderMessageValue pass — {{firstName}} or {{organizerSignature}} typed into the dinner Message box goes out as literal token text to every invitee, and Textarea newlines collapse into one paragraph.

Fix: vars.personalMessage = renderMessageValue(personalMessage, vars, ...) per recipient, matching the other senders.

Med M9 — MCP inviteesTruncated lies under a status filter

src/lib/agent/tools/dinner.ts:98
inviteesTruncated: pagedInvites.length < allInvites.length && !statusFilter,

Failure scenario: 400 invites, 300 PENDING; the agent calls { status: "PENDING" } (limit 200) → 200 rows with inviteesTruncated: false — the agent tells the operator it has the complete pending list while a third is missing. The same "confidently wrong number" class this file's own H4 comment fixed for headcounts, one field over.

Fix: compare against the filtered total (already computed as responded/pending) and drop the !statusFilter clause.

Med M10 — The public submit leaves no trail

src/app/api/public/events/[slug]/rsvp/[token]/route.ts:166–201

Failure scenario: the replace-all deletes + recreates responses and overwrites respondedAt/dietary; the only record is a pino info with a count. A VIP disputes "I confirmed with 3 guests" after anyone holding the (plaintext, roster-visible) token link changed it — zero trail of what changed, when, or from which IP. The house precedent is the public speaker-agreement acceptance, which writes a formal AuditLog row (actor: "SPEAKER", IP) precisely because it's the externally-meaningful event; a dinner RSVP drives paid catering headcounts.

Fix: fire-and-forget AuditLog (action: "RESPOND", entityType: "RSVP_INVITE", userId: null, before→after per dinner + IP).

5. Verified clean (the load-bearing checks that passed)

  • Round-1 fixes hold: the roster GET's denyReviewer + buildEventAccessWhere (H2) are real in code; the B2 full-field select is in place; CSV export logs userId+rowCount (L3) and every cell goes through escapeCsvCell; the MCP headcount H4 fix aggregates over ALL invites.
  • Concurrency core: the public POST's replace-all runs under SELECT … FOR UPDATE in an interactive transaction (single pinned backend through the pooler — the lock genuinely holds); bulk-add uses in-payload dedup + createMany({skipDuplicates}) with the DB's honest count; no P2002 can escape either path.
  • Token security: 192-bit base64url, unique-index lookup then slug-asserted; never logged; public GET returns only the holder's own data; invalid token → uniform 404 on GET and POST (no enumeration oracle); submitted dinnerIds are intersected with the invite's own event's open dinners.
  • Re-submission semantics: un-ticking removes the row; decline-all clears open positives; closed dinners' captured answers survive resubmits; a response cannot land on an inactive dinner.
  • Headcounts: ONE implementation (computeDinnerHeadcounts) feeds console tiles, CSV and MCP; guestCount consistently excludes the invitee everywhere; deleted-dinner responses safely skipped.
  • Cascades: responses cascade from both dinner and invite (schema + migration SQL verified); event delete cascades everything; a dinner deleted mid-submit rolls back cleanly.
  • Logging/audit hygiene: every business 4xx/5xx in all 7 routes logs (bare 401s excluded per the house convention; 403s log inside denyReviewer); all six mutation audits are fire-and-forget with logged catch.
  • Email pipeline: per-event branding, per-recipient personalized link, EmailLog rows with full logContext, per-recipient failure isolation, sends never mutate invite status; the template exists in DEFAULT_TEMPLATES + TEMPLATE_VARIABLES + the SYSTEM_TEMPLATE_SLUGS mirror with the drift test green; invitee names are HTML-escaped (raw keys are only personalMessage/rsvpLink/organizerSignature).
  • Timezone: the console's event-TZ input pair round-trips losslessly at minute precision; console cards and the public form render event-TZ with a label; the invitation email carries no dinner date/time, so there is no email TZ surface.

7. Lessons index

  • Optional config needs default behavior, not just a nullable column (M1).
  • Replace-all needs a freshness precondition — "complete intent" is only true for a fresh form (M2/M3).
  • A "who can see this" decision must be enforced on every read surface, not the one that was reviewed (M5, M4).
  • Cross-cutting protections (send idempotency, message tokens, access-where) added after a module ships must be swept INTO it — a module reviewed once is not reviewed forever (M6/M8/L9).
  • An inverted "accepted-as-is" decision must update the acceptance record in the same change (M12).
