1. Method

Findings below are deduped across lenses; a finding independently confirmed by two lenses is marked ×2. 2. HIGH — both fixed same-day (July 20)

HIGH FIXED Jul 20 H1 — Freshsales re-import: the "EA-edited wins / never overwrite human data" guarantee decayed after ONE import cycle

Where: src/crm/services/crm-import-service.ts (companies + contacts stamp) + src/crm/lib/freshsales-import.ts decideImportAction.

The bug. An EA-born row (hand-created, lastImportedAt: null) correctly took the enrich path on its first import — blanks filled, human data untouched. But the write stamped lastImportedAt:

const stamp = {
  externalSource: FRESHSALES_SOURCE,
  externalId: row.externalId ?? existing?.externalId ?? null,
  lastImportedAt: new Date(),        // ← stamped on ENRICH too
};

On the second import (the advertised "recurring sync via re-uploaded CSVs" workflow), the row now matched by externalId, decideImportAction saw "previously imported, untouched since" (the human's edits predate import 1, so updatedAt never protects them), and took the full Freshsales-wins update path — overwriting the human-typed website/notes/jobTitle/phone, or setting them to NULL where the CSV cell was blank (notes: row.notes ?? null). The code's own comment promised "NEVER overwrite what a human typed"; it held for exactly one cycle. The round-1 test pinned import 1 only.

The fix. The enrich path no longer stamps lastImportedAt — an EA-born row stays enrich-forever (every future import fills blanks only; the externalId is still stamped so matching by id keeps working). Applied to both the companies and contacts importers (deals have no reachable enrich path — they match by externalId only). +2 test pins including a not.toHaveProperty("lastImportedAt") assertion on the enrich write that fails against the old code by construction.

Lesson: a provenance flag ("this row is human-authored") must be durable. If the safe path's own bookkeeping erases the marker that routed you to the safe path, the guarantee only survives one round trip. Test the second cycle, not just the first.

HIGH FIXED Jul 20 ×2 H2 — CrmDealProduct had no unique on (dealId, crmProductId): duplicate line items under concurrency

Where: prisma/schema.prisma (CrmDealProduct) + src/crm/services/crm-product-service.ts addDealProduct.

The bug. The PRODUCT_ALREADY_ON_DEAL guard was a findFirst-then-create with no constraint behind it — the exact class the round-1 seed-race fix (H1, migration 20260716140000) closed for stages/templates/SKUs, reproduced on the line-item join. Two reps with the same deal open (or one fast double-click / two tabs) both pass the check and both create: two identical lines, a double-counted products total, and an ambiguous target for the inline qty edit. Contrast CrmDealContact, which has @@unique([dealId, crmContactId]) and upserts — the product join got the check but not the structural guard.

The fix. @@unique([dealId, crmProductId]) + migration 20260720120000_crm_deal_product_unique (keep-oldest dedup first, then CREATE UNIQUE INDEX IF NOT EXISTS, name byte-matched to Prisma's canonical DDL; the redundant @@index([dealId]) dropped — the unique's leftmost column covers it). Lines whose catalog product was deleted (crmProductId NULL via SetNull) stay exempt — Postgres treats NULLs as distinct. addDealProduct maps the loser's P2002 back to PRODUCT_ALREADY_ON_DEAL (the same business fact, not a 500), with a test pin.

Lesson: a duplicate check without a UNIQUE constraint behind it guards nothing under concurrency. When you fix a bug class (the seed races), sweep for its siblings — this one was one join table away from the fixed ones.

3. MEDIUM — correctness & integrity ALL 12 FIXED Jul 20

MED FIXED Jul 20 ×2 M1 — the archived-record freeze does not cover field edits, notes, or the event-contact link

Round 1's M1 freeze (archivedAt: null in the claim) is real on moveDealStage, closeDeal, completeTask/reopenTask, addDealContact, addDealProduct and the reminder claim — but absent from six paths: updateDeal (deal-service.ts:287-299 — the snapshot doesn't even select archivedAt), updateTask (task-service.ts:221-224 — can re-arm a reminder on an archived task, which then fires on restore), updateCompany, updateCrmContact, createNote's attachment validation, and linkToEventContact; plus updateDealProduct/removeDealProduct don't check the deal. No race needed — editing an archived deal via REST PATCH or MCP update_crm_deal works every time, records an UPDATE history row after the ARCHIVE row, and a re-assign even notifies the new owner of a record they cannot find on any list. Fix direction: add archivedAt: null to the update predicates with the existing *_ARCHIVED discriminator on count===0 (the exact shape moveDealStage already has).

MED FIXED Jul 20 M2 — archive/restore itself is check-then-act

All four set*Archived functions read → guard → unconditional update-by-id (deal-service.ts:670-687 and siblings). Two concurrent archives both pass the "already in state" guard, both write, both record ARCHIVE → duplicate History rows + a shifted archive timestamp; archive-vs-restore crossing is last-write-wins with a misleading trail. The doc comment explicitly promises this can't happen. Fix: conditional claim — updateMany({ where: { id, organizationId, archivedAt: input.archived ? null : { not: null } } }), count===0 → the idempotent no-op return. One line per service.

MED FIXED Jul 20 M3 — last-terminal-stage guard can be raced into a pipeline with zero Won (or Lost) columns

pipeline-service.ts:403-416 (delete) and :264-277 (outcome flip): the sibling count runs outside any transaction. Two admins concurrently deleting the org's two WON-mapped columns both see one sibling, both pass, both delete → every subsequent closeDeal(WON) and won-ward drag refuses with NO_TERMINAL_STAGE. Recoverable by re-adding a mapped column; requires two mapped columns + concurrent admin action. Fix: re-count inside a $transaction with the delete/update.

MED FIXED Jul 20 M4 — import's company resolver doesn't handle the P2002 loser

crm-import-service.ts makeCompanyResolver prefetches once, then bare creates missing names — a company created concurrently (UI or a second import) turns the P2002 into a per-row error ("Another company already uses this name") and the contact/deal row is not imported. Contrast findOrCreateCompany, which re-reads the winner. Convergent on re-run (visible error, never silent corruption). Fix: catch P2002 inside idFor, findUnique by nameKey, reuse.

MED FIXED Jul 20 M5 — task/deal ownership is org-bound but not role-bound: the reminders worker emails CRM content to roles the CRM excludes

task-service.ts:74-79 / deal-service.ts:151-152: ownerId validation is { id, organizationId } only — the route gate checks the caller, nothing checks the assignee. Assign a follow-up to an ONSITE desk temp (org-bound, excluded from the CRM entirely) or a MEMBER (prose- and money-blind), and the reminder worker emails them the deal name, task title and description ("chase Abbott — they countered at AED 480k") — CRM pipeline content delivered outside every CRM gate. Side effect: a MEMBER assignee gets a CrmNotification it can read but never mark read (the PATCH is requireCrmWrite) — a permanent unread badge. Fix: require the owner's role ∈ the CRM roles in validateRelations (one query change per service).

MED FIXED Jul 20 M6 — deal re-import re-stamps wonAt/lostAt to now when the CSV has no Closed-date value

crm-import-service.ts:509-514: wonAt: row.closedDate ?? new Date() also runs on the update path. An export without a Closed-date column re-imported monthly drifts every won deal's close date to the re-import date — "deals won in July" (the dateField=closed filter) drops from 50 to 0. The existing test only re-imports a CSV that carries the date. Fix: on update, only write the stamps when row.closedDate is present.

MED FIXED Jul 20 M7 — an ARCHIVED catalog product can still be added to a deal

crm-product-service.ts:236: the product lookup has no archivedAt: null — a discontinued product is addable at its stale list price from REST or a stale picker. Fix: filter the lookup (or a distinct PRODUCT_ARCHIVED code).

MED FIXED Jul 20 M8 — import dry-run ≠ write for intra-file identity collisions carrying distinct external ids

In-file dedup keys on externalId ?? nameKey, so two rows with different externalIds but the same email/name both pass. Write run: row 1 creates, row 2 finds it → "1 created, 1 updated" (last row wins). Dry run: no write, row 2's lookup misses → "2 created". Breaks the file-header invariant that the confirm screen IS what the write does. Fix: add the nameKey/emailKey to seenKeys alongside the externalId (or simulate creations the way the company resolver's dry: ids already do).

MED FIXED Jul 20 M9 — MCP get_crm_report re-implements a thinner report

agent-tools.ts:490-520 composes its own groupBy shaping (importing only foldMoney) while REST is built on summarizePipeline / computeWinLoss / sortReps. The MCP report omits the open-value rollup, win rate, and the leaderboard, and can drift independently — the cross-caller-duplication rule. Fix: extract a shared report-assembly function in reports.ts.

MED FIXED Jul 20 M10 — the "first non-terminal stage" default is duplicated in two callers, owned by neither

agent-tools.ts:182 and create-deal-dialog.tsx:60 carry the identical expression; REST requires stageId and has no server default. Fix: a defaultOpenStage() in pipeline-service (or default inside createDeal), both callers consume it.

MED FIXED Jul 20 M11 — the one silent failure path in 31 routes

deals/[dealId]/products/route.ts:20: the inline null-check 404 bypasses the crmErrorResponse choke point and logs nothing — every sibling detail GET logs its not-found. One-line fix.

MED FIXED Jul 20 M12 — OWNER DECISION (implemented as recommended — reversible): the prose gate covers notes only; task descriptions and lostReason reach MEMBER

The round-1 M2 rationale ("key-based redaction can't strip a number out of a sentence") applies equally to two channels MEMBER can read: GET /api/crm/tasks?scope=all returns every task's 5,000-char description (also surfaced in History diffs), and the deals list/CSV/WON-LOST activity payloads carry lostReason ("they wanted 300k, we held at 500k") while the Value column is correctly omitted. Same class, smaller: the sponsor-email activity row stores the email subject. A sponsor-side MEMBER can reconstruct negotiation numbers from prose. Options: add description/lostReason to PROSE_KEYS (cheap, but blinds MEMBER to legitimate operational text) or accept-and-record as a deliberate posture. Needs the owner's call either way — today it is drift, not a decision.
