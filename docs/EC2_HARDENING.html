A plain-English walkthrough of what we're doing to the Mumbai EC2, why, and how to refresh your memory later.
Target: events.meetingmindsgroup.com (Mumbai, ap-south-1) Instance: i-0b51ab1213d084640 Owner: Krishna Last updated: 2026-04-23

What we're doing and why

The Mumbai box today is a single EC2, directly exposed to the internet, with port 22 (SSH) open to the whole world and port 443 serving traffic directly from its public IP. There's no CDN/WAF in front, no safety-net snapshot routine, and if this region has an outage (like Dubai did), there's no plan to recover.

This runbook closes those gaps in 6 small steps. Each step is low-risk on its own, reversible, and adds one layer of protection.

Step 1 — EBS snapshot baseline Done (manual)

Goal in one sentence

Take a point-in-time backup of the Mumbai instance's disk so everything we do afterwards is rollback-able.

What an EBS snapshot actually is

An EBS volume is the virtual disk attached to your EC2 (like a hard drive). A snapshot is a complete copy of that disk, stored in S3 (hidden behind the scenes). You can later create a new volume from the snapshot — effectively "restore to this point in time." The first snapshot captures every block; later snapshots only capture blocks that changed, so they're fast.

EBS snapshot vs AMI — quick distinction An EBS snapshot is just the disk contents. An AMI (Amazon Machine Image) bundles the snapshot + instance metadata (type, block device mapping, etc.) so you can "launch a new instance exactly like this one" with one click. For our workflow, the snapshot is enough — the DR plan rebuilds from scratch rather than from an AMI.

How you did it (manual, one-off)

  1. AWS Console → EC2 → Volumes.
  2. Right-click the Mumbai root volume → Create snapshot.
  3. First snapshot takes ~15–30 min; subsequent ones are seconds.

What to add later (optional — recurring backup)

For automated daily snapshots with retention, use AWS Backup:

  1. AWS Console → AWS Backup → Backup plans → Create plan.
  2. Rule: daily at 02:00 IST, 14-day retention.
  3. Resource assignment by tag: Environment=production.
  4. Tag the root volume accordingly so it's picked up.

Verify

aws ec2 describe-snapshots --owner-ids self --region ap-south-1 --query 'Snapshots[?starts_with(Description,`Created`)].[SnapshotId,State,StartTime]' --output table

Step 2 — Cloudflare in front of the box Pending

Goal in one sentence

Route all traffic through Cloudflare so the origin IP is hidden, WAF filters bad requests, and Step 3b can safely lock the SG.

What Cloudflare does for us (free tier)

  • Hides origin IP — users resolve the domain to a Cloudflare edge, not your EC2.
  • WAF — blocks SQL injection, XSS, common exploits before they reach your app.
  • Bot Fight Mode — challenges bots before they burn your rate limits.
  • DDoS absorption — Cloudflare soaks L3/L4 floods.
  • Free rate limiting rule — 1 rule on free tier, backstop to app rate limits.

The 9 sub-steps

  1. Add site — dash.cloudflare.com → Add site → meetingmindsgroup.com → Free plan.
  2. Audit imported DNS — Cloudflare pulls current records from your registrar (GoDaddy/etc.). Critically, check MX / TXT (SPF, DKIM, DMARC) records — if they're missing, org email dies after nameserver switch. Leave events A record on DNS-only (gray cloud) for now.
  3. Change nameservers at registrar — replace current NS with the two Cloudflare-assigned ones. Propagation: 5 min to 24 hr.
  4. Wait for Cloudflare to mark site "Active."
  5. SSL/TLS → Full (strict). The origin (nginx with Let's Encrypt) already has a valid cert.
  6. Flip A record to Proxied (orange cloud). Test that the site still loads; curl -I should show a cf-ray header.
  7. WAF → Managed rules → enable "Cloudflare Managed Ruleset" and "OWASP Core Ruleset" in Block mode.
  8. Security settings → Security Level: Medium, Bot Fight Mode: On, Browser Integrity Check: On.
  9. Rules → Rate limiting → one rule on /api/*, 100 req/min per IP. App-layer rate limits remain authoritative.
Common trap — losing email If you miss MX records during the NS switch, every email to your org stops delivering. Take a screenshot of the registrar's DNS page before switching, compare it to Cloudflare's imported records, and add anything missing manually before flipping nameservers.

Verify

# Traffic should now go through Cloudflare
curl -I https://events.meetingmindsgroup.com
# Look for: server: cloudflare + cf-ray: ...

# Nameservers should be Cloudflare's
dig NS meetingmindsgroup.com +short

Step 3a — AWS SSM Session Manager Done

Goal in one sentence

Get shell access to the Mumbai box without SSH, so we can safely close port 22 in Step 3b.

Why SSM and not just "only allow my IP on 22"

Your laptop's IP changes (ISP reset, travel, mobile hotspot). An allow-list locks you out the moment your IP moves. SSM uses outbound-only HTTPS from the instance to AWS, brokered through IAM credentials — no inbound firewall rule, no IP management, cross-platform (works from any laptop signed into AWS).

How SSM actually works under the hood The amazon-ssm-agent on the box makes an outbound WebSocket to AWS's SSM endpoint. When you run aws ssm start-session --target i-xxx, AWS matches your CLI request to the agent's open socket and proxies a bidirectional TTY stream through it. No public ports. Audited in CloudTrail.

What we did

  1. Verified SSM agent was running on the box (snap.amazon-ssm-agent.amazon-ssm-agent.service).
  2. Created IAM role ea-sys-mumbai-ec2-role with managed policy AmazonSSMManagedInstanceCore.
  3. Attached the role to the instance via EC2 → Actions → Security → Modify IAM role.
  4. Installed session-manager-plugin on your Mac: brew install --cask session-manager-plugin.
  5. Installed awscli on your Mac: brew install awscli, then aws configure.
  6. Tested: aws ssm start-session --target i-0b51ab1213d084640 --region ap-south-1 → shell as ssm-user → sudo -iu ubuntu to become the normal user.

Day-to-day usage

prodshell                         # alias for the SSM start-session command
# Now on the box as ssm-user
sudo -iu ubuntu                   # become ubuntu, same env as old SSH
cd ~/ea-sys
# ...do your thing...
exit                              # leaves ubuntu shell
exit                              # ends SSM session
Security follow-up — rotate off root AWS keys Your Mac's AWS CLI is configured with root account access keys (per aws sts get-caller-identity). Root keys have unrestricted power over your AWS account. Create a dedicated IAM user for CLI use, attach only the policies you actually need, then delete the root access key. Priority: within a week.

Step 3b — Security Group lockdown Reverted — needs CI/CD rewrite first

What happened on April 23, 2026 (lessons learned)
  1. We revoked the SSH 0.0.0.0/0 rule via aws ec2 revoke-security-group-ingress. SSH was closed; SSM remained working. Tested both — SSH timed out as expected, SSM shell worked.
  2. Then we realized the GitHub Actions deploy workflow (.github/workflows/deploy.yml) uses appleboy/ssh-action on port 22 to reach the box. Closing port 22 silently broke CI/CD. Nothing failed immediately because deploys only fire on push to main, but the next push would have failed.
  3. We re-opened port 22 with aws ec2 authorize-security-group-ingress to restore the deploy pipeline. We're back to the starting state on SSH exposure.

The real fix is to convert the GitHub Actions workflow from SSH to AWS SSM (aws ssm send-command). Then port 22 can be closed permanently without breaking CI/CD. Tracked as "Option B" in the deferred list.

Takeaway: before closing any inbound port on a production box, list every system that connects to it — not just humans. CI runners, monitoring probes, scheduled jobs, other internal services. SSH especially is easy to overlook because the thing connecting isn't a human.

Current state

Port 22 is back open to 0.0.0.0/0. SSH still works as before. SSM also works. You now have both access paths — the hardening gain is zero on SSH, but SSM is set up for the day we replace the CI/CD SSH dependency.

Goal in one sentence

Close port 22 entirely and restrict 80/443 to Cloudflare IP ranges, so the only way to reach the box is via Cloudflare or SSM.

Do not run before Step 2 completes If you lock 443 to Cloudflare CIDRs while the A record is still DNS-only (no Cloudflare proxy), every real user's request gets blocked and the site goes down.

What a Security Group is

An AWS SG is a stateful firewall attached to an EC2 (or ENI). It allows/blocks traffic by direction (inbound/outbound), port, protocol, and source (IP range or another SG). "Stateful" means if inbound is allowed, the reply is automatically allowed back out — you configure one direction.

The sub-steps

  1. EC2 → Security Groups → find the SG attached to the Mumbai instance.
  2. Inbound rules — replace entirely:
    • HTTPS 443: source = Cloudflare IPv4 CIDRs from cloudflare.com/ips-v4. One rule per CIDR (~15 rules). Description: Cloudflare-proxy.
    • HTTP 80: same (for ACME cert renewals and HTTP→HTTPS redirects).
    • SSH 22: delete entirely. SSM provides shell.
    • Delete everything else.
  3. Outbound: leave fully open. Box needs to reach Supabase, Stripe, Zoom, Brevo, SSM endpoints, GitHub.

Verify

# From any network — should time out
curl -I --max-time 5 https://<origin-ec2-public-ip>

# Through Cloudflare — should return 200
curl -I https://events.meetingmindsgroup.com/api/health

# SSH — should time out from anywhere
ssh -o ConnectTimeout=5 ubuntu@<origin-ec2-public-ip>

# SSM — should still work
prodshell

Gotcha: Cloudflare IP list drift

Cloudflare occasionally adds new CIDRs (maybe twice a year). When they do, new edge traffic from those CIDRs gets blocked by your SG until you add them. Subscribe to cloudflare.com/ips/ RSS, or later automate with a Lambda. Not in this round.

Step 4 — Unattended OS security updates Pending

Goal in one sentence

Have the box auto-install Ubuntu security patches daily, so you don't have to remember to apt upgrade.

What it does

unattended-upgrades is a Debian/Ubuntu package that runs apt-get upgrade daily, filtered to packages from the security archive only. It won't pull in potentially breaking feature updates. Kernel updates are applied but the box is not rebooted automatically — you reboot in a maintenance window when /var/run/reboot-required exists.

Commands (from inside prodshell → sudo -iu ubuntu)

sudo apt update
sudo apt install -y unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades   # answer Yes

sudo tee /etc/apt/apt.conf.d/20auto-upgrades > /dev/null <<'EOF'
APT::Periodic::Update-Package-Lists "1";
APT::Periodic::Unattended-Upgrade "1";
APT::Periodic::AutocleanInterval "7";
EOF

sudo systemctl is-active unattended-upgrades    # active
sudo systemctl is-active fail2ban               # active (already installed)

fail2ban is already on the box — nothing to install.

Pending reboot flag

After updates apply, if a kernel patch needs activation the file /var/run/reboot-required exists. Check weekly, reboot during low-traffic hours.

Step 5 — Code patch: trust CF-Connecting-IP Done (committed locally)

Goal in one sentence

Make the app's rate limiter read the real client IP from Cloudflare's trusted header, not the spoofable X-Forwarded-For.

The problem we're fixing

When the origin is directly reachable (like today), an attacker could set X-Forwarded-For: 1.2.3.4 on their request. Our getClientIp() trusted the first token, so rate limits would be counted against 1.2.3.4 instead of the attacker. They could burn your entire rate limit, trivially.

After Cloudflare is in front + SG locked to CF CIDRs, all traffic comes from Cloudflare. Cloudflare sets a header called CF-Connecting-IP containing the true client IP. We trust that first; fall back to the old logic as a safety net.

The change

In src/lib/security.ts:

export function getClientIp(req: Request): string {
  // Cloudflare sets CF-Connecting-IP to the true client IP. Trustworthy when
  // the security group only allows 80/443 from Cloudflare ranges — direct
  // connections can't reach the origin to spoof this header.
  const cfIp = req.headers.get("cf-connecting-ip");
  if (cfIp) return cfIp;

  const forwardedFor = req.headers.get("x-forwarded-for");
  if (forwardedFor) {
    return forwardedFor.split(",")[0]?.trim() || "unknown";
  }

  return req.headers.get("x-real-ip") || "unknown";
}

# You should see real client IPs (not Cloudflare edge IPs)

Key concepts (plain English)

Security Group (SG)

A stateful firewall wrapped around your EC2. You say "allow port 443 inbound from these IPs" and AWS blocks everything else. Stateful means replies to allowed inbound are automatically allowed outbound — you don't configure both directions.

IAM role vs IAM user

User = human/application with long-lived access keys. Role = a set of permissions that AWS resources (like EC2s) or users can temporarily assume. Roles are short-lived (tokens auto-refresh) — much safer than baking access keys into EC2s. Our ea-sys-mumbai-ec2-role is a role: the EC2 assumes it to talk to SSM, S3, KMS.

SSM Session Manager

AWS's service for getting a shell on an EC2 without SSH. The agent on the box makes outbound HTTPS to AWS; your CLI makes outbound HTTPS to AWS; AWS proxies a terminal between them. No inbound firewall rules needed. Every session is logged in CloudTrail.

Cloudflare "proxied" (orange cloud)

For a DNS record, "proxied" means Cloudflare owns the public IP that users resolve to. When a user connects, they hit Cloudflare's edge; Cloudflare forwards the request to your origin (over HTTPS). Gray cloud = Cloudflare just serves DNS but doesn't sit in the request path.

CF-Connecting-IP header

When Cloudflare forwards a request to your origin, it adds this header containing the real client IP (e.g., a user's home IP). Trustworthy only if your origin is unreachable directly — otherwise an attacker can send a request directly to the origin with a fake CF-Connecting-IP. That's why Step 3b (SG lockdown) is the thing that makes Step 5's header trust actually safe.

EBS volume vs snapshot vs AMI

Volume = the live disk attached to your EC2. Snapshot = a point-in-time copy of a volume, stored in S3. AMI = a snapshot + metadata (instance type, block device mapping) bundled so you can "launch a new instance just like this." For plain disk-level backup, snapshots are enough.

KMS (Key Management Service)

AWS's managed encryption key service. You create a key; AWS stores and manages it; you reference it to encrypt data. For our DR setup, we use a Singapore KMS key to encrypt the .env backup — so even if the Mumbai KMS key is lost in a regional outage, the Singapore backup is still decryptable.

RTO vs RPO

RTO (Recovery Time Objective): how long until the app is back up after a disaster. Our DR plan targets ~10 minutes. RPO (Recovery Point Objective): how much data you can lose. Our .env is backed up nightly, so RPO is up to 24 hours of .env changes. Since data lives in Supabase (separate system), actual data RPO is Supabase's, not ours.

Terraform 101 — what you'll do when you run it

Terraform is a CLI tool. You write .tf files describing what you want in AWS. Terraform compares "what you wrote" against "what exists in AWS" and makes AWS match your files.

Core commands you'll use

CommandWhat it does
terraform initOne-time per folder. Downloads the AWS + Cloudflare "providers" (plugins) into .terraform/.
terraform planReads your .tf files, checks AWS, prints exactly what it will create/modify/destroy. Nothing changes in AWS yet.
terraform applySame as plan but actually executes. Asks you to type "yes" before touching anything. Or use -auto-approve to skip the prompt.
terraform destroyDeletes every resource Terraform created in this folder. Asks for confirmation.
terraform fmtAuto-formats your .tf files. Cosmetic.
terraform outputPrints the output values from your last apply (e.g., the new public IP).

The state file

Terraform writes terraform.tfstate after every apply. This file is its memory — what it created, with what IDs. Never delete or edit it by hand (Terraform will lose track of your resources). It's in .gitignore for a reason — it can contain secrets.

Troubleshooting

SSM: "TargetNotConnected"

IAM role isn't propagated yet, or SSM agent isn't talking to AWS. Wait 2 min, retry. If still broken: prodshell won't work, so fall back to SSH, then sudo systemctl restart snap.amazon-ssm-agent.amazon-ssm-agent, wait 60s, retry.

SSM: "AccessDeniedException"

Your AWS CLI credentials don't have ssm:StartSession permission. If you're on root keys, this shouldn't happen. If on an IAM user, attach AmazonSSMFullAccess (or a scoped policy).

After SG lockdown: site shows Cloudflare error 522 ("origin unreachable")

Means Cloudflare can't reach your origin — SG isn't letting it through. Two common causes: (1) you didn't add all Cloudflare CIDRs (check list), (2) you still have a conflicting deny rule. Re-check SG → inbound rules.

Site returns 526 "invalid SSL cert"

Cloudflare is in Full (strict) mode but origin cert doesn't match. Check that nginx's Let's Encrypt cert for events.meetingmindsgroup.com is valid: sudo certbot certificates. If expired, sudo certbot renew.

Rate limiter is blocking legitimate users

After the CF-Connecting-IP patch, the limiter counts each user separately (good). If it's blocking real users, either the limit is too strict or a user is behind a shared proxy with many users on the same IP. Check docker compose logs ea-sys-blue | grep rate for the blocked IP and decide.

Cloudflare onboarded but old origin IP still resolving

DNS caches linger. Flush Mac: sudo dscacheutil -flushcache. Check externally: dig events.meetingmindsgroup.com +short. Wait up to a few hours for stubborn resolvers.

Terraform: "Error: No valid credential sources"

AWS CLI credentials not configured for the terraform process. Run aws sts get-caller-identity first in the same shell. Terraform picks up the same credentials via ~/.aws/credentials.

Terraform: cloudflare provider 403 / authentication failed

Cloudflare provider needs a token. Create one at dash.cloudflare.com/profile/api-tokens with "Zone:Read" and "DNS:Edit" on meetingmindsgroup.com. Then export CLOUDFLARE_API_TOKEN=... in your shell before terraform plan.
