Code Review — Survey & Dinner RSVP

4-angle production review · July 13, 2026 · 2 BLOCKER / 5 HIGH / 8 MED / 5 LOW — the most severe result of any domain.
Angles: token security & public-surface auth · RSVP lifecycle & integrity · survey lifecycle & the certificate coupling · RBAC / PII / drift / logging.
Scope: the public token-gated survey + RSVP endpoints, the organizer dinner console, the survey builder and responses, the MCP tool, and survey-thankyou-sweep.

⚠ B1 IS STILL OPEN. The survey share-link lets anyone submit a survey as another attendee, which auto-issues a real CME certificate in that person's name. Every fix changes what an attendee experiences, so it needs an owner decision — see B1 for the three options. Do not run a conference that issues CME certificates off a broadcast survey link until this is closed.
Status — everything else in the Blockers+HIGHs batch is SHIPPED (July 13): 220d421 (B2 dinner-edit wipe + timezone), 1ea4a92 (H2 RSVP token leak, H1 venue-WiFi lockout), c7a2faf (H3 thank-you starvation, H4 MCP headcounts). The MEDs/LOWs are deferred to docs/ROADMAP.md.

The theme: a flag changed meaning, and nobody re-checked who could set it

Two of the worst findings here have the same shape, and it's a shape worth learning because it doesn't look like a bug in any single commit.

June 9 — the survey share link shipped. Its design was explicit: respondents "self-identify by their registered email." At that moment Registration.surveyCompletedAt was just a feedback flag. Trusting a typed email was a reasonable trade: worst case, somebody submits fake feedback.

June 25 — survey-gated certificate auto-issue shipped. Its own header comment says it plainly:

// the survey POST path is UNTOUCHED — it already sets `surveyCompletedAt`.

That sentence was written as reassurance. It is actually the bug report. The new feature promoted an existing flag into a credential-issuing trigger — and never went back to ask who is allowed to set it. A field that was safe to let a stranger write became a field that mints an accredited medical certificate.

💡 The lesson (this is the one to remember)

When you make an existing field load-bearing for something new, re-audit every path that can write it. Security isn't a property of a function — it's a property of the whole graph of who can reach what. Adding a consumer changes the security requirements of the producer, even though the producer's code didn't change and its tests still pass.

Concretely, ask: "what is the weakest authentication on any path that can set this field, and is that strong enough for the strongest thing that now depends on it?" Here the weakest path was "type an email into a public form," and the strongest dependent was "issue a CME certificate." Those two should never have met.

BLOCKER

Blocker B1 — Anyone can submit a survey as another attendee (and mint their certificate) OPEN — needs a decision

src/app/api/public/events/[slug]/survey/route.ts — handleShareSubmit (~:589)

The bug 
// The share token is valid (it's DESIGNED to be broadcast — a slide, WhatsApp,
// an email signature). Now: who is submitting?
const registrations = await db.registration.findMany({
  where: {
    event: { slug },
    status: { notIn: ["CANCELLED"] },
    attendee: { email: normalizedEmail },   // ← whatever they TYPED into the form
  },
});
const target = registrations.find((r) => !r.surveyCompletedAt) ?? registrations[0];

return finalizeSubmission(req, target, rawAnswers, null);
//     └─ writes SurveyResponse, stamps `surveyCompletedAt`, tags the Attendee

Why it's wrong: a typed email is an assertion of identity, not proof of it. Nothing here demonstrates the submitter controls that mailbox. And medical-faculty emails are routinely public.

Failure scenario

The organizer puts the survey QR on the closing slide (its intended use). Someone opens it, types famous.cardiologist@hospital.ae, and submits garbage. Three consequences, none reversible by the victim:

  1. A real CME certificate is auto-issued in their name. surveyCompletedAt is exactly what the auto-issue sweep polls on (auto-issue.ts: surveyCompletedAt: { not: null }), so a serialized, audited certificate is minted and emailed — accreditation evidence manufactured by a stranger.
  2. The real attendee is permanently locked out. SurveyResponse.registrationId is @unique and the flag is now set, so their own link returns alreadyCompleted forever. There is no organizer "reset survey" affordance.
  3. The accreditation dataset is poisoned with forged answers attributed to a named physician.

Why it isn't fixed yet

Every real fix changes the attendee's experience, so it's an owner decision, not an engineering one:

OptionWhat changesTrade
Email them their link (recommended) The share page becomes "enter your registered email → we'll email you your survey link." The emailed link is the existing, secure, per-registration token path. Turns an assertion into a proof — only the mailbox owner can act. Also closes the enumeration oracle (always answer "if that email is registered, we've sent a link"). Costs one step: the attendee checks their phone in the room.
Share = feedback, never a certificate Direct submit stays, but it does not stamp surveyCompletedAt, so it can't mint a certificate. Keeps the frictionless closing-slide flow for feedback — but breaks "fill the survey to get your certificate" for share users.
Retire the share link Only the per-registration emailed token works. Simplest, most secure. Removes a feature organizers use.

💡 What a junior dev should learn

"The user told us who they are" is not authentication. Authentication is proof of control over an identity — a password, a session, a link sent to a mailbox only they can read. A typed email is a claim. It's fine to trust a claim for something harmless (pre-filling a name); it is never fine to trust one for something that issues a credential.

The give-away question: "what does the attacker need that they don't already have?" Here the answer was "the victim's email address, which is on their conference badge." That's not a secret, so it can't be a key.

Blocker B2 — Editing a dinner silently wipes its venue, description and RSVP deadline SHIPPED 220d421

rsvp-invites/route.ts (the select) → dinner/page.tsx (openEditDinner → saveDinner) → dinners/[dinnerId]/route.ts (the PUT)

A three-file chain, where each file is individually reasonable and the combination destroys data.

1. The API doesn't send the fields 
// rsvp-invites/route.ts — the ONLY place the console gets dinners from.
db.rsvpDinner.findMany({
  select: { id: true, name: true, dinnerAt: true },   // ❌ no location / description / rsvpDeadline
})
2. The client reads fields that were never sent 
// dinner/page.tsx
setDinnerForm({
  location:     d.location ?? "",          // undefined → ""
  description:  d.description ?? "",       // undefined → ""
  rsvpDeadline: d.rsvpDeadline ? … : "",   // undefined → ""
});
3. The PUT reads "present but empty" as "clear it" 
// dinners/[dinnerId]/route.ts
...(d.location !== undefined && { location: d.location || null })   // "" is not undefined → writes NULL

Failure scenario

An organizer fixes a typo in a dinner's name and clicks Save. The gala's venue and description are erased — and its RSVP deadline is set to NULL, so RSVP for that dinner never closes. Invitees keep changing their answer and their guest count long after catering has been confirmed to the hotel, and the organizer has no idea, because the dialog shows a blank deadline every time they open it.

Why nobody caught it

interface Dinner {
  location: string | null;       // ← the type SAYS these exist
  rsvpDeadline: string | null;   // ← TypeScript believed it. Nothing checked the API agreed.
}

The interface described the API's contract aspirationally. TypeScript can only check that your code is consistent with the types you wrote — it cannot check that the types are true.

Fix — shipped

The roster GET now selects every field the console edits, and a test pins that select.

💡 What a junior dev should learn

A hand-written type for API data is a wish, not a guarantee. The compiler validated every line of that page against an interface that simply lied. Types protect you from your own inconsistency; they say nothing about what a server actually returns. Anywhere a type crosses the network, something must verify it at runtime (a shared schema, a generated client, or — at minimum — a test that pins the query's select, which is what we added).

And a design rule that would have removed the whole class: "absent" and "empty" must not mean the same thing in a PATCH. If "" means "clear it", then a field the client forgot to load is indistinguishable from a field the user deliberately blanked. Prefer explicit null to clear and treat undefined/absent as "don't touch" — and never let the client send a field it never received.

High H1 — The survey guard locked an entire conference room out SHIPPED 1ea4a92

public/events/[slug]/survey/route.ts — the share-path "soft IP dedup"

The bug 
// Reject if ANY response came from this IP in the last 60 seconds.
const recent = await db.surveyResponse.findFirst({
  where: { eventId, ipHash, submittedAt: { gte: new Date(Date.now() - 60_000) } },
});
if (recent) return NextResponse.json({ error: "A response was just submitted from this device." }, { status: 429 });

It keys on the IP, not on the submitting registration. Everyone on the venue WiFi shares one NAT egress IP — so they all hash to the same ipHash.

Failure scenario

The organizer puts the survey QR on the closing slide. 300 people scan it at once. The first submission lands; every other person in the room gets 429. And because submissions keep arriving, the sliding 60-second window never empties — so the lockout is sustained. They never complete the survey, surveyCompletedAt is never set, and they never get their certificate. The guard fired hardest at the exact moment of intended use.

Two details make this worth studying. First, the code comment said the window was short "so distinct registrants behind one NAT IP aren't blocked" — the author had identified the exact risk, and the guard did the opposite of what the comment claimed. Second, it was redundant: the double-click it was written for is already handled by SurveyResponse.registrationId @unique, whose P2002 is caught and returned as an idempotent 200.

Fix — shipped

Deleted. The @unique is the real guard; ipHash is still recorded for the audit trail.

💡 What a junior dev should learn

A guard that duplicates a database constraint can only add false positives. The DB constraint is exact and race-proof; your application-level approximation of it is neither. Before adding a check, ask what it catches that the constraint doesn't — if the answer is "nothing", you've added a bug surface with no upside.

And: an IP address is not a person. Behind NAT — conference WiFi, offices, mobile carriers, VPNs — thousands of distinct humans share one. Any rule of the form "one X per IP per minute" will punish exactly the crowded, legitimate case you built the feature for. Rate-limit per identity where you have one; use IP limits only for coarse, generous abuse ceilings.

High H2 — The RSVP roster hands out the invite token (an impersonation credential) SHIPPED 1ea4a92

src/app/api/events/[eventId]/rsvp-invites/route.ts — GET

The roster GET returns each invitee's token. That token is the credential: whoever holds it can POST the public rsvp/[token] endpoint with no login and rewrite a named professor's attendance, guest count and dietary note. The response also carries the confidential VIP guest list, and ?export=csv streams the lot.

The bug 
// No denyReviewer. Hand-rolled org filter instead of buildEventAccessWhere.
const event = await db.event.findFirst({
  where: { id: eventId, organizationId: session.user.organizationId! },
});
// … then returns `token: true` for every invitee.

Three org-attached populations could therefore read it: MEMBER (the read-only, sponsor-side observer), ONSITE (org-scoped here, so a desk temp assigned to Event A could pull Event B's roster — the July-7 cross-event class that buildEventAccessWhere exists to close, never carried to the dinner routes), and an internal-domain REGISTRANT — i.e. an attendee account.

Fix — shipped

denyReviewer on the GET (blocks all three) + buildEventAccessWhere for the event lookup. The token stays in the payload — the console's copy-link button needs it — but only the roles who actually run the dinner can reach it. The CSV export now logs who pulled it.

💡 What a junior dev should learn

This is the same finding as the check-in barcode leak (July 11) and the accommodation barcode leak (July 13), in a third costume: a credential sitting in a payload that a read endpoint hands to roles who shouldn't hold it.

Build the habit of asking, for every field you return: "is this data, or is this a key?" A token, a barcode, a signed URL, a password-reset code — these aren't information about the user, they're the ability to act as them. Data can be shown to anyone who may know it. Keys may only be shown to those who may use them, and that's usually a much smaller set.

High H3 — The thank-you sweep starved everyone but the newest 100 SHIPPED c7a2faf

src/lib/certificates/survey-thankyou-sweep.ts

The bug 
const candidates = await db.registration.findMany({
  where: { surveyCompletedAt: { not: null, gte: windowStart } },
  orderBy: { surveyCompletedAt: "desc" },   // newest first
  take: 100,                                // ← take FIRST
});
…
for (const reg of candidates) {
  if (thanked.has(reg.id)) continue;        // ← filter AFTER
  …
}

Why it's wrong: the take is spent before the "already done" rows are removed. Once more than 100 completions exist in the window, the batch is permanently the same newest 100 — all thanked after the first pass — so every tick does zero work, and the older, un-thanked registrations sit below the slice, are never fetched again, and age out of the window unprocessed.

Failure scenario

400 attendees complete the survey. Roughly 300 of them never receive the thank-you — which is the email carrying their certificate — and the sweep cheerfully logs scanned: 100 forever.

Fix — shipped

const candidates = await db.registration.findMany({
  where: {
    surveyCompletedAt: { not: null, gte: windowStart },
    ...(thanked.size && { id: { notIn: [...thanked] } }),   // ✅ exclude in the QUERY
  },
  orderBy: { surveyCompletedAt: "asc" },                    // ✅ oldest first — FIFO, nobody starves
  take: 100,
});

💡 What a junior dev should learn

Filtering after a LIMIT is a queue that never drains. The limit is spent on rows you're about to throw away, so the work you actually need never enters the batch. Whatever decides "does this row still need work?" belongs in the WHERE, not in the loop.

And for any worker that processes a backlog: drain oldest-first. Newest-first plus a limit is a starvation machine — under sustained load, the front of the queue is never reached. The tell is a worker whose logs look healthy ("scanned: 100") while doing no useful work.

High H4 — The agent briefs the caterer with a headcount computed over one page SHIPPED c7a2faf

src/lib/agent/tools/dinner.ts — list_dinner_rsvps

The tool fetched invites with take: limit (default 200) and the optional status filter — then fed that same array to computeDinnerHeadcounts and to summary.totalInvited.

Failure scenario

  1. An event with 260 invitees: the agent reports seats for the oldest 200 only. The caterer is under-ordered by ~25%, with no truncation flag — and the page size is reported as the total.
  2. The operator asks "who hasn't replied to the gala yet?" → status: "PENDING" → headcounts are computed over people who by definition have no responses, so every dinner reports 0 attendees / 0 seats — still presented as the authoritative headcount.

The dashboard aggregates over all invites, so the two surfaces disagreed — and the agent is the one talking to the caterer.

Fix — shipped

Aggregate over the full set; paginate only the returned invitees[]; return inviteesTruncated / inviteesShown so the agent can say "showing 200 of 260" rather than implying it has the whole list.

💡 What a junior dev should learn

Never compute an aggregate from a paginated list. A page is for display; a total is for truth. The moment one query serves both, your sums silently become "the sum of the first page" — and they'll look completely plausible, which is what makes it dangerous.

Extra credit for the second failure: a filter that's correct for the list can be nonsense for the aggregate. "Show me the PENDING invitees" is a sensible list; "the headcount of the PENDING invitees" is meaningless — and the code happily computed it, because both came from the same array.

High H5 — Dinner times drift by the UTC offset on every save SHIPPED 220d421

src/app/(dashboard)/events/[eventId]/dinner/page.tsx

The bug 
// READ:  a UTC instant → the input
dinnerAt: d.dinnerAt.slice(0, 16)                    // "2026-08-01T15:00" — the UTC wall-clock
// WRITE: the input → a UTC instant
dinnerAt: new Date(dinnerForm.dinnerAt).toISOString() // browser parses it as LOCAL

A datetime-local input has no timezone — the browser reads its value as local wall-clock. So slicing the UTC wall-clock into it and then parsing it back as local shifts the instant by your UTC offset — on every single save, compounding.

Failure scenario

A 19:00 Dubai gala opens in the edit dialog showing 15:00 (while the card beside it still says 19:00 — the organizer sees two different times for the same dinner). Save anything at all and the dinner moves to 15:00. Save again → 11:00.

Fix — shipped

New src/lib/datetime-local.ts with the one lossless pair (toLocalDateTimeInput / fromLocalDateTimeInput) and the trap documented. The correct implementation already existed in the scheduled-email dialog; the dinner page had hand-rolled a broken one. Both now share the helper.

💡 What a junior dev should learn

String-slicing a date is always a bug. An ISO string looks like text, so it's tempting to treat it as text — but .slice(0, 16) silently reinterprets the meaning of the value (UTC wall-clock → local wall-clock) while looking like it just trimmed some characters.

The invariant to hold onto: read-back and write-back must be exact inverses. If toInput(toStored(x)) !== x, your form corrupts data a little bit every time someone opens it and clicks Save — and because each save looks like it "worked", nobody notices until the times are hours off. (We pinned exactly this with a test that saves five times and asserts the value didn't move.)

MEDIUM & LOW DEFERRED

  • M1 — the share submit is a registration-enumeration oracle. An unknown email 404s, a known one proceeds — so anyone with the (broadcast) link can test emails against the attendee roster of a named medical conference. Closed automatically if B1 is fixed with the email-a-link option.
  • M2 — a speaker who is also a delegate has two registrations on one email; the share path picks one arbitrarily, so the completion stamp can land on the hidden faculty companion while the delegate row stays "not completed" (and its certificate never issues).
  • M3 — deleting a question in the builder after responses exist makes those answers vanish from the dashboard and the CSV, with no warning. The data survives in the JSON but is unreachable from every operator surface.
  • M4 — a withdrawal is silently partly ignored. The replace-all is scoped to dinners whose deadline hasn't passed, so ticking "I can't attend any of the dinners" leaves a past-deadline Yes +2 intact — while the invitee sees the full "your RSVP has been recorded" thank-you. Three covers get plated for someone who told you they weren't coming.
  • M5 — the import picker includes CANCELLED registrations (no status filter, no status badge), so people who cancelled weeks ago get invited to the gala.
  • M6 — a deactivated dinner still feeds the organizer's headcount (and the CSV) while being hidden from invitees — and because the public replace-all only touches active dinners, those stale responses can never be cleared. Reinstating the dinner resurrects every original "Yes".
  • M7 — 8 unlogged 4xx branches in the SURVEY routes (incl. the share-link org-boundary 404s). Note: the CLAUDE.md claim that "every failure path logs" for Dinner RSVP was independently verified and it HOLDS — zero silent 4xx. Survey is where the rule breaks.
  • M8 — doc drift on a security boundary: both survey-responses routes' comments say "MEMBER is allowed to view (read-only by design)", but denyReviewer 403s MEMBER. The behaviour is the safe one; the comment is a lie in a security-relevant place. (And the denial logs as auth-guard:write-denied on a read route.)
  • LOWs: RSVP tokens never expire and can't be rotated (a permanent bearer credential in a URL path); ?preview=1 is unauthenticated and works for DRAFT events, surviving share-link revocation; the send route shares one 10/hr bucket between bulk and single-invitee sends, so nudging faculty one-by-one 429s on the 11th; target: "all" re-mails everyone including those who already replied (no per-recipient idempotency).

The reviewer's checklist

  • When a field gains a new consumer: "what is the weakest auth on any path that can WRITE this, and is it strong enough for the strongest thing that now READS it?"
  • When you see an identity claim: "what does the attacker need that they don't already have?" If the answer is on the victim's badge, it isn't a key.
  • When you see a hand-written type for API data: "what enforces that the server actually returns this?" Usually: nothing.
  • When you see "" treated as "clear it": "can the client send an empty value it never actually loaded?"
  • When you see a rate limit or dedup keyed on IP: "what happens when 300 people share one NAT?"
  • When you see take/LIMIT: "is anything filtered out AFTER this? Then the limit is being spent on rows I'm about to discard."
  • When you see an aggregate: "is it computed over the full set, or over the page I'm about to display?"
  • When you see a guard next to a DB constraint: "what does this catch that the constraint doesn't?" If nothing — it only adds false positives.
What this domain got right (verified, so the fixes don't regress it): the per-registration survey token is hashed at rest, slug-bound, one-time-use, and its delete + the response write + the completion stamp all happen in one transaction; SurveyResponse.registrationId @unique genuinely prevents a duplicate response (and its P2002 is caught as an idempotent 200), so re-submitting cannot mint a second certificate; answers are strictly validated against the current config, so arbitrary JSON can't land in the response; both CSV exports use escapeCsvCell (no formula injection); the RSVP token is a 192-bit CSPRNG value and is event-slug-bound (no cross-event replay); the share-link compare is timing-safe and correctly ordered (token before expiry); the public RSVP submit is a server-authoritative replace-all under a row lock; bulk-add dedup is race-safe; and — as noted — the Dinner RSVP logging claim in the docs is true.
