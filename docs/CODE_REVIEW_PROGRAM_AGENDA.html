Code Review — Program / Agenda / Sessions

Production-readiness audit · 10 July 2026 · EA-SYS main @ 425f9b4
Four independent review passes: scheduling lifecycle · RBAC & public exposure · concurrency & integrity · drift & failure-visibility.

Two credential leaks are live in production and directly exploitable today. B1 lets any registered attendee retrieve the Zoom host start URL (plus stream key and passcode) for every session, granting host control of your webinars. B2 serves the Zoom recording password and unpublished DRAFT programs to anyone on the internet, unauthenticated and unrate-limited. Both were re-verified in source by the lead reviewer. These should be fixed before anything else in this document.
How to read this. Findings are grouped by theme; where two independent passes hit the same defect that is noted (H1 was found by both the lifecycle and concurrency passes — a strong signal). Known-fixed items from the June 10 "Schedule/Agenda consistency pass" were excluded from scope: REST create+update timezone validation via isSessionWithinEventDates, dashboard honorifics, public agenda topics + session-speaker roles, public times in the event timezone, MCP create_session capacity + topic abstractId, and the 5 restored session tool definitions. Claims that pass verified true are recorded in §8.

1. Executive summary

The scheduling core is reasonable — the event-timezone validation added in June holds across REST and MCP, the FK cascades match what the code assumes, all eight session tools really are exposed to both the in-app agent and MCP, and every Event.settings write in this domain correctly routes through the row-locked updateEventSettings helper. What the review found is concentrated in four seams, and three of them are the same seams that produced this month's other findings:

  1. The public/attendee surface leaks operator credentials. The dashboard sessions list hands the Zoom host startUrl to org-null attendee roles (B1); the public session detail route hands the recording password to anonymous callers and accepts DRAFT events with no rate limit (B2, M3, M4). The two viewing paths — zoom-join (registration-gated) and stream-status (wide open) — enforce contradictory access.
  2. The session PUT destroys child data before it checks the optimistic lock (H1) — so the 409 that says "nothing was saved, reload" is a lie, and the replace isn't transactional. Found independently by two passes.
  3. Zoom is coupled to sessions by convention, not by code. Concurrent creates make two billable remote meetings and orphan one (H2); session DELETE never tells Zoom (H3); retiming a session never re-syncs Zoom (M6); the webinar anchor session is deletable with no guard (H3); and provisionWebinar's documented idempotency doesn't hold under concurrency (M1).
  4. Sessions never got their service. Create/update logic is hand-copied between REST and the MCP executors and has already drifted — a session created via the agent or n8n writes no audit row and pages no admin (H4) — plus the domain's usual unlogged-4xx debt (H5).

3. Credential exposure (fix first)

Blocker   B1 — The dashboard sessions LIST returns the Zoom host start URL, stream key, and passcode to REGISTRANT / SUBMITTER / REVIEWER / MEMBER / ONSITE

src/app/api/events/[eventId]/sessions/route.ts:133-146 (GET) · src/lib/event-access.ts:24-35

zoomMeeting: {
  select: { id: true, zoomMeetingId: true, meetingType: true, status: true,
    joinUrl: true, startUrl: true, passcode: true,          // ← host link + passcode
    liveStreamEnabled: true, streamKey: true, streamStatus: true },
},

Why it's reachable

The GET is guarded only by auth() + buildEventAccessWhere. There is no denyReviewer on it — the denyReviewer(session) at line 182 guards the POST. And buildEventAccessWhere deliberately grants event access to the org-null attendee roles by linkage: 

if (user.role === "REGISTRANT") return { id: eventId, registrations: { some: { userId: user.id } } };
if (user.role === "SUBMITTER")  return { id: eventId, speakers:      { some: { userId: user.id } } };

Exploit

A registrant who paid for the event signs in and calls GET /api/events/{eventId}/sessions directly. The event resolves through their own registration, and the response contains every session's zoomMeeting.startUrl — the Zoom host link, which grants host controls. They can start the webinar, claim host, remove the real host, or lift the raw streamKey and hijack the RTMP stream. Route-level middleware is navigation-only and never guards /api/*. Only ADMIN/ORGANIZER should ever see startUrl/streamKey. (The single-session GET is safe — its select omits zoomMeeting entirely.)

Fix shape

Drop startUrl + streamKey (and likely passcode) from the list select, or gate those three fields on canWrite(role) / an ADMIN-ORGANIZER check and strip them for everyone else — a redaction helper mirroring redactFinancialFields.

Blocker   B2 — The public session-detail route serves the Zoom recording password + unpublished DRAFT programs to anonymous callers, with no rate limit

src/app/api/public/events/[slug]/sessions/[sessionId]/detail/route.ts:17, 91-97, 128

const event = await db.event.findFirst({
  where: { slug, status: { in: ["DRAFT", "PUBLISHED", "LIVE"] } },   // ← DRAFT served publicly
...
zoomMeeting: { select: { recordingUrl: true, recordingPassword: true, recordingStatus: true } },
...
zoomMeeting: session.zoomMeeting,   // returned verbatim

Verified

The route imports no auth at all (imports are NextResponse, db, apiLogger, readSponsors), and it is the only public session route with zero checkRateLimit calls — zoom-join, lobby-status, stream-status and presence each have them.

Exploit

Anyone on the internet GETs the endpoint and reads recordingPassword + recordingUrl — the credential that gates the cloud recording of a paid, CME-accredited session — and can watch it. The same call returns the full agenda, speaker bios/photos, topics and sponsors of DRAFT (unannounced) events, and can be enumerated freely across sessions and events. The contrast is stark: zoom-join requires a signed-in, non-cancelled registration to admit a live viewer, while this route gives the recording away to strangers.

Fix shape

Remove recordingUrl/recordingPassword from the public select (serve the replay through a registration-gated route, as the live path already is); restrict the event filter to PUBLISHED/LIVE (or gate DRAFT on an authenticated org-staff session); add the per-IP rate limit its four siblings have.

Medium   M3 — Public stream-status hands the live HLS URL + stream key to anyone, bypassing the registration gate the SDK path enforces (and an unauthenticated GET performs a DB write)

…/sessions/[sessionId]/stream-status/route.ts:50-104

return NextResponse.json({ status: ..., hlsUrl: isLive ? hlsPlaybackUrl : null,
  hlsOriginUrl: ..., streamKey: zoomMeeting.streamKey });
...
db.zoomMeeting.updateMany({ ... data: { streamStatus } })   // ← public GET mutates state

The two viewing paths disagree on who may watch: zoom-join requires sign-in + a live registration; stream-status is wide open and returns a playable HLS URL (and DRAFT is accepted). Also flags an unauthenticated request mutating streamStatus — cache-throttled and low impact, but it shouldn't be a side effect of a public read.

Medium   M4 — Public session-detail has no rate limit at all (enumeration / scraping)

…/sessions/[sessionId]/detail/route.ts (whole file)

Siblings rate-limit by IP (agenda 120/min, zoom-join 60/hr, lobby-status 600/hr, stream-status 360/hr). detail has none — which is what makes B2's credential-bearing endpoint freely enumerable across every session and event. Fix alongside B2.

Medium   M5 — DRAFT-event exposure is systemic across the public session surface

detail:16 · lobby-status:30 · stream-status:51 · zoom-join:48 · presence:56

All five accept status: { in: ["DRAFT","PUBLISHED","LIVE"] }. Only the agenda route correctly restricts to PUBLISHED/LIVE and honours settings.agendaPublished. The DRAFT allowance exists for organizer end-to-end testing, and for zoom-join/presence it's mitigated by their auth + registration gates — but for detail and stream-status it means "unpublished content plus credentials, to the whole internet." Gate DRAFT on an authenticated org-staff session.

4. The session PUT (data loss)

High   H1 — The session PUT wipes and rebuilds all speakers + topics before it checks the optimistic lock, so a rejected stale write still destroys another editor's data — and the replace isn't transactional

src/app/api/events/[eventId]/sessions/[sessionId]/route.ts:236, 253, 285, 316 — independently reported by the lifecycle and concurrency passes

236  await db.sessionSpeaker.deleteMany({ where: { sessionId } });   // destructive, no tx
253  await db.sessionTopic.deleteMany({ where: { sessionId } });     // cascades TopicSpeaker
256  for (let i = 0; i < data.topics.length; i++) { await db.sessionTopic.create(...) }
...
285  const updateRes = await db.eventSession.updateMany({ where: { id, updatedAt: expected }, ... });
316  if (updateRes.count === 0) { ... code: "STALE_WRITE" ... }      // ← too late, children already gone

Failure scenario

Two organizers open the same session. A saves. B saves a topic tweak with a stale expectedUpdatedAt. B's request deletes and recreates every SessionSpeaker and SessionTopic — overwriting A's just-saved roster — and only then hits the lock, returning 409 "modified by someone else, reload and try again." B believes nothing was saved; A's work is gone. The stale-write guard is exactly inverted.

Compounding it, lines 236-271 are bare statements with no $transaction: any mid-loop failure (e.g. P2002 on SessionTopic.abstractId @unique) leaves the session with its old topics deleted and only a prefix recreated — a truncated agenda plus a 500. Concurrent public readers can observe a session with zero speakers/topics mid-replace. Note the MCP path for the same operation (replaceSessionSpeakers) is wrapped in $transaction with the comment "so the swap is atomic — never an in-between state"; the primary UI path is not.

Fix shape

Claim the lock first (conditional updateMany on updatedAt; count 0 → 409 before any mutation), then perform the parent update and both child replaces inside one $transaction.

5. Zoom ↔ session coupling

High   H2 — Creating a session's Zoom meeting is check-then-act: two concurrent POSTs create two real billable meetings on Zoom, and the loser never deletes its orphan

…/sessions/[sessionId]/zoom/route.ts:127, 170, 189 · prisma/schema.prisma ZoomMeeting.sessionId @unique

127  db.zoomMeeting.findUnique({ where: { sessionId } }),   // existence check
136  if (existingZoom) return 409;
170  zoomResponse = await createZoomMeeting(...);            // ← REAL remote meeting created
189  const zoomMeeting = await db.zoomMeeting.create({ ... }); // @unique blocks the 2nd ROW only

The DB blocks the second row, but only after both requests already called Zoom's API. The loser's create throws P2002 into the generic catch → 500 with no compensating deleteZoomMeeting (that call exists only in the DELETE route, line 333). Result: an orphaned, billable Zoom meeting/webinar on the account with no local row and no way to find it from the app. Fix: claim the row first (insert a placeholder, then call Zoom), or catch P2002 and delete the just-created remote meeting.

High   H3 — Session DELETE never tells Zoom, leaves the webinar anchor pointer dangling, and has no guard against deleting the anchor session at all

…/sessions/[sessionId]/route.ts:385 — the file contains zero references to Zoom · webinar/room/route.ts:73

await db.eventSession.delete({ where: { id: sessionId } });   // that's the whole DELETE

Three consequences, all verified:

  • Orphaned remote meeting. The local ZoomMeeting row cascades away (onDelete: Cascade), but the meeting keeps existing on Zoom — still joinable via any previously-shared joinUrl, still consuming the org's Zoom capacity. The dedicated Zoom DELETE route does call deleteZoomMeeting; session DELETE bypasses it entirely.
  • Dangling anchor pointer. Deleting a WEBINAR event's auto-provisioned anchor session cascades its ZoomMeeting + all attendance / presence / poll / Q&A rows, while settings.webinar.sessionId still points at the dead id. The producer's "Open the room" POST then matches 0 rows → 404 "Webinar session not found". The webinar is dead with no self-heal, and re-running the provisioner would mint a new anchor + new Zoom webinar, invalidating every join link already emailed.
  • No guard. Nothing in the route checks whether the session is settings.webinar.sessionId.

Medium   M1 — provisionWebinar's documented idempotency doesn't hold under concurrency: duplicate anchor sessions + duplicate billable Zoom webinars

src/lib/webinar-provisioner.ts:68-99, 126, 194

The guard reads existingWebinar.sessionId from settings and, if unset, creates the anchor session and a real Zoom webinar — with no lock over the read→create window. It's fired fire-and-forget from POST /api/events and from a manual "Re-run provisioner" control. Two concurrent invocations both see sessionId unset, both create an anchor EventSession, both create a billable Zoom webinar, then both call the (correctly atomic) updateEventSettings — last-write-wins on webinar.sessionId. One anchor + its remote webinar are orphaned and unreachable.

Medium   M6 — Rescheduling a session never re-syncs its Zoom meeting; the two systems silently drift

…/sessions/[sessionId]/route.ts:285-302 · src/lib/agent/tools/sessions.ts:394

Both PUT paths write startTime/endTime and neither touches the ZoomMeeting row or calls updateZoomMeeting/updateZoomWebinar. Move a webinar from 2pm to 4pm and Zoom keeps the original scheduled start and the stored duration goes stale, while lobby logic and the public agenda key off the new times. Nothing reconciles them and the operator is never warned.

6. Drift, duplication, and failure-visibility

High   H4 — There is no session-service: create/update logic is hand-copied REST↔MCP and has already drifted — a session created via the agent or n8n writes no audit row and notifies no admin

…/sessions/route.ts:363, 375 (audits + notifies) vs src/lib/agent/tools/sessions.ts createSession (does neither)

Verified

The MCP file writes auditLog in updateSession (:428), addSpeakerToSession (:678), removeSpeakerFromSession (:713) and replaceSessionSpeakers (:808) — but not in createSession, which ends at refreshEventStats(ctx.eventId); return { success: true, session };. On a platform where the audit trail is load-bearing for RBAC accountability, every agent/n8n-created session is invisible in AuditLog.

Field drift in the same duplication

  • REST create accepts status, session-level abstractId (with an "already assigned to another session" guard) and topic sortOrder; the MCP executor silently drops all three — the agent can't create a DRAFT session, can't link an abstract at creation, can't order topics.
  • Capacity: REST create is z.number().min(1); MCP updateSession does Math.max(0, Number(input.capacity)), so MCP can set a capacity of 0 that REST rejects.

Every other multi-caller domain has a service (accommodation, abstract, speaker, registration, promo-code, billing-account, payment). This is the exact "MUST mirror by hand" smell the services layer exists to kill.

High   H5 — Silent 4xx across the session + public routes (violates the hard "every failure path must log" rule)

sessions/route.ts:240, 251, 259, 264, 268, 273, 279 · [sessionId]/route.ts:153, 157, 180, 197, 205, 212, 229 · presence:31, 42, 60, 78, 87 · detail:32, 102 · zoom-join 404s

Only the Zod-400 and the stale-409 log. Every other business rejection — event/track/abstract/speaker 404s, "end time must be after start time", "session must fall within event dates", "abstract already assigned", and on the public side 401/429/403 NOT_REGISTERED/404 — returns with no apiLogger call. A run of failed agenda saves or failed webinar joins leaves no warn-level trail in /logs. Same class swept for payments (M12) and flagged for abstracts (M11).

Medium   M2 — Session PUT regenerates every topic id, so topic ids are unstable across saves

…/sessions/[sessionId]/route.ts:256-269 (topicSchema accepts id at :18 but the create never uses it)

Editing one topic's title deletes and recreates all topics with fresh cuids. Any stable reference to a topic id — deep links, a future external sync, per-topic abstract mapping — breaks on every session save, and partial topic edits are impossible.

Medium   M7 — Public agenda buckets days in the viewer's timezone while rendering times in the event's — a self-contradictory agenda

src/app/e/[slug]/agenda/page.tsx:85, 134, 448 · src/components/public/speakers-agenda-preview.tsx:125-131, 254

const day = new Date(s.startTime).toLocaleDateString("sv-SE");        // grouping key = VIEWER tz
return new Date(dateStr + "T00:00:00").toLocaleDateString("en-US", …); // heading = viewer-local midnight
...
formatTimeInTz(new Date(session.startTime), timezone)                  // time = EVENT tz

event-time.ts exports localDateInTz/formatDateInTz for exactly this and is bypassed for the date half. An 11 PM event-TZ session (which isSessionWithinEventDates deliberately keeps on the last day) is bucketed under the next calendar day for any viewer west of the event, while its time reads correctly. The preview component also carries a now-false comment claiming the page "renders in viewer tz."

Medium   M8 — The dashboard agenda ignores event.timezone entirely: the list/tooltips render in hardcoded Asia/Dubai while the calendar grid positions cards in browser-local

src/lib/utils.ts:23-24, 62-64 (toDubai fixed offset) · (dashboard)/…/agenda/page.tsx:212 (start.getHours())

Two clocks on one page, neither the event's. For a non-Dubai event (or a travelling organizer) the grid draws a session at one hour and the list labels it at another — the admin builds the agenda against a mis-drawn grid. formatTimeInTz(event.timezone) exists and is unused here. The June-10 pass moved the public rendering onto event-time.ts and left the dashboard behind.

Medium   M9 — Narrowing an event's dates silently orphans out-of-range sessions

src/app/api/events/[eventId]/route.ts:242-243

Shortening an event from 3 days to 2 leaves day-3 sessions in place: isSessionWithinEventDates now rejects any new or edited session on day 3, yet the stale ones persist and still render on the public agenda (which filters on status, not on the event window). The agenda contradicts the event's own dates, with no warning or reconciliation at save time.

Medium   M10 — sortOrder is allocated read-then-insert with no unique constraint → duplicate order keys on concurrent creates

tracks/route.ts:119 (max+1) · src/lib/agent/tools/sessions.ts:229 (count())

Neither Track.sortOrder nor SessionTopic.sortOrder is constrained. Two concurrent track creates — or two add_topic_to_session calls from MCP/n8n on the same session — read the same max/count and insert the same value; orderBy: sortOrder asc then ties and renders non-deterministically. This is the class already fixed for certificate templates via a transactional aggregate+create.

Medium   M11 — SessionRole and SessionStatus have no shared label/colour map; four re-implementations exist and the dashboard tooltip renders raw MODERATOR:

src/lib/speaker-agreement.ts:19 · (dashboard)/…/agenda/page.tsx:181, 1600 · e/[slug]/agenda/page.tsx:518 · e/[slug]/session/[sessionId]/page.tsx:813

The organizer sets "Chairperson" and sees CHAIRPERSON: in the console tooltip but Chairperson on the public page. There is no session-enums.ts analogous to abstract-enums.ts, so a future role/status value must be patched in 4+ hardcoded spots and the build won't catch a miss (an exhaustive Prisma-enum-keyed Record would).

7. Lows

Low   L1 — Removing/replacing a session's speakers orphans their per-topic TopicSpeaker rows

src/lib/agent/tools/sessions.ts:706-708, 794-806 + the REST PUT sessionRoles path

Only SessionSpeaker is deleted. A speaker dropped from a session ("they cancelled") keeps their TopicSpeaker row and still appears on the public agenda under the topic, contradicting the removed state. The tool's own description ("Replace ALL speakers on a session") reinforces the wrong mental model.

Low   L2 — MCP update_session allows a zero-duration session that every REST path forbids

src/lib/agent/tools/sessions.ts:352 (newEnd < newStart) vs REST sessionEnd <= sessionStart

The agent can set endTime === startTime; list_live_sessions_now would mark it live only at a single instant — an effectively never-live agenda entry the dashboard would have blocked.

Low   L3 — Abstract→session uniqueness is check-then-act; the race surfaces as an opaque 500 instead of the intended 400

sessions/route.ts:232, 271, 291 (EventSession.abstractId @unique)

The DB is the real guard, but two concurrent creates both pass the findFirst check and the loser's P2002 falls into catch → 500 "Failed to create session" rather than the friendly "already assigned" 400. Same latent pattern on SessionTopic.abstractId @unique.

Low   L4 — Session/track writes org-scope by hand instead of buildEventAccessWhere; three public "join opens at" timestamps still render viewer-local (CLAUDE.md says two)

sessions/route.ts:220 · tracks/[trackId]/route.ts:34-41 · e/[slug]/session/[sessionId]/page.tsx:682, 683, 1002

The hand-rolled organizationId: session.user.organizationId! isn't exploitable here (denyReviewer fires first) but it's the pattern that produced three IDORs elsewhere this month, and it 404s a SUPER_ADMIN with no org. Separately, line 683 formats the very same instant the header renders in event TZ — so a Dubai 9:00 AM session tells a New York attendee "Join opens at 1:00 AM." CLAUDE.md calls these "two secondary" timestamps; there are three, and one is not secondary.

Low   L5 — Public slug lookups are not org-scoped (pre-multi-tenant), and isSessionWithinEventDates relies on an implicit lexical date-string contract

public/…/agenda/route.ts:30 · src/lib/event-time.ts:44-51

Correct under single-org today, but Event.slug is unique per-org — every public slug lookup becomes a cross-tenant leak the moment multi-tenancy lands (consistent with MULTI_TENANCY_IMPACT.md). And sStart >= evStart && sEnd <= evEnd is a lexical compare on "YYYY-MM-DD" strings — sound only because localDateInTz pins en-CA zero-padded output. Worth documenting as a contract.

9. What checked out clean

  • The June 10 consistency pass held. REST create + update both validate session times via isSessionWithinEventDates in the event timezone, matching MCP.
  • All 8 session tools really are exposed to both surfaces — 8 in SESSION_TOOL_DEFINITIONS, 8 in SESSION_EXECUTORS, all 8 registered in register-mcp-tools.ts. (The identical CLAUDE.md claim for abstracts turned out false; this one is true.)
  • MCP cross-event injection is closed. getOrgIdSecure(eventId, orgId) binds the event to the authenticated org before any executor runs, and every session tool re-filters speakerId/trackId/abstractId by eventId: ctx.eventId.
  • Every Event.settings write in this domain uses the row-locked updateEventSettings — webinar settings, sponsors, and the provisioner. No read-spread-write bypass remains.
  • FK cascades match the code's assumptions: deleting a Speaker cascades into both SessionSpeaker and TopicSpeaker; ZoomMeeting, ZoomAttendance and WebinarPresence cascade from the session. ZoomMeeting.sessionId is genuinely @unique (1:1 holds).
  • The public agenda's publish gate is server-enforced (settings.agendaPublished + PUBLISHED/LIVE only), not UI-only — unlike the registrationOpen no-op fixed in June.
  • list_live_sessions_now is correct — compares absolute instants, excludes CANCELLED.
  • MCP replace_session_speakers is properly transactional, and the presence heartbeat's deliberate no-transaction upsert is sound (the @@unique([sessionId, registrationId]) makes INSERT … ON CONFLICT collision-safe; the only race is a benign joinCount off-by-one on a non-authoritative stat).
