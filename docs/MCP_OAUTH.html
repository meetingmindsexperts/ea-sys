MCP OAuth 2.1 — EA-SYS Implementation Guide • April 2026

This is the guide to how EA-SYS exposes its Model Context Protocol server to browser-based MCP clients — claude.ai web, the Anthropic Console, and anything that can't send a custom HTTP header. It covers the why, the flow, the wire format, the security model, and the operational bits. For the tools themselves (what list_events, create_speaker etc. do), see MCP_REFERENCE.md.

Why this exists

Claude Desktop connects to remote MCP servers via mcp-remote — a Node subprocess that runs on your laptop and can set any HTTP header it wants, including x-api-key: mmg_xxx. Simple.

Claude.ai web runs in a browser sandbox. The connector UI has no "paste your API key here" field, and for good reason: a browser-based app sending a long-lived API key via a header is a security liability. The MCP spec solves this by requiring OAuth 2.1 for all remote MCP servers that want browser clients to connect.

Before this feature, hitting our /api/mcp endpoint from claude.ai failed at three places:

  1. GET /api/mcp returned 401 but no WWW-Authenticate header — so claude.ai had nothing to discover.
  2. GET /.well-known/oauth-protected-resource returned 404 — so even a speculative discovery failed.
  3. OPTIONS /api/mcp with Origin: https://claude.ai returned 204 with no CORS headers — so the browser would have blocked every subsequent request even if auth were solved.

All three are now fixed. Claude Desktop's path (x-api-key header via mcp-remote) is 100% unchanged for backward compatibility.

The big picture

We implement a spec-compliant OAuth 2.1 authorization server with:

  • RFC 9728 Protected Resource Metadata — tells clients where to find the authorization server
  • RFC 8414 Authorization Server Metadata — advertises the /authorize, /token, /register, /revoke endpoints
  • RFC 7591 Dynamic Client Registration — new clients register themselves without human intervention
  • RFC 6749 §4.1 Authorization Code Grant + RFC 7636 mandatory PKCE (S256 only)
  • RFC 6749 §6 Refresh Token Grant with refresh rotation
  • RFC 7009 Token Revocation

Data model

Three Prisma models, all with onDelete: Cascade on their parent relations so cleaning up a client or a user cleans up their OAuth state automatically.

McpOAuthClient

One row per DCR registration. Usually one per Claude user per browser, since claude.ai caches the client_id in localStorage.

FieldPurpose
clientIdRandom UUID issued at registration time. Public — appears in URLs.
clientSecretHashSHA-256 hash of the client secret. null for public clients (token_endpoint_auth_method: "none"), which is what claude.ai uses.
clientNameFree-text display name from the DCR request. Rendered on the consent page.
redirectUrisArray of exact-match redirect URIs. The consent flow will only ever redirect to one of these.
grantTypesDefault ["authorization_code", "refresh_token"]. We don't implement any other grant types.
tokenEndpointAuthMethodEither "none" (public client) or "client_secret_post" (confidential). claude.ai is public.

McpOAuthAuthCode

Short-lived one-time-use authorization code. 10-minute TTL. Stored hashed so a DB leak can't impersonate in-flight flows.

FieldPurpose
codeHashSHA-256 of the raw code. Unique index.
clientIdFK to McpOAuthClient.clientId. The token endpoint verifies the request's client_id matches.
userId, organizationIdThe user who approved the grant and the org it's bound to.
codeChallenge, codeChallengeMethodPKCE state. S256 only. The token endpoint hashes the client's code_verifier and compares to codeChallenge with crypto.timingSafeEqual().
redirectUriThe redirect_uri from the original authorize request. The token endpoint requires the exchange request to pass the exact same value.
expiresAtnow + 10min. Enforced on exchange; cleaned up by the cron.

McpOAuthAccessToken

The issued Bearer token. Also stores the refresh token in the same row (so refresh rotation can revoke the old row and mint a new one atomically).

FieldPurpose
tokenHashSHA-256 of the raw access token. Unique index — this is the lookup key on every MCP request.
refreshHashSHA-256 of the raw refresh token. Nullable, unique when set. Used to locate the row on /token with grant_type=refresh_token.
clientId, userId, organizationIdWho this token belongs to.
expiresAtAccess token expiry: now + 30d. Refresh expiry is createdAt + 90d, checked at refresh time.
revokedAtSet on rotation (refresh flow), explicit revocation (/revoke), or indirectly when the user is deleted (cascade).
lastUsedAtUpdated fire-and-forget on every successful access-token validation so we can tell abandoned tokens apart from live ones.
Raw tokens never touch storage or logs. Every field named *Hash holds only the SHA-256 digest. The raw token exists transiently in RAM during the request that issues or verifies it, and is echoed once to the client. Logs include only the first 10–12 characters for debugging.

The OAuth flow, step by step

1 User tries to add a custom MCP server in claude.ai
They paste https://events.meetingmindsgroup.com/api/mcp. claude.ai GETs it without auth to see what it needs.
2 Server responds 401 with WWW-Authenticate challenge
The header value is Bearer realm="mcp", resource_metadata="https://events.meetingmindsgroup.com/.well-known/oauth-protected-resource". This is the single thing that bootstraps the entire flow — without it, claude.ai just gives up.
3 claude.ai fetches the protected resource metadata
GET /.well-known/oauth-protected-resource returns a JSON document listing authorization_servers: ["https://events.meetingmindsgroup.com"] and supported scopes. claude.ai now knows where the auth server lives.
4 claude.ai fetches the authorization server metadata
GET /.well-known/oauth-authorization-server returns endpoint URLs: authorization_endpoint, token_endpoint, registration_endpoint, revocation_endpoint, plus declared capabilities (S256 PKCE, auth code + refresh grants).
5 claude.ai registers itself via DCR
POST /api/mcp/oauth/register with { client_name: "Claude", redirect_uris: ["https://claude.ai/api/mcp/..."] }. Server creates an McpOAuthClient row and returns a fresh client_id. claude.ai stores it in localStorage for future connections.
6 claude.ai opens the authorize URL in a popup
The URL includes client_id, redirect_uri (one of the registered ones), response_type=code, PKCE code_challenge, code_challenge_method=S256, a random state, and optional scope. Points at /mcp-authorize — our Next.js consent page.
7 User signs in (if needed) and clicks Approve
If there's no NextAuth session, the page redirects to /login?callbackUrl=<encoded full authorize URL>. After login, they're bounced back. The consent page enforces RBAC: only ADMIN, SUPER_ADMIN, and ORGANIZER roles see Approve/Deny buttons. Reviewers, submitters, and registrants get a permission-denied page.
8 Server issues a one-time auth code
The consent form POSTs to /api/mcp/oauth/authorize/decision. The route re-validates client_id + redirect_uri server-side (never trust form fields), issues a code via issueAuthCode(), and 302s to ${redirect_uri}?code=xxx&state=yyy.
9 claude.ai exchanges the code for tokens
POST /api/mcp/oauth/token with grant_type=authorization_code, code, code_verifier, client_id, redirect_uri. Server runs exchangeAuthCode(): verifies PKCE (sha256(verifier) === challenge with timingSafeEqual), verifies redirect_uri matches, verifies client_id matches, verifies not expired, deletes the code row, mints access + refresh tokens. All inside a single transaction so double-use is impossible.
10 claude.ai uses the access token
Every subsequent request to /api/mcp carries Authorization: Bearer mcp_at_xxx. Our authenticate() function tries validateApiKey() first (backward compat), then falls back to validateOAuthAccessToken(), which hashes the Bearer and looks up the McpOAuthAccessToken row. On hit, it returns { organizationId } and the rest of the MCP route proceeds exactly as before.

Discovery endpoints

Both live under /.well-known/ which Next.js serves from src/app/.well-known/*. These routes are public (no auth) and always wrapped with CORS.

/.well-known/oauth-protected-resource

{
  "resource": "https://events.meetingmindsgroup.com/api/mcp",
  "authorization_servers": ["https://events.meetingmindsgroup.com"],
  "scopes_supported": ["mcp"],
  "bearer_methods_supported": ["header"],
  "resource_name": "EA-SYS MCP Server",
  "resource_documentation": "https://events.meetingmindsgroup.com/docs/MCP_REFERENCE.md"
}

/.well-known/oauth-authorization-server

{
  "issuer": "https://events.meetingmindsgroup.com",
  "authorization_endpoint": "https://events.meetingmindsgroup.com/mcp-authorize",
  "token_endpoint": "https://events.meetingmindsgroup.com/api/mcp/oauth/token",
  "registration_endpoint": "https://events.meetingmindsgroup.com/api/mcp/oauth/register",
  "revocation_endpoint": "https://events.meetingmindsgroup.com/api/mcp/oauth/revoke",
  "scopes_supported": ["mcp"],
  "response_types_supported": ["code"],
  "grant_types_supported": ["authorization_code", "refresh_token"],
  "code_challenge_methods_supported": ["S256"],
  "token_endpoint_auth_methods_supported": ["none", "client_secret_post"]
}
Why authorization_endpoint is a page, not an API route. /mcp-authorize is a Next.js server component that needs to render a consent UI with NextAuth session context. Putting it under /api/ would force awkward JSX-from-API-route tricks. A plain page is cleaner.

Dynamic Client Registration

POST /api/mcp/oauth/register, no auth, 10 registrations per hour per IP. Accepts a Zod-validated RFC 7591 request body:

POST /api/mcp/oauth/register
Content-Type: application/json

{
  "client_name": "Claude",
  "redirect_uris": ["https://claude.ai/api/mcp/callback"],
  "grant_types": ["authorization_code", "refresh_token"],
  "token_endpoint_auth_method": "none",
  "scope": "mcp"
}

Response:

201 Created
{
  "client_id": "b3b4e20f-6a2e-4f18-b3c8-....",
  "client_id_issued_at": 1744812345,
  "client_secret_expires_at": 0,
  "client_name": "Claude",
  "redirect_uris": ["https://claude.ai/api/mcp/callback"],
  "grant_types": ["authorization_code", "refresh_token"],
  "token_endpoint_auth_method": "none",
  "scope": "mcp"
}

We default to public clients (token_endpoint_auth_method: "none") because browser-based OAuth flows can't keep a client secret secret. If a confidential client registers with client_secret_post, we mint a secret and return it once in the response; the hash is stored for subsequent /token calls.

We do not restrict redirect_uris to specific hosts. Claude Desktop legitimately uses http://localhost:xxx, claude.ai web uses https://claude.ai/..., and we want to support other remote MCP clients (n8n, Composio, Warp) without hardcoding hosts. The security comes from exact-match validation at authorize time, not from a host allowlist.

Authorization & consent

The authorization endpoint is actually two pieces:

1. /mcp-authorize — the consent UI (page.tsx)

Renders server-side, does all the upfront validation:

  • Validates every OAuth param (client_id, redirect_uri, response_type=code, code_challenge, code_challenge_method=S256).
  • Looks up the client via getClient(). Shows an error page if unknown.
  • Checks client.redirectUris.includes(redirectUri). Exact match. Error if not.
  • Calls auth(). If no session, redirect()s to /login?callbackUrl=<encoded current URL>. NextAuth brings them back after login.
  • Enforces RBAC: ADMIN, SUPER_ADMIN, ORGANIZER only. Other roles see a clean "access denied" page.
  • Loads the user's organization for display, renders the consent card with client name + org name + scope.
  • The form is a plain HTML <form method="POST"> pointing at /api/mcp/oauth/authorize/decision with all OAuth params as hidden inputs plus a decision=approve|deny field.

2. /api/mcp/oauth/authorize/decision — the form POST handler

Re-validates everything server-side. Never trust form fields.

  • Re-looks-up the client and re-verifies redirect_uri match.
  • Re-runs the session + RBAC check.
  • On approve: calls issueAuthCode() which generates a random token, stores its hash in McpOAuthAuthCode with the PKCE challenge and 10-min TTL, then 302s to ${redirect_uri}?code=<raw code>&state=<state>.
  • On deny: 302s to ${redirect_uri}?error=access_denied&state=<state>.
  • Never logs the raw code — only the first 12 characters for debugging.

Token exchange & refresh

POST /api/mcp/oauth/token. Per OAuth 2.1, the request body should be application/x-www-form-urlencoded. Some clients send JSON — we accept both.

Authorization code grant

POST /api/mcp/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=mcp_ac_xxxxxxxxxxxxx
&code_verifier=yyyyyyyyyyy
&client_id=b3b4e20f-6a2e-4f18-b3c8-...
&redirect_uri=https%3A%2F%2Fclaude.ai%2Fapi%2Fmcp%2Fcallback

The exchange runs inside a single Prisma transaction so a code can never be used twice:

  1. Look up McpOAuthAuthCode by codeHash.
  2. Verify not expired, client_id matches, redirect_uri matches, PKCE verifies (via crypto.timingSafeEqual).
  3. db.$transaction([delete(code), create(accessToken)]).
  4. Return the raw access token + refresh token to the client.

Response:

{
  "access_token": "mcp_at_xxxxxxxxxxx",
  "token_type": "Bearer",
  "expires_in": 2592000,
  "refresh_token": "mcp_rt_yyyyyyyyyyy",
  "scope": "mcp"
}

Refresh token grant

POST /api/mcp/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=mcp_rt_yyyyyyyyyyy
&client_id=b3b4e20f-6a2e-4f18-b3c8-...

Refresh rotation:

  1. Look up McpOAuthAccessToken by refreshHash.
  2. Verify not revoked, client_id matches, refresh not past the 90-day window.
  3. db.$transaction([update(old, revokedAt: now), create(new accessToken)]).
  4. Return a fresh access token + a new refresh token.
The old refresh token is revoked but not deleted, so a replay can be detected (looks like a revoked row). Cleanup cron removes revoked rows past the grace window.

Revocation

POST /api/mcp/oauth/revoke. Per RFC 7009, this endpoint always returns 200 regardless of whether the token existed — this prevents attackers from using the endpoint to enumerate valid tokens.

POST /api/mcp/oauth/revoke
Content-Type: application/x-www-form-urlencoded

token=mcp_at_xxxxxxxxxxx
&token_type_hint=access_token

The server hashes the token and runs updateMany({ where: { OR: [{tokenHash}, {refreshHash}] }, data: { revokedAt: now } }). Works for both access and refresh tokens with a single call.

CORS

Every OAuth route and the main /api/mcp route has two things related to CORS:

  1. A dedicated OPTIONS export that calls handlePreflight(req) — returns 204 with the full set of CORS headers for the request's origin.
  2. Every response body is wrapped with withCors(req, res) — adds the same headers to 200/201/401/429/etc responses.

The allowed origins list is tight:

  • https://claude.ai (exact)
  • https://console.anthropic.com (exact)
  • *.claude.ai (subdomain suffix match)
  • *.anthropic.com (subdomain suffix match)
  • http://localhost:3000 / http://localhost:3001 / http://127.0.0.1:3000 — dev only (NODE_ENV !== "production")

Requests from any other origin get a response with a bare Vary: Origin header — no Access-Control-Allow-Origin, so the browser blocks.

Middleware bypass

The app's src/middleware.ts previously intercepted every OPTIONS /api/* preflight and returned 204 with a mobile-app-only CORS allowlist. That allowlist does not include claude.ai, so the middleware was silently swallowing every browser preflight before our route handlers could run.

The fix is three lines, very early in the middleware:

if (pathname.startsWith("/api/mcp")) {
  return NextResponse.next();
}

This lets every /api/mcp/* and /api/mcp/oauth/* request flow through to the route-level handlers, where OPTIONS is exported and handlePreflight() returns the correct headers. The mobile-app CORS is untouched for the rest of the API surface.

API key vs. OAuth: how both work

Claude Desktop, n8n, and anything using mcp-remote send a long-lived API key via x-api-key or Authorization: Bearer mmg_xxx. claude.ai web sends a short-lived OAuth Bearer token via Authorization: Bearer mcp_at_xxx. Both need to work.

The authenticate() function in /api/mcp/route.ts tries both paths:

async function authenticate(req) {
  const key = extractToken(req);          // x-api-key OR Authorization: Bearer
  if (!key) return null;

  // Path 1: existing ApiKey (backward compat)
  const apiKey = await validateApiKey(key);
  if (apiKey) return { organizationId: apiKey.organizationId, keyPrefix: key.slice(0, 12) };

  // Path 2: OAuth 2.1 access token (new)
  const oauth = await validateOAuthAccessToken(key);
  if (oauth) return { organizationId: oauth.organizationId, keyPrefix: "oauth-" + key.slice(0, 10) };

  return null;
}

Both paths return the same { organizationId, keyPrefix } shape. Downstream MCP tools (list_events, create_speaker, etc.) don't know or care which auth method was used — they just see an org ID and scope their queries to it.

Token prefixes make the two distinguishable in logs without exposing the full token:

  • API key: starts with mmg_ (as configured in the existing ApiKey flow)
  • OAuth access token: starts with mcp_at_
  • OAuth refresh token: starts with mcp_rt_
  • OAuth auth code: starts with mcp_ac_

Security model

ThreatMitigation
Replay of auth codeOne-time use enforced inside a transaction: delete(code) + create(token) run atomically. Second exchange attempt finds nothing.
Stolen auth code without code_verifierMandatory PKCE S256. Code can't be exchanged without the verifier.
Man-in-the-middle on redirectRedirect URI exact-match against registered URIs. No substring or prefix.
Token leak via logsNever log raw tokens. Only first 10–12 characters for debugging. Hashed storage means DB dumps don't leak live tokens.
Token leak via DBSHA-256 hashed storage, unique indexes only on hashes. Raw tokens never persisted.
Timing attack on PKCE verifiercrypto.timingSafeEqual().
Timing attack on client secretcrypto.timingSafeEqual() in verifyClientSecret().
Open redirector via redirect_uriExact-match validation against the registered URIs on BOTH the authorize page AND the decision route. Never trust form fields.
Privilege escalation via consentRBAC on the consent page: only ADMIN, SUPER_ADMIN, ORGANIZER can approve. Reviewers, submitters, registrants can't grant MCP access.
Cross-org data accessTokens are bound to the approving user's organizationId. MCP tools scope all queries to this org via the existing getOrgIdSecure() helper.
Refresh token replayRotation on every refresh: old row flipped to revokedAt, new row created. A replay sees the revoked row and fails.
Refresh token lifetime abuse90-day hard limit from createdAt. After that, user must re-auth.
DCR abuse10/hr/IP rate limit on /register.
Token endpoint abuse60/hr/client_id rate limit on /token.
Enumeration via revocationRevoke always returns 200 regardless of whether the token existed (RFC 7009 compliant).

Cleanup cron

GET|POST /api/cron/mcp-oauth-cleanup with Authorization: Bearer $CRON_SECRET. Runs hourly. Deletes:

  • McpOAuthAuthCode rows with expiresAt < now() — catches abandoned authorize flows.
  • McpOAuthAccessToken rows with expiresAt < now() - 7 days — 7-day grace period so clients can refresh before their row disappears.

EC2 crontab entry:

0 * * * * curl -s -X POST \
  -H "Authorization: Bearer $CRON_SECRET" \
  https://events.meetingmindsgroup.com/api/cron/mcp-oauth-cleanup

Troubleshooting

"Failed to connect" in claude.ai immediately after pasting the URL

  • Check GET /.well-known/oauth-protected-resource returns 200 JSON (not 404).
  • Check curl -sI https://events.meetingmindsgroup.com/api/mcp shows the WWW-Authenticate header.
  • Check browser DevTools → Network for the CORS preflight response; it should have Access-Control-Allow-Origin: https://claude.ai.

The consent popup opens but shows "Unknown client"

  • The DCR registration step failed. Check server logs for mcp-oauth:register-failed.
  • Make sure you haven't exceeded 10 registrations/hour from the same IP.

Approve button works but redirect lands on an error page

  • Check logs for mcp-oauth:redirect-uri-mismatch — the client's redirect_uri doesn't exactly match what was registered. This is normal for Claude Desktop's localhost flows (port numbers change) but should never happen for claude.ai web.
  • Check the code_verifier matches the code_challenge. PKCE mismatches log as mcp-oauth:code-exchange-failed with error: invalid_grant.

Tools work once but fail after 30 days

Access token expired. The client should automatically POST /api/mcp/oauth/token with grant_type=refresh_token. If refresh itself fails, the user has to re-authorize. The refresh token has a 90-day hard limit.

"Server not initialized" error on curl tests

This is not an OAuth issue — it's a missing clientInfo in your initialize request body. See MCP_REFERENCE.md for a working curl example.
