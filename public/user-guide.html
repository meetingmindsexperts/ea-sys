Getting Started
Logging in, finding your event, reading the dashboard.
All roles
Logging in

Open the system URL and sign in with the email and password your organisation issued. Forgotten password? Use the “Forgot password?” link — you’ll receive a reset link by email within a minute.

Dashboard overview

After login you land on the main dashboard. It shows the events you have access to and your recent activity. Different roles see different things here — admins see all events in the org, reviewers see only events they’re assigned to.

Finding and opening your event

The Events page lists every event in your organisation. Click any row to open that event’s workspace.

Reading the event landing page

Opening an event takes you to its landing page. Four stat tiles at the top (Registrations, Speakers, Sessions, Hotels), quick actions for what’s urgent, and a live activity feed for the event.

Conference vs. Webinar — what’s different
ℹ️
Event type affects available features
Webinar events get extra tools: an auto-provisioned Zoom webinar, attendance tracking, engagement reports (polls + Q&A), and a dedicated Webinar Console. Conference events get fuller accommodation management, day-of check-in, and badge printing. Hybrid gets both. See Section 11 for webinar-specific guidance.

Speakers & KOLs
Adding, managing, briefing speakers; agreements; day-of swaps.
Speaker & KOL Management · Event Coordination
Speakers list

The speakers tab lists every faculty member added to the event. Each row shows the speaker’s name, title, organisation, email, and current status. Filter by status, search by name or email, or bulk-select to email a group.

Adding a speaker
Speakers tab → Add Speaker
Fill in Title, First name, Last name, Email
Add optional details: Organisation, Job Title, Specialty, Bio, Photo
Set initial Status — default is INVITED
Click Save
✅
Import existing attendees as speakers
If a speaker has already registered as an attendee, use Import from Registrations rather than adding manually — preserves the link between their Registration and Speaker rows.
Speaker statuses
StatusMeaningNext action
INVITEDAdded; invitation sent or pendingSend invitation email; chase for confirmation
CONFIRMEDSpeaker has acceptedAssign to session; send agreement
DECLINEDSpeaker has declinedFind replacement; update programme
CANCELLEDWas confirmed but has cancelledDay-of swap (see below)
Speaker agreements

The system supports two routes for the speaker agreement — pick whichever fits your event.

Option A — Inline HTML (default)
Edit the agreement text once in Event → Content → Speaker Agreement. The system generates a personalised PDF for each speaker at send time. Default content is the MMG Invited Faculty Participation Agreement.
Option B — Upload .docx template
Upload a Word template with {{token}} merge fields in Event Settings → Email Branding. The system mail-merges per-speaker values and attaches the personalised .docx.
Set up the agreement once (either route above)
From the speakers list: bulk-select confirmed speakers → Send Agreement
Each speaker receives an email with the personalised agreement attached and a link to accept online
Track signatures from the speakers list — the agreement column shows Signed / Unsigned / Not sent
⚠️
Chase unsigned agreements 2 weeks before the event
Filter the speakers list by “Unsigned”, bulk-select, and use the agreement-reminder email template. The system attaches a fresh copy of the personalised agreement.
Assigning speakers to sessions with roles

Open a session detail page to assign speakers. Each speaker can take one of four roles per session:

Speaker
Delivering a presentation in the session
Moderator
Facilitating the session — managing time and Q&A
Chairperson
Leading the session; academic chair
Panelist
Panel discussion participant

You can also assign speakers to specific topics within a session (e.g. one speaker per 15-minute topic block in a 60-minute session).

Day-of speaker swap

When a speaker cancels on the day of the event, you don’t need to delete and recreate sessions. The system preserves the session ID, topics, and delegate registrations during a speaker swap.

Open the session → Speakers panel
Click Remove next to the cancelling speaker
Click Add Speaker → search for the replacement
Assign the same role
✅
Replacing multiple speakers at once
For a full swap of multiple speakers, use Replace All Speakers — define the entire new speaker list for a session in one atomic step. No half-state where the session has neither the old nor the new speakers.

Registration & Ticketing
Ticket types & pricing tiers, registering attendees, statuses, CSV import, day-of check-in.
Delegate Support · Event Coordination · Finance
Registration types & pricing tiers

A registration type defines an attendee category — Physician, Allied Health, Student, Resident, Member, Complimentary. Each type can have pricing tiers (Early Bird, Standard, Onsite) with their own prices and sales windows.

⚠️
Ticket pricing is set in the dashboard only
New ticket types start at $0 with inactive pricing tiers. To set actual prices, edit each tier and activate it. AI agents cannot set or change pricing — intentional safety boundary.
Registrations list
Registering an attendee manually
Registrations tab → Add Registration
Fill in the attendee details (Title, First name, Last name, Email, Organisation)
Pick a Registration Type
If the type has tiers, pick a Pricing Tier (Early Bird / Standard / Onsite). This drives reporting.
Set Payment Status: UNASSIGNED for paid tickets pending follow-up, UNPAID if the registrant should self-pay via Stripe, PAID if already settled, COMPLIMENTARY for VIP/no-charge, INCLUSIVE for sponsor-paid (see Section 5).
Click Register. The system generates a QR code, serial ID, and (for outstanding-payment registrations) sends a confirmation email with quote PDF attached.
Public registration page

Every event has a public registration URL: /e/[event-slug]/register. Share this with prospective attendees. The page redirects to the first active pricing tier automatically.

Payment statuses
StatusMeaning
UNASSIGNEDAdmin-created, payment pending (no Stripe expected yet)
UNPAIDSelf-registered via the public form, Stripe payment due

ℹ️
Stripe-driven states are webhook-owned
PENDING, REFUNDED, and FAILED are set by the Stripe webhook automatically. Admins cannot set these manually — protects against accidental drift between Stripe and the system.
Bulk import from CSV

Got a list of registrants from a sponsor or partner? Import them in one shot. Required columns: email, firstName, lastName. Optional columns include organization, jobTitle, phone, title, specialty, registrationType, tags, plus three newer columns for finer control:

registrationStatus
PENDING / CONFIRMED / WAITLISTED / CANCELLED. Defaults to CONFIRMED.
paymentStatus
UNASSIGNED / UNPAID / PAID / COMPLIMENTARY / INCLUSIVE. Defaults match the “Add” form.
sponsor
Sponsor name (matched against the event’s sponsor list). Required when paymentStatus=INCLUSIVE.
✅
Existing templates keep working
Templates without the three new columns continue to import — they fall back to the prior defaults (CONFIRMED status, UNASSIGNED/COMPLIMENTARY payment based on price, no sponsor).
Bulk actions on the list

Tick the checkboxes on multiple registrations to enable the bulk-action bar at the top of the list:

Change type
Move selected registrations to a different registration type. Auto-adjusts soldCount on both old and new types.
Add / remove tags
Useful for grouping VIPs, dietary requirements, etc.
Send email
Opens the bulk-email dialog pre-filtered to the selected registrations.
Export CSV
Downloads all (or selected) registrations with the full attendee + tier + payment data.

Sponsor-Paid Registrations
Tagging registrations to a sponsor when the sponsor pays a lump sum out-of-band.
Delegate Support · Sponsorship & BD · Finance
When to use INCLUSIVE

When a sponsor (e.g. Abbott, Pfizer) contracts for a block of N attendees and pays MMG directly — either by wire transfer or per the sponsorship agreement — those attendees don’t owe anything individually. Tag their registrations as INCLUSIVE so:

  • Payment-reminder emails skip them (they don’t owe)
  • The sponsor attribution is visible in reports (“87 attendees were brought in by Abbott”)
  • Check-in works without a payment check
  • Finance can reconcile against the sponsor invoice rather than per-attendee
💡
Why not just use COMPLIMENTARY?
COMPLIMENTARY means “no money changes hands at all” (VIPs, speakers, in-kind). INCLUSIVE means “the sponsor paid” — money DID change hands, just not from the attendee. Same UI behaviour (no chase emails, check-in OK) but separate for reporting and reconciliation.
Workflow prerequisite — add the sponsor first

Before you can tag registrations as INCLUSIVE, the sponsor must exist in the event’s sponsor list:

Event → Sponsors → Add Sponsor
Enter the sponsor’s name (use the exact full name — the CSV import resolves by name)
Optionally upload a logo, set a tier (Platinum / Gold / Silver / Bronze / Partner / Exhibitor), add a website URL
Save. The sponsor is now available in the tier dropdown when editing registrations and in the CSV import sponsor column.
Importing a sponsor’s attendee list (CSV)

The recommended flow when a sponsor sends you their attendee list:

Open the sponsor’s list in a spreadsheet. Make sure you have email, firstName, lastName columns.
Add three columns: registrationStatus = CONFIRMED, paymentStatus = INCLUSIVE, sponsor = the sponsor’s exact name (e.g. “Abbott”)
Optionally add a registrationType column — pick which type the sponsor’s attendees should land in (Physician, Allied Health, etc.)
Save as CSV. Registrations tab → Import CSV → pick your file
Preview the first 5 rows. Click Import. The system creates all registrations as INCLUSIVE + tagged to your sponsor in one transaction.
ℹ️
Mixing sponsors in one CSV
You can mix sponsors in a single import — just put different sponsor names in the sponsor column per row. Most workflows are one-sponsor-per-CSV (Abbott’s 100 separately from Pfizer’s 50), which is simplest.
Tagging a single registration manually

For one-off cases, open the registration detail sheet:

Open the registration. Click Edit.
Change Payment Status to INCLUSIVE.
A Sponsored by dropdown appears below. Pick the sponsor.
Save. The Payment Summary now shows a violet “Sponsored by: Abbott” pill.
✅
Sponsor attribution survives status flips
If you later flip the payment status away from INCLUSIVE (e.g. to PAID), the sponsor tag is preserved as historical context. The only way to clear the sponsor is to explicitly remove it.
Reporting on sponsor-paid attendees

Finance and BD typically need two views:

By sponsor
Filter the registrations list by Payment Status = INCLUSIVE and Sponsor = Abbott. Export to CSV for reconciliation against the sponsor invoice.
Total inclusive count
The Payment Status filter on the registrations list shows the count alongside each status — one-click view of all INCLUSIVE attendees across all sponsors.
⚠️
Communicating with sponsor-paid attendees
When sending bulk emails to INCLUSIVE attendees (e.g. pre-event briefing), filter by Payment Status = INCLUSIVE in the bulk-email dialog. Sponsor-paid attendees still get registration confirmations and event reminders — they just don’t get payment-chase emails.

Adding hotels & room types

Setting up review criteria

Before reviews start, set up scoring criteria. Each criterion has a name and a weight (1–10). Examples: Scientific Novelty (weight 8), Clinical Relevance (weight 9), Presentation Quality (weight 7). Reviewers score each criterion 0–10; the system computes a weighted overall score automatically.

Configure criteria in Event Settings → Review Criteria.

Assigning reviewers

Reviewers are org-independent — the same reviewer can review across multiple events without being part of your organisation. Per-abstract assignment is optional; reviewers in the event pool can pick up any abstract.

Reviewing & deciding on abstracts

Once you have at least the required number of reviews (set per-event in Event Settings, default 1), the status action buttons become active. Pick Accept, Reject, or Request Revision. The author is notified by email automatically with the aggregated reviewer notes.

Chair override
⚠️
Force a decision before all reviews are in
When an urgent programme decision is needed before the required review count is met, use Force Decision. Every override is logged with timestamp, the chair’s identity, and the prior review count — visible in the audit trail.
Reviewer self-service portal

Reviewers have their own portal at /my-reviews that lists every abstract assigned to them across all events. They submit scores from there. See Section 13.

Finance & Invoicing
Invoice numbering, types, viewing & downloading, manual payment capture, refunds.
Finance & Operations · Leadership
How invoice numbers work

Invoice numbers are generated automatically using the event code. Format: [EVENT_CODE]-INV-001, [EVENT_CODE]-INV-002, etc. Event codes are derived from the event name (“Heart Failure Forum 2026” → HFF2026). Receipts use the same prefix: HFF2026-REC-001. Credit notes: HFF2026-CN-001.

ℹ️
Quote vs. Invoice
When a registration is created (admin or public) with outstanding payment, the system generates a Quote PDF and emails it. When the registrant pays via Stripe, the quote is promoted in place to a status-PAID Invoice — no duplicate numbers, the same record flows through both stages.
Invoice types
Quote
Generated at registration time when payment is outstanding. The payment request sent to the registrant.
Invoice (PAID)
Generated automatically when Stripe payment succeeds (promotes the existing Quote) or when finance manually records a payment. PDF includes “Paid on X via Visa ending 4242” line.
Credit Note
Issued for refunds or partial credits. Created alongside the refund transaction.
Viewing & downloading invoices
Manual payment capture (bank transfer, cash, card on-site)

For onsite events and bank-transfer payments where Stripe isn’t involved, finance records the payment manually:

Open the registration. Billing & Payments tab → Record Payment
Pick a Method: Bank Transfer, Card On-Site, or Cash
Fill the method-specific fields:
  • Bank Transfer: bank reference + upload receipt URL (optional)
  • Card On-Site: card last-4 digits + receipt number
  • Cash: receiver name + receipt number
Set amount + currency (12 supported, defaults to USD). Add notes if needed.
Submit. The system creates a Payment record, flips the registration to PAID, and emails an Invoice (status PAID) with PDF attached. Same artefacts as a Stripe payment.
✅
Recovery path for legacy registrations
If a registration was marked PAID manually before this feature existed (no Payment row), you can still record the payment retroactively — the system detects the missing Payment record and allows the capture without 409’ing.
Promo codes
⚠️
Delete is soft-delete
Deleting a promo code marks it inactive — it stops working for new registrations but existing registrations that used it are not affected. Reactivate any time.
Chasing unpaid registrations

The communications hub (Section 9) lets you filter recipients by payment status. To send a payment reminder to everyone who owes money:

Communications → Send Email → recipient type = Registrations
Set Payment Status Filter = UNPAID (or PENDING for in-flight Stripe sessions)
Pick the “Payment Reminder” template, review the recipient count, send.
ℹ️
PAID, COMPLIMENTARY, and INCLUSIVE are auto-excluded
The system never chases people who don’t owe money — PAID, COMPLIMENTARY, INCLUSIVE, and REFUNDED statuses are filtered out automatically from payment-reminder bulk emails.
Refunds
🔴
Refunds must be processed via the dashboard
Open the registration → click Refund. This triggers the actual Stripe refund, updates payment status to REFUNDED, creates a credit note, and emails the registrant — all in one step. AI agents cannot initiate refunds. Intentional safety boundary.

Custom Notification
General-purpose email
Customising a template

Each template can be customised per-event without affecting the default. Event Settings → Email Templates → pick a template → edit subject + body using the WYSIWYG editor. Template variables like {{eventName}}, {{attendeeName}}, {{quoteAmount}} are filled in at send time.

Sending a bulk email
Communications → Send Email
Recipient type: Registrations, Speakers, or Abstracts
For registrations: optionally set the Status Filter (CONFIRMED only, etc.) and the Payment Status Filter (UNPAID only, etc.)
Pick a template or compose freely. Subject + body support variables.
Review the recipient count. Send now, or pick a date/time to schedule for later.
⚠️
Rate limit: 10 bulk sends per event per hour
Plan campaigns accordingly. Avoid sending back-to-back in quick succession — you’ll get a 429 with a retry-after time.
Scheduled emails

Future-dated sends. The bulk email dialog has a “Send now / Schedule for later” toggle. Scheduled emails appear in the Scheduled tab where you can edit, cancel, or retry failed sends. Recipient filters are re-evaluated at fire time — so a “remind unpaid” scheduled email naturally excludes anyone who paid in the meantime.

Footer image

Webinar Features
Webinar Console, attendance tracking, engagement reports, recordings.
Virtual Events & IT · Medical Education
When you create a WEBINAR-type event

The system auto-provisions everything you need: an anchor session, a Zoom webinar via your org’s Zoom credentials, a 5-stage email sequence (confirmation, reminders at 24h/1h, “Live now”, post-event thank-you). The sidebar swaps out Accommodation / Check-In / Promo Codes / Abstracts and adds the Webinar Console.

Webinar Console
Attendance tracking

Attendance is updated every 10 minutes for 24 hours post-event. Export to CSV for CME reporting.

Engagement — polls & Q&A
Recording & replay

Cloud recordings are fetched automatically every 5 minutes after the webinar ends. Status flow: NOT_REQUESTED → PENDING → AVAILABLE (typically within 30 minutes). When AVAILABLE, the public session page displays a “Watch Replay” button and the post-event thank-you email surfaces it too.

⚠️
Recordings expire
Zoom retains cloud recordings for 7 days by default. After that the status flips to EXPIRED and the replay button disappears. Download from the Webinar Console while the recording is AVAILABLE if you need a long-term archive.

Day-of Check-in
QR scanning, manual search, badge printing, walk-up handling.
On-site team · Event Coordination
Check-in scanner
ℹ️
Hardware scanners are supported
Bluetooth/USB barcode scanners work in the manual search field — just keep the field focused. The system has a 2-second debounce to prevent double-scans.
Handling cancelled registrations (walk-ups)
Option A — Reinstate First
Open the registration. Change status from CANCELLED to CONFIRMED. Then check in normally. Correct if the cancellation was an error.
Option B — Walk-up Override
Use the Allow Cancelled Check-in override in the scanner. Logged in the audit trail. Use for genuine walk-ups where reinstating first is impractical.
Badge printing

From the registrations list, select rows and click Print Badges. Opens a PDF in a new tab; the browser’s print dialog handles the rest. One badge per A4 page, horizontally centred. Badge layout: name, country, barcode, badge type (large), registration number. Adjust the vertical offset in Event Settings → General if your physical badge stock needs alignment.

Editing Conflicts & Email Changes
What happens when two people edit the same record — or when you need to change an email.
All roles
Concurrent edits — the “stale write” error

The system uses optimistic locking on registrations, speakers, sessions, accommodations, and abstracts. If you opened a record at 10:00, someone else edited and saved at 10:02, and you try to save your changes at 10:03, you’ll see a clear error:

⚠️
“This record was updated by someone else. Refresh and retry.”
Click Refresh to pull the latest version. Re-apply your changes. Save again. The system would rather warn you than silently overwrite a colleague’s edit.
Changing a registrant’s or speaker’s email

Email addresses are immutable via the regular edit form. Changing them silently would split identity across the Registration / Attendee / User / Contact tables — finance reconciliation gets messy. Use the dedicated Change Email dialog:

Open the speaker/registration/contact detail. The email field is read-only.
Click Change next to the email field.
Enter the new email + confirm. The system checks for collisions across all related tables.
On success, the email is updated everywhere atomically: Speaker/Registration/Attendee/User/Contact all flip together. Audit-logged.

Known Limitations
Intentional boundaries and features not yet built.
Reference

Your AI agents are aware of all of these and will direct you to the right alternative when they apply.

⛔
Ticket pricing — dashboard onlySet prices in Event Settings → Registration Types. AI agents cannot change prices.
⛔
Refunds — dashboard + Stripe onlyRefunds must be initiated from the registration detail sheet. AI agents cannot issue Stripe refunds. Deliberate safety boundary.
⛔
Media upload — dashboard onlyAll file uploads happen via the Media Library. Agents can reference existing files but not upload new ones.
⛔
Sponsor list — replace-all, not mergeUpdating sponsors via the API replaces the entire list. The dashboard sponsors page handles add/remove individually — use that if you want incremental changes.
⛔
Email address changes — dedicated flowThe regular edit form blocks email changes. Use the Change Email dialog (Section 16).
⛔
Event slug, dates, timezone, type — not editable via APIThese are dashboard-only because changing them cascades to public URLs, scheduled emails, and Zoom webinar provisioning.
⚠️
Webinar recordings expire after 7 daysZoom’s default retention. Download from the Webinar Console while available if you need a long-term archive.
⚠️
claude.ai integration tool list is cachedWhen new MCP tools are added to the system, you need to disconnect and re-add the EA-SYS integration in claude.ai for the new tools to appear. Spec-required behaviour; not a bug.
⚠️
Single-organisation modeThe system supports one organisation at a time. Multi-org will land in a future release.
