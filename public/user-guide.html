Getting Started
Logging in, finding your event, reading the dashboard.
All roles
Logging in

Open the system URL and sign in with the email and password your organisation issued. Forgotten password? Use the “Forgot password?” link — you’ll receive a reset link by email within a minute.

Dashboard overview

After login you land on the main dashboard. It shows the events you have access to and your recent activity. Different roles see different things here — admins see all events in the org, reviewers see only events they’re assigned to.

Finding and opening your event

The Events page lists every event in your organisation. Click any row to open that event’s workspace.

Reading the event landing page

Opening an event takes you to its landing page. Four stat tiles at the top (Registrations, Speakers, Sessions, Hotels), quick actions for what’s urgent, and a live activity feed for the event.

Conference vs. Webinar — what’s different
ℹ️
Event type affects available features
Webinar events get extra tools: an auto-provisioned Zoom webinar, attendance tracking, engagement reports (polls + Q&A), and a dedicated Webinar Console. Conference events get fuller accommodation management, day-of check-in, and badge printing. Hybrid gets both. See Section 11 for webinar-specific guidance.
✨ Crown jewel · Available in every event

The MMG AI Agent

Run your event in plain English. Type what you want done — add a speaker, send a payment reminder to everyone unpaid under the Standard ticket, summarise registrations for next week — and the agent does it, with every action logged. Open any event → AI Agent in the sidebar.

What you can ask the agent to do

The agent has access to most read + create operations across the system. You don’t need to learn syntax — phrase requests the way you’d ask a colleague.

Add a speaker: Dr. Khaled Al Qawasmeh, kawasmeh@tawamhospital.ae, organisation Tawam Hospital. Set status to CONFIRMED.
Send a payment reminder to all unpaid CONFIRMED registrations under the Standard ticket type.
Create a session called “Opening Keynote”, start 9 AM Dubai time, end 10 AM, track Clinical, location Main Hall.
Show me a registration summary for the next 14 days — group by ticket type and payment status.
List the abstracts that are UNDER_REVIEW and have fewer than two reviewer submissions.
Import these 12 sponsor attendees as INCLUSIVE registrations tagged to Abbott. Here’s the list…
Mark Dr. Khaled’s registration as checked-in.
Cancel registration #042 and notify the attendee.
What the agent can and can’t do
✅ The agent CAN
  • Add speakers, sessions, registrations, accommodations, abstracts, contacts, promo codes
  • Update statuses, assign reviewers, change ticket types, swap session speakers
  • Send bulk emails with status + payment-status filters
  • Read every list and detail in the system
  • Tag INCLUSIVE registrations to sponsors (with the sponsor already in your sponsor list)
  • Run multiple operations in sequence (“create the speaker, then send the agreement”)
⛔ The agent CANNOT
  • Issue Stripe refunds — dashboard-only with explicit consent
  • Delete records hard — only soft-delete equivalents (CANCELLED, isActive=false)
  • Change ticket pricing or pricing-tier prices
  • Upload media files (it can reference existing media in the library)
  • Modify event slug, dates, timezone, or event type
  • Change someone’s email address (use the Change Email dialog — Section 16)
  • Issue INTERNAL-tier API keys
Limits & safety
Rate limit
20 agent requests per hour per user. Heavy bulk work? Use CSV import — faster and better error reports.
Per-request create cap
Up to 20 create operations per single prompt, so “create 100 speakers” won’t go through — intentional brake. Use CSV import for big batches.
Audit trail
Every action the agent performs is logged with source: “agent” — visible in /logs and on each record’s audit history.
Role-gated
ADMIN, ORGANIZER, and SUPER_ADMIN get full agent access. MEMBER gets read-only agent access (lookups + reporting; write tools refused). Reviewers, submitters, and registrants can’t use it.
How a conversation works

The agent streams its reasoning as it works — you can see what it’s about to do before it acts. For larger or potentially-destructive operations, it asks for confirmation. Multi-step asks (“create the speaker, then send the agreement”) chain through automatically without you having to send each one separately.

✅
Be specific for best results
“Send a payment reminder to UNPAID, CONFIRMED registrations under the Standard ticket” works better than “send a reminder”. The agent will ask for clarification if it’s ambiguous, but specifying filters up front saves a round-trip.
Same tools, external access

The exact same tool set is also exposed externally for power users + integrations:

claude.ai web
Add EA-SYS as a connector in claude.ai Settings → Integrations. Goes through OAuth — one click consent per organisation.
Claude Desktop
Install the EA-SYS MCP via mcp-remote with your API key. See Settings → API Keys for setup.

ℹ️
When to use which
In-app agent is best for quick day-to-day work. External MCP (n8n, claude.ai web) is best for workflows that span MMG + other systems (e.g. “when a new lead comes into HubSpot, create a registration”). Same backend, same audit trail, same rate-limit tier model.

Speaker & KOL Management · Event Coordination
Speakers list

The speakers tab lists every faculty member added to the event. Each row shows the speaker’s name, title, organisation, email, and current status. Filter by status, search by name or email, or bulk-select to email a group.

Who to add where — speakers, committee, moderators

The same person can wear several hats, and the right place to add them depends on one question: are they presenting in any way? Anyone with a session role (speaker, moderator, chairperson, panelist) is faculty — add them under Speakers. Someone who is only on a committee (not presenting) is added under Registrations. “Committee” itself is never a separate person — it is just a committee tag (plus a Committee badge) layered on top of whichever record they already have.

The person is…Add underThen do this
A speaker / presenter (even if not yet confirmed) Speakers tab → Add Speaker Set status INVITED (move to CONFIRMED once they accept). Their Faculty registration — badge, barcode, check-in, survey, certificate — is created automatically.
A moderator (or chairperson / panelist) Speakers tab → Add Speaker A moderator is a session role, not a separate kind of person. Add them as a speaker, then open the session and assign them the Moderator role.
Committee only (not presenting) Registrations tab → Add Registration Set payment Complimentary, badge type Committee (or Organizing / Scientific Committee), and add the tag committee (plus committee-organizing / committee-scientific if you split them). That comp registration gives them badge, check-in, and certificate.
Committee and moderator Speakers tab → Add Speaker Add them as a speaker and assign the Moderator session role, then add the committee tag on their (auto-created) registration and set its badge to Committee if you want that printed. One person, both hats.
Committee and speaker Speakers tab → Add Speaker Add them as a speaker as usual, then add the committee tag on their auto-created registration (and set the badge to Committee if desired). Do not create a second registration for them.
✅
Rule of thumb
Presenting in any way → Speakers. Committee-only → Registrations. Committee is always just a committee tag added on top — never a separate person or a ticket type — so a committee member who is also faculty stays one record and shows up wherever you filter by committee. Setting the tag and the Committee badge is a manual step today (in the registration’s Tags and Badge Type fields).
Adding a speaker
Speakers tab → Add Speaker
Fill in Title, First name, Last name, Email
Add optional details: Organisation, Job Title, Specialty, Bio, Photo
Set initial Status — default is INVITED
Click Save
✅
Import existing attendees as speakers
If a speaker has already registered as an attendee, use Import from Registrations rather than adding manually — preserves the link between their Registration and Speaker rows.
✅
Every speaker is automatically attend-ready
When you add a speaker — manually, by CSV, or by import — the system automatically creates a linked Faculty registration for them behind the scenes. So every speaker gets a badge, an entry barcode, a DTCM barcode, check-in, and the post-event survey, exactly like a delegate — nothing extra to do. Faculty appear in both the Speakers list and the Registrations list — with their profession as the registration type (Physician, Nurse, etc., taken from the speaker’s details) and a Faculty badge marking them as faculty. (“Faculty” is a badge/role, not a registration type, so it never appears in the type column.) They are complimentary and are excluded from your delegate counts and don’t take a paid seat — so your “Registrations” tile stays a true delegate number. If a speaker also registered separately as a paying delegate, that real registration still counts as a delegate. Where to see it: open any speaker’s detail page and look at the Registration card — it shows their registration number, badge type, whether their entry barcode is issued, and their check-in and survey status, so you don’t have to dig through the Registrations list. (The same summary appears in the speaker quick-view slide-out from the Speakers list.)

Speaker agreements

The system supports two routes for the speaker agreement — pick whichever fits your event.

Option A — Inline HTML (default)
Edit the agreement text once in Event → Content → Speaker Agreement. The system generates a personalised PDF for each speaker at send time. Default content is the MMG Invited Faculty Participation Agreement.
Option B — Upload .docx template
Upload a Word template with {{token}} merge fields in Event Settings → Email Branding. The system mail-merges per-speaker values and attaches the personalised .docx.
Set up the agreement once (either route above)
From the speakers list: bulk-select confirmed speakers → Send Agreement
Each speaker receives an email with the personalised agreement attached and a link to accept online
Track signatures from the speakers list — the agreement column shows Signed / Unsigned / Not sent
⚠️
Chase unsigned agreements 2 weeks before the event
Filter the speakers list by “Unsigned”, bulk-select, and use the agreement-reminder email template. The system attaches a fresh copy of the personalised agreement.
📎
Sending the agreement with OTHER emails — two tokens, and why a customised invitation may not include it (July 2026)
Besides the dedicated Send Agreement action (which always attaches), the agreement can ride along on any speaker email — the invitation, a custom send — but only when the template opts in with a token:

{{agreementBlock}} — renders a one-liner + a Review & Agree button AND attaches the personalised agreement for speakers who haven’t signed yet. Speakers who already signed get a green “already accepted” note instead (no re-ask, no attachment).
{{agreementAttachment}} — invisible marker: attaches the personalised PDF/.docx only, with no button and no online link. Use this when speakers should print or e-sign the attached PDF (Adobe Fill & Sign) and return it — upload the returned copy on their Documents card.

⚠️ The gotcha: the default Speaker Invitation template already contains {{agreementBlock}} — but if you customised your invitation template and it contains neither token, invitations send without any agreement link or PDF, no matter what. If “the agreement PDF isn’t being sent”, check your template for one of these tokens first — add whichever behaviour you want and re-send.

Revoking an acceptance: the Revoke button on the speaker’s page clears their acceptance record — the speaker counts as unsigned again, so the next send with a token (or a strict Send Agreement) attaches a fresh personalised copy and re-asks. Revoking alone does not send anything.
Assigning speakers to sessions with roles

Open a session detail page to assign speakers. Each speaker can take one of four roles per session:

Speaker
Delivering a presentation in the session
Moderator
Facilitating the session — managing time and Q&A
Chairperson
Leading the session; academic chair
Panelist
Panel discussion participant

You can also assign speakers to specific topics within a session (e.g. one speaker per 15-minute topic block in a 60-minute session).

📅
Briefing emails: {{presentationDetails}} for speakers, {{moderatorDetails}} for moderators (July 2026)
Two per-recipient variables build each person’s schedule into any speaker email (templates, the Brief Moderators tile, or tokens typed straight into the message box):

{{presentationDetails}} — the recipient’s own schedule table: their session(s), their assigned topics each with its own start–end time and length (computed from the topic order and durations you set on the session), the session’s date, start–end time and duration on three separate lines, and the track. The role is deliberately not shown — you send to moderators and speakers separately, so the email never needs to say which they are.
{{moderatorDetails}} — for moderators: every session they moderate with the full run-sheet — a Time · Topic · Speaker(s) · Duration table where each topic’s start–end is computed from the session start plus the topic lengths. Renders nothing for non-moderators, so it’s safe to include in any speaker template. All times show in the event’s timezone.

Tip: give every topic a duration on the session editor — topics without one show “—” instead of times.
Day-of speaker swap

When a speaker cancels on the day of the event, you don’t need to delete and recreate sessions. The system preserves the session ID, topics, and delegate registrations during a speaker swap.

Open the session → Speakers panel

Replacing multiple speakers at once
For a full swap of multiple speakers, use Replace All Speakers — define the entire new speaker list for a session in one atomic step. No half-state where the session has neither the old nor the new speakers.

Registration types & pricing tiers

A registration type defines an attendee category — Physician, Allied Health, Student, Resident, Member, Complimentary. Each type can have pricing tiers (Early Bird, Standard, Onsite) with their own prices and sales windows.

⚠️
Ticket pricing is set in the dashboard only
New ticket types start at $0 with inactive pricing tiers. To set actual prices, edit each tier and activate it. AI agents cannot set or change pricing — intentional safety boundary.
Registrations list
Registering an attendee manually
Registrations tab → Add Registration
Fill in the attendee details (Title, First name, Last name, Email, Organisation)
Pick a Registration Type
If the type has tiers, pick a Pricing Tier (Early Bird / Standard / Onsite). This drives reporting.
Set Payment Status: UNASSIGNED for paid tickets pending follow-up, UNPAID if the registrant should self-pay via Stripe, PAID if already settled, COMPLIMENTARY for VIP/no-charge, INCLUSIVE for sponsor-paid (see Section 5).
(Optional) Charge to another account — if a hospital, company, or grant is paying instead of the attendee, pick a payer under Charge to (or click + Add a new payer to create one inline). Full details in Finance & Invoicing → Charging to a third-party payer.
Click Register. The system generates a QR code, serial ID, and (for outstanding-payment registrations) sends a confirmation email with quote PDF attached.
Public registration page

Every event has a public registration URL: /e/[event-slug]/register. Share this with prospective attendees. The page redirects to the first active pricing tier automatically.

Editing a registration — Edit, then Save

Open any registration to see its detail panel. To change anything, click Edit, make your changes, then click Save — nothing is saved until you click Save. This covers the attendee's personal details, Registration Type, Badge Type, Attendance, Registration Status, Payment Status, Sponsor, and the billing/payer fields, all committed together in one Save. While you're just viewing (not editing), these fields are read-only, so an accidental click never changes anything.

ℹ️
One exception: the Pricing Tier has its own "Apply" button
On the Billing & Payments tab, the Pricing Tier picker (used to give an unpaid registrant a courtesy rate, e.g. Early Bird) is committed by its own Apply button right there — you don't need to enter Edit mode for it. The promo code Apply/Remove buttons next to it work the same way. Everything else on the registration is Edit → Save.

ℹ️
When does a registration become PENDING? — approval-gated tiers
A registration is PENDING only when it’s created on a pricing tier that has Requires Approval switched on. That flag turns off the normal auto-confirm, so instead of landing as CONFIRMED the sign-up waits for you to review it — useful for member-only rates, VIP/faculty seats, or any ticket you want to vet before accepting. It applies to every sign-up path (public form, CSV import, EventsAir import, manual add).

Where to find the switch: open Tickets → on a registration type, add or edit a Pricing Tier → in the tier dialog, tick “Requires Approval” (just below Active). It lives inside the tier editor, not on the ticket-type form — so if a type has no pricing tiers, there’s nowhere to enable it yet. It’s off by default: leave it unticked and every registration is CONFIRMED instantly.

To approve a PENDING registration: open it, and change the Status from PENDING to CONFIRMED (or CANCELLED to turn them away). There’s no separate “Approve” button — the status change is the approval.

In-person vs virtual attendance (Hybrid events)

On Hybrid events, each registration is either In-person or Virtual. A virtual attendee gets no entry barcode or badge, does not consume a venue seat (so a sold-out room still accepts virtual signups), and is priced by the ticket type’s virtual price.

Where to change it. Open the registration → on the detail sheet’s Details tab, use the Attendance dropdown (shown on Hybrid events only) to switch between In-person and Virtual:

Virtual → In-person: an entry barcode is issued automatically and the registration claims a venue seat. If that ticket type is sold out, the switch is blocked and the registration stays virtual — nobody is stranded.
In-person → Virtual: the venue seat is released and badge / check-in are suppressed. The existing barcode is kept for audit.
⚠️
Switching attendance does not re-price the registration
The Attendance toggle moves the venue seat and the barcode, but it does not recalculate the amount owed (in-person price vs virtual price). If the price should change, adjust the payment separately on the Billing & Payments tab.

Sponsor-Paid Registrations
Tagging registrations to a sponsor when the sponsor pays a lump sum out-of-band.
Delegate Support · Sponsorship & BD · Finance
When to use INCLUSIVE

When a sponsor (e.g. Abbott, Pfizer) contracts for a block of N attendees and pays MMG directly — either by wire transfer or per the sponsorship agreement — those attendees don’t owe anything individually. Tag their registrations as INCLUSIVE so:

  • Payment-reminder emails skip them (they don’t owe)
  • The sponsor attribution is visible in reports (“87 attendees were brought in by Abbott”)
  • Check-in works without a payment check
  • Finance can reconcile against the sponsor invoice rather than per-attendee
✅
Sponsor-paid attendees can’t be double-charged (July 10, 2026)
An INCLUSIVE registrant’s portal shows “Sponsored registration — no payment required” instead of a Pay Now button, and the checkout API refuses a charge on INCLUSIVE (and REFUNDED) registrations outright — so a sponsor-paid attendee can never accidentally pay a second time.
💡
Why not just use COMPLIMENTARY?
COMPLIMENTARY means “no money changes hands at all” (VIPs, speakers, in-kind). INCLUSIVE means “the sponsor paid” — money DID change hands, just not from the attendee. Same UI behaviour (no chase emails, check-in OK) but separate for reporting and reconciliation.
Workflow prerequisite — add the sponsor first

Before you can tag registrations as INCLUSIVE, the sponsor must exist in the event’s sponsor list:

Event → Sponsors → Add Sponsor
Enter the sponsor’s name (use the exact full name — the CSV import resolves by name)
Optionally upload a logo, set a tier (Platinum / Gold / Silver / Bronze / Partner / Exhibitor), add a website URL
Save. The sponsor is now available in the tier dropdown when editing registrations and in the CSV import sponsor column.
Importing a sponsor’s attendee list (CSV)

The recommended flow when a sponsor sends you their attendee list:

Open the sponsor’s list in a spreadsheet. Make sure you have email, firstName, lastName columns.
Add three columns: registrationStatus = CONFIRMED, paymentStatus = INCLUSIVE, sponsor = the sponsor’s exact name (e.g. “Abbott”)
Optionally add a registrationType column — pick which type the sponsor’s attendees should land in (Physician, Allied Health, etc.)
Save as CSV. Registrations tab → Import CSV → pick your file
Preview the first 5 rows. Click Import. The system creates all registrations as INCLUSIVE + tagged to your sponsor in one transaction.
ℹ️
Mixing sponsors in one CSV
You can mix sponsors in a single import — just put different sponsor names in the sponsor column per row. Most workflows are one-sponsor-per-CSV (Abbott’s 100 separately from Pfizer’s 50), which is simplest.

How invoice numbers work

Invoice numbers are generated automatically using the event code. Format: [EVENT_CODE]-INV-001, [EVENT_CODE]-INV-002, etc. Event codes are derived from the event name (“Heart Failure Forum 2026” → HFF2026). Receipts use the same prefix: HFF2026-REC-001. Credit notes: HFF2026-CN-001.

ℹ️
Quote vs. Invoice
When a registration is created (admin or public) with outstanding payment, the system generates a Quote PDF and emails it. When the registrant pays via Stripe, the quote is promoted in place to a status-PAID Invoice — no duplicate numbers, the same record flows through both stages.
Invoice types
Quote
Generated at registration time when payment is outstanding. The payment request sent to the registrant.
Invoice (PAID)
Generated automatically when Stripe payment succeeds (promotes the existing Quote) or when finance manually records a payment. PDF includes “Paid on X via Visa ending 4242” line.
Credit Note
Issued for refunds — full or partial. The organiser issues it first (it unlocks the Refund button); several partial credit notes can be issued against one registration, capped at the amount paid.
Charging a registration to a third-party payer

Why. Faculty and delegates are often paid for by someone other than themselves — a hospital sponsoring its doctor, or a pharma grant covering a specific HCP. The invoice needs to be addressed to (and chased from) that institution, not the attendee, even though the attendee is the person attending.

What. A payer (the “Charge to another account” option) is a reusable billing party — name, address, VAT/tax number, contact — stored once at the organisation level and attached to whichever events use it, so “Cleveland Clinic” is a single record, never duplicated per event. Setting a payer on a registration changes who the invoice is billed to: the PDF’s bill-to becomes the payer, the attendee drops to a reference line, and the attendee’s own “Pay now” button is suppressed. It is orthogonal to payment status — the money is still owed and the registration stays UNPAID until the payer settles.

💡
Payer vs. Sponsor-paid (INCLUSIVE) — which one?
Use a payer when an invoice will be sent and money is still owed (bill the hospital, then collect). Use INCLUSIVE (Section 5) when a sponsor has already pre-paid a block out-of-band and the attendee owes nothing. Payer = “who gets the bill”; INCLUSIVE = “already settled by a sponsor”.

How — add a payer inline while registering:

On the Add Registration form, under Charge to, pick an existing payer — or click + Add a new payer.
Enter the payer’s name (required) and any details you have (type, contact, email, address, tax/VAT number). You can complete or edit these later in Settings → Billing.
Save. The payer is created at the organisation level and attached to this event in one step, then selected on the registration — no trip to Settings.
Optionally add a PO / grant reference and tick attendee is guarantor (the attendee backstops the bill if the payer doesn’t settle).
Click Register. The quote/invoice is now billed to the payer.
✅
Same name = same payer (consolidated org-wide)
Typing a payer name that already exists (any capitalisation) reuses the existing organisation record rather than creating a duplicate — just like Contacts. A near-duplicate name (e.g. “Cleveland Clinic” vs “Cleveland Clinic Foundation”) is still created but flagged for an admin to review/merge in Settings → Billing. To change a payer’s address or tax number, edit the single record in Settings → Billing — it updates everywhere that payer is used.

Refunds (full or partial)

A refund is a two-step, credit-note-gated flow — both full and partial refunds are supported. Open the registration → Billing & Payments tab → Refunds & credit notes:

Issue Credit Note first — enter the amount (defaults to the full remaining, type a smaller number for a partial), an optional reason, and tick Email the credit note to the attendee if you want them to receive it. This is required: the Issue Refund button stays disabled until a credit note exists.
Click Issue Refund and enter the refund amount — capped at what’s still refundable and never more than the total credited (since July 11, 2026 the server enforces that cumulative refunds can’t exceed the credit-note total; issue another credit note first to unlock a larger refund). Stripe payments are reversed automatically (a partial amount is sent to Stripe); offline payments (cash / bank transfer / card-on-site) must be returned to the attendee manually — the system only records the reversal.
A partial refund keeps the registration PAID and shows “Refunded X of Y · Z remaining”; you can refund again later up to the balance. Once the full amount is returned the status flips to REFUNDED.
If the registration was paid through several payments (e.g. a Stripe charge plus a cash top-up), the refund is split automatically: each Stripe charge is reversed up to what it holds, and any offline remainder is recorded for you to return manually — the response lists exactly which payment each part came from.
ℹ️
No automatic email to the registrant — from either path
Issuing a refund does not auto-email the attendee — you communicate the refund yourself (and send the credit note from the Issue Credit Note dialog if you want them to have it). A Stripe refund made directly in the Stripe Dashboard is reconciled back automatically — the credit note + refunded total update on their own, admins get an in-app notification and an audit entry, and (since July 10, 2026) the attendee is not emailed for Dashboard refunds either — one policy for both paths.
⚠️
“Refund could not be confirmed” — wait, don’t retry blindly
If a refund attempt ends with a message that the state could not be confirmed with Stripe, the system reconciles it automatically within ~10 minutes (it verifies against Stripe and either confirms the refund or releases the booking, with an admin notification either way). Check the Stripe Dashboard before retrying — retrying immediately can’t double-refund, but waiting avoids confusion.
🔴
Dashboard only — AI agents cannot refund
Refunds and credit notes are issued from the dashboard by finance/admin roles. AI agents cannot initiate them — an intentional safety boundary.
Cancelling a registration

Open the registration → Edit → set Registration Status to Cancelled → Save. What happens next depends on whether it was paid:

Unpaid registration → cancels immediately. The amount owed becomes 0 (it’s no longer chased by payment reminders), and the seat is released back to availability. The original price stays on the record for history.
Paid registration → Save opens a confirmation with three choices:
  • Cancel & refund — issues a credit note and refunds the remaining balance (Stripe reverses automatically; offline payments must be returned to the attendee manually), then cancels. Irreversible.
  • Just cancel, no refund — cancels and keeps the money (e.g. a no-refund policy). Any other edits you staged are saved too.
  • Keep registration — dismiss, nothing changes.
ℹ️
Refund happens before the cancel
On Cancel & refund, the money is returned first; only then is the registration cancelled. If the refund can’t be processed, the registration is not cancelled — so you can fix the issue and retry, with nothing left half-done. An already-partly-refunded registration only refunds what’s left.
⚠️
If the balance didn’t load, reload first
The refund choice needs the registration’s balance, which loads when you open the detail. If it failed to load, cancelling a paid registration is blocked with a “reload the page” message — so a paid attendee is never cancelled without the refund decision being offered.

Scheduled emails — and reaching people who register later

Why this matters. A common need is “send the agenda to everyone next Monday” or “remind unpaid registrants three days before the event.” Between now and then, more people register. You almost always want those late registrations to receive the email too — without having to remember to re-send.

What happens now. When you schedule an email, its audience is worked out at the moment it sends, not the moment you schedule it. So a reminder you queue today will reach anyone who registers (and still matches your filters) before it goes out. A “remind unpaid” send likewise drops anyone who has paid by then. Both immediate “Send now” and future-dated sends appear in the Scheduled tab, where you can edit, cancel, or retry failed sends and watch per-send success/failure counts.

How to use it.

Open an audience card (e.g. Registrations) → Send / Schedule. Set any Filter recipients options (payment status, registration type, badge, tags, exclude faculty) — these define who matches.

Common questions

Does everyone get the same link?
No. Every invitee has their own private link. When they open it, it shows their name and remembers their answer. Never forward one person’s link to someone else — add them to the list and send them their own.

How is a bulk email personalized, then?
Each email in a bulk send is built individually with that person’s own link and name (first/last/email). A bulk send is exactly like sending one at a time — just automated. There is never a shared link.

Can I email just one person?
Yes — click the paper-plane Send button on their row in the roster. It uses the same template.

The email didn’t seem to send — what do I do?
The Send button tells you if it failed and why (for example, the event’s sender address isn’t verified — check Settings → Email Branding). Either way you can always Copy link from the roster and send it yourself.

Can invitees change their answer?
Yes — any time before the deadline. Reopening their link shows their current answer, which they can update.

What does the RSVP deadline do?
After a dinner’s deadline that dinner locks on the form (it can’t be changed), while any other dinners stay open. Leave the deadline blank for no cut-off.

What if someone can’t attend at all?
They tick “I won’t be able to attend”. That’s recorded as a real decline (it shows in the roster) — not left sitting as “pending”.

Do guests count in the headcount?
Yes. Each attendee’s guest count is added to that dinner’s total (e.g. “42 attending, +8 guests”), and there’s a guest column in the CSV for catering.

My event only has one dinner — does the wording still read right?
Yes. The form and the email automatically use the singular (“dinner”) when there’s one and the plural (“dinners”) when there are several.

Where do I change the email wording or branding?
Communications → Email Templates → Dinner RSVP Invitation. Always Preview before sending.

Common questions

The speaker made a mistake (e.g. a wrong IBAN) — can they fix it?
Yes. Open their submission (console or profile card → Open console) and click Reopen for edits. The form unlocks with everything they entered still there; they correct it and resubmit. Don’t process a payment while a form is reopened.

Does everyone get the same link?
No — every speaker has their own private link, pre-filled with their details. Never forward one speaker’s link to another.

Which currencies can they claim in?
USD, AED or SAR, per line. Mixed-currency claims show a separate total per currency — they are never added together.

When does the speaker get paid?
The declaration (and the confirmation email) state payment by bank wire within 45 days of receiving the completed form and all supporting documents. The actual payment is processed by finance outside the system — use the CSV export.

Can I change the email before sending?
Yes — type a subject/personal message in the send dialog for a one-off change, or click Edit the template to change the wording for every future send. Always Preview first.

Where do the uploaded receipts and passport copies go?
They’re stored privately — only Admins/Organizers can open them, from the submission view. They are never publicly accessible, even with a direct URL.

How certificates work in EA-SYS

The certificate system is built around the PDF-overlay model: your designer hands you a finished cert visual (PDF, JPG, or PNG — banner, borders, signatures, footer logos all baked in), you upload it, and then drag text boxes onto a canvas where the recipient name + event details + serial number get printed at issue time.

There are two cert categories:

  • Certificate of Attendance — for registrations (people who attended).
  • Certificate of Appreciation — for speakers, moderators, chairpersons, and abstract / poster submitters.

Within each category you can have multiple templates — e.g. “Standard Attendance” for general attendees, “VIP Attendance” for sponsor guests, “Chairman Appreciation” for the conference chair. Each template has its own background + its own text-box layout + its own cover-email defaults.

ℹ️
One cert per recipient per template — so one person can hold several role certs
A person can hold one certificate per template. This means someone who fills several roles can receive several role-specific certificates — e.g. a person who is on the Organizing Committee and a Speaker and a Moderator gets all three certificates, one from each template. Re-running the same template skips anyone who already holds it (no double-issue of that template), but different templates can each be issued to the same person. (This replaces the old “one per category” rule.) To give one person three certs, make three templates (one per role), tag the person for each, and issue each template.

💡

The system does not figure out who is a moderator, speaker, or delegate. You decide, by pairing a template with a tag. The simplest way to think about it:

One certificate = one template. A tag is that template’s guest list. The Role field is just the word that prints on it.

So for a conference with delegates, speakers, and moderators you make one template per certificate and give each its own tag:

The Issue flow is tag-driven manual selection — no auto-eligibility based on check-in status or session role. You tag people in advance from the existing Registrations + Speakers pages, then at Issue time you pick which tag gets which template.

This puts you in full control: a Chairman cert goes to people tagged “Chairman”; a Poster Presenter cert goes to people tagged “Poster Presenter”. Nothing fires until you tell it to, and only the tagged people get certs.

Tags live on:

  • Registrations — via the attendee detail sheet’s Tags field. Drives Attendance cert eligibility.
  • Speakers — via the speaker detail sheet’s Tags field. Drives Appreciation cert eligibility for both formal speakers and abstract submitters (submitters are stored as speakers).
✅
Pre-event tagging is the planning step
Before you start issuing, run through your speaker + registration lists and tag people by what cert they should receive. “Speaker”, “Moderator”, “Chairperson”, “VIP”, “Sponsor Guest”, “Poster Presenter” — whatever your event needs. The tag value is freeform; you decide what makes sense.
🔑
Tag rules — read this before you tag
  • Freeform value — any label you like (Speaker, VIP, Poster Presenter). No prefix or format required.
  • Matched EXACTLY, case-sensitive — speaker, Speaker and SPEAKER are three different tags. The tag on the person must be spelled + cased identically to the tag on the template. Avoid trailing spaces.
  • Tag the right list — an Attendance template reads the attendee’s tags (on the Registration); an Appreciation template reads the speaker’s tags (on the Speaker). Tag someone in the wrong list and the template simply won’t match them — no error, no certificate.
  • The tag decides WHO; it does NOT print on the certificate. What prints as the person’s role is the template’s own Role field (the {{role}} token) — see the next note.
ℹ️
Tag vs. Role — two different knobs on a template
These are set separately and do different jobs:
  • Tag = who gets this certificate (the selection filter, matched against people’s tags).
  • Role field = what the certificate says. It fills the {{role}} token with a fixed label for the whole template — it is NOT looked up per person from their session role. Every cert from that template prints the same role text (blank if you leave Role empty).
So to give people role-specific certificates, make one template per role: e.g. a Speaker template (Role = “Speaker”, tag speaker) and a Moderator template (Role = “Moderator”, tag moderator). Someone who is both gets both certs — just give them both tags. Keep each template’s Role and tag aligned so “who gets it” and “what it says” agree.

Not the same as a person’s profession. This template Role field is the designation printed on the certificate — it is not the attendee/speaker Role field you see in Registrations & Speakers (Physician, Nurse, Pharma, Student…, the person’s profession). That profession field never appears on certificates and is not used to pick who gets one; only tags do.

ℹ️
CME vs non-CME events — surveys are universal, certificates aren’t
You send surveys for every event, but that doesn’t mean every event issues certificates or CME. Two separate things: (1) Does the event issue a certificate? Only if you create a template (and, for hands-free, flag it + tag people) — so a non-CME event with no template issues nothing on survey completion. (2) Is the event CME-accredited? That’s set under the CME tab (CME hours + accrediting bodies). The top of the Certificates page shows a CME accredited / Not CME-accredited badge so you can tell at a glance. If an event has no CME, you don’t send any CME — correct. The {{cmeHours}} and accreditation tokens render blank when there’s no CME, so a non-CME certificate simply shows the attendance/appreciation text with no CME hours. Just don’t place a {{cmeHours}} box on a non-CME template (or leave it — it’ll print empty).
The Review gate — before emails go out (manual issue)

Click Confirm & issue and the run starts as PENDING. A background worker drains the run in two phases:

  1. RENDERING — the worker produces one PDF per recipient, with tokens substituted, and stores them on the server. Roughly 50 certs per minute. For 300 attendees that’s ~6 minutes; for 5000 it’s ~1.5 hours.
  2. AWAITING_REVIEW — the worker stops here. NO emails have been sent yet. You can spot-check the rendered PDFs from the Issue tab’s run panel before triggering Send.

When you’re happy, click Send emails on the run panel. The worker enters the SENDING phase, fires emails in batches of 25 with the cert PDF attached (compliant with AWS SES’s 14/sec ceiling), and transitions to COMPLETED when done.

⚠️
The review gate exists for a reason
Once you click Send, those emails are out the door. Spot-check a few rendered PDFs at AWAITING_REVIEW. If something’s wrong (typo in a token, box in the wrong place, etc.), click Cancel instead, fix the template, and re-issue.

Spot-checking large runs (300, 500, 5000 recipients): the dashboard shows the first 20 rendered certs as clickable links. You’re not expected to visually inspect 5000 PDFs — humans physically can’t, and the rendering is deterministic per recipient. If the 20 samples look right (token substitution working, box positions correct, fonts and colors as designed), the remaining 4,980 will too. Edge cases that DO break (very long names overflowing a box, characters that can’t encode in WinAnsi, missing recipient data) land in the red Failures panel above with the exact error — you Retry or accept the partial run from there. So the workflow at scale is: sample 5 from the spot-check list visually, check the Failures count is acceptable (e.g. < 1%), click Send.

ℹ️
Local-dev caveat (operators on prod can ignore)
If you’re testing on a local dev server that points at the production Supabase database, clicking a spot-check link may return 404. The cert PDF was likely rendered by the prod EC2 cron (which shares your database but writes to a different disk than your laptop). For local end-to-end UAT, either disable the EC2 crontab temporarily and fire the cron locally with curl, or simply test on production directly. In normal production usage this never happens — the same EC2 box renders and serves.
Run progress + status panel

The Issue tab keeps polling the run every 4 seconds while it’s active. You see live counters for total / rendered / emailed / failed plus a progress bar. Status badges:

  • PENDING — queued; the cron worker picks up within 60 seconds.
  • RENDERING — producing PDFs.
  • AWAITING_REVIEW — PDFs ready, waiting for you to click Send.
  • SENDING — emails are going out.
  • COMPLETED — done. The run row stays for audit.
  • FAILED — something blocked the run. Check Logs — the failure reason is logged.
  • CANCELLED — you cancelled before completion. Already-emailed recipients keep their certs; the rest don’t fire.
Re-issuing & the audit trail

Every issued cert is captured in an immutable IssuedCertificate row with a frozen recipient snapshot, a unique serial, the PDF URL, and a link to the template that produced it. This means:

  • A reprint months later renders byte-for-byte the same PDF — the snapshot doesn’t drift with the underlying Registration / Speaker / Event records.
  • Deleting a template is BLOCKED if any cert was issued from it. You can rename to mark as retired, but the audit chain stays linked.
  • You can’t accidentally double-issue someone the same template — the eligibility query excludes anyone who already holds a cert from that template.
Resending & issuing from a person’s detail sheet

Open any Registration or Speaker → their detail sheet has a Certificates card listing every certificate they hold. From there you can, per person, without touching tags or the bulk Issue flow:

  • Resend latest version (the per-row Resend button) — re-renders the certificate from the current template (so a corrected greeting, a fixed design, or any template edit is picked up) and emails it again. The serial stays the same; the PDF is refreshed. This is the button to use after you’ve fixed something on a template — it pushes the corrected certificate to that person.
  • Issue certificate — tick one or several templates (checkboxes) and issue them to this one person on the spot, delivered as one email with one PDF per certificate. The “no tag, no certificate” rule applies here too: only templates whose tag this person holds can be ticked (non-matching ones are greyed out with the reason, and an untagged person can’t be issued anything — tag them first). A template they already hold is simply re-attached (same serial, flagged in amber in the picker) — never duplicated. When several certificates go out together, the email text comes from the editable “Certificate Delivery (Multiple Certificates)” template under Communications → Email Templates — edit it there once and every bundle email uses your wording.
  • Preview before resending — both the per-row Resend and Resend all confirmations show the exact email (recipient, subject, and the fully branded body) before anything goes out — what you see is byte-for-byte what is sent. And every certificate email’s final HTML is stored: the person’s Activity timeline shows a View email link on certificate emails, opening the exact copy that was delivered.
  • Resend all (N) — appears when the person holds two or more certs (e.g. Committee + Speaker + Moderator). One click re-renders + resends all of them from their current templates, as separate emails (one PDF each), and tells you if any failed.
ℹ️
“Resend” now means “resend the latest version”
Resending always re-renders from the current template, so if you correct a mistake (a wrong greeting token, a design tweak) and then hit Resend, the recipient gets the fixed certificate — you don’t have to revoke + re-issue. The card shows “Reissued N times (latest template)” so you can see how many times a refreshed copy went out.
✅
Fixed a template after some certs went out? Resend the fixed version to everyone in one click
If you corrected a template after issuing, go to Certificates → Issue, pick the template, and click “Resend to everyone (N)” — it re-renders every certificate already issued from that template (with your fix) and re-emails all N recipients. Each recipient keeps their original serial; only the PDF and the email are refreshed. Progress shows in the run panel just like a normal Issue run. (To resend to just one person instead, use Resend latest on that person’s Certificates card.)
Common patterns

Conference with 300 attendees + 30 speakers

  • Two templates: “Attendance” (with your designer’s attendance PDF) + “Speaker Appreciation” (with the appreciation PDF).
  • Pre-event: tag every registration with “Attendee”; every speaker with “Speaker”.
  • Issue twice: Attendance + tag “Attendee” (300 fires), then Appreciation + tag “Speaker” (30 fires).

Event with VIP segmentation

  • Three templates: “Standard Attendance”, “VIP Attendance”, “Speaker Appreciation”.
  • Tag your VIPs with “VIP” in addition to “Attendee”.
  • Issue VIP Attendance + tag “VIP” FIRST. Then issue Standard Attendance + tag “Attendee” — the eligibility query auto-excludes the VIPs (they already hold an Attendance cert), so they don’t get the standard one too.

CME-accredited event with poster session

  • Two templates: “Attendance (CME)” with the {{cmeHours}} + {{accreditationBody}} tokens in its text boxes; “Poster Appreciation” with {{abstractTitle}} in the cover email body.
  • Set CME hours + accreditations once in the CME / CPD tab.
  • Tag poster presenters with “Poster Presenter”.
  • Issue Attendance + tag “Attendee”, then Poster Appreciation + tag “Poster Presenter”.

CRM — Sales Pipeline
Track sponsorship & exhibitor deals against your events — companies, contacts, a drag-and-drop pipeline, follow-ups, and reports.
Sales / Admin / Organizer · Org-wide

The CRM is where your team tracks money coming IN — sponsors and exhibitors, the deals you’re working, and who you’re talking to. It lives at CRM in the left sidebar (one entry; the sections below are tabs across the top). It is org-wide, not tied to a single event — but every deal can be linked to the event it’s being sold against, which is the whole point.

Two kinds of “contacts” — and why they’re separate

This trips people up, so read it once: there are two different address books, deliberately kept apart.

  • Event Contacts (the Contacts item in the main sidebar) — these are your HCPs: doctors, nurses, allied-health attendees. This list feeds event communications and the marketing mailing list.
  • CRM Contacts (the Contacts tab inside CRM) — the business people you negotiate with: a pharma rep, an exhibitor’s sales manager, a procurement officer.

Keep a pharma rep in CRM Contacts, never in Event Contacts — otherwise they’d get marketed to as if they were a doctor. If a rep also attends your conference, that’s fine: open their CRM contact and link it to their event registration — one person, one record, two roles. You never enter them twice.

Companies (accounts)

A Company is the account — Abbott, Cleveland Clinic, a society. Deals and contacts hang off it. On the Companies tab you can add one, search, and filter by industry. When you type a company name that already exists (even with different capitalisation), the system links you to the existing one rather than creating a duplicate — the toast tells you which happened. If a new name looks close to an existing account (“Cleveland Clinic” vs “Cleveland Clinic Foundation”), it’s created but flagged Needs review so you can confirm whether they’re genuinely different or should be merged.

Deals — the pipeline board

The Deals tab is a kanban board: one column per pipeline stage (New → Proposal → Negotiation → Contract Signed → Purchase Order → Invoice Sent → Won / Lost). Drag a card to move a deal along. If a colleague moved the same deal while you were dragging, you’ll get a “someone else moved this” message and the board refreshes — nobody’s change is silently lost.

  • New deal — give it a name (“Abbott — BRIDGES 2026 Gold”), a company, a value, and — required — the event (project) it’s being sold against, chosen from the dropdown. A deal must always be tied to an event; you can re-point it to a different one later but not remove it. Filter the board by event to see just that conference’s pipeline.
  • Click a deal to open its own page (its own web address, so you can bookmark it or send a colleague the link). There you can edit its fields, link the people on it (the rep, plus marketing or procurement) each with their role, email those people (see below), log notes (“called Dr Khan, wants Gold, decides after the board meets”), add follow-ups, and close it Won or Lost. A History timeline at the bottom records every change.
  • Won a sponsorship? Close it Won — the value counts toward your win reports. (Won and Lost are the last two columns; dragging a card into them, or using the Mark won / Mark lost buttons, does the same thing.)

Companies and CRM contacts each open their own page the same way — click a row to see the account (with all its deals and people) or the contact (with the deals they’re on), edit it, and see its History.

Products & line items

The Products tab is your service catalog — every service you sell (Sponsorship - Gold, Content - Prospectus, Virtual - Live Webinar, …), each with a category (the functional group: Content, Sponsorship, Design, CME…), a SKU, whether it’s In-House or Out-Sourced, and a list price. Your catalog is seeded with Meeting Minds’ standard list the first time you open it, and you can search it, filter by category, and add / edit / archive products (archiving hides one from the catalog and the deal picker; admins & CRM Users can restore it).

On a deal, open it and use the Products card to itemize what you’re selling: pick a product from the catalog, set the quantity and the unit price (it pre-fills from the catalog list price, but you set the actual price on the deal), and the card shows each line total and a products total. This total is for reference — the deal’s headline Value stays whatever you typed; adding products doesn’t change it. Adding or removing a product is recorded in the deal’s History. Members (read-only) see the products but not the prices (shown as “—”), same as deal values.

Emailing sponsors & deal contacts

You can send a personalised email — a sponsorship prospectus, a follow-up — straight from the CRM, with an attachment:

  • Email all sponsors of an event — pick an event in the board’s filter, then click Email sponsors. It gathers everyone on that event’s active deals (prospects + won, never lost), removes duplicates, and shows you the list so you can un-tick anyone before sending.
  • Email one deal’s contacts — open a deal and click Email to write to just the people on that deal.

In the compose window you can start from a saved template to pre-fill the subject and body, personalise with {{firstName}} / {{companyName}} / {{eventName}} (a greeting is added for you), and attach files (the prospectus PDF — up to 5 files, 10 MB total). Whatever you pick is just a starting point — edit the subject and body freely before sending. Each send is logged on that person’s Email History and on the contact’s CRM History.

Managing templates. Templates live on the Templates tab inside the CRM. Your org starts with three ready-made ones (Sponsorship prospectus / Follow-up / Thank you) that you can edit, and you can create your own — give it a name, a subject, and a body (with the same {{…}} tokens). Anyone with CRM write access can add and edit templates; archiving one (hiding it from the picker) is limited to Admins and CRM Users, and archived templates can be restored.

Tasks & reminders

Add a follow-up from any deal (“chase Abbott about the Gold package”) with a due date. Give it a due date and you’ll get an email reminder when it falls due. The Tasks tab lists your follow-ups with overdue ones surfaced first; toggle between your own and everyone’s, and filter by rep or due-date range.

Filtering

The board, tasks and reports all filter by sales rep, date range (expected close / created / closed), status, value range, and event. Filters live in the page address, so a filtered view is shareable — copy the URL to send a colleague “every open Abbott deal closing this quarter”. Use Clear to reset.

Reports & export

The Reports tab shows, for whatever you’ve filtered to:

  • Pipeline by stage — how many deals and how much value sit in each stage.
  • Win / loss — won vs lost counts and values, and your win rate over the chosen period.
  • By sales rep — a leaderboard of open and won deals per person.

Click Export deals (CSV) to download the filtered deal list as a spreadsheet — it honours whatever filters are active, so you export exactly what you’re looking at.

Who can see and do what
  • Admins & Organizers — full access: own deals, edit everything, see values.
  • CRM User (a dedicated sales-team role) — full run of the CRM (own deals, edit, see deal values), but walled off from the rest of the platform: no events, registrations, invoices or settings. Invite one from Settings → Users for a sponsorship salesperson who should only ever see the pipeline.
  • Members (read-only leadership / auditors) — can view the board and reports but not change anything, and deal values are hidden from them (shown as “—”, and the CSV they export has no value columns). This is deliberate: a member account handed to a sponsor-side stakeholder must not reveal every rival’s deal size.
  • Registration-desk (Onsite) staff, reviewers, submitters and registrants have no CRM access.

Editing vs archiving. Anyone with write access (Admins, Organizers, CRM Users) can edit records. Archiving a record — hiding it from the lists — is limited to Admins and CRM Users; an Organizer can edit but not archive.

Not in the CRM (v1)

Two-way email inbox sync and lead scoring are intentionally out of scope for now. Nothing is ever hard-deleted either: a mistaken deal is closed Lost or archived (hidden from the lists but restorable from the “Show archived” toggle), and a duplicate company is merged — so history is never destroyed. Every record keeps a full History log of who changed what and when.

A purpose-built, fully-owned platform

EA-SYS is a single, integrated event management system built in-house for MeetingMinds Group. It replaces fragmented third-party tools (EventsAir, spreadsheets, manual workflows) with one platform we fully own and control.

Below is a snapshot of the platform's scope and engineering investment as of June 2026 — the kind of summary you'd want when a customer, auditor, or potential hire asks “what is this?”

The hand-written code that powers EA-SYS (excluding open-source libraries and build artifacts):

  • ~149,000 lines of TypeScript across 622 files — the application code
  • ~33,000 lines of API code (188 backend route handlers covering registration, payments, speakers, abstracts, accommodation, certificates, webinars, communications, MCP, surveys, AI agent …)
  • ~31,000 lines of dashboard UI (62 admin pages plus 90 reusable React components)
  • ~7,200 lines of public-facing pages (the /e/[slug]/… registration + completion + survey flows)

Industry context: a typical SaaS at this code volume usually has 5–15 engineers. EA-SYS is built by a small in-house team — the surface area reflects the breadth of what the platform does, not headcount.

Sidebar layout — what each section is for

The event-scoped sidebar is organised around how often you touch each item. Daily-use items live in the main sections; configure-once items live behind the Setup entry to keep the sidebar uncluttered for teams that don’t need them every day.

  • Overview — Dashboard, plus Webinar Console when the event is a webinar.
  • Manage — daily-use items the registration / program / check-in teams use. Registrations, Check-In, Speakers, Agenda, Accommodation.
  • Abstracts — Abstracts list + Reviewers. Only relevant for events with a CFP.
  • Tools — ongoing operational tools. Communications (sending bulk emails, scheduled sends) and AI Agent.
  • Setup — Event Setup hub + Settings. The Event Setup card grid is where you’ll find Registration Types, Content, Email Templates, Sponsors, Survey, Certificates, and Media Library — all the items that are configured once per event (or occasionally tweaked) rather than touched daily.
  • Analytics — sits at the bottom, visually separated. Different persona (reports viewers / MEMBER role) than the daily operational sections above.

The Setup hub shows a status pill on each card (“Configured” / “N configured” / “Not configured”) so you can scan event-readiness at a glance. Each card click-throughs to the same page that existed before the reorg — just relocated, not renamed.

Why this matters operationally

The scale numbers above translate into three things that matter day-to-day:

  • No vendor lock-in. Every feature, every workflow, every email template lives in code we control. When EventsAir or any other vendor changes pricing, deprecates a feature, or has an outage, we’re insulated.
  • Fast iteration. When you ask “can we add a field for X?” or “can we tweak the certificate layout?”, the answer is days — not vendor support tickets that take weeks.
  • Operational visibility. The activity log, email history, audit trail, and admin-alert pipeline mean problems surface quickly. When something goes wrong, you know — usually before the registrant does.
